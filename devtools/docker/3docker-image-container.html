Section 3-1

イメージとコンテナの関係

Docker を理解する上で最も重要な概念が「イメージ」と「コンテナ」の関係です。プログラミングにおける「クラス」と「インスタンス」の関係に似ており、イメージが設計図、コンテナがその実体にあたります。

📜 イメージとコンテナの関係
📜 Dockerfile 設計図を記述
→
📦 イメージ 読み取り専用テンプレート
→
💻 コンテナ A 実行中インスタンス
💻 コンテナ B 実行中インスタンス
💻 コンテナ C 実行中インスタンス

イメージとは

イメージ（Image）は、アプリケーションの実行環境をパッケージ化した読み取り専用のテンプレートです。OS のベース環境、ライブラリ、アプリケーションコード、設定ファイルなどが含まれます。

  • 読み取り専用（変更不可）で、一度作成されたイメージは変わらない
  • レイヤー構造を持ち、差分で効率的に管理される
  • Docker Hub からダウンロードするか、Dockerfile から自作できる
  • タグでバージョンを管理する（例：node:18-alpine、nginx:latest）

コンテナとは

コンテナ（Container）は、イメージから作成された実行可能なインスタンスです。イメージの上に書き込み可能なレイヤーが追加され、アプリケーションが隔離された環境で動作します。

  • イメージをもとに作成され、独立した実行環境を持つ
  • 1つのイメージから複数のコンテナを作成できる
  • コンテナ内の変更はそのコンテナにのみ反映される
  • コンテナを削除すると、書き込みレイヤーのデータは失われる
POINT

「イメージ = 設計図（クラス）」「コンテナ = 実体（インスタンス）」と覚えましょう。イメージは共有・配布可能な不変のテンプレートで、コンテナはそこから生成された動くプロセスです。

Section 3-2

イメージの取得と管理

Docker イメージは Docker Hub などのレジストリからダウンロード（pull）して利用するのが基本です。イメージの取得・一覧表示・削除の基本コマンドを学びましょう。

イメージの取得（docker pull）

イメージのダウンロード Bash 
# 最新版の nginx イメージを取得
$ docker pull nginx
Using default tag: latest
latest: Pulling from library/nginx
...
Status: Downloaded newer image for nginx:latest

# 特定バージョンを指定して取得
$ docker pull node:18-alpine

# 特定のプラットフォーム向けイメージを取得
$ docker pull --platform linux/amd64 python:3.12

イメージの一覧表示（docker images）

ローカルのイメージ一覧 Bash 
$ docker images
REPOSITORY   TAG          IMAGE ID       CREATED        SIZE
nginx        latest       a6bd71f48f68   2 days ago     187MB
node         18-alpine    7f2f7e3c2b1a   5 days ago     117MB
python       3.12         c5d7f8e9a0b2   1 week ago     1.02GB

# 特定のリポジトリのイメージだけ表示
$ docker images nginx

イメージの削除（docker rmi）

イメージの削除 Bash 
# イメージ名で削除
$ docker rmi nginx:latest

# イメージIDで削除
$ docker rmi a6bd71f48f68

# 使用されていないイメージを一括削除
$ docker image prune

# すべての未使用イメージを削除（確認あり）
$ docker image prune -a
注意

コンテナが使用中のイメージは削除できません。先にコンテナを停止・削除してからイメージを削除してください。docker image prune -a は未使用のすべてのイメージを削除するため、必要なイメージまで消えないよう注意しましょう。

イメージの検索（docker search）

Docker Hub でイメージを検索 Bash 
# Docker Hub でイメージを検索
$ docker search mysql
NAME           DESCRIPTION                                     STARS   OFFICIAL
mysql          MySQL is a widely used...                        15000   [OK]
mariadb        MariaDB Server is a...                           6000    [OK]
percona        Percona Server is a...                           600     [OK]

# 公式イメージのみ検索
$ docker search --filter is-official=true mysql
Section 3-3

コンテナの作成と実行

コンテナの作成・実行は docker run コマンドで行います。このコマンドは Docker で最も頻繁に使うコマンドの一つで、多くのオプションがあります。基本的な使い方をマスターしましょう。

基本的な docker run

docker run の基本構文 Bash 
# 基本構文
docker run [オプション] イメージ名[:タグ] [コマンド]

# 例：Ubuntu コンテナを起動して bash を実行
$ docker run -it ubuntu bash

# 例：Nginx をバックグラウンドで起動
$ docker run -d -p 8080:80 --name web nginx

# 例：コンテナを実行して終了後に自動削除
$ docker run --rm alpine echo "Hello, Docker!"

よく使うオプション

オプション 説明 使用例
-d バックグラウンド（デタッチ）モードで実行 docker run -d nginx
-it 対話型モード（ターミナルに接続） docker run -it ubuntu bash
-p ポートマッピング（ホスト:コンテナ） docker run -p 3000:3000 node-app
--name コンテナに名前を付ける docker run --name myapp nginx
--rm コンテナ終了時に自動削除 docker run --rm alpine ls
-v ボリュームのマウント docker run -v ./data:/app/data nginx
-e 環境変数の設定 docker run -e DB_HOST=localhost app

実行中のコンテナの確認

コンテナの一覧表示 Bash 
# 実行中のコンテナを一覧表示
$ docker ps
CONTAINER ID   IMAGE   COMMAND   CREATED   STATUS   PORTS   NAMES

# 停止中を含む全コンテナを表示
$ docker ps -a

# コンテナIDのみ表示
$ docker ps -q
Section 3-4

コンテナのライフサイクル

コンテナには「作成・起動・一時停止・停止・削除」という一連のライフサイクルがあります。各状態と、状態を遷移させるコマンドを理解することが重要です。

🔄 コンテナのライフサイクル
📦 Created 作成済み
→
▶ Running 実行中
↔
⏸ Paused 一時停止
■ Stopped 停止
→
🗑 Deleted 削除済み

ライフサイクル管理コマンド

コンテナの状態管理コマンド Bash 
# コンテナの作成（起動はしない）
$ docker create --name myapp nginx

# コンテナの起動
$ docker start myapp

# コンテナの停止（GracefulにSIGTERMを送信）
$ docker stop myapp

# コンテナの強制停止（SIGKILLを送信）
$ docker kill myapp

# コンテナの一時停止
$ docker pause myapp

# 一時停止の解除
$ docker unpause myapp

# コンテナの再起動
$ docker restart myapp

# コンテナの削除（停止済みのもの）
$ docker rm myapp

# 実行中のコンテナを強制削除
$ docker rm -f myapp

# 停止中の全コンテナを一括削除
$ docker container prune
POINT

docker run は docker create + docker start を同時に実行するコマンドです。通常は docker run を使いますが、設定を事前に行ってから起動したい場合は create と start を分けて実行します。

コンテナの情報確認

コンテナの詳細情報を取得 Bash 
# コンテナの詳細情報（JSON形式）
$ docker inspect myapp

# コンテナのログを確認
$ docker logs myapp

# リアルタイムでログを表示（tail -f のように）
$ docker logs -f myapp

# コンテナのリソース使用量を表示
$ docker stats myapp
Section 3-5

コンテナ内部の操作

実行中のコンテナの内部に入って操作する方法を学びましょう。デバッグや設定確認などで頻繁に使うテクニックです。

docker exec でコンテナ内にアクセス

docker exec は、実行中のコンテナ内でコマンドを実行するためのコマンドです。最もよく使われるのが、シェル（bash / sh）を起動してコンテナ内部を操作するパターンです。

コンテナ内部への接続 Bash 
# コンテナ内で bash を起動（対話モード）
$ docker exec -it myapp bash
root@a1b2c3d4e5f6:/# ls
bin  boot  dev  etc  home  lib  ...

# bash がない場合は sh を使う（alpine 系など）
$ docker exec -it myapp sh

# コンテナ内で特定のコマンドだけ実行
$ docker exec myapp cat /etc/nginx/nginx.conf

# 環境変数を確認
$ docker exec myapp env

# コンテナ内部から抜ける
root@a1b2c3d4e5f6:/# exit

コンテナとホスト間のファイルコピー

docker cp でファイルをコピー Bash 
# ホスト → コンテナへのコピー
$ docker cp ./index.html myapp:/usr/share/nginx/html/

# コンテナ → ホストへのコピー
$ docker cp myapp:/var/log/nginx/access.log ./access.log

# ディレクトリごとコピー
$ docker cp ./config/ myapp:/etc/nginx/conf.d/
POINT

docker cp は停止中のコンテナに対しても使用できます。デバッグ時にコンテナ内のログファイルをホストに取り出したり、設定ファイルを差し替えたりする際に便利です。

実践例：Nginx コンテナを操作する

Nginx コンテナの操作フロー Bash 
# 1. Nginx コンテナを起動
$ docker run -d -p 8080:80 --name web nginx

# 2. コンテナ内に入る
$ docker exec -it web bash

# 3. デフォルトの HTML ファイルを確認
root@xxx:/# cat /usr/share/nginx/html/index.html

# 4. HTML を書き換え
root@xxx:/# echo "<h1>Hello Docker!</h1>" > /usr/share/nginx/html/index.html

# 5. コンテナから抜ける
root@xxx:/# exit

# 6. ブラウザで http://localhost:8080 にアクセスして変更を確認

# 7. ログを確認
$ docker logs web

# 8. 後片付け
$ docker stop web && docker rm web
注意

コンテナ内で行った変更は、コンテナを削除すると失われます。永続的な変更が必要な場合は、Dockerfile でイメージに組み込むか、ボリュームを使用してデータを永続化しましょう（第6章で詳しく解説します）。

Exercise

3章 練習問題

Q1

Docker における「イメージ」と「コンテナ」の関係として正しいものはどれですか？

  1. イメージは実行中のプロセスで、コンテナは設計図である
  2. イメージは読み取り専用のテンプレートで、コンテナはそこから作成された実行可能なインスタンスである
  3. イメージとコンテナは同じもので、呼び方が違うだけである
  4. 1つのイメージからは1つのコンテナしか作成できない
Q2

実行中のコンテナ内で bash シェルを起動するコマンドとして正しいものはどれですか？

  1. docker run -it myapp bash
  2. docker exec -it myapp bash
  3. docker start -it myapp bash
  4. docker attach myapp bash
Q3

docker run --rm alpine echo "test" を実行した後、このコンテナはどうなりますか？

  1. バックグラウンドで実行を続ける
  2. 停止状態（Stopped）で残る
  3. 一時停止状態（Paused）になる
  4. 自動的に削除される
Answer

3章 解説・解答

Q1の解答: B

イメージは読み取り専用のテンプレートで、アプリケーションの実行に必要なすべてのファイルを含みます。コンテナはイメージから作成された実行可能なインスタンスで、書き込み可能なレイヤーが追加されます。1つのイメージから複数のコンテナを作成できる点も重要な特徴です。

Q2の解答: B

docker exec -it myapp bash が正解です。docker exec は既に実行中のコンテナ内でコマンドを実行するためのコマンドです。-i は標準入力を維持、-t は疑似TTYを割り当てるオプションで、対話的なシェル操作に必要です。docker run は新しいコンテナを作成するコマンドなので異なります。

Q3の解答: D

--rm オプションを指定すると、コンテナの実行が終了した時点で自動的にコンテナが削除されます。使い捨てのコンテナ（ワンショット実行）を行う際に便利なオプションで、不要なコンテナが溜まることを防げます。
