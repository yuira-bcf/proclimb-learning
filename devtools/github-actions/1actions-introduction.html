CI/CDとGitHub Actions入門

ソフトウェア開発を加速させる継続的インテグレーション・デリバリーの世界と、GitHub Actionsの基本を理解しましょう。

sync CI/CDとは何か

CI/CDは、現代のソフトウェア開発において不可欠な自動化プラクティスです。手動で行っていたテストやデプロイ作業を自動化することで、開発チームの生産性とソフトウェア品質を大幅に向上させます。

CI（継続的インテグレーション）とは

Continuous Integration（継続的インテグレーション）とは、開発者がコードの変更を頻繁にメインブランチへ統合し、その都度自動的にビルドとテストを実行するプラクティスです。

  • コードの変更を頻繁に（1日に何度も）共有リポジトリにマージ
  • マージのたびに自動ビルド・自動テストを実行
  • 問題を早期に発見し、修正コストを最小化
  • 「インテグレーション地獄」（統合時の大量のコンフリクト）を回避

CD（継続的デリバリー / 継続的デプロイメント）とは

Continuous Delivery（継続的デリバリー）は、CIで検証されたコードを本番環境にリリースできる状態に常に保つプラクティスです。Continuous Deployment（継続的デプロイメント）は、さらに進んで本番環境への反映まで自動化します。

手動テスト・デプロイの問題点

CI/CDを導入していない開発現場では、以下のような問題が日常的に発生します。

問題 具体例 CI/CDによる解決
テスト漏れ 手動テストでは全パターンの確認が困難 毎回同じテストを自動実行
デプロイミス 手順書の読み飛ばし、環境設定ミス デプロイ手順をコード化
統合の遅れ リリース直前に大量のコンフリクト発生 頻繁な統合で問題を早期発見
リリース頻度の低下 手動作業が面倒でリリースを避ける ワンクリック（または自動）でリリース
属人化 「あの人しかデプロイできない」状態 手順がコードとして共有される
lightbulb 重要ポイント

CI/CDの本質は「自動化」と「頻度の向上」です。小さな変更を頻繁に統合・テスト・デプロイすることで、問題の早期発見と修正コストの削減を実現します。

hub GitHub Actionsの概要

GitHub Actionsは、GitHubが提供するCI/CDプラットフォームです。リポジトリ内にYAMLファイルを配置するだけで、コードのビルド、テスト、デプロイを自動化できます。

基本的な仕組み

GitHub Actionsは以下の流れで動作します。

  1. リポジトリの .github/workflows/ ディレクトリにYAMLファイルを配置
  2. 指定したイベント（push、pull_request等）が発生
  3. GitHub側のランナー（仮想マシン）でワークフローが実行
  4. 結果がActionsタブに表示される
# 最小限のワークフロー例
name: My First Workflow

on: push

jobs:
  greet:
    runs-on: ubuntu-latest
    steps:
      - name: Hello World
        run: echo "Hello, GitHub Actions!"

GitHub Actionsを選ぶ場面

  • GitHubでソースコードを管理している場合 → 最もスムーズに導入可能
  • 小〜中規模のプロジェクト → 無料枠で十分対応可能
  • 複数OS/言語でテストが必要 → マトリックスビルドが強力
  • 素早く導入したい → サーバー構築不要、YAML記述だけでスタート

  • ソースコードはGitHubで管理

GitHub Actionsが最適です。理由:

  • GitHubとネイティブに統合されており、追加のサーバー構築やツール設定が不要
  • パブリックリポジトリなら無制限、プライベートでも月2,000分の無料枠がある
  • Node.jsのセットアップアクション（actions/setup-node）が公式提供されている
  • 小規模チームでの運用コスト（サーバー管理等）が発生しない

star GitHub Actionsのメリット

1. GitHubとのシームレスな統合

GitHub Actionsは、GitHubのイベントシステムと完全に統合されています。push、pull_request、issue作成、リリース公開など、GitHub上のあらゆるイベントをトリガーにワークフローを実行できます。

# Pull Requestのイベントに反応する例
name: PR Check

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  lint-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm run lint
      - run: npm test

この例のように、Pull Requestが作成・更新された際に自動的にリントチェックとテストを実行し、結果をPull Request上に表示できます。

2. 豊富なMarketplace

GitHub Marketplace には20,000以上のアクションが公開されています。よくある処理は既存のアクションを uses キーワードで呼び出すだけで利用可能です。

# Marketplaceのアクションを活用した実践例
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      # リポジトリをチェックアウト
      - uses: actions/checkout@v4

      # Node.jsセットアップ
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      # 依存関係インストール・ビルド
      - run: npm ci
      - run: npm run build

      # S3にデプロイ（Marketplaceアクション）
      - uses: jakejarvis/s3-sync-action@master
        with:
          args: --delete
        env:
          AWS_S3_BUCKET: ${{ secrets.AWS_S3_BUCKET }}
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}

3. マトリックスビルド

複数のOS・言語バージョンの組み合わせでテストを自動実行できます。

# マトリックスビルドの例
name: Cross-platform Test

on: push

jobs:
  test:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        node-version: [18, 20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
      - run: npm ci
      - run: npm test

4. その他のメリット

  • セットアップが容易: YAMLファイルを配置するだけ。サーバー構築不要
  • Secrets管理: APIキーやパスワードを安全に管理・利用可能
  • キャッシュ機能: 依存関係のキャッシュでビルド時間を短縮
  • アーティファクト: ビルド成果物の保存・共有が可能
  • 再利用可能ワークフロー: 共通処理をテンプレート化して複数リポジトリで再利用
  • 環境保護ルール: デプロイ前の承認フロー設定が可能

「Node.jsのライブラリを開発しており、Node.js 18, 20, 22の3バージョンと、Ubuntu、Windows、macOSの3つのOSでテストを実行したい。」

解答例を見る

マトリックスビルドが最も活かされます。

strategy.matrixでosとnode-versionを定義するだけで、3 x 3 = 9通りの組み合わせテストが自動的に並列実行されます。手動で9つのジョブを書く必要はありません。
