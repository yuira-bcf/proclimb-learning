Section 3-1

リポジトリの初期化（git init）

Gitでバージョン管理を始めるには、まずリポジトリを作成する必要があります。git init コマンドを使って、プロジェクトフォルダをGitリポジトリに変換しましょう。

1.1 リポジトリの作成

新しいプロジェクトを始める Bash 
# プロジェクト用のフォルダを作成
mkdir my-project
cd my-project

# Gitリポジトリを初期化
git init

# 出力:
# Initialized empty Git repository in /Users/username/my-project/.git/

git init を実行すると、フォルダ内に .git という隠しディレクトリが作成されます。ここにGitが変更履歴などの全ての情報を保存します。

.gitディレクトリを確認 Bash 
# 隠しファイルを含む一覧を表示
ls -la

# 出力:
# drwxr-xr-x  3 user  staff  96  1 15 10:00 .
# drwxr-xr-x  5 user  staff 160  1 15 10:00 ..
# drwxr-xr-x 10 user  staff 320  1 15 10:00 .git
warning .gitフォルダは触らない

.git フォルダにはGitの内部データが格納されています。このフォルダを手動で編集・削除すると、リポジトリが壊れてしまいます。.git フォルダを削除すると、全ての変更履歴が失われるので注意してください。

1.2 既存のプロジェクトを管理対象にする

すでにファイルがあるフォルダでも git init できます。既存のファイルはGitの追跡対象になっていないため、あとから git add で追加します。

既存プロジェクトにGitを導入 Bash 
# 既存のプロジェクトフォルダに移動
cd existing-project

# Git管理を開始
git init

# 既存のファイルを全てGitの追跡対象にする
git add .

# 最初のコミットを作成
git commit -m "Initial commit"
Section 3-2

ファイルの追跡（git add）

git add は、変更したファイルを「ステージングエリア」に追加するコマンドです。コミットする前の「準備段階」として、どのファイルの変更をコミットに含めるかを選択します。

🛠 git init → add → commit のフロー
📁 git init リポジトリ作成
→
📝 ファイル編集 コードを書く
→
📦 git add ステージングに追加
→
📸 git commit 履歴に記録

2.1 ファイルのステージング

git add の基本的な使い方 Bash 
# 特定のファイルをステージングに追加
git add index.html

# 複数のファイルをまとめて追加
git add index.html style.css script.js

# カレントディレクトリ以下の全ての変更を追加
git add .

# 特定の拡張子のファイルを全て追加
git add *.html

# 特定のフォルダ内の全ファイルを追加
git add src/
lightbulb なぜ add が必要なのか

git add があることで、変更したファイルの中から「今回のコミットに含めたいもの」だけを選べます。例えば、機能Aと機能Bを同時に開発していた場合、機能Aに関するファイルだけをステージングしてコミットし、機能Bは別のコミットにする、といった使い方ができます。

2.2 ステージングの取り消し

間違えてステージングしてしまった場合は、取り消すことができます。

ステージングの取り消し Bash 
# 特定のファイルのステージングを取り消す
git restore --staged index.html

# 全てのステージングを取り消す
git restore --staged .
warning git add . は慎重に

git add . は便利ですが、意図しないファイル（パスワード情報、ログファイル、node_modulesなど）まで追加してしまう危険があります。.gitignore ファイルを適切に設定するか、ファイル名を指定して git add するのが安全です。

Section 3-3

変更の記録（git commit）

git commit は、ステージングエリアにある変更をリポジトリの履歴に記録するコマンドです。各コミットには一意のID（ハッシュ値）が付与され、いつでもその時点の状態に戻ることができます。

3.1 コミットの作成

コミットの基本 Bash 
# コミットメッセージ付きでコミット（最も一般的な方法）
git commit -m "ヘッダーのデザインを修正"

# 出力例:
# [main abc1234] ヘッダーのデザインを修正
#  1 file changed, 5 insertions(+), 2 deletions(-)

# 複数行のコミットメッセージを書く場合
git commit -m "ヘッダーのデザインを修正

- ロゴの位置を調整
- ナビゲーションの色を変更
- レスポンシブ対応を追加"

3.2 良いコミットメッセージの書き方

コミットメッセージは、将来の自分やチームメンバーが変更内容を理解するための重要な手がかりです。以下のルールを守りましょう。

ルール 良い例 悪い例
何をしたかを簡潔に ログイン機能を追加 修正
具体的に書く メール認証のバグを修正 バグ修正
1コミット=1変更 ヘッダーのレイアウトを変更 いろいろ変更
現在形で書く READMEを更新 READMEを更新した
lightbulb コミットメッセージのプレフィックス

チーム開発では、コミットメッセージにプレフィックスを付ける慣習があります。例えば: feat: ログイン機能を追加、fix: メール送信のバグを修正、docs: READMEを更新、style: インデントを統一 など。これによりコミットの種類が一目で分かります。

3.3 add と commit をまとめて実行

git commit -am（追跡済みファイルのみ） Bash 
# 追跡済みファイルの変更を一括 add + commit
git commit -am "スタイルシートを更新"

# 注意: 新しく作成したファイル（未追跡ファイル）は含まれない
# 未追跡ファイルは先に git add で追加する必要がある
warning -am オプションの注意点

git commit -am は便利ですが、新規ファイル（まだ一度も git add されていないファイル）は含まれません。新規ファイルがある場合は git add で明示的に追加する必要があります。

Section 3-4

ステータスの確認（git status）

git status は、リポジトリの現在の状態を確認するコマンドです。どのファイルが変更されたか、どのファイルがステージングされているかを教えてくれます。最も頻繁に使うコマンドの一つです。

4.1 git status の読み方

git status の出力例 Bash 
$ git status

On branch main

Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
        modified:   index.html          # ステージング済み

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   style.css           # 変更あり（未ステージング）

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        script.js                       # 新規ファイル（未追跡）
カテゴリ 意味 次のアクション
Changes to be committed ステージング済み。次のコミットに含まれる git commit で記録
Changes not staged 変更はあるがステージングされていない git add で追加
Untracked files Gitがまだ追跡していない新規ファイル git add で追跡開始
短縮表示（git status -s） Bash 
$ git status -s

M  index.html       # M（左）= ステージング済み
 M style.css        # M（右）= 変更あり（未ステージング）
?? script.js        # ?? = 未追跡ファイル
A  new-file.html    # A = 新規追加（ステージング済み）
lightbulb こまめに git status を確認しよう

Git操作で迷ったときは、まず git status を実行しましょう。現在の状態が分かるだけでなく、次に何をすればよいかのヒントも表示してくれます。操作に慣れるまでは、何かコマンドを実行するたびに git status で確認する習慣をつけましょう。

Section 3-5

ワーキングツリー・ステージング・リポジトリ

Gitには3つのエリアがあり、ファイルの変更はこの3つのエリアを段階的に移動していきます。この概念を理解することが、Git操作を正しく行う鍵です。

🎯 Gitの3つのエリア
📝 ワーキングツリー 実際のファイルを
編集する場所
git add
→
📦 ステージングエリア コミットする変更を
準備する場所
git commit
→
📁 リポジトリ 変更履歴が
永続的に記録される

5.1 各エリアの役割

📝

ワーキングツリー（Working Tree）

実際にファイルを編集する作業領域です。プロジェクトフォルダそのものがワーキングツリーです。エディタで開いてコードを書いたり修正したりする場所です。

📦

ステージングエリア（Staging Area / Index）

次のコミットに含める変更を一時的に置いておく場所です。「インデックス」とも呼ばれます。git add でファイルをここに移動させます。

📁

リポジトリ（Repository）

コミットされた変更が永続的に保存される場所です。.git フォルダ内に格納されます。過去の任意のコミットに戻ることができます。

5.2 実際のワークフロー

3つのエリアを意識した操作フロー Bash 
# 1. ワーキングツリーでファイルを作成・編集
echo "<h1>Hello World</h1>" > index.html

# 2. git status で状態を確認
git status
# → Untracked files: index.html（ワーキングツリーのみ）

# 3. ステージングエリアに追加
git add index.html

# 4. git status で再確認
git status
# → Changes to be committed: index.html（ステージングに移動）

# 5. リポジトリにコミット
git commit -m "index.htmlを作成"

# 6. git status で最終確認
git status
# → nothing to commit, working tree clean（全エリアが同期）
lightbulb ステージングエリアの利点

「ステージングエリアって面倒では？」と感じるかもしれませんが、この仕組みのおかげで「コミットに含める変更を細かく制御」できます。10個のファイルを変更したとしても、関連する3つだけを選んでコミットし、残り7つは別のコミットにする、というように整理できます。きれいなコミット履歴はチーム開発で非常に重要です。

Exercise

3章 練習問題

Q1

新しいGitリポジトリを作成するコマンドはどれですか？

  1. git create
  2. git new
  3. git init
  4. git start
Q2

Gitの3つのエリアを正しい順序で並べたものはどれですか？

  1. リポジトリ → ステージング → ワーキングツリー
  2. ワーキングツリー → リポジトリ → ステージング
  3. ステージング → ワーキングツリー → リポジトリ
  4. ワーキングツリー → ステージング → リポジトリ
Q3

git status で「Untracked files」と表示されたファイルの状態として正しいのはどれですか？

  1. ステージングエリアに追加済みのファイル
  2. すでにコミット済みで変更のないファイル
  3. Gitがまだ追跡していない新しいファイル
  4. 削除されたファイル
Answer

3章 解説・解答

Q1の解答: C

git init は「initialize（初期化）」の略で、カレントディレクトリに新しいGitリポジトリを作成するコマンドです。実行すると .git ディレクトリが作成され、バージョン管理が開始されます。

Q2の解答: D

Gitの3つのエリアは「ワーキングツリー → ステージングエリア → リポジトリ」の順番で変更が流れます。ワーキングツリーで編集し、git add でステージングに移し、git commit でリポジトリに記録します。

Q3の解答: C

「Untracked files」は、Gitがまだ追跡（トラッキング）していない新しいファイルを意味します。git add コマンドで追跡を開始する必要があります。一度追跡を開始すると、その後の変更はGitが自動的に検出します。
