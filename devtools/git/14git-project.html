Section 14-1

演習の概要と準備

この章では、これまで学んだGit & GitHubの知識を総動員して、チームポートフォリオサイトを作成する実践演習を行います。3〜4人のチームで、GitHub Flowを使った共同開発を体験しましょう。

🚀 プロジェクト演習の全体フロー
📝 Step 1 計画・役割分担
→
🛠 Step 2 リポジトリ準備
→
🌱 Step 3 ブランチで開発
→
💬 Step 4 PR & レビュー
→
🎉 Step 5 完成 & 公開

1.1 演習の目標

  • GitHub Flowに基づいたチーム開発の流れを体験する
  • ブランチの作成・切り替え・マージを実践する
  • プルリクエストの作成とコードレビューを行う
  • コンフリクトが発生した場合に解決できるようになる
  • 完成したサイトをGitHub Pagesで公開する

1.2 プロジェクト概要：チームポートフォリオサイト

チームメンバー全員の自己紹介を1つのWebサイトにまとめたチームポートフォリオサイトを作ります。

項目 内容
サイト構成 トップページ（index.html） + 各メンバーの紹介ページ
使用技術 HTML / CSS
ワークフロー GitHub Flow

公開方法 GitHub Pages

GitHub Pages設定
Section 14-2

リポジトリのセットアップ

まず、リーダーがGitHub上にリポジトリを作成し、チームメンバーを招待します。

2.1 リーダーの作業：リポジトリ作成

  1. GitHubで「New repository」をクリック
  2. リポジトリ名をteam-portfolioに設定
  3. 「Public」を選択（GitHub Pagesで公開するため）
  4. 「Add a README file」にチェックを入れる
  5. 「Add .gitignore」でNode等の適切なテンプレートを選択
  6. 「Create repository」をクリック
リーダー：初期ファイルの作成とpush Terminal 
# リポジトリをクローン
$ git clone https://github.com/leader-name/team-portfolio.git
$ cd team-portfolio

# プロジェクトの基本構造を作成
$ mkdir css images members

# index.htmlの雛形を作成
# (エディタで編集)

# 共通CSSファイルを作成
# (エディタで編集)

# 初期コミット
$ git add .
$ git commit -m "feat: add initial project structure"
$ git push origin main

2.2 リーダーの作業：メンバーの招待

  1. リポジトリのSettingsタブを開く
  2. 左メニューのCollaboratorsを選択
  3. 「Add people」をクリック
  4. メンバーのGitHubユーザー名またはメールアドレスを入力して招待
lightbulb 招待を受ける側の操作

招待されたメンバーは、メールまたはGitHubの通知で招待を確認し、「Accept invitation」をクリックして承認します。承認後、リポジトリへのpush権限が付与されます。

2.3 メンバーの作業：リポジトリのクローン

各メンバー：リポジトリをクローン Terminal 
# リポジトリをクローン
$ git clone https://github.com/leader-name/team-portfolio.git
$ cd team-portfolio

# ファイル構成を確認
$ ls
README.md  css/  images/  index.html  members/

2.4 Issueの作成

各メンバーの担当タスクをIssueとして登録します。これにより、誰が何をやっているかをチーム全体で把握できます。

Issueの例 テキスト 
Issue #1: トップページ（index.html）を作成する  → リーダー
Issue #2: メンバーA の自己紹介ページを作成する   → メンバーA
Issue #3: メンバーB の自己紹介ページを作成する   → メンバーB
Issue #4: メンバーC の自己紹介ページを作成する   → メンバーC
Issue #5: 共通CSSを整える                       → メンバーA
Issue #6: GitHub Pagesを設定して公開する         → メンバーC
Section 14-3

ブランチ戦略の実践

GitHub Flowに従い、各メンバーはそれぞれのブランチで作業を進めます。

3.1 ブランチの作成と開発

メンバーA の作業例 Terminal 
# mainを最新にする
$ git checkout main
$ git pull origin main

# 作業ブランチを作成
$ git checkout -b feature/member-a-page

# 自己紹介ページを作成
# members/member-a.html をエディタで作成

# 変更をコミット
$ git add members/member-a.html
$ git commit -m "feat: add member A introduction page (#2)"

# リモートにpush
$ git push -u origin feature/member-a-page
自己紹介ページのテンプレート（members/member-a.html） HTML 
<!DOCTYPE html>
<html lang="ja">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>メンバーA - チームポートフォリオ</title>
    <link rel="stylesheet" href="../css/style.css">
</head>
<body>
    <header>
        <nav>
            <a href="../../index.html">ホーム</a>
        </nav>
    </header>
    <main>
        <section class="profile">
            <h1>メンバーA</h1>
            <img src="../images/member-a.jpg" alt="メンバーAの写真">
            <h2>自己紹介</h2>
            <p>こんにちは！Web開発を学んでいます。</p>
            <h2>スキル</h2>
            <ul>
                <li>HTML / CSS</li>
                <li>JavaScript</li>
                <li>Git / GitHub</li>
            </ul>
        </section>
    </main>
</body>
</html>
warning mainブランチに直接コミットしない

GitHub Flowでは、mainブランチには直接コミットしません。必ずfeatureブランチで作業し、プルリクエストを通じてmainにマージします。これにより、コードレビューの機会が生まれ、品質を保てます。

3.2 作業中のmainの更新を取り込む

他のメンバーの変更がmainにマージされた場合、自分のブランチにも最新の変更を取り込みましょう。

mainの最新変更を自分のブランチに取り込む Terminal 
# mainの最新を取得
$ git checkout main
$ git pull origin main

# 自分のブランチに戻ってmainをマージ
$ git checkout feature/member-a-page
$ git merge main

# コンフリクトがあれば解決してコミット
# コンフリクトがなければそのまま作業を続行
Section 14-4

プルリクエストとコードレビュー実践

featureブランチでの作業が完了したら、プルリクエスト（PR）を作成してコードレビューを受けましょう。

4.1 プルリクエストの作成

  1. GitHubリポジトリのページを開く
  2. 「Compare & pull request」ボタンをクリック（pushした直後に表示される）
  3. PRのタイトルと説明を記入
  4. Reviewersにレビュー担当者を指定
  5. 「Create pull request」をクリック
プルリクエストの説明文テンプレート Markdown 
## 概要
メンバーAの自己紹介ページを作成しました。

## 変更内容
- `members/member-a.html` を新規作成
- 自己紹介、スキル一覧、プロフィール画像を追加

## 関連Issue
Closes #2

## 確認してほしいポイント
- HTMLの構造は適切か
- リンクは正しく動作するか
- レスポンシブ対応は問題ないか

## スクリーンショット
（画面キャプチャを添付）

4.2 コードレビューの実践

レビュー担当者は、以下の観点でコードを確認します。

🔎

コードの正確性

HTMLのタグは正しく閉じられているか、リンクのパスは正確かなどを確認します。

📄

コードの可読性

インデントは揃っているか、クラス名は分かりやすいかなどを確認します。

🎯

プロジェクトとの一貫性

他のメンバーのページと構造やスタイルが統一されているかを確認します。

lightbulb レビューコメントの書き方

レビューコメントは建設的で具体的に書きましょう。「ここがダメ」ではなく、「このクラス名をprofile-imageに変更すると、他のページとの一貫性が保てると思います」のように改善提案を含めるのが良いレビューです。

4.3 レビュー後のマージ

  1. レビュアーが「Approve」（承認）を選択
  2. PRページの「Merge pull request」をクリック
  3. 「Confirm merge」でマージを確定
  4. マージ後、featureブランチを削除（「Delete branch」ボタン）
マージ後のローカル同期 Terminal 
# mainを最新にする
$ git checkout main
$ git pull origin main

# マージ済みのローカルブランチを削除
$ git branch -d feature/member-a-page
Section 14-5

プロジェクト完成とふりかえり

全メンバーのページがマージされたら、最終調整を行い、GitHub Pagesでサイトを公開しましょう。

5.1 GitHub Pagesでの公開

  1. リポジトリのSettingsタブを開く
  2. 左メニューのPagesを選択
  3. Sourceで「Deploy from a branch」を選択
  4. Branchで「main」、フォルダで「/ (root)」を選択
  5. 「Save」をクリック
  6. 数分後にURLが発行される

5.2 ふりかえり（KPT法）

プロジェクト終了後、チームでKPT法を使ってふりかえりを行いましょう。

カテゴリ 意味 例
Keep（続けたいこと） うまくいったこと、続けるべき習慣 PRのテンプレートが分かりやすかった
Problem（問題点） 困ったこと、改善が必要な点 コンフリクトの解決に時間がかかった
Try（次に試したいこと） 次回のプロジェクトで試したいこと こまめにpullして差分を小さくする
lightbulb 演習のアレンジ

この演習に慣れたら、GitHub ActionsでHTMLのリンクチェックを自動化したり、CSSフレームワーク（Bootstrap等）を導入してデザインを改善したり、さらにチャレンジしてみましょう。

Exercise

14章 練習問題

Q1

GitHub Flowでは、新しい機能を開発する際にまず何をしますか？

  1. mainブランチに直接コミットする
  2. mainブランチから新しいfeatureブランチを作成する
  3. developブランチから新しいfeatureブランチを作成する
  4. リモートリポジトリを削除して作り直す
Q2

GitHubでコラボレーターを追加するには、リポジトリのどのページから操作しますか？

  1. Code タブ
  2. Issues タブ
  3. Settings タブの Collaborators
  4. Actions タブ
Q3

プルリクエストの説明文に「Closes #2」と書くとどうなりますか？

  1. Issue #2 が自動的に作成される
  2. PRがマージされると Issue #2 が自動的にクローズされる
  3. Issue #2 にコメントが自動投稿される
  4. Issue #2 の担当者が変更される
Answer

14章 解説・解答

Q1の解答: B

GitHub Flowでは、新しい作業を始める際にmainブランチから新しいfeatureブランチを作成します。Git Flowとは異なり、developブランチは使いません。mainブランチは常にデプロイ可能な状態を保ちます。

Q2の解答: C

コラボレーターの追加は、リポジトリのSettingsタブ内にあるCollaboratorsのページから行います。「Add people」ボタンをクリックして、ユーザー名やメールアドレスで招待できます。

Q3の解答: B

PRの説明文にCloses #2と記述すると、そのPRがmainにマージされた際にIssue #2が自動的にクローズされます。同様に Fixes #2、Resolves #2 でも同じ効果があります。これにより、タスク管理が効率化されます。
