Section 10-1

プルリクエストとは

プルリクエスト（Pull Request、略してPR）は、GitHubの最も重要な機能の一つです。自分のブランチでの変更を、別のブランチ（通常はmain）にマージしてほしいとリクエストする仕組みです。単にコードをマージするだけでなく、コードレビューの場としても機能します。

1.1 なぜプルリクエストが必要なのか

🔍

コードレビュー

他のメンバーが変更内容を確認し、バグや改善点を指摘できます。コードの品質向上に直結します。

💬

議論の場

コード行ごとにコメントを付けたり、全体的なフィードバックを共有したりできます。技術的な議論の記録が残ります。

🛡

品質の担保

mainブランチへの直接プッシュを禁止し、PRを通じてのみマージを許可するルールを設定できます（ブランチ保護）。

📈

変更履歴の可視化

どの変更がいつ、誰によって、なぜ行われたかが明確に記録されます。

Section 10-2

プルリクエストの作成手順

プルリクエストの作成には、Gitコマンドでの準備とGitHub上での操作の2段階があります。全体の流れを理解しましょう。

2.1 PRワークフロー全体像

🔄 プルリクエスト ワークフロー
🌿 1. ブランチ作成 git switch -c
→
📝 2. コミット 変更を記録
→
🚀 3. プッシュ git push -u
→
📄 4. PR作成 GitHub上で作成
→
🔍 5. レビュー コードの確認
→
✅ 6. マージ mainに統合

2.2 Step 1-3: ターミナルでの準備

PR作成のためのGit操作 Terminal 
# 1. mainブランチを最新にする
git switch main
git pull origin main

# 2. 新しいブランチを作成して切り替え
git switch -c feature/add-contact-page

# 3. ファイルを作成・編集
# ... 開発作業 ...

# 4. 変更をコミット
git add contact.html style.css
git commit -m "お問い合わせページを追加"

# 必要に応じて追加のコミット
git add contact.html
git commit -m "お問い合わせフォームのバリデーションを追加"

# 5. リモートにプッシュ
git push -u origin feature/add-contact-page

2.3 Step 4: GitHub上でのPR作成

  1. プッシュ後、GitHubのリポジトリページに「Compare & pull request」ボタンが表示されるのでクリック
  2. base（マージ先）がmain、compare（マージ元）が自分のブランチであることを確認
  3. タイトルを入力（変更内容を簡潔に表現する）
  4. 説明（Description）を記入（変更の詳細、背景、テスト方法など）
  5. 必要に応じてReviewers（レビュー担当者）、Assignees（担当者）、Labels（ラベル）を設定
  6. Create pull requestボタンをクリック
PRの説明文テンプレート例 Markdown 
## 概要
お問い合わせページを新規追加しました。

## 変更内容
- お問い合わせフォーム（名前、メール、本文）を作成
- フォームのバリデーションを実装
- レスポンシブデザインに対応

## 関連Issue
Closes #42

## テスト方法
1. contact.html をブラウザで開く
2. 空の状態で送信ボタンを押し、エラーメッセージが表示されることを確認
3. 正しい値を入力して送信できることを確認

## スクリーンショット
（必要に応じてスクリーンショットを添付）
lightbulb 良いPRの書き方

PRの説明は丁寧に書きましょう。「何を変更したか」だけでなく、「なぜ変更したか」「どうテストしたか」を含めると、レビューアーの理解が格段に深まります。変更が大きい場合はスクリーンショットや動画を添付するとさらに効果的です。

2.4 マージ方法の種類

PRをマージする際、GitHubでは3つのマージ方法を選択できます。

方法 説明 コミット履歴
Create a merge commit マージコミットを作成して統合 ブランチの全コミット + マージコミットが残る
Squash and merge 複数のコミットを1つにまとめてマージ 1つのコミットに圧縮される
Rebase and merge リベースしてからFast-forwardマージ 一直線の履歴になる
lightbulb Squash and mergeが便利

開発中に「typo修正」「デバッグコード削除」などの細かいコミットを重ねた場合、Squash and mergeを使えば1つのきれいなコミットにまとめられます。mainブランチの履歴がすっきりして見やすくなるため、多くのチームで採用されています。

Section 10-3

コードレビューの進め方

コードレビューは、PRを通じて他のメンバーがコードを確認し、フィードバックを行うプロセスです。バグの発見だけでなく、チーム内の知識共有やコード品質の向上に大きく貢献します。

3.1 レビューの種類

GitHubでは、PRに対して3つの種類のレビューを送信できます。

レビュー種類 意味 アイコン
Approve 変更を承認する（マージOK） 緑のチェックマーク
Request changes 修正を要求する（修正後に再レビュー） 赤のバツマーク
Comment コメントのみ（承認でも却下でもない） グレーのコメントアイコン

3.2 レビュアーとしてのベストプラクティス

👉

建設的なフィードバック

「ここがダメ」ではなく、「こうするとさらに良くなります」という表現を心がけましょう。改善案を具体的に示すとなお良いです。

💡

nits / suggestion の使い分け

「nit:」は些細な指摘（命名の提案など、必須ではない）、「suggestion:」は改善提案を意味します。重要度を明確にすると、作者が優先順位を判断しやすくなります。

⏰

レスポンスは早めに

PRが長時間放置されると開発が滞ります。24時間以内にレビューするのが理想的です。すぐにレビューできない場合は、その旨をコメントしましょう。

warning 自分のPRを自分でApproveしない

コードレビューの目的は「第三者の目」で品質を確認することです。自分が作成したPRを自分でApproveすることは、レビューの意味がなくなるため避けましょう。チームでは必ず別のメンバーにレビューを依頼してください。

3.3 レビュー後の対応

レビューで修正を求められた場合は、同じブランチで修正してプッシュすれば、PRに自動的に反映されます。

レビュー指摘への対応 Terminal 
# レビュー指摘に基づいてファイルを修正
# ... 修正作業 ...

# 修正をコミット
git add contact.html
git commit -m "レビュー指摘: バリデーションメッセージを修正"

# 同じブランチにプッシュ（PRに自動反映される）
git push origin feature/add-contact-page
Section 10-4

Issueの活用方法

Issue（イシュー）は、GitHubのプロジェクト管理機能です。バグ報告、機能要望、タスク管理、質問など、プロジェクトに関するあらゆるトピックをIssueとして記録・追跡できます。

4.1 Issueの構成要素

📋 Issue管理の要素
🏷 ラベル（Labels） bug, feature, docs など
Issueの分類に使用
🎯 マイルストーン v1.0, Sprint 3 など
リリース単位で管理
👤 アサイニー 担当者の割り当て
誰が対応するか明確に
📄 テンプレート バグ報告テンプレート
統一されたフォーマット

4.2 Issueの作成

  1. リポジトリのIssuesタブを開く
  2. New issueボタンをクリック
  3. タイトルを入力（問題や要望を簡潔に記述）
  4. 説明をMarkdownで記入
  5. 右サイドバーでLabels、Assignees、Milestoneを設定
  6. Submit new issueをクリック
バグ報告Issueの例 Markdown 
## バグの概要
お問い合わせフォームの送信後に確認メッセージが表示されない

## 再現手順
1. /contact にアクセス
2. フォームに正しい値を入力
3. 「送信」ボタンをクリック
4. ページが遷移するが確認メッセージが表示されない

## 期待される動作
送信後に「送信が完了しました」というメッセージが表示される

## 実際の動作
何も表示されずに空白ページが表示される

## 環境
- ブラウザ: Chrome 120
- OS: macOS 14.2
- 画面サイズ: 1920x1080

4.3 ラベルの活用

ラベルを使うとIssueを分類でき、フィルタリングや一覧表示が便利になります。

ラベル名 色の例 用途
bug 赤 バグの報告
feature 緑 新機能の要望
enhancement 青 既存機能の改善
documentation 紫 ドキュメントの修正・追加
good first issue 黄 初心者向けのタスク
help wanted オレンジ 助けが必要なタスク
Section 10-5

IssueとPRの連携

GitHubでは、IssueとPRを連携させることで、「どのIssueがどのPRで解決されたか」を追跡できます。PRがマージされた際に、関連するIssueを自動的にクローズすることも可能です。

5.1 PRからIssueを自動クローズする

PRの説明文やコミットメッセージに特定のキーワードとIssue番号を記載すると、PRのマージ時にIssueが自動的にクローズされます。

Issueを自動クローズするキーワード Markdown 
# PRの説明文やコミットメッセージで使用可能なキーワード

Closes #42         # Issue #42を自動クローズ
Fixes #42          # Issue #42を自動クローズ
Resolves #42       # Issue #42を自動クローズ

# 複数のIssueを同時にクローズ
Closes #42, Closes #43

# コミットメッセージで使う場合
git commit -m "お問い合わせフォームを修正 Fixes #42"
lightbulb IssueとPRの連携フロー

推奨されるワークフローは以下の通りです。(1) 作業の前にIssueを作成する (2) Issueに基づいてブランチを作成する（例: feature/42-contact-form） (3) PRの説明文にCloses #42を記載する (4) PRがマージされるとIssue #42が自動的にクローズされる。これにより、全ての変更がIssueとして追跡可能になります。

5.2 Issueからブランチを作成する

GitHubのIssueページから直接ブランチを作成することもできます。Issueのサイドバーにある「Create a branch」リンクをクリックすると、Issue番号を含むブランチが自動的に作成されます。

Issueベースのブランチ運用例 Terminal 
# Issue #42 "お問い合わせフォームの追加" に対応するブランチ
git switch -c feature/42-add-contact-form

# 作業してコミット
git add .
git commit -m "お問い合わせフォームを追加 #42"
# → #42 でIssueへのリンクが自動的に生成される

# プッシュしてPRを作成
git push -u origin feature/42-add-contact-form
# PRの説明文に "Closes #42" を記載

5.3 プロジェクト管理の全体像

IssueとPRを適切に連携させることで、プロジェクト全体の進捗を可視化できます。

📋

GitHub Projects

カンバンボード形式でIssueやPRを「Todo」「In Progress」「Done」のように管理できます。チームの作業状況が一目で分かります。

🎯

Milestone（マイルストーン）

Issueをマイルストーン（例: v1.0リリース）にグループ化して、リリースに必要なタスクの進捗を追跡できます。進捗バーで完了率が表示されます。

warning Issueを作らずにいきなりコードを書かない

チーム開発では、コードを書き始める前にまずIssueを作成する習慣をつけましょう。Issueを通じてチームメンバーと方針を共有し、合意を得てから開発を始めることで、手戻りを防ぎ、効率的に開発を進められます。「Issue駆動開発」はチーム開発の基本です。

Exercise

10章 練習問題

Q1

PRの説明文に記載すると、マージ時にIssue #15を自動的にクローズするキーワードはどれですか？

  1. Close #15（先頭が大文字のClose）
  2. Closes #15
  3. Related #15
  4. Done #15
Q2

コードレビューで修正を求められた場合、正しい対応方法はどれですか？

  1. 新しいPRを作成し直す
  2. 同じブランチで修正してプッシュすれば、PRに自動反映される
  3. mainブランチで直接修正してプッシュする
  4. PRを一度クローズしてから修正する
Q3

PRのマージ方法のうち、複数のコミットを1つにまとめてマージするのはどれですか？

  1. Create a merge commit
  2. Rebase and merge
  3. Squash and merge
  4. Fast-forward merge
Answer

10章 解説・解答

Q1の解答: B

Closes #15、Fixes #15、Resolves #15がIssueの自動クローズに対応するキーワードです。実はAのClose #15も機能しますが、一般的にはCloses（三人称単数形）が最も広く使われています。RelatedやDoneはGitHubの自動クローズキーワードには含まれません。

Q2の解答: B

レビュー指摘を受けた場合、同じブランチで修正をコミットし、git pushするだけで、既存のPRに変更が自動的に反映されます。新しいPRを作り直す必要はありません。レビュアーは追加されたコミットの差分を確認して、再度レビューを行います。

Q3の解答: C

Squash and mergeは、PRに含まれる全てのコミットを1つのコミットに圧縮してからmainブランチにマージします。開発中の細かいコミット（typo修正、デバッグコード削除など）をまとめてくれるため、mainブランチの履歴がきれいに保たれます。
