Section 9-1

リモートリポジトリとは

リモートリポジトリとは、ネットワーク上（インターネットやイントラネット）に存在するGitリポジトリのことです。GitHubやGitLabなどのサービス上に作成され、複数の開発者がコードを共有するための中央の保管場所として機能します。

1.1 ローカルとリモートの関係

Gitは分散型バージョン管理システムです。各開発者のPCにある「ローカルリポジトリ」と、GitHubなどにある「リモートリポジトリ」を同期しながら開発を進めます。

💻

ローカルリポジトリ

自分のPC上に存在するリポジトリ。オフラインでもコミットや履歴の確認ができます。直接ファイルを編集して作業します。

☁

リモートリポジトリ

GitHub等のサーバー上に存在するリポジトリ。チームメンバー全員がアクセスできる共有の場所です。バックアップとしても機能します。

🔄 ローカル ↔ リモートの操作フロー
💻
ローカルリポジトリ
あなたのPC
git push →
← git pull
← git fetch
← git clone
☁
リモートリポジトリ
GitHub

push=ローカルからリモートへ送信 / pull=リモートからローカルへ取得してマージ / fetch=取得のみ / clone=初回コピー

lightbulb originとは

リモートリポジトリにはエイリアス（ニックネーム）を付けて管理します。最も一般的な名前がoriginです。git cloneでリポジトリをコピーした場合、クローン元のリモートリポジトリが自動的にoriginという名前で登録されます。

Section 9-2

リモートの追加と確認（git remote）

git remoteコマンドを使って、リモートリポジトリの登録、確認、変更、削除を行います。

2.1 リモートの確認

リモートの確認コマンド Terminal 
# 登録されているリモート名の一覧
git remote
# 出力例: origin

# URLも含めて詳細表示
git remote -v
# 出力例:
# origin  git@github.com:username/my-project.git (fetch)
# origin  git@github.com:username/my-project.git (push)

# 特定のリモートの詳細情報
git remote show origin
# 出力例:
# * remote origin
#   Fetch URL: git@github.com:username/my-project.git
#   Push  URL: git@github.com:username/my-project.git
#   HEAD branch: main
#   Remote branches:
#     main tracked
#     feature/login tracked

2.2 リモートの追加・変更・削除

リモートの操作 Terminal 
# リモートを追加
git remote add origin git@github.com:username/my-project.git

# 複数のリモートを登録することも可能
git remote add upstream git@github.com:original-author/project.git

# リモート名を変更
git remote rename origin github

# リモートURLを変更
git remote set-url origin git@github.com:username/new-project.git

# リモートを削除
git remote remove upstream
lightbulb upstreamリモートの使い方

OSSにコントリビュートする場合、フォーク元のリポジトリをupstreamとして登録することがよくあります。これにより、自分のフォーク（origin）とオリジナルのリポジトリ（upstream）の両方を管理できます。git fetch upstreamで最新の変更を取得できます。

Section 9-3

プッシュとプル（git push, git pull）

リモートリポジトリとの間でデータをやり取りする最も基本的な操作が、プッシュ（push）とプル（pull）です。

3.1 git push - ローカルの変更をリモートに送信

git pushの使い方 Terminal 
# 基本的なプッシュ
git push origin main
# → originリモートのmainブランチにプッシュ

# 初回プッシュ（上流ブランチの設定）
git push -u origin main
# → -u オプションで上流ブランチを設定
# → 以降は git push だけでプッシュ可能

# 現在のブランチをプッシュ（上流ブランチ設定済みの場合）
git push

# 新しいブランチをリモートにプッシュ
git switch -c feature/new-page
# ... 作業してコミット ...
git push -u origin feature/new-page

# リモートのブランチを削除
git push origin --delete feature/old-branch
warning force pushは危険

git push --force（または-f）はリモートの履歴を強制的に上書きします。他のメンバーの作業が消える可能性があるため、原則として使用しないでください。どうしても必要な場合はgit push --force-with-leaseを使うと、他の人が変更を加えていた場合にプッシュが拒否されるため、より安全です。

3.2 git pull - リモートの変更をローカルに取得

git pullは、リモートリポジトリの最新の変更を取得し、ローカルブランチにマージする操作です。内部的にはgit fetch+git mergeと同等の動作をします。

git pullの使い方 Terminal 
# リモートの最新を取得してマージ
git pull origin main

# 上流ブランチが設定されていれば省略可能
git pull

# rebaseモードでpull（マージコミットを作らない）
git pull --rebase origin main

# pullのデフォルトをrebaseに設定
git config --global pull.rebase true
lightbulb pullする前にcommitしよう

git pullを実行する前に、ローカルの変更をコミットしておくことをおすすめします。未コミットの変更がある状態でpullすると、コンフリクトが発生した場合に複雑になることがあります。git stashで一時退避するのも一つの方法です。

Section 9-4

クローンとフェッチ（git clone, git fetch）

リモートリポジトリからの初回のコピーにはgit cloneを、最新情報の取得にはgit fetchを使います。

4.1 git clone - リポジトリのコピー

git cloneは、リモートリポジトリの全てのコミット履歴を含む完全なコピーをローカルに作成します。

git cloneの使い方 Terminal 
# SSH URLでクローン（推奨）
git clone git@github.com:username/my-project.git

# HTTPS URLでクローン
git clone https://github.com/username/my-project.git

# ディレクトリ名を指定してクローン
git clone git@github.com:username/my-project.git my-folder

# 特定のブランチだけクローン
git clone -b develop git@github.com:username/my-project.git

# 浅いクローン（最新のコミットだけ取得、高速）
git clone --depth 1 git@github.com:username/my-project.git

4.2 git fetch - リモートの情報を取得

git fetchは、リモートリポジトリの最新情報を取得しますが、ローカルのブランチには統合しません。取得した情報はorigin/mainのようなリモート追跡ブランチに保存されます。

git fetchの使い方 Terminal 
# リモートの最新情報を取得（マージしない）
git fetch origin

# 取得した情報を確認
git log origin/main --oneline
# リモートのmainブランチの履歴が確認できる

# ローカルとリモートの差分を確認
git diff main origin/main

# 問題なければ手動でマージ
git merge origin/main

# 不要になったリモート追跡ブランチを削除
git fetch --prune
コマンド 取得 マージ 安全性
git fetch リモートの情報を取得する マージしない 安全（ローカルを変更しない）
git pull リモートの情報を取得する 自動でマージする コンフリクトの可能性あり
lightbulb fetchを先に使う習慣

チーム開発では、いきなりgit pullするのではなく、まずgit fetchで変更内容を確認してから手動でマージする方が安全です。特に大規模な変更がある場合や、コンフリクトが予想される場合は、git fetch + git diff + git mergeの手順がおすすめです。

Section 9-5

コラボレーターの招待と権限管理

GitHubでは、リポジトリに他の開発者をコラボレーターとして招待し、共同で開発を進めることができます。権限レベルを設定することで、メンバーごとにできる操作を制限できます。

5.1 コラボレーターの招待方法

  1. リポジトリのSettingsタブを開く
  2. 左メニューからCollaborators（またはManage access）をクリック
  3. Add peopleボタンをクリック
  4. 招待したい人のGitHubユーザー名またはメールアドレスを入力
  5. 権限レベルを選択して招待を送信
  6. 相手が招待を承認すれば、コラボレーターとして参加完了

5.2 権限レベル

← 権限が低い 　　　　 権限が高い →

権限レベル できること 適する人
Read コード閲覧、Issue作成、PR作成 外部のレビュアー、閲覧者

Admin 全ての操作（削除含む） リポジトリオーナー
warning 最小権限の原則

セキュリティの観点から、各メンバーには必要最小限の権限を付与しましょう。全員にAdmin権限を与えてしまうと、誤ってリポジトリを削除したり、設定を変更してしまうリスクがあります。一般の開発メンバーにはWrite権限で十分です。

Exercise

9章 練習問題

Q1

git pullは内部的にどの2つのコマンドと同等の動作をしますか？

  1. git clone + git merge
  2. git fetch + git merge
  3. git fetch + git rebase
  4. git push + git merge
Q2

初めてローカルブランチをリモートにプッシュする際、上流ブランチを設定するオプションはどれですか？

  1. git push --force origin main
  2. git push -u origin main
  3. git push --set origin main
  4. git push --track origin main
Q3

git fetchとgit pullの違いとして正しいものはどれですか？

  1. fetchはリモートの変更を取得しマージまで行い、pullは取得のみ
  2. fetchはリモートの変更を取得するのみで、pullは取得とマージを行う
  3. fetchとpullは全く同じ動作をする
  4. fetchはローカルの変更をリモートに送信する
Answer

9章 解説・解答

Q1の解答: B

git pullは、git fetch（リモートの情報を取得）とgit merge（ローカルブランチに統合）を一度に行うコマンドです。git pull --rebaseオプションを使うと、merge の代わりに rebase で統合します。

Q2の解答: B

-u（--set-upstreamの短縮形）オプションは、ローカルブランチとリモートブランチの紐付け（上流ブランチの設定）を行います。一度設定すれば、以降はgit pushやgit pullでブランチ名を省略できるようになります。

Q3の解答: B

git fetchはリモートの最新情報をリモート追跡ブランチ（origin/mainなど）に取得するだけで、ローカルのブランチには変更を加えません。git pullは取得と同時にローカルブランチにマージまで行います。安全に確認したい場合はgit fetch+git diff+git mergeの手順がおすすめです。
