Section 13-1

GitHub Pages（静的サイトホスティング）

GitHub Pagesは、GitHubリポジトリから直接Webサイトを無料で公開できるホスティングサービスです。ポートフォリオサイトやプロジェクトのドキュメントサイトなど、静的サイトの公開に最適です。

1.1 GitHub Pagesの特徴

💰

無料で利用可能

パブリックリポジトリなら無料でWebサイトを公開できます。独自ドメインの設定も可能です。

🚀

自動デプロイ

リポジトリにpushするだけで自動的にサイトが更新されます。手動でサーバーにアップロードする必要はありません。

🔒

HTTPS対応

デフォルトでHTTPS（SSL証明書）が適用されるため、安全な通信が保証されます。

1.2 GitHub Pagesの設定手順

  1. GitHubリポジトリのページを開く
  2. Settingsタブをクリック
  3. 左メニューのPagesを選択
  4. 「Source」で公開するブランチ（通常はmain）を選択
  5. 公開フォルダを選択（/ (root) または /docs）
  6. Saveをクリック

数分後、https://ユーザー名.github.io/リポジトリ名/ でサイトが公開されます。

GitHub Pages用の基本的なindex.html HTML 
<!DOCTYPE html>
<html lang="ja">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>My Portfolio</title>
    <link rel="stylesheet" href="style.css">
</head>
<body>
    <h1>Welcome to My Portfolio</h1>
    <p>GitHub Pagesで公開しています！</p>
</body>
</html>
lightbulb ユーザーサイトとプロジェクトサイト

ユーザー名.github.ioという名前のリポジトリを作ると、https://ユーザー名.github.io/ でアクセスできるユーザーサイトになります。それ以外のリポジトリはプロジェクトサイトとして/リポジトリ名のパス配下で公開されます。

Section 13-2

GitHub Actions（CI/CD）

GitHub Actionsは、GitHubに組み込まれたCI/CD（継続的インテグレーション/継続的デリバリー）プラットフォームです。コードのプッシュやプルリクエストなどのイベントをトリガーに、テストの実行やデプロイなどを自動化できます。

⚙ GitHub Actions のワークフロー構造
⚡ Event push, PR, schedule
→
📄 Workflow .yml ファイル
→
💼 Job 実行単位（並列可）
→
👉 Step 個別のコマンド

2.1 GitHub Actionsの基本概念

用語 説明
Event（イベント） ワークフローを起動するトリガー（push、pull_request、scheduleなど）
Workflow（ワークフロー） 自動化プロセス全体の定義。YAMLファイルで記述する
Job（ジョブ） ワークフロー内の実行単位。複数のジョブは並列実行が可能
Step（ステップ） ジョブ内の個別の実行コマンドやアクション
Runner（ランナー） ワークフローを実行するサーバー（Ubuntu、Windows、macOS）

2.2 ワークフローファイルの作成

ワークフローは、リポジトリの.github/workflows/ディレクトリにYAMLファイルとして配置します。

.github/workflows/ci.yml - 基本的なCIワークフロー YAML 
name: CI

# トリガー: mainブランチへのpushまたはPR作成時
on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    # 実行環境
    runs-on: ubuntu-latest

    steps:
      # リポジトリのコードをチェックアウト
      - uses: actions/checkout@v4

      # Node.jsのセットアップ
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      # 依存パッケージのインストール
      - name: Install dependencies
        run: npm ci

      # テストの実行
      - name: Run tests
        run: npm test

      # ビルド
      - name: Build
        run: npm run build
.github/workflows/deploy.yml - GitHub Pagesへの自動デプロイ YAML 
name: Deploy to GitHub Pages

on:
  push:
    branches: [ main ]

# GitHub Pagesへのデプロイ権限
permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}

    steps:
      - uses: actions/checkout@v4

      - name: Setup Pages
        uses: actions/configure-pages@v4

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: '.'

      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
lightbulb Actionsの実行結果を確認する

ワークフローの実行結果は、リポジトリのActionsタブから確認できます。成功は緑色のチェックマーク、失敗は赤色のバツ印で表示されます。失敗した場合はログを確認してエラーの原因を特定しましょう。

Section 13-3

GitHub Codespaces（クラウド開発環境）

GitHub Codespacesは、ブラウザ上でVSCodeと同じ開発環境を利用できるクラウドサービスです。ローカル環境のセットアップなしで、すぐに開発を始められます。

3.1 Codespacesの起動方法

  1. GitHubリポジトリのページを開く
  2. 緑色の「Code」ボタンをクリック
  3. 「Codespaces」タブを選択
  4. 「Create codespace on main」をクリック
  5. 数秒〜数分でブラウザ版VSCodeが起動する

3.2 Codespacesの利点

💻

環境構築不要

Node.js、Python、Javaなどのランタイムが事前にインストールされており、すぐに開発を始められます。

👥

統一された開発環境

.devcontainer/devcontainer.jsonで環境を定義すれば、チーム全員が同じ環境で開発できます。「自分の環境では動くのに」問題を解消します。

🌐

どこからでもアクセス

ブラウザさえあれば、どのPCからでも同じ開発環境にアクセスできます。iPad等のタブレットからも作業可能です。

.devcontainer/devcontainer.json の例 JSON 
{
  "name": "Node.js Development",
  "image": "mcr.microsoft.com/devcontainers/javascript-node:20",
  "features": {
    "ghcr.io/devcontainers/features/git:1": {}
  },
  "postCreateCommand": "npm install",
  "customizations": {
    "vscode": {
      "extensions": [
        "esbenp.prettier-vscode",
        "dbaeumer.vscode-eslint"
      ]
    }
  },
  "forwardPorts": [3000]
}
lightbulb Codespacesの無料枠

GitHub Freeプランでは、月120コア時間（2コアマシンで約60時間）まで無料で利用できます。学習用途であれば十分な量です。使わない時は必ずCodespaceを停止して、時間を節約しましょう。

Section 13-4

Wiki・Discussions・Projects

GitHubにはコード管理以外にも、チームのコラボレーションを支援する多くの機能があります。

4.1 Wiki（ドキュメント管理）

GitHub Wikiは、リポジトリに付属するドキュメント作成スペースです。

  • Markdownでドキュメントを作成・編集できる
  • 複数ページに分けてプロジェクトの説明書を作成
  • サイドバーでページ間のナビゲーションを設定可能
  • Git管理されるため、履歴の確認や復元ができる

4.2 Discussions（ディスカッション）

GitHub Discussionsは、フォーラム形式のコミュニケーション機能です。Issueがバグや機能要望など具体的なタスクに向いているのに対し、Discussionsはより自由な議論に適しています。

  • Q&A：質問と回答（ベストアンサー機能あり）
  • Announcements：プロジェクトのお知らせ
  • General：一般的な議論
  • Ideas：アイデアの提案と投票
  • Show and tell：成果の共有

4.3 Projects（プロジェクト管理）

GitHub Projectsは、カンバンボード形式のタスク管理ツールです。IssueやPull Requestと連携して、開発の進捗を視覚的に管理できます。

  • ボードビュー：カンバン形式でタスクを「Todo」「In Progress」「Done」に分類
  • テーブルビュー：スプレッドシートのような表形式で管理
  • ロードマップビュー：タイムライン形式で計画を可視化
  • カスタムフィールドで優先度や担当者を設定可能
lightbulb IssueとProjectsの連携

Issueに「Closes #123」のようなキーワードをPRに含めると、PRがマージされた時にIssueが自動的にクローズされます。Projectsのボード上でもタスクが自動的に「Done」に移動するため、管理が楽になります。

Section 13-5

GitHub Marketplace と統合機能

GitHub Marketplaceは、GitHubと連携するアプリケーションやActionsを検索・導入できるプラットフォームです。開発ワークフローを拡張するツールが豊富に揃っています。

🌎 GitHubの機能エコシステム
🌐 Pages 静的サイトホスティング
⚙ Actions CI/CD自動化
💻 Codespaces クラウド開発環境
📖 Wiki ドキュメント管理
📊 Projects タスク管理ボード
💬 Discussions コミュニティフォーラム
🛒 Marketplace ツール・拡張機能
🛡 Security 脆弱性スキャン

5.1 人気のMarketplaceアプリ

アプリ名 カテゴリ 機能
Dependabot セキュリティ 依存パッケージの脆弱性検知と自動更新PR作成
CodeQL セキュリティ コードの脆弱性を自動スキャン
Codecov テスト テストカバレッジの計測とレポート
Vercel デプロイ フロントエンドアプリの自動デプロイとプレビュー
Slack Integration 通知 GitHubの通知をSlackに連携
lightbulb GitHub Copilot

GitHub CopilotはAIペアプログラマーで、コードの補完やチャットによるコーディング支援を提供します。VSCodeやJetBrains IDEなどのエディタに拡張機能として導入でき、GitHub Actionsのワークフロー作成の補助にも活用できます。学生は無料で利用可能です。

13章 練習問題

Q1

GitHub Actionsのワークフローファイルを配置するディレクトリはどれですか？

  1. .github/actions/
  2. .github/workflows/
  3. .actions/
  4. workflows/
Q2

GitHub Pagesで公開されるサイトのURLとして正しいものはどれですか？（ユーザー名: tanaka、リポジトリ名: portfolio）

  1. https://github.com/tanaka/portfolio
  2. https://tanaka.github.io/portfolio/
  3. https://pages.github.com/tanaka/portfolio
  4. https://tanaka.github.com/portfolio/
Q3

GitHub Codespacesの説明として正しいものはどれですか？

  1. ローカルにIDEをインストールする必要がある
  2. ブラウザ上でVSCodeと同じ開発環境を利用できる
  3. Gitのバージョン管理機能を提供する
  4. 静的サイトのホスティングサービスである
Answer

13章 解説・解答

Q1の解答: B

GitHub Actionsのワークフローファイルは、リポジトリの.github/workflows/ディレクトリに配置します。YAML形式（.ymlまたは.yaml）で記述し、ファイル名は自由に設定できます。

Q2の解答: B

GitHub PagesのURLは https://ユーザー名.github.io/リポジトリ名/ の形式になります。ユーザー名が「tanaka」、リポジトリ名が「portfolio」の場合、https://tanaka.github.io/portfolio/ でアクセスできます。

Q3の解答: B

GitHub Codespacesはブラウザ上でVSCodeと同じ開発環境を利用できるクラウドサービスです。ローカルへのIDEインストールが不要で、どの端末からでも同じ環境で開発できます。
