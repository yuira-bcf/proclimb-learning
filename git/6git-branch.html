Section 6-1

ブランチとは

ブランチ（branch）は、Gitにおける最も強力な機能の一つです。「branch」は英語で「枝」を意味し、コミット履歴の流れから枝分かれして、独立した開発ラインを作ることができます。これにより、メインのコードに影響を与えることなく、新機能の開発やバグ修正を安全に進められます。

1.1 ブランチの仕組み

Gitのブランチは、特定のコミットを指すポインタ（参照）です。ブランチを作成しても、ファイルのコピーが作られるわけではありません。単に「このコミットから開発を続ける」という印をつけるだけなので、非常に軽量で高速です。

🌳

軽量なポインタ

ブランチはコミットのSHA-1ハッシュを記録した40バイトのファイルに過ぎません。作成も削除もほぼ一瞬で完了します。

🔒

独立した開発ライン

あるブランチでの変更は、他のブランチに一切影響しません。安心して実験的な開発ができます。

👥

並行開発の実現

複数人で同時に異なる機能を開発する場合、それぞれが専用のブランチで作業できます。

🌳 ブランチの分岐イメージ
📍 main メインブランチ
→
📝 commit A 初期コミット
→
📝 commit B 機能追加
↗
🌿 feature/login ログイン機能開発

mainブランチのcommit Bから feature/login ブランチが分岐して独立した開発を行う

lightbulb HEADとは

HEADは「今どのブランチで作業しているか」を示す特別なポインタです。git switchでブランチを切り替えると、HEADが指すブランチも変わります。git log --onelineでHEAD -> mainのように表示され、現在のブランチを確認できます。

Section 6-2

ブランチの作成と切り替え

ブランチの作成と切り替えは、Git操作の中で最も頻繁に行う操作の一つです。ここではgit branchとgit switchコマンドを使った基本操作を学びます。

2.1 ブランチの作成（git branch）

新しいブランチを作成するには、git branchコマンドにブランチ名を指定します。ただし、このコマンドはブランチを作成するだけで、切り替えは行いません。

ブランチの作成 Terminal 
# 新しいブランチを作成（現在のブランチはそのまま）
git branch feature/login

# 作成されたブランチを確認
git branch
# 出力例:
#   feature/login
# * main          ← * が付いているのが現在のブランチ

2.2 ブランチの切り替え（git switch）

git switchはGit 2.23で導入された、ブランチの切り替え専用コマンドです。従来はgit checkoutが使われていましたが、git switchの方が直感的で安全です。

ブランチの切り替え Terminal 
# 既存のブランチに切り替え
git switch feature/login
# 出力: Switched to branch 'feature/login'

# 現在のブランチを確認
git branch
# 出力例:
# * feature/login   ← 現在のブランチ
#   main

2.3 ブランチの作成と切り替えを同時に行う

実際の開発では、ブランチを作成したらすぐに切り替えることがほとんどです。git switch -c（create の略）を使えば、一つのコマンドで両方を同時に実行できます。

ブランチの作成と同時切り替え（推奨） Terminal 
# ブランチを作成して同時に切り替え（最もよく使うパターン）
git switch -c feature/signup
# 出力: Switched to a new branch 'feature/signup'

# 従来のコマンド（git checkout -b）でも同じことができる
git checkout -b feature/signup
lightbulb git switch と git checkout の違い

git checkoutはブランチの切り替え以外にも、ファイルの復元など多くの機能を持つ万能コマンドでした。混乱を防ぐため、Git 2.23以降ではブランチ操作にはgit switch、ファイルの復元にはgit restoreと、役割が分離されました。新しいプロジェクトではgit switchの使用が推奨されています。

warning 未コミットの変更がある場合の注意

ワーキングディレクトリに未コミットの変更がある状態でブランチを切り替えると、変更内容が切り替え先のブランチに持ち越されます。コンフリクトが発生する場合は切り替えがブロックされます。切り替え前にgit stashで一時退避するか、git commitで変更を保存してから切り替えましょう。

Section 6-3

ブランチの一覧と削除

開発が進むとブランチの数が増えていきます。不要になったブランチを放置するとリポジトリが散らかってしまうため、定期的な整理が重要です。

3.1 ブランチの一覧表示

ブランチ一覧の確認 Terminal 
# ローカルブランチの一覧を表示
git branch
# 出力例:
#   bugfix/header-layout
#   feature/login
# * main                   ← 現在のブランチ
#   feature/signup

# リモートブランチも含めて表示
git branch -a
# 出力例:
#   bugfix/header-layout
#   feature/login
# * main
#   remotes/origin/main
#   remotes/origin/feature/login

# 各ブランチの最新コミット情報も表示
git branch -v
# 出力例:
#   bugfix/header-layout  a1b2c3d ヘッダーレイアウト修正
#   feature/login         e4f5g6h ログイン画面追加
# * main                  h7i8j9k 初期リリース

3.2 ブランチの削除

マージが完了して不要になったブランチは削除しましょう。Gitには安全な削除と強制削除の2種類があります。

ブランチの削除 Terminal 
# 安全な削除（マージ済みのブランチのみ削除可能）
git branch -d feature/login
# 出力: Deleted branch feature/login (was e4f5g6h).

# 強制削除（マージされていないブランチも削除）
git branch -D feature/experimental
# 出力: Deleted branch feature/experimental (was k1l2m3n).

# 現在チェックアウトしているブランチは削除できない
git switch main    # まずmainに切り替えてから
git branch -d feature/login  # 削除する
warning -D（大文字）は慎重に使おう

-d（小文字）はマージ済みのブランチのみ削除するため安全です。一方、-D（大文字）はマージされていないブランチも強制的に削除します。まだマージしていない重要な変更が失われる可能性があるため、使用前に必ず内容を確認してください。

3.3 マージ済みブランチのまとめて削除

マージ済みブランチの確認と一括削除 Terminal 
# mainにマージ済みのブランチを確認
git branch --merged main
# 出力例:
#   feature/login
#   bugfix/header-layout
# * main

# mainにマージされていないブランチを確認
git branch --no-merged main
# 出力例:
#   feature/signup
Section 6-4

ブランチの命名規則

ブランチ名はチーム開発において重要なコミュニケーション手段です。一貫した命名規則を守ることで、「このブランチが何のためのものか」が一目で分かるようになります。

4.1 一般的なプレフィックス

📌 ブランチ命名規則の例
✨ feature/ 新機能の開発
例: feature/user-auth
🐛 bugfix/ バグの修正
例: bugfix/login-error
🔥 hotfix/ 緊急修正
例: hotfix/security-patch
📦 release/ リリース準備
例: release/v2.1.0
🔬 experiment/ 実験的な試み
例: experiment/new-ui
🛠 chore/ 雑務・設定変更
例: chore/update-deps
プレフィックス 用途 例
feature/ 新しい機能の追加 feature/user-profile
bugfix/ バグの修正 bugfix/cart-total-error
hotfix/ 本番環境の緊急修正 hotfix/sql-injection
release/ リリース準備 release/v1.2.0

4.2 命名のベストプラクティス

  • 小文字とハイフンを使う: feature/user-login（スペースやアンダースコアは避ける）
  • 短く分かりやすい名前にする: 何の作業かが一目で分かる長さ
  • Issue番号を含める: feature/123-user-auth（トラッキングしやすくなる）
  • 日本語は使わない: 環境によって文字化けする可能性がある
  • スラッシュでカテゴリを分ける: feature/、bugfix/のようにプレフィックスを付ける
良い命名例と悪い命名例 Terminal 
# 良い命名例
feature/user-authentication
bugfix/fix-header-overlap
hotfix/patch-xss-vulnerability
release/v2.0.0

# 悪い命名例
new-feature          # プレフィックスがない
Feature/Login        # 大文字を使っている
feature/ログイン機能  # 日本語を使っている
my-branch            # 何の作業か分からない
temp                 # 意味が不明確
Section 6-5

ブランチの活用パターン

ブランチをどのように運用するかは、チームやプロジェクトの規模によって異なります。ここでは代表的なブランチ戦略を紹介します。

5.1 GitHub Flow

GitHub Flowは最もシンプルなブランチ戦略で、小規模チームや頻繁にデプロイするプロジェクトに適しています。

  • mainブランチは常にデプロイ可能な状態を維持する
  • 新しい作業を始めるときはmainからトピックブランチを作成する
  • 作業が完了したらプルリクエストを作成してレビューを受ける
  • レビュー後にmainへマージしてデプロイする
GitHub Flow の基本的な流れ Terminal 
# 1. mainブランチを最新にする
git switch main
git pull origin main

# 2. 新しいブランチを作成して切り替え
git switch -c feature/add-search

# 3. 開発作業とコミット
# ... ファイルを編集 ...
git add .
git commit -m "検索機能を追加"

# 4. リモートにプッシュ
git push origin feature/add-search

# 5. GitHubでプルリクエストを作成
# 6. レビュー後にマージ
# 7. ローカルのブランチを削除
git switch main
git pull origin main
git branch -d feature/add-search

5.2 Git Flow

Git Flowはより厳密なブランチ戦略で、リリースサイクルが明確なプロジェクトに適しています。常時2つのメインブランチ（mainとdevelop）を維持します。

ブランチ 役割 ライフサイクル
main 本番環境と同じ状態 永続的
develop 次回リリースの開発ベース 永続的
feature/* 個別の機能開発 一時的（develop から分岐、develop へマージ）
release/* リリース準備 一時的（develop から分岐、main と develop へマージ）
hotfix/* 緊急バグ修正 一時的（main から分岐、main と develop へマージ）
lightbulb 初学者へのおすすめ

最初のうちはGitHub Flowから始めるのがおすすめです。シンプルで理解しやすく、個人開発でも小規模チームでも使えます。Git Flowはプロジェクトが大きくなり、リリース管理が必要になった段階で検討しましょう。

Exercise

6章 練習問題

Q1

新しいブランチ「feature/contact-form」を作成して、同時にそのブランチに切り替えるコマンドはどれですか？

  1. git branch feature/contact-form
  2. git switch feature/contact-form
  3. git switch -c feature/contact-form
  4. git branch -c feature/contact-form
Q2

マージ済みのブランチを安全に削除するコマンドのオプションはどれですか？

  1. git branch -D
  2. git branch -d
  3. git branch --remove
  4. git branch --delete-force
Q3

ブランチの命名規則として適切なものはどれですか？

  1. Feature/UserLogin
  2. feature/user-login
  3. feature/ユーザーログイン
  4. my new feature
Answer

6章 解説・解答

Q1の解答: C

git switch -cは、ブランチの作成（create）と切り替えを同時に行うコマンドです。Aはブランチの作成のみ、Bは既存ブランチへの切り替えのみを行います。Dのgit branch -cはブランチのコピー（copy）を作成するオプションであり、切り替えは行いません。

Q2の解答: B

git branch -d（小文字のd）は、マージ済みのブランチのみを削除する安全なオプションです。マージされていないブランチを削除しようとするとエラーになります。-D（大文字のD）はマージの有無に関わらず強制削除するため、慎重に使う必要があります。

Q3の解答: B

feature/user-loginが正しい命名規則に従っています。小文字を使い、単語の区切りにハイフンを使い、プレフィックス（feature/）でブランチの目的を示しています。Aは大文字を含み、Cは日本語を使用しており、Dはプレフィックスがなくスペースも含まれています。
