Section 1-1

バージョン管理とは

バージョン管理（Version Control）とは、ファイルの変更履歴を記録し、必要に応じて過去の状態に戻したり、変更内容を比較したりできる仕組みのことです。ソフトウェア開発において最も重要なツールの一つです。

1.1 なぜバージョン管理が必要なのか

プログラミングをしていると、次のような経験をしたことはありませんか？

  • 動いていたコードを修正したら動かなくなった。元に戻したいが、どこを変えたか覚えていない
  • ファイルを「最終版」「最終版2」「最終版_本当の最終」と増やしてしまった
  • チームで同じファイルを編集して、誰かの変更を上書きしてしまった
  • いつ、誰が、なぜその変更をしたのか分からなくなった

バージョン管理システム（VCS: Version Control System）を使えば、これらの問題をすべて解決できます。

lightbulb バージョン管理は「タイムマシン」

バージョン管理を使えば、プロジェクトの任意の時点の状態に戻ることができます。まるでタイムマシンのように、過去のどの時点にも自由に移動できるのです。ゲームの「セーブポイント」をイメージすると分かりやすいでしょう。

1.2 バージョン管理でできること

📅

変更履歴の記録

いつ、誰が、何を変更したかを自動的に記録します。コードレビューやバグ追跡に欠かせません。

⏪

過去の状態への復元

間違った変更をしてしまっても、いつでも過去の正しい状態に戻すことができます。

👥

チームでの共同作業

複数の開発者が同時に同じプロジェクトで作業でき、変更の衝突も安全に解決できます。

🔌

並行開発（ブランチ）

本番環境に影響を与えず、新機能の開発やバグ修正を並行して進められます。

Section 1-2

バージョン管理の歴史

バージョン管理の概念は、ソフトウェア開発の歴史とともに進化してきました。その変遷を知ることで、現在のGitがなぜこのような設計になっているかが理解できます。

2.1 ローカル型バージョン管理

初期のバージョン管理は、個人のコンピュータ内でファイルのコピーを管理する方式でした。代表的なシステムにRCS（Revision Control System、1982年）があります。

  • ファイルの差分をローカルに保存
  • 1人での開発には十分だが、チーム開発には不向き
  • コンピュータが壊れたらデータを失う危険性

2.2 集中型バージョン管理

チーム開発の需要から、中央サーバーでバージョンを一元管理する方式が登場しました。CVS（1990年）やSubversion（SVN、2000年）が代表例です。

  • 中央サーバーに全ての履歴を集中管理
  • チームメンバーはサーバーからファイルを取得して作業
  • サーバーがダウンすると全員が作業不能になる弱点

2.3 分散型バージョン管理

2005年、Linuxカーネルの開発者リーナス・トーバルズがGitを開発しました。各開発者がリポジトリの完全なコピーを持つ「分散型」の仕組みです。

  • 各開発者が履歴を含む完全なリポジトリのコピーを保持
  • オフラインでも作業可能
  • サーバーが壊れても、各開発者のコピーから復元可能
  • 高速な操作（ほとんどの操作がローカルで完結）
lightbulb Gitの誕生秘話

Gitは、Linuxカーネルの開発で使っていた商用ツール（BitKeeper）が無料で使えなくなったことをきっかけに、リーナス・トーバルズがわずか2週間で基本部分を開発しました。「速度」「シンプルな設計」「大規模プロジェクトへの対応」を重視して設計されています。

Section 1-3

集中型と分散型

バージョン管理システムの2大方式である「集中型」と「分散型」の違いを詳しく見ていきましょう。これを理解することがGitの仕組みを理解する第一歩です。

📊 集中型 vs 分散型 バージョン管理
集中型（SVNなど）
🖥 中央サーバー 全履歴を一元管理
↓
💻 開発者A 最新版のみ
💻 開発者B 最新版のみ
分散型（Gitなど）
🖥 リモートリポジトリ 共有用リポジトリ
↕
💻 開発者A 完全な履歴
💻 開発者B 完全な履歴

3.1 集中型の特徴

項目 集中型（SVN等） 分散型（Git等）
リポジトリ サーバーに1つだけ 各開発者が完全なコピーを所有
オフライン作業 不可（サーバー接続が必須） 可能（ローカルで完結）
速度 ネットワーク速度に依存 高速（ほとんどローカル処理）
障害耐性 サーバー故障で全停止 各コピーから復元可能
ブランチ コストが高い 高速・軽量
warning 集中型の落とし穴

集中型バージョン管理では、中央サーバーがダウンすると全ての開発者が作業できなくなります。また、サーバーのハードディスクが壊れた場合、バックアップがなければ全ての履歴が失われます。分散型のGitではこのリスクが大幅に軽減されます。

Section 1-4

Gitが選ばれる理由

現在、ソフトウェア開発の現場ではGitがバージョン管理のデファクトスタンダード（事実上の標準）です。なぜGitがここまで広く普及したのでしょうか。

4.1 Gitの主な利点

⚡

圧倒的な速度

ほとんどの操作がローカルで完結するため、ネットワーク遅延の影響を受けません。ブランチの作成やコミットは一瞬で完了します。

🔒

データの完全性

Gitは全てのデータにSHA-1ハッシュを使用しており、データの改ざんや破損を検出できます。

🔌

強力なブランチ機能

軽量なブランチの作成・マージが可能で、並行開発やフィーチャーブランチ戦略を簡単に実現できます。

🌐

GitHubとの連携

GitHub、GitLab、Bitbucketなどのホスティングサービスとシームレスに連携し、チーム開発を強力にサポートします。

4.2 Gitの普及状況

Stack Overflowの開発者調査によると、プロの開発者の90%以上がGitを使用しています。Web開発、モバイル開発、データサイエンスなど、あらゆる分野でGitは必須スキルとなっています。

Git 93%
その他 7%
lightbulb Gitはプログラマーだけのものではない

最近ではデザイナー、ライター、研究者など、プログラマー以外の人もGitを活用しています。ドキュメントの変更管理や、データ分析のコード管理など、バージョン管理が必要なあらゆる場面でGitが使われています。

Section 1-5

バージョン管理の基本用語

Gitを学ぶ前に、バージョン管理で頻繁に使われる基本用語を押さえておきましょう。これらの用語は今後の章で繰り返し登場します。

📚 Gitの基本用語マップ
📁 リポジトリ プロジェクトの保管庫。変更履歴を全て記録する
📸 コミット 変更のスナップショットを記録する操作
🔌 ブランチ 作業を分岐させて並行開発する仕組み
🔀 マージ 分岐した作業を統合する操作
📥 クローン リモートリポジトリの完全なコピーを作成
📤 プッシュ / プル リモートとローカルの同期操作

5.1 用語の詳細解説

用語 英語 説明
リポジトリ Repository プロジェクトのファイルと変更履歴を保存する場所。「リポ」や「レポ」と略すことも
コミット Commit 変更内容を記録するスナップショット。メッセージ付きで履歴に保存される
ブランチ Branch メインの開発ラインから分岐した独立した作業ライン
マージ Merge 別のブランチの変更を現在のブランチに統合する操作
クローン Clone リモートリポジトリの完全なコピーをローカルに作成する操作
プッシュ Push ローカルの変更をリモートリポジトリに送信する操作
プル Pull リモートリポジトリの変更をローカルに取り込む操作
コンフリクト Conflict 同じ箇所を複数人が変更した場合に発生する衝突
lightbulb 用語の覚え方

最初は用語が多くて圧倒されるかもしれませんが、心配ありません。実際にGitを使いながら自然に覚えていきましょう。特にリポジトリ、コミット、ブランチの3つは最も重要な概念なので、まずこの3つをしっかり理解することを目指しましょう。

Exercise

1章 練習問題

Q1

バージョン管理システムの主な目的として、最も適切でないものはどれですか？

  1. ファイルの変更履歴を記録する
  2. 過去の状態に戻すことができる
  3. ファイルを自動的にバックアップサーバーに保存する
  4. チームでの共同作業を支援する
Q2

分散型バージョン管理（Git）の特徴として正しいものはどれですか？

  1. 中央サーバーに接続しないと一切の作業ができない
  2. 各開発者がリポジトリの完全なコピーを持つ
  3. ブランチの作成にはサーバー管理者の承認が必要
  4. ファイルの最新版のみをローカルに保存する
Q3

Gitにおいて、変更内容をスナップショットとして履歴に記録する操作を何と呼びますか？

  1. プッシュ（Push）
  2. マージ（Merge）
  3. コミット（Commit）
  4. クローン（Clone）
Answer

1章 解説・解答

Q1の解答: C

バージョン管理システムの主な目的は「変更履歴の記録」「過去の状態への復元」「チームでの共同作業の支援」です。自動バックアップは副次的な効果であり、VCSの主な目的ではありません。バックアップはあくまでリモートリポジトリにプッシュした場合の結果です。

Q2の解答: B

分散型バージョン管理では、各開発者がリポジトリの完全なコピー（全履歴を含む）をローカルに保持します。そのためオフラインでもコミットやブランチ作成が可能です。Aは集中型の特徴、Cは誤り（Gitではローカルで自由にブランチ作成可能）、Dも集中型の特徴です。

Q3の解答: C

コミット（Commit）は、変更内容をスナップショットとしてリポジトリの履歴に記録する操作です。プッシュはリモートへの送信、マージはブランチの統合、クローンはリポジトリのコピー作成です。
