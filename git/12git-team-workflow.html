Section 12-1

チーム開発ワークフロー

チーム開発では、複数の開発者が同時に作業を進めます。混乱なくコードを管理するためには、ブランチ戦略と呼ばれるワークフロー（作業の進め方のルール）を決めておくことが重要です。

1.1 ワークフローが必要な理由

ワークフローを決めずにチーム開発を行うと、以下のような問題が起きやすくなります。

  • mainブランチに直接コミットしてしまい、不安定なコードが本番に反映される
  • 誰がどのブランチで何をしているか分からなくなる
  • コンフリクトが頻発し、解決に時間がかかる
  • リリースのタイミングが管理できない
lightbulb 代表的なワークフロー

現在、広く使われているGitのワークフローは主にGit FlowとGitHub Flowの2つです。プロジェクトの規模やリリースサイクルに応じて選択します。

Section 12-2

Git Flow

Git Flowは、2010年にVincent Driessenが提唱したブランチモデルです。リリースサイクルが明確なプロジェクト（モバイルアプリ、パッケージソフトウェアなど）に適しています。

🌳 Git Flow のブランチ構成
★ main 本番リリース用
常に安定した状態
🛠 develop 開発の統合ブランチ
次のリリースの準備
🌱 feature/* 新機能の開発用
developから分岐
📦 release/* リリース準備用
developから分岐
🔥 hotfix/* 緊急バグ修正用
mainから分岐

2.1 各ブランチの役割

ブランチ 分岐元 マージ先 役割
main - - 本番環境のコード。常にリリース可能な状態を保つ
develop main release 開発の中心。全てのfeatureブランチがここに統合される
feature/* develop develop 個別の機能開発。機能が完成したらdevelopにマージ
release/* develop main, develop リリース前の最終調整。バグ修正やバージョン番号の更新
hotfix/* main main, develop 本番の緊急バグ修正。修正後はmainとdevelopの両方に反映

2.2 Git Flowの実践コマンド

feature ブランチの作業フロー Terminal 
# developブランチから新しいfeatureブランチを作成
$ git checkout develop
$ git checkout -b feature/login-form

# 機能を開発してコミット
$ git add .
$ git commit -m "Add login form UI"

# developにマージ
$ git checkout develop
$ git merge feature/login-form

# featureブランチを削除
$ git branch -d feature/login-form
release ブランチの作業フロー Terminal 
# developからreleaseブランチを作成
$ git checkout develop
$ git checkout -b release/1.0.0

# バグ修正やバージョン番号の更新
$ git commit -m "Bump version to 1.0.0"

# mainにマージしてタグを付ける
$ git checkout main
$ git merge release/1.0.0
$ git tag -a v1.0.0 -m "Version 1.0.0"

# developにもマージ
$ git checkout develop
$ git merge release/1.0.0

# releaseブランチを削除
$ git branch -d release/1.0.0
lightbulb Git Flowの向き・不向き

Git Flowは構造がしっかりしている反面、ブランチの数が多く管理が複雑になります。定期的なリリースサイクルがあるプロジェクトには最適ですが、頻繁にデプロイするWebアプリケーションには向かない場合もあります。

Section 12-3

GitHub Flow

GitHub Flowは、GitHub社が提唱するシンプルなワークフローです。継続的デプロイを行うWebアプリケーション開発に最適で、Git Flowよりもルールが少なく運用しやすいのが特徴です。

🚀 GitHub Flow のシンプルなフロー
★ main 常にデプロイ可能
→
🌱 ブランチ作成 feature/xxx
→
📝 開発 & コミット こまめにpush
→
💬 プルリクエスト レビュー依頼
→
🚀 マージ & デプロイ mainに統合

3.1 GitHub Flowの6つのルール

  1. mainブランチは常にデプロイ可能な状態を維持する
  2. 新しい作業はmainからブランチを切って開始する
  3. ブランチには分かりやすい名前をつける（例: feature/add-search）
  4. 定期的にリモートにpushしてバックアップを取る
  5. マージ前に必ずプルリクエストを作成し、コードレビューを受ける
  6. レビュー承認後にmainにマージしてデプロイする

3.2 GitHub Flowの実践コマンド

GitHub Flow の典型的な作業フロー Terminal 
# 1. mainブランチを最新にする
$ git checkout main
$ git pull origin main

# 2. 作業用ブランチを作成
$ git checkout -b feature/add-search

# 3. 開発してこまめにコミット
$ git add .
$ git commit -m "Add search bar component"
$ git commit -m "Add search functionality"

# 4. リモートにpush
$ git push -u origin feature/add-search

# 5. GitHubでプルリクエストを作成
#    → ブラウザでGitHubを開き、PRを作成

# 6. レビュー後、GitHubでマージ
#    → PRページの「Merge pull request」ボタン

# 7. ローカルのmainを更新してブランチを削除
$ git checkout main
$ git pull origin main
$ git branch -d feature/add-search
lightbulb Git Flow vs GitHub Flow

Git Flowはリリースバージョン管理が必要な大規模プロジェクト向け、GitHub Flowは継続的デプロイを行うWebアプリ向けです。初心者やスタートアップは、シンプルなGitHub Flowから始めるのがおすすめです。

Section 12-4

.gitignoreの設定

.gitignoreファイルは、Gitの管理対象から除外するファイルやディレクトリを指定するための設定ファイルです。不要なファイルをリポジトリに含めないことで、リポジトリをクリーンに保てます。

4.1 .gitignoreが必要な理由

以下のようなファイルは、Gitで管理すべきではありません。

🔒

秘密情報・環境変数

.envファイルやAPIキー、パスワードなどをリポジトリに含めると、セキュリティリスクになります。

📦

依存パッケージ

node_modules/などの依存パッケージは、package.jsonから復元できるため、リポジトリに含める必要はありません。

💻

ビルド生成物

dist/、build/などのビルド成果物は、ソースコードから再生成できるため不要です。

🛠

エディタ・OS固有のファイル

.DS_Store（macOS）、Thumbs.db（Windows）、.vscode/（VSCode設定）などは個人環境に依存するファイルです。

warning 一度コミットしたファイルは.gitignoreで無視できない

.gitignoreはまだ追跡されていないファイルにのみ有効です。既にコミットしたファイルを無視したい場合は、git rm --cached ファイル名で追跡を解除してから.gitignoreに追加する必要があります。

4.2 .gitignoreのパターン記法

パターン 意味 具体例
ファイル名 指定したファイルを無視 .env → .envファイルを無視
ディレクトリ/ 指定したディレクトリ全体を無視 node_modules/ → フォルダごと無視
*.拡張子 指定した拡張子のファイルを全て無視 *.log → 全ての.logファイル
!ファイル名 除外ルールの例外（無視しない） !important.log → このファイルだけは追跡
**/ディレクトリ 任意の階層にあるディレクトリを無視 **/__pycache__ → どの階層でも無視
# コメント コメント行（無視される） # 環境変数ファイル

4.3 実践的な.gitignoreの例

Webプロジェクト向け .gitignore .gitignore

# ========================================

# ========================================

# ========================================

# ビルド生成物
# ========================================

lightbulb gitignore.io を活用しよう

gitignore.ioは、使用する言語やフレームワークを選ぶだけで、最適な.gitignoreを自動生成してくれるWebサービスです。また、GitHubでリポジトリを作成する際にもテンプレートを選択できます。

4.4 追跡済みファイルの除外方法

既にGitで追跡しているファイルを後から.gitignoreに追加したい場合は、以下の手順が必要です。

追跡済みファイルを.gitignoreに追加する手順 Terminal 
# 1. Gitの追跡を解除（ファイルは残る）
$ git rm --cached .env

# 2. .gitignoreに追加
$ echo ".env" >> .gitignore

# 3. 変更をコミット
$ git add .gitignore
$ git commit -m "Add .env to .gitignore"
warning Gitの履歴には残る

git rm --cachedで追跡を解除しても、過去のコミット履歴にはそのファイルが残っています。もし秘密情報を誤ってコミットしてしまった場合は、git filter-branchやBFG Repo-Cleanerなどのツールで履歴からも削除する必要があります。最初から.gitignoreを正しく設定しておくことが最善です。

Section 12-5

Git管理のベストプラクティス

チーム開発をスムーズに進めるために、日頃から心がけたいGit管理のベストプラクティスをまとめます。

5.1 コミットのベストプラクティス

📝

意味のあるコミットメッセージを書く

「fix」「update」だけではなく、何を・なぜ変更したのかが分かるメッセージにしましょう。例: Fix login form validation error on empty email

🎯

1コミット1変更の原則

1つのコミットには関連する1つの変更のみを含めます。バグ修正と新機能追加を同じコミットに入れないようにしましょう。

🛠

動く状態でコミットする

コンパイルエラーやテスト失敗の状態でコミットしないようにしましょう。各コミットは常に動作する状態を保つことが理想です。

5.2 コミットメッセージの規約

チームで統一的なコミットメッセージを書くために、Conventional Commitsという規約がよく使われます。

Conventional Commits の形式 テキスト 
# 基本形式
type(scope): description

# 具体例
feat(auth): add Google OAuth login
fix(cart): resolve quantity update bug
docs(readme): update installation guide
style(header): adjust responsive layout
refactor(api): simplify error handling logic
test(user): add unit tests for signup flow
chore(deps): update dependencies
type 用途
feat新機能の追加
fixバグの修正
docsドキュメントの変更
styleコードの書式変更（機能変更なし）
refactorリファクタリング
testテストの追加・修正
choreビルド設定や依存関係の更新
Exercise

12章 練習問題

Q1

Git Flowにおいて、緊急のバグ修正に使用するブランチはどれですか？

  1. feature ブランチ
  2. release ブランチ
  3. hotfix ブランチ
  4. develop ブランチ
Q2

.gitignoreの記述で、全ての.logファイルを無視するパターンはどれですか？

  1. .log
  2. *.log
  3. log.*
  4. log/
Q3

GitHub Flowの特徴として正しいものはどれですか？

  1. developブランチを必ず使用する
  2. mainブランチに直接コミットすることを推奨する
  3. プルリクエストを使ったコードレビューが必須
  4. releaseブランチでバージョン管理を行う
Answer

12章 解説・解答

Q1の解答: C

Git Flowでは、本番環境で発見された緊急バグの修正にはhotfixブランチを使用します。hotfixブランチはmainブランチから直接分岐し、修正後はmainとdevelopの両方にマージされます。

Q2の解答: B

*.log のアスタリスク（*）はワイルドカードで、「任意のファイル名」を意味します。つまり *.log は error.log、access.log など、拡張子が.logの全てのファイルにマッチします。

Q3の解答: C

GitHub Flowでは、mainブランチにマージする前に必ずプルリクエストを作成してコードレビューを受けることが基本ルールです。developブランチやreleaseブランチは使わず、シンプルにmainとfeatureブランチだけで運用します。
