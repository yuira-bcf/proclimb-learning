Section 2-1

情報設計とは何か

情報設計（Information Architecture、IA）とは、ユーザーが必要な情報に迷わずたどり着くための「情報の構造」を設計するプロセスです。UI/UXの領域では、IAは"UX 5層モデルの構造層（Structure Layer）"に該当し、サービス全体の使いやすさを左右する非常に重要な要素です。

IAは見た目のデザインではない

IAは見た目のデザインではありません。サービスの中にある情報や機能を整理し、どのような階層構造を持たせ、どの画面からどの画面に行けるべきか、ユーザーはどの順序で情報に触れるべきかといった"目に見えない設計"を担当します。

情報設計が不十分だと...
  • 見たい情報に辿り着けない
  • 迷う
  • 意図とは違う画面に移動してしまう
  • 同じ情報が複数の場所に散乱している

逆にIAが優れていると、ユーザーは直感的に目的を達成でき、ストレスのない体験が実現します。

Section 2-2

なぜ情報設計がUXで重要なのか

UI/UXでは「見た目が綺麗」よりも「迷わず使えること」のほうが価値が高いとされています。その基盤を作るのが情報設計です。

1. ユーザーは「全部読む」のではなく「探す」

実際のユーザーは、トップページから順番に全ての文字を読んでくれるわけではありません。「自分の目的に関係ありそうな場所を"ざっと眺めながら探す"という行動をとります。

ユーザーの"目的"の例
  • 「お支払い方法ってどこに書いてある？」
  • 「退会ってどうやるんだろう？」
  • 「このサービスの料金プランは？」

IAが適切であれば、ユーザーは数秒のうちに「ここを見ればよさそうだ」と当たりを付けることができます。

逆に構造がぐちゃぐちゃな場合、

  • 関連情報がバラバラの場所にある
  • メニュー名から内容が想像できない
  • 画面遷移が複雑で元の場所に戻れない

といった問題が発生し、ユーザーは強いストレスを感じます。

「内容は良いのに使われないサービス」は、多くの場合このIAの問題を抱えています。

2. ナビゲーション（導線）に一貫性を与える

ユーザーは、一度学んだ操作・導線を次の画面でも期待します。

トップ → カテゴリ → 詳細 → 行動（購入・予約・応募）

マイページ → 設定 → 各種変更 → 完了画面

のように、"お決まりのパターン"があることで、ユーザーは毎回ゼロから考えなくても動けるようになります。

IAはこうした導線パターンを意図的に設計し、「どの階層に何があるか」「どのページからどのページに遷移できるのか」を整理します。

予想通りに画面が遷移することは、それだけでUXを大きく向上させます。

3. サービスの拡張性を高める

リリース直後のシンプルなサービスでは、あまりIAを意識しなくても運用できることがあります。しかし、機能・コンテンツが増えるにつれて、

  • 何となく似た内容のページが量産される
  • 開発のたびに新しいメニューが追加され、メニューが肥大化する
  • どこに何を追加すれば良いのか、チーム内で判断がばらつく

といった問題が発生します。よくある状況として、「とりあえず"その他"メニューの配下に放り込んでおこう」という判断が増えると、やがて「その他」が"何でもありのゴミ箱"になり、ユーザーも開発者も「ここには何が入っているのか」分からなくなってしまいます。

IAをきちんと設計しておけば...
  • 新しい機能やページを追加する際に「置き場所」を判断しやすい
  • サイト構造の一貫性が保たれる
  • サービスの拡張とともにUXが崩壊することを防げる
Section 2-3

IAを構成する主要要素

IAは、主に次のような要素の組み合わせとして考えることができます。

1

コンテンツの棚卸し・分類（タクソノミー）

2

階層構造（サイトマップ・ツリー構造）

3

ナビゲーション設計（導線・メニュー）

4

ラベリング（名前のつけ方）

5

検索・フィルタ機能との連携

1. コンテンツの棚卸し・分類（Taxonomy）

最初のステップは、「どんな情報が存在するのか」をすべて書き出すことです。これをコンテンツインベントリ（Content Inventory）と呼ぶこともあります。

例：ECサイトの場合
  • 商品情報（商品名、価格、説明、画像、在庫）
  • カテゴリ情報（メンズ／レディース／キッズ など）
  • キャンペーン情報
  • 会員情報（ポイント、購入履歴）
  • ヘルプ・ガイド（送料、返品、よくある質問）
  • 企業情報（会社概要、採用情報）

このように「何があるか」を出し切ったうえで、共通点・関係性を見つけながらグループ化していきます。

POINT!!

いきなり「メニュー構成」を考えない。まずはコンテンツの"中身"を洗い出してから、構造やメニューを設計する。

2. 階層構造（Hierarchy）

分類した情報を「どの階層に配置するか」を決めるのが階層構造の設計です。一般的にはツリー構造（木構造）として図示され、サイトマップとして表現されます。

階層が浅すぎる場合
  • トップページに項目が多すぎる
  • ユーザーがスクロールや目視で"全部から探す"必要が出てくる
  • 重要な情報の優先度が曖昧になる
階層が深すぎる場合
  • 「カテゴリを選んで → サブカテゴリを選んで → さらに条件を選んで → …」と、目的のページにたどり着くまでのクリック数が増えすぎる
  • 戻ったり進んだりしているうちに、ユーザーが現在地を見失う

UXの観点では、「迷わずにたどり着ける範囲で階層を分ける」バランス感覚が必要です。

3. ナビゲーション設計（Navigation）

ナビゲーションは、設計された階層構造をユーザーに"見せるためのUI"です。主な種類は次のとおりです。

ナビゲーションの種類 説明
グローバルナビゲーション サイト全体で共通のメニュー（ヘッダーなど）
ローカルナビゲーション 特定のセクション内での移動を助けるメニュー
パンくずリスト 現在地までの階層を表示し、上位階層への移動を可能にする
フッターナビゲーション サイト全体の補足リンク（規約・会社情報・ポリシーなど）

ナビゲーションの「位置」「ラベル」「数」には一貫性が必要であり、IAがしっかりしているサービスほど、ナビゲーションもシンプルで分かりやすくなります。

4. ラベリング（Labeling）

ラベリングは、「メニュー名」「ボタン名」「リンクテキスト」などに、どのような言葉を使うかを決めるプロセスです。

良いラベルの条件
  • ユーザーの言葉で書かれている（専門用語を避ける）
  • クリックした先にある内容が想像できる
  • 他のラベルと意味が重複していない
例

「サポート」 → 抽象的

「お問い合わせ」「よくある質問」「使い方ガイド」 → 具体的

ラベルが曖昧だと、いくら構造が良くてもユーザーはその入口を見つけられません。IAにおいて、ラベリングは"最後の一押し"として非常に重要です。

5. 検索・フィルタ機能との連携

情報量が多いサービス（EC、求人、ニュースなど）では、ナビゲーションだけではなく検索・絞り込み（フィルタ）がIAの重要な要素になります。

  • カテゴリやタグの設計
  • 検索結果の並べ方（関連順、新着順など）
  • 絞り込み項目の項目名・順番

これらも「ユーザーが目的の情報を探す」という観点からIAに含めて考えると理解しやすくなります。

Section 2-4

情報設計のための調査手法（概念）

現場のUXデザインでは、IAを「勘」だけで決めるのではなく、ユーザー調査の結果を反映します。ここでは代表的な手法を、概念レベルで整理します。

1. コンテンツインベントリ（Content Inventory）

既存サイトを改善する場合にまず行うのが、"棚卸し"する作業です。

  • どんなページがあるのか
  • どんな情報がどこに置かれているのか
  • 重複している情報はないか
  • 古くなって役に立っていない情報はないか

スプレッドシート等を用いて一覧化し、それぞれを残すべきか・統合すべきか・削除すべきか判断していきます。

項目 説明
URLページのアドレス
ページタイトルページの名前
内容の概要何が書かれているか
更新日時最後に更新された日
担当部署管理責任者
アクセス数閲覧数（分かる場合）

2. カードソーティング（Card Sorting）

ユーザーに「情報のカード」を渡し、自然なグループに分けてもらう手法です。

  • 「この中で似ていると思うものを同じグループにしてください」
  • 「そのグループに名前をつけるとしたら何と呼びますか？」

といった形で行い、ユーザーの頭の中での分類方法（メンタルモデル）を探ります。

例

ユーザーは「お支払い方法」「送料」「返品ポリシー」などをまとめて「ご利用ガイド」と呼ぶかもしれない。

サイト運営側の分類とユーザーの分類が大きくズレている場合、IAをユーザー視点に合わせて再設計する必要があることが分かります。

3. ツリー検証（Tree Testing）

設計した階層構造・メニュー構成が、実際に"迷わず使えるか"をテストする方法です。画面デザインは用意せず、テキストベースのメニュー（ツリー）だけで実施できるのが特徴です。

  • 「このサイトで"配送日時の変更方法"を探してみてください」
  • 「あなたならどのメニューを辿りますか？」

といったタスクを与え、ユーザーが選択した経路・かかった時間などを記録します。

成功率が低かったり、遠回りの経路を選ぶユーザーが多い場合は、

  • 階層構造
  • メニューのラベル
  • カテゴリ分類

のいずれかに問題がある可能性が高くなります。

Section 2-5

代表的なサービスのIAパターン

ここでは、代表的なWebサービスごとに「よくあるIAパターン」を整理します。実務では、これらのパターンをベースにしつつ、自サービスの特性に合わせてカスタマイズしていきます。BtoBの業務システムでは、IAの良し悪しがそのまま「業務効率」や「教育コスト」に影響するため、UXデザインの中でも特に重要視される領域です。

1. ECサイトのIA

特徴
  • 商品数が多く、カテゴリ構造が深くなりがち
  • 絞り込み条件（ブランド、価格帯、サイズ、カラーなど）がUXに直結
  • 「カート → 注文確認 → 決済 → 完了」という決済フローをいかにわかりやすくするかが重要
典型的な構成
  • トップ
  • 商品カテゴリ（メンズ / レディース / キッズなど）
  • ランキング・特集
  • カート / マイページ
  • ガイド（送料・返品・支払い方法）

2. 予約サービスのIA（ホテル・飲食・チケット など）

特徴
  • 日付・時間・人数・場所など、複数条件の組み合わせで検索する
  • 在庫（空き状況）と価格が日々変動する
  • 予約完了までのステップ数をいかに削るかが鍵
典型的な構成
  • トップ / 日付選択
  • 検索結果一覧（絞り込み可能）
  • 詳細ページ（プラン内容・口コミ）
  • 予約フォーム → 確認 → 完了
  • マイページ（予約確認・変更・キャンセル）

3. 求人サイトのIA

特徴
  • 条件検索（職種、勤務地、給与、雇用形態）が中心機能
  • 求人詳細ページから応募フォームへの導線設計が重要
  • 閲覧履歴・保存リスト（お気に入り）もよく使われる
典型的な構成
  • トップ（検索フォーム）
  • 求人一覧（条件絞り込み）
  • 求人詳細ページ
  • 応募フォーム
  • マイページ（応募履歴、保存した求人、プロフィール）

4. メディア・ニュースサイトのIA

特徴
  • 記事コンテンツが大量に蓄積されていく
  • カテゴリ（政治、経済、テクノロジーなど）の構造が重要
  • トレンド記事・ランキングの見せ方が回遊性を生む
典型的な構成
  • トップ（最新記事 + 注目記事）
  • カテゴリトップ
  • 記事詳細ページ
  • 著者ページ、タグページ
  • 特集・連載

5. SaaS・業務システムのIA

特徴
  • 機能数が多く、画面も複雑になりがち
  • ダッシュボードが「入口」として重要
  • サイドメニュー（左ナビ）＋上部メニューの組み合わせが多い
典型的な構成
  • ダッシュボード
  • データ一覧・詳細
  • 設定画面（ユーザー・権限・通知など）
  • ヘルプ・チュートリアル
Section 2-6

IAが破綻するケースとその原因

ケース1：追加・追加で「メニューだらけ」になる

新機能が追加されるたびにグローバルナビにメニューを足す。結果として、ユーザーがどれを押せばいいか分からない状態になる。

原因
  • 情報を体系として考えず、「とりあえず足す」運用になっている
  • 既存の構造に統合するルールがない

ケース2：社内組織構造で分類してしまう

「営業部」「企画部」「管理部」など社内の部署名をそのままメニュー名に使う。ユーザーには部署構造が分からないため、何がどこにあるか想像できない。

原因
  • ユーザーではなく組織都合でIAを作ってしまっている
  • ペルソナやユーザーの行動シナリオが共有されていない

ケース3：ラベルが抽象的・似通っている

「サポート」「ヘルプ」「ガイド」「お問い合わせ」など似た言葉が並ぶ。どのメニューを押せばいいか分からず、試行錯誤させてしまう。

原因
  • 情報のグルーピングとラベリングが曖昧
  • "このラベルを押したら何があるか"がチーム内で言語化されていない

POINT!!

IAの失敗は画面デザインのレベルでは気づきにくく、「ユーザーが迷う」「サポート問い合わせが増える」ことで表面化する。だからこそ、早い段階でIAを設計し、検証することが重要。

Section 2-7

IA設計の基本プロセス

情報設計を行う際の、一般的な流れを整理しておきます。「IA → ワイヤーフレーム → 画面デザイン」という流れの"IAの部分"を丁寧に意識できるようになることが重要です。

1

現状の把握・コンテンツ棚卸し

2

ユーザー理解（ペルソナ・シナリオの確認）

3

情報の分類・グルーピング

4

階層構造・サイトマップ作成

5

ナビゲーション・ラベルの設計

6

プロトタイプ作成（ワイヤーフレームレベル）

7

ユーザーによる検証（ツリー検証等）

8

改善・確定

Section 2-8

まとめ

情報設計（IA）は、情報を整理し、構造をつくり、導線を整え、ラベルで分かりやすくすることで、ユーザーが「迷わず・不安なく」サービスを使えるようにするための基盤です。

この章のポイント

  • コンテンツを分類し、
  • 適切な階層構造を設計し、
  • ナビゲーションとラベルで"入口"を用意し、
  • 必要に応じて検索やフィルタを組み合わせる。

こうしたIAの設計を意識できるようになると、「なぜこのサービスは使いやすいのか／使いにくいのか」を構造レベルで説明できるようになります。
