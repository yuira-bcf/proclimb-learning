Section 1-1

コンテナ技術とは

コンテナ技術とは、アプリケーションとその実行に必要なライブラリや設定ファイルをひとまとめにパッケージ化し、どの環境でも同じように動作させるための仮想化技術です。従来のサーバー管理の課題を解決する革新的なアプローチとして、現代のソフトウェア開発に欠かせない存在となっています。

なぜコンテナが必要なのか

ソフトウェア開発において、最もよく起こる問題の一つが「自分のパソコンでは動くのに、サーバーでは動かない」という環境差異の問題です。これは、OSのバージョン、ライブラリのバージョン、設定ファイルの違いなどが原因で発生します。

コンテナ技術はこの問題を根本的に解決します。アプリケーションに必要なすべての依存関係を1つのパッケージ（コンテナ）にまとめることで、開発環境・テスト環境・本番環境のどこでも同一の動作を保証します。

POINT

コンテナの核心は「環境ごと持ち運べる」こと。アプリケーション本体だけでなく、OS層の一部・ライブラリ・設定ファイルをすべて含めてパッケージ化します。

コンテナの仕組み

コンテナは、ホストOS（実際に動いているOS）のカーネルを共有しながら、プロセスやネットワーク、ファイルシステムを隔離する技術です。Linux の namespace（名前空間）と cgroup（リソース制御）という2つのカーネル機能を基盤としています。

  • namespace：プロセスID、ネットワーク、ファイルシステムなどをコンテナごとに隔離する
  • cgroup：CPU、メモリ、ディスクI/Oなどのリソース使用量を制限・管理する

これにより、1台のサーバー上で複数のコンテナを安全かつ効率的に実行できます。

コンテナ技術の歴史的背景

コンテナの概念は新しいものではありません。2000年代初頭から Linux には類似の仕組みが存在していました。

年代 技術 概要
1979年chrootファイルシステムのルートディレクトリを変更する Unix 機能
2000年FreeBSD Jailプロセスとファイルシステムの隔離を実現
2006年cgroupsGoogle が開発したリソース制御の仕組み
2008年LXCLinux カーネルのコンテナ機能を利用した仮想化
2013年Dockerコンテナを誰でも簡単に使えるツールとして普及

Docker の登場以前にもコンテナ技術は存在していましたが、使いこなすには高度な Linux の知識が必要でした。Docker は直感的なコマンドとイメージの共有基盤（Docker Hub）を提供することで、コンテナ技術を爆発的に普及させました。

Section 1-2

仮想マシンとコンテナの違い

サーバー仮想化の手法として、従来は仮想マシン（VM）が主流でした。コンテナと仮想マシンは似ているようで、アーキテクチャが大きく異なります。両者の違いを正しく理解しましょう。

💻 仮想マシン vs コンテナ アーキテクチャ比較
仮想マシン（VM）
App A アプリケーション
ゲストOS Ubuntu / CentOS など
ハイパーバイザー VMware / VirtualBox
ホストOS / ハードウェア
コンテナ
App A アプリ + ライブラリ
コンテナランタイム Docker Engine
ホストOS / ハードウェア カーネルを共有

比較表

項目 仮想マシン（VM） コンテナ
起動速度 数分 数秒
サイズ 数GB〜数十GB 数MB〜数百MB
OS ゲストOS が必要 ホストOS のカーネルを共有
隔離レベル 高い（完全なOS分離） 中程度（プロセスレベル分離）
リソース効率 低い（オーバーヘッド大） 高い（軽量）
可搬性 やや低い 非常に高い
POINT

コンテナは「軽量・高速・ポータブル」、VMは「高い隔離性・独立したOS」が特徴です。用途に応じて使い分けることが重要で、両方を組み合わせるケースも増えています。

Section 1-3

Dockerの歴史と特徴

Docker（ドッカー）は、2013年にDotCloud社（現Docker社）のソロモン・ハイクスによって開発されたコンテナプラットフォームです。コンテナ技術自体はLinuxに以前から存在していましたが、Dockerはそれを誰でも簡単に使えるツールとして世界中に普及させました。

Dockerの年表

年 出来事
2013年Docker 初版リリース（PyCon で発表）
2014年Docker Hub 公開、Docker 1.0 リリース
2015年Docker Compose、Docker Swarm 登場
2016年Docker for Mac / Windows リリース
2017年Moby プロジェクト発足、Kubernetes との連携強化
2019年Docker Desktop の本格展開
2021年Docker Desktop の有料化（大企業向け）
2023年〜AI 連携機能、Docker Init、MCP対応など進化

Dockerの主な特徴

  • 簡単なコマンド操作：docker run 一発でコンテナを起動できるシンプルさ
  • Dockerfile によるコード化：環境構築手順をテキストファイルとして管理・共有できる
  • Docker Hub：公式イメージや世界中の開発者が作成したイメージを無料で利用可能
  • レイヤーシステム：イメージを差分で管理し、効率的にビルド・配布できる
  • 豊富なエコシステム：Docker Compose、Kubernetes、CI/CD ツールとの連携

Section 1-4

コンテナのユースケース

コンテナ技術は、開発・テスト・運用のあらゆる場面で活用されています。代表的なユースケースを見ていきましょう。

1. 開発環境の統一

チーム開発において、メンバー全員が同じ環境で開発できることは非常に重要です。Dockerを使えば、docker compose up 一発で、データベース・バックエンド・フロントエンドの開発環境を立ち上げられます。

開発環境の起動例 Bash 
# プロジェクトをクローンして環境構築
git clone https://github.com/example/project.git
cd project
docker compose up -d

# これだけで全員同じ環境が手に入る！

2. CI/CD パイプライン

継続的インテグレーション/デリバリー（CI/CD）では、テスト環境を毎回クリーンに構築する必要があります。コンテナなら数秒で環境を用意し、テスト後に即座に破棄できます。GitHub Actions や Jenkins でもDockerが標準的に利用されています。

3. マイクロサービスアーキテクチャ

1つの大きなアプリケーションを複数の小さなサービスに分割するマイクロサービスでは、各サービスを独立したコンテナとして運用します。サービスごとに異なる言語やフレームワークを使うことも可能です。

4. 本番環境のデプロイ

AWS ECS、Google Cloud Run、Azure Container Instances など、主要なクラウドサービスはすべてコンテナベースのデプロイをサポートしています。開発環境と同じコンテナを本番にそのまま展開できるため、デプロイの信頼性が大幅に向上します。

5. レガシーアプリケーションの移行

古い技術で構築されたアプリケーションをコンテナ化することで、モダンなインフラ上で運用を継続できます。アプリケーションの動作環境をそのままコンテナに封じ込めるため、ホストOSのアップデートに影響されにくくなります。

6. 教育・学習環境

プログラミング学習や研修では、受講者全員が同じ環境を使えることが重要です。Docker を使えば docker run コマンドひとつで、講師と同じ開発環境を瞬時に構築できます。環境構築でつまずくことなく、本来の学習内容に集中できます。

POINT

コンテナ技術は「インフラエンジニアだけのもの」ではありません。フロントエンド開発者、バックエンド開発者、データサイエンティストなど、ソフトウェアに関わるすべての人にとって有用なスキルです。

Section 1-5

Docker用語集

Dockerの学習を進める上で、頻出する重要な用語を整理しておきましょう。各用語は後の章で詳しく解説しますが、ここで概要を把握しておくと理解がスムーズになります。

用語 説明
イメージ（Image） コンテナの設計図。アプリケーションの実行に必要なファイルやライブラリをまとめた読み取り専用のテンプレート
コンテナ（Container） イメージから作成された実行中のインスタンス。隔離された環境でアプリケーションが動作する
Dockerfile イメージを作成するための命令を記述したテキストファイル。環境構築手順をコード化する
Docker Hub Dockerイメージの共有・配布を行うクラウドレジストリ。公式イメージが多数公開されている
Docker Engine コンテナの作成・実行を行うランタイム。Docker のコア部分
Docker Compose 複数のコンテナを定義・管理するためのツール。YAML ファイルで構成を記述する
レジストリ（Registry） Docker イメージを保管・配布するためのサービス。Docker Hub は公式レジストリ
ボリューム（Volume） コンテナのデータを永続化するための仕組み。コンテナを削除してもデータが残る
ネットワーク（Network） コンテナ間の通信を管理する仕組み。コンテナ同士をつなげたり隔離したりできる
タグ（Tag） イメージのバージョンを識別するためのラベル。例：nginx:latest、node:18-alpine
注意

「イメージ」と「コンテナ」を混同しないようにしましょう。イメージは「設計図（クラス）」、コンテナは「実体（インスタンス）」と考えるとわかりやすいです。1つのイメージから複数のコンテナを作成できます。

Exercise

1章 練習問題

Q1

仮想マシン（VM）とコンテナの最も大きな違いとして正しいものはどれですか？

  1. コンテナはインターネット接続が不要だが、VMは必要
  2. コンテナはホストOSのカーネルを共有するが、VMはゲストOSを持つ
  3. VMは起動が速く、コンテナは起動が遅い
  4. コンテナはWindows上では動作しない
Q2

Docker Hub の説明として正しいものはどれですか？

  1. Dockerのソースコードを管理するGitリポジトリ
  2. Dockerイメージを共有・配布するクラウドレジストリ
  3. Docker Composeの設定ファイルを保存するサービス
  4. コンテナの実行ログを保存するストレージサービス
Q3

コンテナ技術の利点として正しくないものはどれですか？

  1. 環境差異をなくし、どこでも同じ動作が保証される
  2. 起動が高速で、数秒でアプリケーションを立ち上げられる
  3. 仮想マシンより常にセキュリティが高い
  4. リソース効率が高く、1台のサーバーで多くのコンテナを動かせる
Answer

1章 解説・解答

Q1の解答: B

コンテナはホストOSのカーネルを共有し、プロセスレベルで隔離を行います。一方、仮想マシンはハイパーバイザー上にゲストOSを丸ごとインストールして動作します。この違いにより、コンテナはVMに比べて起動が速く、リソース消費も少なくなります。

Q2の解答: B

Docker Hub は、Dockerイメージの共有・配布を行うクラウド型のレジストリサービスです。公式イメージ（nginx、node、python など）のほか、コミュニティが作成したイメージも多数公開されています。無料プランと有料プランがあります。

Q3の解答: C

コンテナは仮想マシンに比べて隔離レベルがやや低いため、「常にセキュリティが高い」とは言えません。VMはOS全体を隔離するため、セキュリティの隔離レベルはVMの方が一般的に高いとされています。コンテナのセキュリティは適切な設定と管理が重要です。
