Section 5-1

エンドポイント設計

エンドポイントとは、APIにアクセスするためのURLのことです。適切なエンドポイント設計は、APIの使いやすさに直結します。

1.1 URL命名規則

ルール 良い例 悪い例
リソース名は複数形 /users /user
小文字を使う /users /Users
複合語はハイフンで区切る /user-profiles /userProfiles
動詞を使わない POST /users POST /createUser
末尾にスラッシュを付けない /users /users/

1.2 バージョニング

APIの互換性を保つために、URLにバージョン番号を含めるのが一般的です。

バージョニングの例 URI 
https://api.example.com/v1/users    ← バージョン1
https://api.example.com/v2/users    ← バージョン2（新しい仕様）
lightbulb バージョニングの方式

URLパスに含める方式（/v1/）が最も一般的です。他にもヘッダーで指定する方式（Accept: application/vnd.api+json;version=1）もありますが、URLパス方式が直感的で広く採用されています。

Section 5-3

クエリパラメータとパスパラメータ

APIに追加情報を渡す方法として、パスパラメータとクエリパラメータの2つがあります。それぞれの使い分けを理解しましょう。

3.1 パスパラメータ

特定のリソースを識別するために使います。URLパスの一部として埋め込みます。

パスパラメータの例 URI 
GET /users/123           ← ユーザーID: 123
GET /users/123/orders/456 ← ユーザー123の注文456
GET /products/abc-widget  ← 商品スラッグ: abc-widget

3.2 クエリパラメータ

フィルタリング、ソート、検索、ページネーションなど、コレクションの取得条件を指定するために使います。

クエリパラメータの例 URI 
GET /users?role=admin                  ← roleで絞り込み
GET /users?sort=name&order=asc         ← 名前の昇順でソート
GET /products?category=electronics&min_price=1000
                                       ← カテゴリと最低価格で絞り込み
GET /articles?q=REST+API               ← キーワード検索
GET /users?page=2&per_page=20          ← ページネーション

3.3 使い分けのガイドライン

条件 使うべきパラメータ 例
特定のリソースを指定する パスパラメータ /users/123
省略可能な条件を付ける クエリパラメータ /users?role=admin
ソートや並び替え クエリパラメータ /users?sort=name
ページネーション クエリパラメータ /users?page=2
Section 5-4

ページネーション

大量のデータを一度に返すとパフォーマンスが低下します。ページネーション（ページ分割）を使って、データを分割して返す設計が必要です。

4.1 オフセット方式

📖 ページネーションのイメージ
Page 1
1〜10件
Page 2
11〜20件
Page 3
21〜30件
...
...
Page 5
41〜50件
全50件のデータを10件ずつ5ページに分割

ページ番号と1ページあたりの件数を指定する、最も一般的な方式です。

リクエスト HTTP 
GET /v1/articles?page=2&per_page=10
レスポンス（メタ情報付き） JSON 
{
  "data": [
    { "id": 11, "title": "記事11" },
    { "id": 12, "title": "記事12" }
  ],
  "meta": {
    "currentPage": 2,
    "perPage": 10,
    "totalPages": 5,
    "totalCount": 50
  }
}

4.2 カーソル方式

前回のレスポンスに含まれるカーソル（位置情報）を使って次のページを取得する方式です。リアルタイムに更新されるデータに向いています。

カーソル方式の例 JSON 
{
  "data": [...],
  "pagination": {
    "nextCursor": "eyJpZCI6MjB9",
    "hasMore": true
  }
}

// 次のページ取得:
// GET /v1/articles?cursor=eyJpZCI6MjB9&limit=10
lightbulb 方式の選び方

オフセット方式はシンプルで実装が簡単ですが、大量データでは性能が低下します。カーソル方式は高性能ですが実装が複雑です。一般的なWebアプリではオフセット方式で十分なケースが多いです。

Section 5-5

エラーレスポンスの設計

APIがエラーを返すとき、クライアントが何が問題だったかを理解できるように、統一された形式でエラー情報を返すことが重要です。

5.1 エラーレスポンスの標準形式

バリデーションエラー（400 Bad Request） JSON 
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力内容に誤りがあります",
    "details": [
      {
        "field": "email",
        "message": "メールアドレスの形式が正しくありません"
      },
      {
        "field": "name",
        "message": "名前は必須項目です"
      }
    ]
  }
}
認証エラー（401 Unauthorized） JSON 
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "認証トークンが無効または期限切れです"
  }
}
リソース未発見（404 Not Found） JSON 
{
  "error": {
    "code": "NOT_FOUND",
    "message": "指定されたユーザーが見つかりません",
    "resource": "User",
    "id": 999
  }
}

5.2 エラー設計のポイント

  • HTTPステータスコードとエラー内容を一致させる（400番台はクライアント起因、500番台はサーバー起因）
  • エラーの原因と対処法がわかるメッセージを返す
  • バリデーションエラーではどのフィールドが問題かを具体的に示す
  • 内部のスタックトレースなどの機密情報は含めない
  • 全エラーレスポンスで統一した形式を使う
warning セキュリティ上の注意

本番環境のエラーレスポンスに、データベースのエラーメッセージやファイルパス、スタックトレースなどの内部情報を含めないようにしましょう。攻撃者にシステムの内部構造を知られるリスクがあります。

Exercise

5章 練習問題

Q1

ユーザー一覧をroleで絞り込み、nameの昇順で取得するURLとして適切なものはどれですか？

  1. GET /users/role/admin/sort/name
  2. GET /users?role=admin&sort=name&order=asc
  3. GET /getUsers?filterRole=admin
  4. POST /users/filter { "role": "admin" }
Q2

POSTリクエストで新しいリソースが正常に作成された場合の適切なステータスコードはどれですか？

  1. 200 OK
  2. 201 Created
  3. 204 No Content
  4. 301 Moved Permanently
Q3

REST APIのURLバージョニングとして適切な形式はどれですか？

  1. /users?version=1
  2. /v1/users
  3. /users_v1
  4. /version1/users
Answer

5章 解説・解答

Q1の解答: B

フィルタリングやソートなどの条件は、クエリパラメータで指定するのがRESTfulな設計です。GET /users?role=admin&sort=name&order=ascが適切です。

Q2の解答: B

新しいリソースが作成された場合は201 Createdを返すのが適切です。200 OKは一般的な成功、204 No Contentは成功だがレスポンスボディなし（DELETE時など）に使います。

Q3の解答: B

URLパスにバージョン番号を含める/v1/形式が最も広く採用されています。シンプルで直感的であり、GitHub API、Twitter APIなど多くの有名APIがこの方式を採用しています。
