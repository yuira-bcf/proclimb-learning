Section 3-1

HTTPプロトコルとは

HTTP（HyperText Transfer Protocol）は、Webにおけるデータ通信の基盤となるプロトコル（通信規約）です。1991年にTim Berners-Leeが考案し、ブラウザとサーバー間の通信をはじめ、API通信にも広く利用されています。

1.1 HTTPの基本的な特徴

📜

テキストベースのプロトコル

HTTPメッセージは人間が読めるテキスト形式で構成されています。開発者ツールで簡単に内容を確認できます。

🔃

リクエスト-レスポンスモデル

クライアントがリクエストを送り、サーバーがレスポンスを返す、一問一答形式の通信です。

📄

ステートレス

各リクエストは独立しており、サーバーは前回のリクエスト情報を覚えていません。毎回必要な情報を全て送る必要があります。

1.2 HTTPとHTTPSの違い

項目 HTTP HTTPS
暗号化 なし（平文通信） SSL/TLSで暗号化
ポート番号 80 443
安全性 盗聴・改ざんのリスクあり 通信内容が保護される
URL http://example.com https://example.com
lightbulb HTTPSが必須の時代

現在のWebではHTTPSが標準です。Google Chromeなどのブラウザは、HTTPサイトに「保護されていない通信」と警告を表示します。API通信でも、認証情報や個人データを扱う場合はHTTPSが必須です。

Section 3-2

リクエストの構造

HTTPリクエストは、クライアントからサーバーへ送られるメッセージです。メソッド、URL、ヘッダー、ボディの4つの要素で構成されます。

2.1 リクエストの4要素

📦 HTTPリクエストの4つの構成要素
sendメソッドGET / POST
+
linkURL/api/users
+
descriptionヘッダーメタ情報
+
data_objectボディJSON等
要素 説明 例
メソッド 何をしたいかを示す動詞 GET, POST, PUT, DELETE
URL リソースの場所（アドレス） /api/users/1
ヘッダー リクエストのメタ情報 Content-Type, Authorization
ボディ 送信するデータ本体（POST/PUT時） JSON形式のデータ

2.2 GETリクエストの例

GETリクエスト（データ取得） HTTP 
GET /api/users/1 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer token123

GETリクエストにはボディがありません。必要な情報はURLとヘッダーで送ります。

2.3 POSTリクエストの例

POSTリクエスト（データ作成） HTTP 
POST /api/users HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer token123

{
  "name": "田中太郎",
  "email": "tanaka@example.com",
  "role": "member"
}

POSTリクエストではボディにJSON形式のデータを含めます。Content-Type: application/json ヘッダーで、データ形式がJSONであることをサーバーに伝えます。

Section 3-3

HTTPメソッド

HTTPメソッドは、リソースに対して「何をしたいか」を表す動詞です。REST APIでは、主に5つのメソッドを使い分けます。

3.1 主要なHTTPメソッド

メソッド 用途 ボディ 冪等性 安全性
GET データの取得 なし あり 安全
POST データの新規作成 あり なし 非安全
PUT データの全体更新 あり あり 非安全
PATCH データの部分更新 あり なし 非安全
DELETE データの削除 なし あり 非安全
lightbulb 冪等性（べきとうせい）とは

同じリクエストを何度送っても、結果が同じになる性質のことです。例えば、GETでユーザー情報を10回取得しても同じデータが返り、DELETEで同じリソースを何度削除しても結果は「削除済み」です。一方、POSTは送るたびに新しいリソースが作成されるため、冪等ではありません。

warning 注意: GETリクエストにボディを含めない

HTTP仕様上、GETリクエストにボディを含めることは推奨されていません。検索条件などはURLのクエリパラメータ（例: ?name=tanaka&age=28）で送るのが正しい方法です。

3.2 PUTとPATCHの違い

PUT: リソース全体を置き換え HTTP 
PUT /api/users/1
Content-Type: application/json

{
  "name": "田中太郎",
  "email": "tanaka-new@example.com",
  "role": "admin"
}
PATCH: 指定フィールドのみ更新 HTTP 
PATCH /api/users/1
Content-Type: application/json

{
  "email": "tanaka-new@example.com"
}

PUTはリソース全体を送信して置き換え、PATCHは変更したいフィールドだけを送信します。

Section 3-4

ステータスコード

ステータスコードは、サーバーがリクエストの処理結果をクライアントに伝える3桁の数字です。5つのカテゴリに分類されます。

4.1 ステータスコードのカテゴリ

🎮 ステータスコードの5つのカテゴリ
1xx
情報
2xx
成功
3xx
転送
4xx
クライアント
エラー
5xx
サーバー
エラー
200 OK成功
201 Created作成成功
400 Bad Request不正リクエスト
401 Unauthorized認証必要
404 Not Foundリソース無し
500 Server Errorサーバー障害
範囲 カテゴリ 意味
1xx 情報 リクエストを受け付け、処理中
2xx 成功 リクエストが正常に処理された
3xx リダイレクト 別のURLへの転送が必要
4xx クライアントエラー リクエスト側に問題がある
5xx サーバーエラー サーバー側に問題がある

4.2 よく使うステータスコード

コード 名前 意味 よくある場面
200 OK 成功 GET/PUT/PATCHが成功した時
201 Created 作成成功 POSTで新規リソースを作成した時
204 No Content 成功（返却データなし） DELETEが成功した時
400 Bad Request 不正なリクエスト バリデーションエラー、JSON構文エラー
401 Unauthorized 認証が必要 トークンが未送信・期限切れ
403 Forbidden アクセス権限なし 認証済みだが権限が不足
404 Not Found リソースが見つからない 存在しないURLへのリクエスト
429 Too Many Requests リクエスト過多 レート制限に達した時
500 Internal Server Error サーバー内部エラー サーバー側のバグや障害
503 Service Unavailable サービス利用不可 メンテナンス中・過負荷
lightbulb 401と403の違い

401 Unauthorizedは「誰だか分からない」（認証されていない）、403 Forbiddenは「誰だか分かるが許可されていない」（認可されていない）という違いがあります。ログインしていなければ401、ログイン済みだが管理者権限がなければ403が返ります。

Section 3-5

リクエスト/レスポンスヘッダー

HTTPヘッダーは、リクエストやレスポンスに付加されるメタ情報です。データ形式、認証情報、キャッシュ制御など、通信に関する重要な情報を伝えます。

5.1 よく使うヘッダー

ヘッダー名 用途 値の例
Content-Type 送信データの形式を指定 application/json
Accept 受け入れ可能なデータ形式 application/json
Authorization 認証情報（トークン等） Bearer eyJhbGc...
Cache-Control キャッシュの制御 no-cache, max-age=3600
Content-Length ボディのサイズ（バイト） 256
User-Agent クライアントの種類 Mozilla/5.0 ...

5.2 レスポンス全体の例

完全なHTTPレスポンス HTTP 
HTTP/1.1 200 OK
Content-Type: application/json; charset=utf-8
Content-Length: 128
Cache-Control: no-cache
X-Request-Id: abc-123-def

{
  "id": 1,
  "name": "田中太郎",
  "email": "tanaka@example.com",
  "createdAt": "2025-04-01T10:30:00Z"
}

レスポンスは「ステータスライン → ヘッダー → 空行 → ボディ」の順で構成されます。ヘッダーとボディの間には必ず空行が入ります。

Exercise

3章 練習問題

Q1

新しいユーザーを作成するために使用するHTTPメソッドとして最も適切なものはどれですか？

  1. GET
  2. POST
  3. PUT
  4. DELETE
Q2

ステータスコード「404」の意味として正しいものはどれですか？

  1. 認証が必要
  2. リクエストが不正
  3. リソースが見つからない
  4. サーバー内部エラー
Q3

Content-Type ヘッダーの役割として正しいものはどれですか？

  1. 認証トークンを送信する
  2. 送信データの形式を指定する
  3. キャッシュの有効期限を設定する
  4. リクエストのサイズを制限する
Answer

3章 解説・解答

Q1の解答: B

新しいリソースの作成にはPOSTメソッドを使用します。GETはデータ取得、PUTは既存データの更新、DELETEはデータの削除に使います。

Q2の解答: C

404 Not Foundは、指定されたURLにリソースが存在しないことを意味します。401が認証必要、400が不正リクエスト、500がサーバー内部エラーです。

Q3の解答: B

Content-Typeヘッダーは、リクエストボディやレスポンスボディのデータ形式（MIMEタイプ）を指定します。例えばapplication/jsonはJSON形式であることを示します。
