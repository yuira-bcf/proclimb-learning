Core Web Vitals最適化テンプレート

verified
学習ポイント

Core Web Vitals（LCP, FID, CLS）を最適化するHTML設計が重要です。クリティカルCSSのインライン化、フォントの font-display: swap、画像の fetchpriority などの手法があります。

assignment課題

以下のパフォーマンス最適化要件を満たすHTMLテンプレートを設計してください。

  • クリティカルCSS（ファーストビューに必要な最小CSS）を <style> でインライン化
  • 非クリティカルCSSを preload + onload で非同期読み込み
  • LCP対象の画像に fetchpriority="high" を設定
  • フォールドビューの画像に loading="lazy" と decoding="async" を設定
  • 全画像に width / height を設定しCLSを防止
  • <link rel="dns-prefetch"> でサードパーティドメインを事前解決
lightbulbヒント

fetchpriority="high" はLCP要素（ヒーロー画像など）に使い、ブラウザにダウンロード優先度を伝えます。decoding="async" は画像のデコード処理をメインスレッドからオフロードします。

check_circle解答・解説 
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>パフォーマンス最適化ページ</title>

  <link rel="dns-prefetch" href="https://cdn.example.com">
  <link rel="preconnect" href="https://fonts.googleapis.com">
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

  <!-- クリティカルCSS（インライン） -->
  <style>
    body { margin: 0; font-family: sans-serif; }
    .hero { width: 100%; min-height: 60vh; }
    .hero img { display: block; width: 100%; height: auto; }
  </style>

  <!-- 非クリティカルCSS（非同期読み込み） -->
  <link rel="preload" href="style.css" as="style"
    onload="this.onload=null;this.rel='stylesheet'">
  <noscript><link rel="stylesheet" href="style.css"></noscript>

  <link href="https://fonts.googleapis.com/css2?family=Noto+Sans+JP:wght@400;700&display=swap" rel="stylesheet">
</head>
<body>
  <header>
    <nav>
      <a href="/">ホーム</a>
    </nav>
  </header>

  <main>
    <section class="hero">
      <h1>高速表示を実現するHTML設計</h1>
      <!-- LCP対象: fetchpriorityでダウンロード優先 -->
      <img src="hero.webp" alt="ヒーロー画像"
        width="1200" height="600"
        fetchpriority="high">
    </section>

    <section>
      <h2>コンテンツ</h2>
      <!-- 遅延読み込み + 非同期デコード -->
      <img src="content1.webp" alt="コンテンツ画像1"
        width="800" height="450"
        loading="lazy" decoding="async">
      <img src="content2.webp" alt="コンテンツ画像2"
        width="800" height="450"
        loading="lazy" decoding="async">
    </section>
  </main>
</body>
</html>
解説
  • クリティカルCSSのインライン化により、外部CSSの読み込みを待たずにファーストビューを表示できます。
  • fetchpriority="high" はLCP指標の改善に直結します。LCP対象以外の画像には使わないでください。
  • loading="lazy" と decoding="async" の併用でスクロール下の画像のパフォーマンスを最適化します。
  • 画像に width / height を明示することでCLS（レイアウトシフト）を防ぎます。

PWA対応マニフェスト設計

verified
学習ポイント

PWA（Progressive Web App）は <link rel="manifest"> でマニフェストファイルを指定し、meta タグでテーマカラーやApple対応を設定します。

assignment課題

以下の仕様でPWA対応のHTMLテンプレートを設計してください。

  • Webアプリマニフェスト（manifest.json）へのリンク
  • meta name="theme-color" の設定
  • Apple用メタタグ（apple-mobile-web-app-capable、apple-touch-icon）
  • ファビコン各サイズ（16x16, 32x32, 192x192, 512x512）の設定
  • Service Worker の登録スクリプト（<script> 内でインライン記述）
lightbulbヒント

Service Worker の登録は if ('serviceWorker' in navigator) { navigator.serviceWorker.register('/sw.js'); } で行います。apple-mobile-web-app-capable は content="yes" にします。

check_circle解答・解説 
<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>My PWA App</title>

  <!-- PWA マニフェスト -->
  <link rel="manifest" href="/manifest.json">

  <!-- テーマカラー -->
  <meta name="theme-color" content="#007AFF">

  <!-- Apple PWA 対応 -->
  <meta name="apple-mobile-web-app-capable" content="yes">
  <meta name="apple-mobile-web-app-status-bar-style" content="default">
  <meta name="apple-mobile-web-app-title" content="My PWA">
  <link rel="apple-touch-icon" href="/icons/icon-192x192.png">

  <!-- ファビコン -->
  <link rel="icon" type="image/png" sizes="16x16" href="/icons/icon-16x16.png">
  <link rel="icon" type="image/png" sizes="32x32" href="/icons/icon-32x32.png">
  <link rel="icon" type="image/png" sizes="192x192" href="/icons/icon-192x192.png">
  <link rel="icon" type="image/png" sizes="512x512" href="/icons/icon-512x512.png">

  <link rel="stylesheet" href="style.css">
</head>
<body>
  <header>
    <h1>My PWA App</h1>
  </header>
  <main>
    <p>このアプリはPWAに対応しています。</p>
  </main>

  <script>
    if ('serviceWorker' in navigator) {
      window.addEventListener('load', function() {
        navigator.serviceWorker.register('/sw.js')
          .then(function(registration) {
            console.log('SW registered:', registration.scope);
          })
          .catch(function(error) {
            console.log('SW registration failed:', error);
          });
      });
    }
  </script>
</body>
</html>
解説
  • manifest.json にはアプリ名、アイコン、開始URL、表示モードなどを定義します。
  • theme-color はブラウザのアドレスバーやタスクバーの色を設定します。
  • Apple デバイスは Web App Manifest を完全にサポートしていないため、独自のメタタグが必要です。
  • Service Worker は HTTPS 環境でのみ動作します（localhost は例外）。

WCAG 2.1準拠のテキストコンテンツ

verified
学習ポイント

WCAG 2.1 では、テキストリンクは文脈から目的が分かること、省略語は正式名称を提供すること、見出し階層は論理的であることが求められます。

assignment課題

以下のアクセシビリティ要件を満たすニュースサイトのコンテンツページを作成してください。

  • リンクテキストは「こちら」「詳しくはこちら」ではなく、リンク先が分かる具体的な文言にする
  • 略語は全て <abbr> で初出時に正式名称を提供
  • 引用は出典を <cite> で明示し、<blockquote cite="URL"> でソースURLを設定
  • 日時は全て <time datetime="..."> で機械可読にする
  • ページの読み上げ順序が論理的になるようマークアップする
lightbulbヒント

リンクテキストの例: × 「こちらをクリック」→ ○ 「WCAG 2.1ガイドラインの全文を読む」。<blockquote cite="https://..."> の cite 属性はブラウザには表示されませんが、機械可読です。

check_circle解答・解説 
<article>
  <header>
    <h1>Web<abbr title="アクセシビリティ">a11y</abbr>の最新動向</h1>
    <p>公開日: <time datetime="2026-03-01">2026年3月1日</time></p>
  </header>

  <section>
    <h2><abbr title="Web Content Accessibility Guidelines">WCAG</abbr> 2.1の概要</h2>
    <p>WCAG 2.1はWebコンテンツのアクセシビリティガイドラインです。
    <abbr title="World Wide Web Consortium">W3C</abbr>が
    <time datetime="2018-06-05">2018年6月5日</time>に勧告しました。</p>

    <blockquote cite="https://www.w3.org/TR/WCAG21/">
      <p>Webコンテンツをよりアクセシブルにするための幅広い推奨事項を網羅しています。</p>
      <footer>
        <cite><a href="https://www.w3.org/TR/WCAG21/" target="_blank" rel="noopener noreferrer">WCAG 2.1 — W3C勧告</a></cite>
      </footer>
    </blockquote>
  </section>

  <section>
    <h2>関連リソース</h2>
    <ul>
      <li><a href="https://www.w3.org/TR/WCAG21/" target="_blank" rel="noopener noreferrer">WCAG 2.1ガイドラインの全文を読む</a></li>
      <li><a href="https://www.w3.org/WAI/" target="_blank" rel="noopener noreferrer">W3C WAIのアクセシビリティリソース一覧</a></li>
    </ul>
  </section>
</article>
解説
  • 「こちら」「ここをクリック」はスクリーンリーダーのリンク一覧でリンク先が不明になります。具体的なリンクテキストが必須です。
  • <abbr> は初出時に title で正式名称を提供し、以降は省略可能です。
  • <blockquote cite="URL"> の cite 属性は視覚的には表示されませんが、引用元を機械的に示します。

多機能ナビゲーション設計

verified
学習ポイント

大規模サイトのナビゲーションでは、スキップリンク、キーボード操作、ランドマーク、動的メニュー状態の ARIA 属性が必要です。

assignment課題

以下のアクセシビリティ要件を満たすナビゲーションシステムを設計してください。

  • スキップリンク（「メインコンテンツへ」）を最初に配置
  • メガメニュー構造（カテゴリ > サブカテゴリ）に aria-expanded, aria-haspopup を設定
  • パンくずリスト（aria-label + 構造化データ対応）
  • 現在位置を aria-current="page" で明示
  • モバイルメニューの開閉状態を aria-expanded で管理
lightbulbヒント

スキップリンクは <a href="#main-content" class="skip-link"> で、CSSで視覚的に隠し、フォーカス時に表示します。aria-haspopup="true" はサブメニューを持つ項目に付けます。

check_circle解答・解説 
<!-- スキップリンク -->
<a href="#main-content" class="skip-link">メインコンテンツへスキップ</a>

<header>
  <!-- モバイルメニューボタン -->
  <button aria-expanded="false" aria-controls="main-nav" aria-label="メニューを開く">
    <span aria-hidden="true">&#9776;</span>
  </button>

  <nav id="main-nav" aria-label="メインナビゲーション">
    <ul>
      <li><a href="/" aria-current="page">ホーム</a></li>
      <li>
        <a href="/products" aria-haspopup="true" aria-expanded="false">製品</a>
        <ul>
          <li><a href="/products/software">ソフトウェア</a></li>
          <li><a href="/products/hardware">ハードウェア</a></li>
        </ul>
      </li>
      <li><a href="/about">会社概要</a></li>
    </ul>
  </nav>
</header>

<!-- パンくずリスト -->
<nav aria-label="パンくず">
  <ol>
    <li><a href="/">ホーム</a> <span aria-hidden="true">/</span></li>
    <li><a href="/products">製品</a> <span aria-hidden="true">/</span></li>
    <li><a href="/products/software" aria-current="page">ソフトウェア</a></li>
  </ol>
</nav>

<main id="main-content">
  <h1>ソフトウェア製品一覧</h1>
</main>
解説
  • スキップリンクはキーボードユーザーがナビを飛ばしてメインコンテンツに直接アクセスするための仕組みです。
  • aria-expanded はメニューの開閉状態を伝え、JavaScriptでtrue/falseを切り替えます。
  • aria-controls は操作対象の要素を id で関連付けます。
  • aria-haspopup="true" はサブメニューの存在をスクリーンリーダーに伝えます。

レスポンシブテーブル設計

verified
学習ポイント

テーブルのレスポンシブ対応は横スクロール方式と、data-* 属性を使ったカード表示方式があります。どちらもアクセシビリティを維持する必要があります。

assignment課題

以下の仕様で2種類のレスポンシブテーブルを設計してください。

  • 方式A: 横スクロール方式 — <div role="region" aria-label="..." tabindex="0"> で囲み、overflow-x: auto を適用
  • 方式B: カード表示方式 — 各 <td> に data-label 属性で列名を設定し、CSSでモバイル時にカード表示に切り替え
  • 両方式とも <caption> と scope 属性を維持
  • 5列×4行以上のデータテーブルで実装
lightbulbヒント

方式Aの tabindex="0" でキーボードユーザーもスクロール操作が可能になります。方式Bでは data-label の値をCSS ::before の content: attr(data-label) で表示します。

check_circle解答・解説

方式A: 横スクロール

<div role="region" aria-label="社員一覧テーブル" tabindex="0" style="overflow-x: auto;">
  <table>
    <caption>社員一覧</caption>
    <thead>
      <tr>
        <th scope="col">ID</th>
        <th scope="col">氏名</th>
        <th scope="col">部署</th>
        <th scope="col">役職</th>
        <th scope="col">入社日</th>
      </tr>
    </thead>
    <tbody>
      <tr>
        <td>001</td><td>山田太郎</td><td>開発部</td><td>主任</td>
        <td><time datetime="2020-04-01">2020/04/01</time></td>
      </tr>
      <!-- 他の行も同様 -->
    </tbody>
  </table>
</div>

方式B: カード表示

<table class="responsive-card">
  <caption>社員一覧</caption>
  <thead>
    <tr>
      <th scope="col">ID</th>
      <th scope="col">氏名</th>
      <th scope="col">部署</th>
      <th scope="col">役職</th>
      <th scope="col">入社日</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td data-label="ID">001</td>
      <td data-label="氏名">山田太郎</td>
      <td data-label="部署">開発部</td>
      <td data-label="役職">主任</td>
      <td data-label="入社日">2020/04/01</td>
    </tr>
    <!-- 他の行も同様 -->
  </tbody>
</table>
<!-- CSS: @media (max-width: 768px) で thead を非表示、
  td::before { content: attr(data-label); } でラベル表示 -->
解説
  • 方式Aはテーブル構造を維持するためアクセシビリティへの影響が少ないです。role="region" と aria-label でスクロール領域の意味を伝えます。
  • 方式Bはモバイルで見やすいですが、CSSで <thead> を非表示にするためスクリーンリーダーとの互換性に注意が必要です。
  • data-* 属性はカスタムデータを要素に付与するHTML5の仕組みで、CSSやJavaScriptからアクセスできます。

ソート・フィルター対応テーブル設計

verified
学習ポイント

インタラクティブなテーブルでは aria-sort でソート状態を伝え、aria-live で動的な変更をスクリーンリーダーに通知します。

assignment課題

以下の仕様でインタラクティブなデータテーブルのHTMLを設計してください。

  • ヘッダーをクリックでソート切替 — <th> 内に <button> を配置
  • aria-sort でソート状態（ascending / descending / none）を設定
  • テーブル上にフィルター用入力欄を配置し、aria-controls でテーブルと関連付け
  • 結果件数を aria-live="polite" でリアルタイム通知
lightbulbヒント

<th aria-sort="ascending"><button>氏名 ▲</button></th> でソートボタンを実装します。aria-live="polite" はページの読み込みを妨げずにスクリーンリーダーに通知します。

check_circle解答・解説 
<div>
  <label for="table-filter">検索:</label>
  <input type="text" id="table-filter" aria-controls="data-table" placeholder="名前で検索...">
  <p aria-live="polite" id="result-count">表示中: 4件</p>
</div>

<table id="data-table">
  <caption>商品データ</caption>
  <thead>
    <tr>
      <th scope="col" aria-sort="ascending">
        <button>商品名 <span aria-hidden="true">&#9650;</span></button>
      </th>
      <th scope="col" aria-sort="none">
        <button>価格 <span aria-hidden="true">&#9650;&#9660;</span></button>
      </th>
      <th scope="col" aria-sort="none">
        <button>在庫 <span aria-hidden="true">&#9650;&#9660;</span></button>
      </th>
    </tr>
  </thead>
  <tbody>
    <tr><td>キーボード</td><td>8,500</td><td>12</td></tr>
    <tr><td>マウス</td><td>3,200</td><td>25</td></tr>
    <tr><td>モニター</td><td>35,000</td><td>3</td></tr>
    <tr><td>ヘッドホン</td><td>12,000</td><td>8</td></tr>
  </tbody>
</table>
解説
  • aria-sort の値は ascending（昇順）/ descending（降順）/ none（未ソート）です。
  • ソートボタンは <th> 内の <button> に配置し、キーボード操作を可能にします。
  • aria-controls はフィルター入力と操作対象テーブルを関連付けます。
  • aria-live="polite" は現在の読み上げが終わった後に変更を通知します。

T4 フォーム・入力要素

WCAG準拠フォームバリデーション

verified
学習ポイント

アクセシブルなフォームバリデーションでは、エラーメッセージを aria-describedby で関連付け、aria-invalid でエラー状態を示し、エラーサマリーを aria-live で通知します。

assignment課題

以下のアクセシビリティ要件を満たす注文フォームを設計してください。

  • 各入力欄にヘルプテキスト（aria-describedby で関連付け）
  • エラー時に aria-invalid="true" を設定
  • エラーメッセージを role="alert" で即座に通知
  • フォーム上部にエラーサマリー（エラー一覧＋該当フィールドへのリンク）
  • 必須フィールドに aria-required="true" と視覚的な「*」マーク
lightbulbヒント

aria-describedby は複数の id をスペース区切りで指定可能です。ヘルプテキストとエラーメッセージの両方を関連付けるには aria-describedby="help-name error-name" とします。

check_circle解答・解説 
<form action="/order" method="post" novalidate>
  <!-- エラーサマリー -->
  <div role="alert" id="error-summary" hidden>
    <h2>入力エラーがあります</h2>
    <ul>
      <li><a href="#name">氏名は必須です</a></li>
      <li><a href="#email">有効なメールアドレスを入力してください</a></li>
    </ul>
  </div>

  <div>
    <label for="name">氏名 <span aria-hidden="true">*</span></label>
    <input type="text" id="name" name="name"
      aria-required="true" required
      aria-describedby="help-name error-name"
      aria-invalid="false">
    <span id="help-name">姓と名をスペース区切りで入力</span>
    <span id="error-name" role="alert" hidden>氏名は必須です</span>
  </div>

  <div>
    <label for="email">メール <span aria-hidden="true">*</span></label>
    <input type="email" id="email" name="email"
      aria-required="true" required
      aria-describedby="help-email error-email"
      aria-invalid="false">
    <span id="help-email">確認メールを送信します</span>
    <span id="error-email" role="alert" hidden>有効なメールアドレスを入力してください</span>
  </div>

  <button type="submit">注文する</button>
</form>
解説
  • novalidate でブラウザデフォルトのバリデーションを無効にし、カスタムバリデーションを使用します。
  • role="alert" はエラーメッセージ表示時にスクリーンリーダーが即座に読み上げます。
  • エラーサマリーの各項目が該当フィールドへのリンクになっているのは、キーボードユーザーの利便性のためです。
  • 「*」マークは aria-hidden="true" で隠し、aria-required で必須情報を伝えます。

アクセシブルなダイアログフォーム

verified
学習ポイント

verified
学習ポイント

Schema.orgのJSON-LDはページの種類に応じて使い分けます。Article、BreadcrumbList、FAQPage、HowTo など、適切なスキーマを選択することがリッチスニペット表示の鍵です。

assignment課題

以下の4つの構造化データパターンを1つのページに実装してください。

  • Article: 記事タイトル、著者、公開日、更新日、画像
  • BreadcrumbList: ホーム > カテゴリ > 記事タイトル の3階層
  • FAQPage: 3つの質問と回答
  • HowTo: 3ステップの手順（各ステップに名前・テキスト・画像）
visibility 表示結果
ブログ記事ページ。Google検索ではパンくずリスト、FAQ（アコーディオン形式）、手順（ステップ表示）のリッチスニペットが表示される可能性がある。記事情報は著者名・公開日付きで表示。
lightbulbヒント

BreadcrumbList は itemListElement 配列で ListItem を並べます。FAQPage は mainEntity 配列に Question 型を並べます。HowTo は step 配列に HowToStep を並べます。

verified

ダッシュボードUIでは、複数のウィジェット、データテーブル、フォームコントロール、ライブデータを統合し、WAI-ARIAで全体のアクセシビリティを確保する設計力が求められます。

assignment課題

以下の仕様で管理ダッシュボードのHTMLを設計してください。

  • サイドバーナビ: <nav aria-label="ダッシュボードメニュー">、現在ページに aria-current
  • 統計カード×4: 売上/ユーザー数/注文数/コンバージョン率、各カードに aria-label
  • データテーブル: ソート可能ヘッダー（aria-sort）、結果件数（aria-live）

  • 全体を <main> + <aside> のレイアウト構造で構築

assignment課題
