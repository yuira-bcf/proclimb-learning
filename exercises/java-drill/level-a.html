T1 変数・データ型・演算子
A-01
T1 変数・データ型・演算子

暗黙の型変換トラップ

verified
JavaSilverポイント

byte + byte の演算結果は int に昇格します。その結果を byte 変数に代入しようとすると、narrowing conversion が必要になりコンパイルエラーになります。試験では「問題なく動く」と思わせるダミー選択肢が頻出です。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question01 {
    public static void main(String[] args) {
        byte a = 10;
        byte b = 20;
        byte c = a + b;
        System.out.println(c);
    }
}
  • A. 30
  • B. 0
  • C. コンパイルエラー
  • D. 実行時に ArithmeticException が発生する
lightbulbヒント
byte 同士の加算結果の型は何になるか考えてみましょう。Java では byte, short, char の算術演算結果は自動的に int に昇格します。
check_circle解答・解説

正解: C（コンパイルエラー）

a + b の結果は int 型に昇格します。int を byte に代入するには明示的なキャストが必要です。byte c = (byte)(a + b); と書けば 30 が出力されます。

A が間違いの理由: キャストなしでは代入できないため、そもそも実行に到達しません。

B が間違いの理由: オーバーフローの話ではなく、型変換エラーです。

D が間違いの理由: 実行時エラーではなくコンパイル時のエラーです。

Silver頻出パターン: byte/short/char 同士の演算結果が int になる点は最頻出です。変数同士の演算とリテラル演算（byte c = 10 + 20; はOK）の違いにも注意しましょう。

A-02
T1 変数・データ型・演算子

演算子優先順位と短絡評価のトラップ

verified
JavaSilverポイント

&& は短絡評価（short-circuit evaluation）を行います。左辺が false なら右辺は評価されません。副作用のあるコード（インクリメントなど）が右辺にある場合、実行されるかどうかが問われます。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question02 {
    public static void main(String[] args) {
        int x = 5;
        int y = 10;
        boolean result = (x > 6) && (++y > 10);
        System.out.println("y=" + y + ", result=" + result);
    }
}
  • A. y=11, result=false
  • B. y=10, result=false
  • C. y=11, result=true
  • D. y=10, result=true
lightbulbヒント
x > 6 は false です。&& の短絡評価で右辺 ++y > 10 は評価されるでしょうか？
check_circle解答・解説

正解: B（y=10, result=false）

x > 6 は 5 > 6 で false。&& の短絡評価により右辺の ++y > 10 は一切評価されません。よって y は 10 のまま、result は false です。

A が間違いの理由: 短絡評価を見落とし、++y が実行されると考えた場合の誤答です。

C が間違いの理由: 左辺が false なので result が true になることはありません。

D が間違いの理由: y=10 は正しいですが result=true はあり得ません。

Silver頻出パターン: &&（短絡AND）と &（非短絡AND）の違いは頻出です。& なら両辺とも必ず評価されるため ++y は実行されます。|| と | も同様のパターンで出題されます。

T2 文字列操作
A-03
T2 文字列操作

String Pool とインターン化のトラップ

verified
JavaSilverポイント

文字列リテラルは String Pool に格納され、同じ内容なら同一オブジェクトを参照します。一方 new String() は常に新しいオブジェクトを作成します。== と equals() の違いは最頻出テーマの一つです。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question03 {
    public static void main(String[] args) {
        String s1 = "Hello";
        String s2 = new String("Hello");
        String s3 = s2.intern();

        System.out.println(s1 == s2);
        System.out.println(s1 == s3);
        System.out.println(s2 == s3);
    }
}
  • A. true true true
  • B. false true false
  • C. false false false
  • D. false true true
lightbulbヒント
intern() メソッドは String Pool にある同じ内容の文字列の参照を返します。s3 は Pool 内の "Hello" と同一の参照になります。
check_circle解答・解説

正解: B（false true false）

s1 == s2: s1 は Pool 内、s2 は new で作ったヒープ上の別オブジェクト → false

s1 == s3: s3 = s2.intern() は Pool 内の "Hello" の参照を返す。s1 も Pool 内の "Hello" → true

s2 == s3: s2 はヒープ上、s3 は Pool 内の別オブジェクト → false

A が間違いの理由: new String() は Pool とは別のオブジェクトなので s1 == s2 は false です。

C が間違いの理由: intern() の効果を理解していれば s1 == s3 は true です。

D が間違いの理由: s2 と s3 は別のオブジェクト参照です。intern() は新しい参照を返すだけで s2 自体は変わりません。

Silver頻出パターン: == は参照比較、equals() は値比較。intern() は Pool の参照を返す。この3点セットで必ず出題されます。

A-04
T2 文字列操作

Stringの不変性とメソッドチェーンのトラップ

verified
JavaSilverポイント

String は不変（immutable）オブジェクトです。replace()、toUpperCase()、substring() 等のメソッドは元の文字列を変更せず、新しい String オブジェクトを返します。戻り値を使わなければ何も変わりません。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question04 {
    public static void main(String[] args) {
        String s = "Hello World";
        s.replace("World", "Java");
        s.toUpperCase();
        System.out.println(s);
        System.out.println(s.substring(0, 5));
        System.out.println(s.substring(6));
    }
}
  • A. HELLO JAVA / HELLO / JAVA
  • B. Hello World / Hello / World
  • C. Hello Java / Hello / Java
  • D. コンパイルエラー
lightbulbヒント
s.replace(...) と s.toUpperCase() の戻り値はどこに代入されていますか？ String は不変オブジェクトであることを思い出してください。
check_circle解答・解説

正解: B（Hello World / Hello / World）

s.replace("World", "Java") は新しい String "Hello Java" を返しますが、戻り値を s に再代入していないため s は "Hello World" のままです。s.toUpperCase() も同様に戻り値が捨てられています。

s.substring(0, 5) → "Hello"（インデックス 0〜4）

s.substring(6) → "World"（インデックス 6 以降）

A が間違いの理由: replace と toUpperCase の結果を再代入していないため、元の s は変わりません。

C が間違いの理由: replace の結果を再代入していないため s は "Hello World" のままです。

D が間違いの理由: 戻り値を捨てること自体はコンパイルエラーにはなりません。

Silver頻出パターン: String の不変性を利用した「戻り値を受け取り忘れる」パターンは定番です。StringBuilder との対比（こちらは破壊的変更）もセットで出題されます。

T3 配列
A-05
T3 配列

配列の参照渡しトラップ

verified
JavaSilverポイント

Java はすべて「値渡し」ですが、配列変数の値は参照（アドレス）です。メソッド内で配列の要素を変更すると呼び出し元にも影響しますが、配列変数自体を別の配列に再代入しても呼び出し元は変わりません。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question05 {
    static void modify(int[] arr) {
        arr[0] = 99;
        arr = new int[]{-1, -2, -3};
        arr[1] = 100;
    }

    public static void main(String[] args) {
        int[] data = {1, 2, 3};
        modify(data);
        System.out.println(data[0] + ", " + data[1] + ", " + data[2]);
    }
}
  • A. 1, 2, 3
  • B. 99, 2, 3
  • C. -1, 100, -3
  • D. 99, 100, 3
lightbulbヒント
arr[0] = 99; は元の配列を変更します。その後 arr = new int[]{...} で arr は新しい配列を指すようになります。この再代入は呼び出し元の data に影響するでしょうか？
check_circle解答・解説

正解: B（99, 2, 3）

処理の流れ:

1. arr[0] = 99; → arr は data と同じ配列を指しているので、data[0] が 99 に変更される

2. arr = new int[]{-1, -2, -3}; → arr が新しい配列を指すようになる。data はまだ元の配列を指している

3. arr[1] = 100; → 新しい配列の要素を変更。data には影響しない

A が間違いの理由: arr[0] = 99; の時点では arr と data は同じ配列を指しているため data[0] は変更されます。

C が間違いの理由: arr = new int[]{...} は arr ローカル変数の再代入であり、data には影響しません。

D が間違いの理由: arr[1] = 100; は再代入後の新しい配列への操作であり、data[1] は変わりません。

Silver頻出パターン: 「参照のコピーが渡される」仕組みを正確に理解しているかが問われます。要素の変更は反映されるが、参照の再代入は反映されない — この区別は必須知識です。

A-06
T3 配列

多次元配列と ArrayIndexOutOfBoundsException のトラップ

verified
JavaSilverポイント

Java の多次元配列は「配列の配列」であり、各行の長さが異なる不揃い（jagged）配列が可能です。arr.length は行数、arr[i].length は各行の列数で、混同すると ArrayIndexOutOfBoundsException が発生します。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question06 {
    public static void main(String[] args) {
        int[][] matrix = new int[3][];
        matrix[0] = new int[]{1, 2};
        matrix[1] = new int[]{3, 4, 5};
        matrix[2] = new int[]{6};

        int sum = 0;
        for (int i = 0; i < matrix.length; i++) {
            sum += matrix[i][matrix[i].length - 1];
        }
        System.out.println(sum);
    }
}
  • A. 13
  • B. 11
  • C. 6
  • D. 実行時に ArrayIndexOutOfBoundsException が発生する
lightbulbヒント
matrix[i][matrix[i].length - 1] は各行の最後の要素を取り出します。各行の最後の要素は何ですか？
check_circle解答・解説

正解: A（13）

各行の最後の要素を取得します:

・matrix[0] = {1, 2} → 最後の要素: 2

・matrix[1] = {3, 4, 5} → 最後の要素: 5

・matrix[2] = {6} → 最後の要素: 6

sum = 2 + 5 + 6 = 13

B が間違いの理由: 各行の先頭要素の合計（1+3+6=10）や他の計算と混同しています。

C が間違いの理由: matrix[2] の要素だけを見た場合の値です。

D が間違いの理由: matrix[i].length - 1 は各行の有効な最終インデックスなので例外は発生しません。不揃い配列でも各行ごとに length を使えば安全です。

Silver頻出パターン: 不揃い配列で matrix.length（行数）と matrix[i].length（各行の列数）を混同させる問題が出ます。matrix[0].length を全行に適用して例外を起こすパターンが典型です。

T4 フロー制御
A-07
T4 フロー制御

switch文のfall-throughトラップ

verified
JavaSilverポイント

switch 文で break を省略すると、次の case に「落ちる」（fall-through）します。また default の位置は先頭や中間にも置けますが、fall-through の影響範囲が変わります。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question07 {
    public static void main(String[] args) {
        int val = 2;
        String result = "";
        switch (val) {
            case 1: result += "A";
            case 2: result += "B";
            case 3: result += "C";
                    break;
            default: result += "D";
            case 4: result += "E";
        }
        System.out.println(result);
    }
}
  • A. B
  • B. BC
  • C. BCD
  • D. BCDE
lightbulbヒント
val は 2 なので case 2 にマッチします。break が出現するまで何が実行されるか、上から順に追ってみましょう。
check_circle解答・解説

正解: B（BC）

val = 2 なので case 2 にマッチし、result += "B" が実行されます。break がないので case 3 に fall-through し、result += "C" が実行されます。case 3 に break; があるのでここで switch を抜けます。

A が間違いの理由: case 2 の後に break がないため、case 3 にも fall-through します。

C が間違いの理由: case 3 に break があるため、default には到達しません。

D が間違いの理由: break で switch を抜けるため default と case 4 は実行されません。

Silver頻出パターン: fall-through の範囲を正確に追う問題は定番です。default を中間に配置して混乱させるパターンもよく出ます。break の位置を1つ1つ確認しましょう。

A-08
T4 フロー制御

ラベル付きbreak/continueとループスコープのトラップ

verified
JavaSilverポイント

ラベル付き break は指定したラベルのループ全体を抜けます。ラベル付き continue は指定したラベルのループの次の反復に進みます。ラベルなしの break/continue との違いを正確に区別する必要があります。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question08 {
    public static void main(String[] args) {
        int count = 0;
        OUTER:
        for (int i = 0; i < 3; i++) {
            for (int j = 0; j < 3; j++) {
                if (j == 1) continue OUTER;
                count++;
            }
        }
        System.out.println(count);
    }
}
  • A. 3
  • B. 6
  • C. 9
  • D. コンパイルエラー
lightbulbヒント
continue OUTER は内側のループを中断し、外側ループの次の反復に進みます。j が 1 になった時点で外側に戻るので、各 i の反復で count++ が何回実行されるか考えてみましょう。
check_circle解答・解説

正解: A（3）

各反復の流れ:

・i=0: j=0 → count++（1）、j=1 → continue OUTER で外側へ

・i=1: j=0 → count++（2）、j=1 → continue OUTER で外側へ

・i=2: j=0 → count++（3）、j=1 → continue OUTER で外側へ

各 i の反復で count++ は1回だけ実行されるため、合計 3 回です。

B が間違いの理由: ラベルなしの continue（内側ループのみ継続）と混同した場合、j=0 と j=2 で count++ が実行され 6 になりますが、continue OUTER は外側に戻るため j=2 には到達しません。

C が間違いの理由: continue がなかった場合の結果（3×3=9）です。

D が間違いの理由: ラベル付き continue は正当な Java 構文です。

Silver頻出パターン: ラベル付き break/continue はそれだけで出題される頻出テーマです。ラベルなしとの動作の違い、特に「どのループに対して作用するか」を正確に把握しましょう。

T5 クラス・メソッド
A-09
T5 クラス・メソッド

メソッドオーバーロードの解決ルールトラップ

verified
JavaSilverポイント

オーバーロード解決は「拡大変換 > autoboxing > 可変長引数」の優先順位で行われます。int を渡した場合、long を受け取るメソッドが Integer を受け取るメソッドより優先されます。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question09 {
    static String test(long x) {
        return "long";
    }
    static String test(Integer x) {
        return "Integer";
    }
    static String test(int... x) {
        return "varargs";
    }

    public static void main(String[] args) {
        System.out.println(test(10));
    }
}
  • A. long
  • B. Integer
  • C. varargs
  • D. コンパイルエラー（曖昧な呼び出し）
lightbulbヒント
オーバーロード解決の優先順位を思い出しましょう。int → long は拡大変換、int → Integer は autoboxing、int → int... は可変長引数です。どれが最優先でしょうか？
check_circle解答・解説

正解: A（long）

Java のオーバーロード解決は以下の優先順位で行われます:

1. 拡大変換（widening）: int → long（最優先）

2. autoboxing: int → Integer

3. 可変長引数（varargs）: int → int...

10 は int リテラルで、拡大変換により long を受け取るメソッドが選択されます。

B が間違いの理由: autoboxing は拡大変換より低い優先度です。

C が間違いの理由: 可変長引数は最も低い優先度で、他にマッチするものがない場合にのみ使われます。

D が間違いの理由: 優先順位が明確に定義されているため、曖昧にはなりません。

Silver頻出パターン: 「拡大 → autoboxing → varargs」の優先順位は必ず覚えましょう。さらに「autoboxing + 拡大」（int → Integer → Number）は不可という制約も出題されます。

A-10
T5 クラス・メソッド

staticメンバとインスタンスメンバの混同トラップ

verified
JavaSilverポイント

static メソッドから this やインスタンスメンバに直接アクセスすることはできません。static 初期化ブロック内でもインスタンスメンバは使えません。一方、インスタンスメソッドから static メンバにはアクセス可能です。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question10 {
    int count = 0;
    static int total = 0;

    static {
        total = 10;
        // count = 5;  // もしこの行がコメントを外されたら？
    }

    static void add(int n) {
        total += n;
        // count += n;  // もしこの行がコメントを外されたら？
    }

    public static void main(String[] args) {
        add(5);
        Question10 obj = new Question10();
        obj.count = 3;
        System.out.println(total + ", " + obj.count);
    }
}
  • A. 15, 3
  • B. 10, 3
  • C. 15, 0
  • D. コンパイルエラー
lightbulbヒント
コメントアウトされた行は実行されません。static ブロックで total = 10 に初期化され、add(5) で total は 15 になります。obj.count はインスタンスフィールドで別管理です。
check_circle解答・解説

正解: A（15, 3）

処理の流れ:

1. クラスロード時に static ブロックが実行: total = 10

2. add(5): total += 5 → total = 15

3. new Question10(): count は 0 で初期化される（インスタンスフィールド）

4. obj.count = 3: obj のインスタンスの count が 3 に

5. 出力: 15, 3

B が間違いの理由: add(5) の呼び出しにより total は 10 + 5 = 15 です。

C が間違いの理由: obj.count = 3; で明示的に代入しているため 0 ではありません。

D が間違いの理由: コメント行は実行されないため、現状のコードにエラーはありません。もしコメントを外すと、static コンテキストからインスタンスフィールド count にアクセスするためコンパイルエラーになります。

Silver頻出パターン: 「static から non-static にアクセス不可」「コメントを外したらどうなるか」という形式が頻出です。static 初期化ブロックの実行タイミング（クラスロード時）も問われます。

T6 継承・IF・ポリモーフィズム
A-11
T6 継承・IF・ポリモーフィズム

コンストラクタチェーンとオーバーライドのトラップ

verified
JavaSilverポイント

子クラスのコンストラクタは暗黙的に super() を呼びます。親クラスに引数なしコンストラクタがない場合、明示的に super(引数) を呼ばないとコンパイルエラーになります。また、オーバーライド時にアクセス修飾子をより狭くすることはできません。

assignment課題

以下のコードの出力として正しいものを選んでください。

class Parent {
    String name;
    Parent(String name) {
        this.name = name;
        System.out.print("P:" + name + " ");
    }
    public String greet() {
        return "Hello";
    }
}

class Child extends Parent {
    Child() {
        super("child");
        System.out.print("C ");
    }
    @Override
    protected String greet() {
        return "Hi";
    }
}

public class Question11 {
    public static void main(String[] args) {
        Parent p = new Child();
        System.out.print(p.greet());
    }
}
  • A. P:child C Hi
  • B. C P:child Hello
  • C. P:child C Hello
  • D. コンパイルエラー
lightbulbヒント
オーバーライド時にアクセス修飾子を変更していますが、public を protected にすることは許されるでしょうか？ オーバーライドのルールは「同じかより広い」アクセスです。
check_circle解答・解説

正解: D（コンパイルエラー）

親クラスの greet() は public ですが、子クラスで protected にオーバーライドしています。オーバーライド時にアクセス修飾子をより狭くすることはできません（public → protected は不可）。

A が間違いの理由: コンパイルエラーになるため、出力には到達しません。

B が間違いの理由: コンストラクタの実行順序は正しく super() が先ですが、そもそもコンパイルが通りません。

C が間違いの理由: ポリモーフィズムにより実行時には Child の greet() が呼ばれますが、コンパイルエラーのためそこまで到達しません。

Silver頻出パターン: オーバーライドの制約は3つ覚えましょう。(1) アクセス修飾子は同じかより広い、(2) 戻り値型は同じか共変（サブタイプ）、(3) throws は同じかより狭い例外。このルールに違反するとコンパイルエラーです。

インタフェースのdefault/staticメソッドとダイヤモンド継承のトラップ

verified
JavaSilverポイント

Java 8 以降、インタフェースに default メソッドを定義できます。2つのインタフェースが同じシグネチャの default メソッドを持ち、クラスが両方を実装する場合、実装クラスでオーバーライドしなければコンパイルエラーになります。

assignment課題

以下のコードの出力として正しいものを選んでください。

interface Flyable {
    default String move() { return "fly"; }
}

interface Swimmable {
    default String move() { return "swim"; }
}

class Duck implements Flyable, Swimmable {
    @Override
    public String move() {
        return Flyable.super.move() + " & " + Swimmable.super.move();
    }
}

public class Question12 {
    public static void main(String[] args) {
        Duck d = new Duck();
        System.out.println(d.move());
    }
}
  • A. fly
  • B. swim
  • C. fly & swim
  • D. コンパイルエラー
lightbulbヒント
Duck クラスは move() をオーバーライドしています。Flyable.super.move() は Flyable の default 実装を呼び出す構文です。
check_circle解答・解説

正解: C（fly & swim）

Duck クラスは両方のインタフェースの move() メソッドをオーバーライドして衝突を解決しています。オーバーライド内で Flyable.super.move() と Swimmable.super.move() を明示的に呼び出し、結果を結合しています。

A, B が間違いの理由: 両方の default 実装を呼び出して結合しているため、片方だけにはなりません。

D が間違いの理由: もしオーバーライドがなければコンパイルエラーですが、Duck は move() をオーバーライドしているため問題ありません。インタフェース名.super.メソッド() は正当な構文です。

Silver頻出パターン: ダイヤモンド問題の解決方法（実装クラスでのオーバーライド必須）と、インタフェース名.super.メソッド() の構文は頻出です。オーバーライドしないパターンで「コンパイルエラー」が正解になる問題もあります。

例外の伝播とcatch順序のトラップ

verified
JavaSilverポイント

catch ブロックは上から順に評価されます。親クラスの例外を先に catch すると、子クラスの catch に到達できないためコンパイルエラーになります。また、finally 内で return すると try/catch の return を上書きします。

assignment課題

以下のコードの出力として正しいものを選んでください。

public class Question13 {
    static int calc() {
        try {
            int[] arr = new int[3];
            arr[5] = 10;
            return 1;
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.print("catch ");
            return 2;
        } finally {
            System.out.print("finally ");
            return 3;
        }
    }

    public static void main(String[] args) {
        System.out.print(calc());
    }
}
  • A. catch finally 2
  • B. catch finally 3
  • C. finally 3
  • D. コンパイルエラー
lightbulbヒント
arr[5] = 10; で例外が発生し catch ブロックに入ります。finally ブロックは必ず実行されます。finally 内の return は catch の return を上書きするでしょうか？
check_circle解答・解説

正解: B（catch finally 3）

処理の流れ:

1. arr[5] = 10; → ArrayIndexOutOfBoundsException 発生

2. catch ブロック: "catch " を出力、return 2 を保留

3. finally ブロック: "finally " を出力、return 3 でcatch の return 2 を上書き

4. 最終的に 3 が返される

A が間違いの理由: finally 内の return が catch 内の return を上書きするため、2 ではなく 3 が返されます。

C が間違いの理由: 例外が発生するため catch ブロックは実行されます。

D が間違いの理由: finally 内の return は警告は出ますがコンパイルエラーにはなりません。

Silver頻出パターン: finally 内の return による上書きは頻出トラップです。実務では finally 内で return を書くのは避けるべきですが、試験では理解を問うために出題されます。catch と finally の実行順序を正確に追いましょう。

try-with-resourcesとAutoCloseableのトラップ

verified
JavaSilverポイント

try-with-resources で宣言されたリソースは、try ブロック終了後に宣言と逆順で close() が呼ばれます。close() で例外が発生した場合は suppressed exception として記録されます。finally ブロックは close() の後に実行されます。

assignment課題

以下のコードの出力として正しいものを選んでください。

class ResourceA implements AutoCloseable {
    ResourceA() { System.out.print("openA "); }
    @Override
    public void close() { System.out.print("closeA "); }
}

class ResourceB implements AutoCloseable {
    ResourceB() { System.out.print("openB "); }
    @Override
    public void close() { System.out.print("closeB "); }
}

public class Question14 {
    public static void main(String[] args) {
        try (ResourceA a = new ResourceA();
             ResourceB b = new ResourceB()) {
            System.out.print("use ");
        } finally {
            System.out.print("done");
        }
    }
}
  • A. openA openB use closeA closeB done
  • B. openA openB use closeB closeA done
  • C. openA openB use done closeB closeA
  • D. openA openB use closeB closeA
lightbulbヒント
リソースは宣言順に open され、逆順に close されます。finally ブロックは close() の後に実行されます。
check_circle解答・解説

正解: B（openA openB use closeB closeA done）

実行順序:

1. ResourceA のコンストラクタ: "openA "

2. ResourceB のコンストラクタ: "openB "

3. try ブロック本体: "use "

4. ResourceB の close(): "closeB "（後に宣言されたものが先にクローズ）

5. ResourceA の close(): "closeA "

6. finally ブロック: "done"

A が間違いの理由: close() は宣言の逆順で呼ばれるため、closeA が先ではありません。

C が間違いの理由: finally は close() の後に実行されます。close() の前ではありません。

D が間違いの理由: finally ブロックの出力 "done" が含まれていません。

Silver頻出パターン: try-with-resources の close() 順序（逆順）と、finally との実行順序（close → finally）は頻出です。close() で例外が発生するパターン（suppressed exception）も出題されます。

T8 コレクション・ラムダ・日時

ジェネリクスの型消去とワイルドカードのトラップ

verified
JavaSilverポイント

Java のジェネリクスはコンパイル時にのみ型チェックされ、実行時には型消去（type erasure）されます。List<? extends Number> は読み取り専用で要素追加不可、List<? super Integer> は Integer を追加可能です（PECS原則）。

assignment課題

以下のコードの出力として正しいものを選んでください。

import java.util.ArrayList;
import java.util.List;

public class Question15 {
    static void addNumber(List<? super Integer> list) {
        list.add(42);
    }

    static void readNumber(List<? extends Number> list) {
        // list.add(42);  // この行がコメント解除されたら？
        Number n = list.get(0);
        System.out.print(n + " ");
    }

    public static void main(String[] args) {
        List<Number> nums = new ArrayList<>();
        addNumber(nums);
        addNumber(nums);
        readNumber(nums);
        System.out.print("size=" + nums.size());
    }
}
  • A. 42 size=2
  • B. 42 42 size=2
  • C. コンパイルエラー
  • D. 実行時に ClassCastException が発生する
lightbulbヒント
List<? super Integer> に Integer を add するのは合法です。readNumber は list.get(0) を1回だけ呼び出しています。コメントアウトされた行は実行されないことに注意してください。
check_circle解答・解説

正解: A（42 size=2）

処理の流れ:

1. addNumber(nums) × 2回: nums に 42 が2つ追加される → [42, 42]

2. readNumber(nums): list.get(0) → 42 を取得して "42 " を出力（1回のみ）

3. nums.size() → 2 を出力

B が間違いの理由: readNumber 内で get(0) を1回しか呼んでいないため "42" は1回だけ出力されます。

C が間違いの理由: List<? super Integer> への Integer の add は合法です。コメント行は実行されないため影響しません。もしコメントを外すと、List<? extends Number> への add はコンパイルエラーになります。

D が間違いの理由: 型安全な操作しか行っていないため実行時例外は発生しません。

Silver頻出パターン: PECS（Producer Extends, Consumer Super）は必須知識です。? extends は読み取り専用（addは不可、nullのみ可）、? super は書き込み可能（ただし読み取り時の型は Object）。この制約をコンパイルエラーと絡めて出題されます。

ラムダ式のスコープとeffectively finalのトラップ

verified
JavaSilverポイント

ラムダ式の中からローカル変数にアクセスする場合、その変数は final または effectively final（一度も再代入されない）でなければなりません。違反するとコンパイルエラーになります。また、ラムダ内の this はラムダを囲むクラスのインスタンスを参照します。

assignment課題

以下のコードの出力として正しいものを選んでください。

import java.util.Arrays;
import java.util.List;

public class Question16 {
    String name = "outer";

    void run() {
        String prefix = "Hello";
        // prefix = "Hi";  // この行がコメント解除されたら？
        List<String> names = Arrays.asList("Alice", "Bob");

        names.forEach(n -> {
            System.out.print(prefix + " " + n + " ");
            System.out.print(this.name + " ");
        });
    }

    public static void main(String[] args) {
        new Question16().run();
    }
}
  • A. Hello Alice outer Hello Bob outer
  • B. Hello Alice Hello Bob
  • C. コンパイルエラー（effectively final 違反）
  • D. コンパイルエラー（ラムダ内で this は使用不可）
lightbulbヒント
コメントアウトされた prefix = "Hi"; は実行されません。prefix はこのコードでは一度も再代入されないため effectively final です。ラムダ内の this は Question16 のインスタンスを指します。
check_circle解答・解説

正解: A（Hello Alice outer Hello Bob outer ）

prefix は一度も再代入されていないため effectively final であり、ラムダ内から参照可能です。ラムダ内の this は Question16 のインスタンスを指すため、this.name は "outer" です。

・forEach で "Alice" → "Hello Alice outer " を出力

・forEach で "Bob" → "Hello Bob outer " を出力

B が間違いの理由: this.name の出力 "outer " が含まれていません。

C が間違いの理由: prefix = "Hi"; はコメントアウトされているため、prefix は effectively final です。もしコメントを外すと再代入されるため effectively final ではなくなり、コンパイルエラーになります。

D が間違いの理由: ラムダ内で this は使用可能です。ラムダの this は囲むクラスのインスタンスを指します（匿名クラスの this とは異なります）。

Silver頻出パターン: effectively final の判定（コメントを外す/外さないで結果が変わる）と、ラムダ内の this の挙動は頻出です。匿名クラスでは this が匿名クラス自身を指す点との対比も問われます。
