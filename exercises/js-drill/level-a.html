イミュータブルな状態管理ヘルパーを設計

verified
学習ポイント

ReduxやuseReducerで求められるイミュータブルな状態更新パターンを、ヘルパー関数として設計します。

assignment課題

ネストしたオブジェクトをイミュータブルに更新する updateIn 関数を設計・実装してください。

function updateIn(obj, path, updater) → object
- obj: 更新対象のオブジェクト（元のオブジェクトは変更しない）
- path: ドット区切りのパス文字列（例: 'user.address.city'）
- updater: 現在値を受け取り、新しい値を返す関数
- パス上のオブジェクトをすべてシャローコピーしながら更新 
const state = {
  user: { name: '田中', address: { city: '東京', zip: '100-0001' } },
  settings: { theme: 'dark' }
};

const newState = updateIn(state, 'user.address.city', () => '大阪');
console.log(newState.user.address.city); // '大阪'
console.log(state.user.address.city);    // '東京'（元は不変）
console.log(newState.settings === state.settings); // true（変更なしは共有）
lightbulbヒント

パスを split('.') で配列にし、再帰またはループでスプレッド構文を使いながらコピーを作ります。変更が必要な部分だけコピーし、それ以外は参照を共有します。

check_circle解答・解説 
function updateIn(obj, path, updater) {
  const keys = path.split('.');

  function helper(current, index) {
    if (index === keys.length) {
      return updater(current);
    }
    const key = keys[index];
    return {
      ...current,
      [key]: helper(current[key], index + 1)
    };
  }

  return helper(obj, 0);
}
解説
  • 再帰の各ステップでスプレッド構文のシャローコピーを作り、変更するキーだけを上書きします。
  • 変更パス上にないプロパティ（settings）は元の参照がそのまま共有されるため、メモリ効率が良いです。
  • Immer ライブラリはこの概念をProxy で自動化し、ミュータブルな書き方でイミュータブル更新を実現しています。

スキーマバリデーターを設計

verified
学習ポイント

Zodのようなスキーマバリデーションの仕組みを自前で設計することで、型安全なデータ処理の理解を深めます。

assignment課題

以下のAPIで使えるスキーマバリデーターを設計・実装してください。

const userSchema = schema({
  name: string().minLength(1),
  age: number().min(0).max(150),
  email: string().matches(/@/),
  role: oneOf(['admin', 'user', 'guest'])
});

const result1 = userSchema.validate({
  name: '田中', age: 25, email: 'tanaka@example.com', role: 'admin'
});
console.log(result1); // { valid: true, errors: [] }

const result2 = userSchema.validate({
  name: '', age: -1, email: 'invalid', role: 'superadmin'
});
console.log(result2.errors);
// ['name: 1文字以上', 'age: 0以上', 'email: パターン不一致', 'role: admin,user,guest のいずれか']
lightbulbヒント

各バリデーター（string(), number()等）はルール配列を保持するオブジェクトを返し、メソッドチェーンで return this します。schema() は各フィールドのバリデーターを実行してエラーを集約します。

check_circle解答・解説 
function string() {
  const rules = [{ check: v => typeof v === 'string', msg: '文字列ではありません' }];
  const self = {
    minLength(n) { rules.push({ check: v => v.length >= n, msg: `${n}文字以上` }); return self; },
    maxLength(n) { rules.push({ check: v => v.length <= n, msg: `${n}文字以下` }); return self; },
    matches(regex) { rules.push({ check: v => regex.test(v), msg: 'パターン不一致' }); return self; },
    _validate(value) { return rules.filter(r => !r.check(value)).map(r => r.msg); }
  };
  return self;
}

function number() {
  const rules = [{ check: v => typeof v === 'number', msg: '数値ではありません' }];
  const self = {
    min(n) { rules.push({ check: v => v >= n, msg: `${n}以上` }); return self; },
    max(n) { rules.push({ check: v => v <= n, msg: `${n}以下` }); return self; },
    _validate(value) { return rules.filter(r => !r.check(value)).map(r => r.msg); }
  };
  return self;
}

function oneOf(options) {
  return {
    _validate(value) {
      return options.includes(value) ? [] : [`${options.join(',')} のいずれか`];
    }
  };
}

function schema(definition) {
  return {
    validate(data) {
      const errors = [];
      for (const [key, validator] of Object.entries(definition)) {
        const fieldErrors = validator._validate(data[key]);
        fieldErrors.forEach(msg => errors.push(`${key}: ${msg}`));
      }
      return { valid: errors.length === 0, errors };
    }
  };
}
解説
  • 各バリデーター関数はクロージャで rules 配列を保持し、メソッドチェーンでルールを追加します。
  • _validate メソッドで全ルールを実行し、失敗したルールのメッセージを返します。
  • Zodライブラリの基本構造と同じ設計原理です。実務ではZodを使いますが、内部の仕組みを理解しておくとデバッグに役立ちます。

依存性注入コンテナを設計

verified
学習ポイント

DI（依存性注入）はテスタビリティと疎結合を実現する設計パターンです。クロージャとMapを使って簡易DIコンテナを実装します。

assignment課題

以下のAPIで使えるDIコンテナを設計・実装してください。

const container = createContainer();

// サービスの登録
container.register('logger', () => ({
  log: (msg) => console.log(`[LOG] ${msg}`)
}));

container.register('userService', (c) => ({
  getUser: (id) => {
    c.resolve('logger').log(`Fetching user ${id}`);
    return { id, name: 'User ' + id };
  }
}));

// 解決
const userService = container.resolve('userService');
userService.getUser(1); // "[LOG] Fetching user 1"
lightbulbヒント

Map でファクトリ関数を保持し、resolve 時にファクトリを実行します。ファクトリにコンテナ自身を渡すことで他のサービスを解決できます。

check_circle解答・解説 
function createContainer() {
  const factories = new Map();
  const instances = new Map();

  const container = {
    register(name, factory) {
      factories.set(name, factory);
    },
    resolve(name) {
      if (instances.has(name)) return instances.get(name);
      const factory = factories.get(name);
      if (!factory) throw new Error(`Service not found: ${name}`);
      const instance = factory(container);
      instances.set(name, instance);
      return instance;
    }
  };

  return container;
}
解説
  • instances マップでシングルトンキャッシュを実現しています。同じサービスを何度 resolve しても同一インスタンスが返されます。
  • ファクトリ関数にコンテナを渡すことで、サービス間の依存関係を解決できます。
  • NestJS、Angular、InversifyJS などのフレームワークが採用しているDIパターンの簡易版です。

リアクティブシグナルを設計

verified
学習ポイント

SolidJS や Preact のシグナル（Signal）パターンは、クロージャで状態と購読者を管理する軽量なリアクティブシステムです。

assignment課題

以下のAPIで使えるシグナルシステムを実装してください。

const [count, setCount] = createSignal(0);
const [doubled, setDoubled] = createSignal(0);

// エフェクトの登録
createEffect(() => {
  setDoubled(count() * 2);
});

console.log(count());   // 0
console.log(doubled()); // 0
setCount(5);
console.log(count());   // 5
console.log(doubled()); // 10
lightbulbヒント

createSignal はゲッター関数とセッター関数のタプルを返します。グローバルな currentEffect 変数を使って、ゲッター呼び出し時に依存関係を自動追跡します。

check_circle解答・解説 
let currentEffect = null;

function createSignal(initialValue) {
  let value = initialValue;
  const subscribers = new Set();

  const getter = () => {
    if (currentEffect) subscribers.add(currentEffect);
    return value;
  };

  const setter = (newValue) => {
    value = newValue;
    subscribers.forEach(fn => fn());
  };

  return [getter, setter];
}

function createEffect(fn) {
  currentEffect = fn;
  fn();
  currentEffect = null;
}
解説
  • createEffect 実行中にゲッターが呼ばれると、そのエフェクト関数が購読者として登録されます（自動依存追跡）。
  • セッターが呼ばれると全購読者が再実行され、リアクティブに値が伝播します。
  • SolidJS の createSignal / createEffect の基本原理そのものです。

LINQ風クエリビルダーを設計

verified
学習ポイント

遅延評価とメソッドチェーンを組み合わせたクエリビルダーパターンは、ORMやデータ処理ライブラリの基本設計です。

assignment課題

以下のAPIで使えるクエリビルダーを実装してください。操作は execute() を呼ぶまで実行されない（遅延評価）ようにしてください。

const data = [
  { name: '田中', age: 30, dept: '開発' },
  { name: '鈴木', age: 25, dept: '営業' },
  { name: '佐藤', age: 35, dept: '開発' },
  { name: '高橋', age: 28, dept: '開発' }
];

const result = query(data)
  .where(p => p.dept === '開発')
  .orderBy('age', 'desc')
  .select('name', 'age')
  .execute();

// [{ name: '佐藤', age: 35 }, { name: '田中', age: 30 }, { name: '高橋', age: 28 }]
lightbulbヒント

操作を関数の配列として蓄積し、execute() で順番に適用します。各メソッドは操作を追加して return this します。

check_circle解答・解説 
function query(data) {
  const operations = [];

  const builder = {
    where(predicate) {
      operations.push(d => d.filter(predicate));
      return builder;
    },
    orderBy(key, direction = 'asc') {
      operations.push(d => [...d].sort((a, b) => {
        const cmp = a[key] > b[key] ? 1 : a[key] < b[key] ? -1 : 0;
        return direction === 'desc' ? -cmp : cmp;
      }));
      return builder;
    },
    select(...keys) {
      operations.push(d => d.map(item => {
        const result = {};
        keys.forEach(k => { result[k] = item[k]; });
        return result;
      }));
      return builder;
    },
    execute() {
      return operations.reduce((result, op) => op(result), [...data]);
    }
  };

  return builder;
}
解説
  • 遅延評価パターンにより、操作の組み立てと実行を分離しています。最適化やデバッグが容易になります。
  • execute() で reduce を使い、蓄積した操作関数を順番に適用しています。
  • Knex.js、Prisma、TypeORM などのクエリビルダーの基本設計原理です。

差分検出アルゴリズムを設計

verified
学習ポイント

仮想DOMやGitの差分検出の基礎となるオブジェクト比較アルゴリズムを実装することで、差分更新の仕組みを理解します。

assignment課題

2つのオブジェクトを比較し、差分を検出する diff 関数を実装してください。

function diff(oldObj, newObj) → array of changes
- 各変更は { type: 'add'|'remove'|'update', path, value?, oldValue? }
- ネストしたオブジェクトも再帰的に比較 
const old = { name: '田中', age: 30, address: { city: '東京' } };
const updated = { name: '田中', age: 31, address: { city: '大阪' }, email: 'a@b.com' };

console.log(diff(old, updated));
// [
//   { type: 'update', path: 'age', oldValue: 30, value: 31 },
//   { type: 'update', path: 'address.city', oldValue: '東京', value: '大阪' },
//   { type: 'add', path: 'email', value: 'a@b.com' }
// ]
lightbulbヒント

再帰関数で両方のキーを走査します。oldObj にしかないキーは remove、newObj にしかないキーは add、両方にあるが値が違うものは update またはネスト比較します。

check_circle解答・解説 
function diff(oldObj, newObj, prefix = '') {
  const changes = [];
  const allKeys = new Set([...Object.keys(oldObj), ...Object.keys(newObj)]);

  for (const key of allKeys) {
    const path = prefix ? `${prefix}.${key}` : key;
    const oldVal = oldObj[key];
    const newVal = newObj[key];

    if (!(key in oldObj)) {
      changes.push({ type: 'add', path, value: newVal });
    } else if (!(key in newObj)) {
      changes.push({ type: 'remove', path, oldValue: oldVal });
    } else if (typeof oldVal === 'object' && typeof newVal === 'object'
               && oldVal !== null && newVal !== null
               && !Array.isArray(oldVal) && !Array.isArray(newVal)) {
      changes.push(...diff(oldVal, newVal, path));
    } else if (oldVal !== newVal) {
      changes.push({ type: 'update', path, oldValue: oldVal, value: newVal });
    }
  }

  return changes;
}
解説
  • Set で両オブジェクトのキーをマージし、各キーの状態を判定しています。
  • オブジェクト同士の場合は再帰呼び出しでネストを走査し、パスを prefix.key で構築しています。
  • React の仮想DOM差分やImmerの変更検出の基本原理です。

仮想DOMの簡易実装

verified
学習ポイント

Reactの仮想DOMの基本原理を理解するために、JSX風のオブジェクトからDOM要素を生成し、差分更新を行う仕組みを実装します。

assignment課題

以下の3つの関数を実装してください。

function h(tag, props, ...children) → vnode
- 仮想DOMノードを作成（{ tag, props, children }）

function render(vnode) → HTMLElement
- vnodeからDOM要素を生成

function mount(vnode, container) → void
- containerにレンダリング結果を追加 
const app = h('div', { class: 'app' },
  h('h1', null, 'Hello Virtual DOM'),
  h('ul', null,
    h('li', null, 'Item 1'),
    h('li', null, 'Item 2')
  )
);

mount(app, document.getElementById('root'));
lightbulbヒント

h() はオブジェクトを返すだけ。render() は createElement と再帰で子要素を処理します。文字列の子要素は createTextNode で扱います。

check_circle解答・解説 
function h(tag, props, ...children) {
  return { tag, props: props || {}, children: children.flat() };
}

function render(vnode) {
  if (typeof vnode === 'string' || typeof vnode === 'number') {
    return document.createTextNode(String(vnode));
  }

  const el = document.createElement(vnode.tag);

  for (const [key, value] of Object.entries(vnode.props)) {
    if (key.startsWith('on')) {
      el.addEventListener(key.slice(2).toLowerCase(), value);
    } else {
      el.setAttribute(key, value);
    }
  }

  vnode.children.forEach(child => {
    el.appendChild(render(child));
  });

  return el;
}

function mount(vnode, container) {
  container.appendChild(render(vnode));
}
解説
  • h() は React の createElement（JSXのコンパイル先）に相当します。
  • render() は再帰的にvnodeツリーを走査し、DOM要素を構築しています。
  • イベントハンドラは on プレフィクスで判定し、addEventListener で登録しています。

SPAルーターを設計

verified
学習ポイント

History API を使ったSPAルーティングの仕組みを理解することで、React RouterやNext.jsのルーティングの基礎を学びます。

assignment課題

以下のAPIで使えるSPAルーターを実装してください。

  • history.pushState でURLを変更（ページリロードなし）
  • :param 形式の動的ルートパラメータに対応
  • ブラウザの戻る/進むボタン（popstate イベント）に対応
lightbulbヒント

ルートパターン /user/:id を正規表現 /^\/user\/([^/]+)$/ に変換し、マッチ時にキャプチャグループからパラメータを抽出します。

check_circle解答・解説 
function createRouter(containerId = 'app') {
  const routes = [];

  function pathToRegex(path) {
    const paramNames = [];
    const pattern = path.replace(/:(\w+)/g, (_, name) => {
      paramNames.push(name);
      return '([^/]+)';
    });
    return { regex: new RegExp(`^${pattern}$`), paramNames };
  }

  function matchRoute(path) {
    for (const route of routes) {
      const match = path.match(route.regex);
      if (match) {
        const params = {};
        route.paramNames.forEach((name, i) => { params[name] = match[i + 1]; });
        return { handler: route.handler, params };
      }
    }
    return null;
  }

  function render(path) {
    const matched = matchRoute(path);
    const container = document.getElementById(containerId);
    if (matched && container) {
      container.innerHTML = matched.handler(matched.params);
    }
  }

  window.addEventListener('popstate', () => render(location.pathname));

  return {
    addRoute(path, handler) {
      const { regex, paramNames } = pathToRegex(path);
      routes.push({ regex, paramNames, handler });
    },
    navigate(path) {
      history.pushState(null, '', path);
      render(path);
    }
  };
}
解説
  • :param を正規表現のキャプチャグループに変換し、URLマッチングとパラメータ抽出を同時に行っています。
  • history.pushState でブラウザのURLを変更し、popstate イベントで戻る/進む操作を検知します。
  • React Router、Vue Router、Next.js のファイルベースルーティングの基本原理です。

Pub/Sub イベントバスを設計

verified
学習ポイント

Publish/Subscribeパターンはコンポーネント間通信の基盤設計です。ワイルドカードマッチ、once購読、名前空間を設計判断として扱います。

assignment課題

以下のAPIで使える高機能イベントバスを設計・実装してください。

const bus = createEventBus();

// 通常購読
bus.on('user:login', (data) => console.log('ログイン:', data.name));

// ワイルドカード購読（user: で始まるすべてのイベントを受信）
bus.on('user:*', (data, eventName) => {
  console.log(`[user系イベント] ${eventName}:`, data);
});

// once購読（1回だけ）
bus.once('app:init', () => console.log('初期化完了'));

bus.emit('user:login', { name: '田中' });
// ログイン: 田中
// [user系イベント] user:login: { name: '田中' }

bus.emit('user:logout', { name: '田中' });
// [user系イベント] user:logout: { name: '田中' }

bus.emit('app:init'); // 初期化完了
bus.emit('app:init'); // （2回目は発火しない）
terminal期待される出力
ログイン: 田中
[user系イベント] user:login: { name: '田中' }
[user系イベント] user:logout: { name: '田中' }
初期化完了
lightbulbヒント

リスナーを Map で管理し、emit 時にワイルドカードパターン（user:*）もマッチ判定します。once はラッパー関数を作り、呼び出し後に自身を off します。

check_circle解答・解説 
function createEventBus() {
  const listeners = new Map();

  function matchPattern(pattern, eventName) {
    if (pattern === eventName) return true;
    if (pattern.endsWith(':*')) {
      const prefix = pattern.slice(0, -1);
      return eventName.startsWith(prefix);
    }
    return false;
  }

  const bus = {
    on(event, fn) {
      if (!listeners.has(event)) listeners.set(event, []);
      listeners.get(event).push(fn);
      return () => bus.off(event, fn);
    },

    once(event, fn) {
      const wrapper = (...args) => {
        bus.off(event, wrapper);
        fn(...args);
      };
      return bus.on(event, wrapper);
    },

    off(event, fn) {
      const fns = listeners.get(event);
      if (!fns) return;
      const idx = fns.indexOf(fn);
      if (idx > -1) fns.splice(idx, 1);
    },

    emit(event, ...args) {
      for (const [pattern, fns] of listeners) {
        if (matchPattern(pattern, event)) {
          fns.forEach(fn => fn(...args, event));
        }
      }
    }
  };

  return bus;
}
解説
  • ワイルドカード :* はプレフィクス一致で判定し、名前空間ベースのイベント階層を実現しています。
  • once はラッパー関数を作り、初回呼び出し時に自身を登録解除する高階関数パターンです。
  • on がアンサブスクライブ関数を返すのは、メモリリーク防止の定番設計です。

function createGestureRecognizer(element) {
  const handlers = { dragstart: [], dragmove: [], dragend: [] };
  let dragging = false;
  let startX, startY, startTime;

  function getPos(e) {
    const touch = e.touches?.[0] || e.changedTouches?.[0];
    return touch ? { x: touch.clientX, y: touch.clientY }
                 : { x: e.clientX, y: e.clientY };
  }

  function emit(event, data) {
    handlers[event]?.forEach(fn => fn(data));
  }

  function onStart(e) {
    dragging = true;
    const pos = getPos(e);
    startX = pos.x;
    startY = pos.y;
    startTime = Date.now();
    emit('dragstart', { x: pos.x, y: pos.y });
  }

  function onMove(e) {
    if (!dragging) return;
    e.preventDefault();
    const pos = getPos(e);
    emit('dragmove', {
      x: pos.x, y: pos.y,
      dx: pos.x - startX, dy: pos.y - startY,
      startX, startY
    });
  }

  function onEnd(e) {
    if (!dragging) return;
    dragging = false;
    const pos = getPos(e);
    const dx = pos.x - startX;
    const dy = pos.y - startY;
    emit('dragend', {
      x: pos.x, y: pos.y, dx, dy,
      distance: Math.sqrt(dx * dx + dy * dy),
      duration: Date.now() - startTime
    });
  }

  element.addEventListener('mousedown', onStart);
  element.addEventListener('touchstart', onStart, { passive: false });
  document.addEventListener('mousemove', onMove);
  document.addEventListener('touchmove', onMove, { passive: false });
  document.addEventListener('mouseup', onEnd);
  document.addEventListener('touchend', onEnd);

  return {
    on(event, fn) { handlers[event]?.push(fn); },
    destroy() {
      element.removeEventListener('mousedown', onStart);
      element.removeEventListener('touchstart', onStart);
      document.removeEventListener('mousemove', onMove);
      document.removeEventListener('touchmove', onMove);
      document.removeEventListener('mouseup', onEnd);
      document.removeEventListener('touchend', onEnd);
    }
  };
}
解説
  • マウスとタッチを getPos で統一し、同じロジックで両方のデバイスに対応しています。
  • mousemove / mouseup は document に登録することで、要素外へのドラッグにも対応しています。
  • { passive: false } でタッチイベントの preventDefault を有効化しています。
  • @use-gesture（React）、Hammer.js の基本設計原理です。

function createTaskQueue({ concurrency = 1 } = {}) {
  const queue = [];
  const events = {};
  let running = 0;
  let paused = true;
  let drainResolve = null;

  function emit(event, ...args) {
    (events[event] || []).forEach(fn => fn(...args));
  }

  function runNext() {
    if (paused) return;
    while (running < concurrency && queue.length > 0) {
      const { task } = queue.shift();
      running++;
      task()
        .then(result => { emit('taskComplete', result); })
        .catch(error => { emit('taskError', error); })
        .finally(() => {
          running--;
          if (queue.length === 0 && running === 0) {
            emit('drain');
            drainResolve?.();
          }
          runNext();
        });
    }
  }

  return {
    add(task, { priority = 0 } = {}) {
      queue.push({ task, priority });
      queue.sort((a, b) => b.priority - a.priority);
      if (!paused) runNext();
    },
    start() {
      paused = false;
      runNext();
      if (queue.length === 0 && running === 0) return Promise.resolve();
      return new Promise(resolve => { drainResolve = resolve; });
    },
    pause() {
      paused = true;
    },
    on(event, fn) {
      (events[event] ||= []).push(fn);
    }
  };
}
解説
  • 優先度で降順ソートし、shift() で常に最高優先度のタスクを取り出しています。
  • finally ブロックで成功・失敗に関わらず次のタスクを起動し、全完了時に drain を通知しています。
  • start() が返す Promise で全タスク完了を await できる設計です。
  • BullMQ、p-queue などのジョブキューライブラリの基本アーキテクチャです。

キャッシュに { value, expiry, promise } を保存します。実行中の Promise も保存することで、同じキーの同時リクエストが同じ Promise を待つようにします（deduplication）。

function asyncMemoize(fn, { ttl = Infinity, maxSize = Infinity } = {}) {
  const cache = new Map();
  const pending = new Map();

  function evictOldest() {
    if (cache.size >= maxSize) {
      const oldest = cache.keys().next().value;
      cache.delete(oldest);
    }
  }

  return async function (...args) {
    const key = JSON.stringify(args);

    // キャッシュヒット判定
    if (cache.has(key)) {
      const entry = cache.get(key);
      if (Date.now() < entry.expiry) {
        return entry.value;
      }
      cache.delete(key);
    }

    // 同一キーの実行中 Promise を共有（重複排除）
    if (pending.has(key)) {
      return pending.get(key);
    }

    const promise = fn(...args)
      .then(value => {
        evictOldest();
        cache.set(key, { value, expiry: Date.now() + ttl });
        return value;
      })
      .finally(() => {
        pending.delete(key);
      });

    pending.set(key, promise);
    return promise;
  };
}
解説
  • pending マップで実行中の Promise を管理し、同じキーの同時リクエストを1つの Promise にまとめています（deduplication）。
  • Map は挿入順を保持するため、keys().next().value で最も古いエントリを取得して LRU 的な削除ができます。
  • TTL で期限切れのキャッシュを自動無効化し、maxSize でメモリ使用量を制限しています。
  • SWR、React Query、DataLoader の基本設計原理です。

function createCircuitBreaker(fn, options = {}) {
  const { failureThreshold = 5, resetTimeout = 30000 } = options;
  let state = 'closed';
  let failureCount = 0;
  let nextAttempt = 0;

  return async function (...args) {
    if (state === 'open') {
      if (Date.now() < nextAttempt) {
        throw new Error('Circuit breaker is OPEN');
      }
      state = 'half-open';
    }

    try {
      const result = await fn(...args);
      // 成功 → リセット
      failureCount = 0;
      state = 'closed';
      return result;
    } catch (error) {
      failureCount++;
      if (failureCount >= failureThreshold || state === 'half-open') {
        state = 'open';
        nextAttempt = Date.now() + resetTimeout;
      }
      throw error;
    }
  };
}
解説
  • 3つの状態（closed/open/half-open）をクロージャで管理し、失敗率に応じて遷移する有限ステートマシンです。

  • nextAttempt のタイムスタンプで HalfOpen 遷移のタイミングを制御しています。

function createAdvancedContainer() {
  const registrations = new Map();
  const singletons = new Map();
  const disposables = [];

  const container = {
    singleton(name, factory) {
      registrations.set(name, { factory, lifecycle: 'singleton' });
    },

    transient(name, factory) {
      registrations.set(name, { factory, lifecycle: 'transient' });
    },

    factory(name, factory) {
      registrations.set(name, { factory, lifecycle: 'factory' });
    },

    resolve(name) {
      const reg = registrations.get(name);
      if (!reg) throw new Error(`Service not found: ${name}`);

      if (reg.lifecycle === 'singleton') {
        if (!singletons.has(name)) {
          const instance = reg.factory(container);
          singletons.set(name, instance);
          if (instance?.dispose) disposables.push(instance);
        }
        return singletons.get(name);
      }

      const instance = reg.factory(container);
      if (instance?.dispose) disposables.push(instance);
      return instance;
    },

    create(name, ...args) {
      const reg = registrations.get(name);
      if (!reg) throw new Error(`Factory not found: ${name}`);
      return reg.factory(container, ...args);
    },

    async dispose() {
      for (const instance of disposables.reverse()) {
        await instance.dispose();
      }
      singletons.clear();
      disposables.length = 0;
    }
  };

  return container;
}
解説
  • ライフサイクル（singleton/transient/factory）を登録時に指定し、resolve の挙動を切り替えています。
  • disposables 配列で dispose メソッドを持つインスタンスを追跡し、逆順で破棄しています。
  • create メソッドで追加引数を渡せるファクトリパターンをサポートしています。
  • NestJS、tsyringe、InversifyJS のDIコンテナの設計原理です。

function createNotifier() {
  const channels = new Map();
  const rules = [];

  return {
    addChannel(name, strategy) {
      channels.set(name, strategy);
    },

    addRule(name, rule) {
      rules.push({ name, ...rule });
    },

    notify(event) {
      for (const rule of rules) {
        if (!rule.match(event)) continue;

        const message = rule.transform(event);
        for (const channelName of rule.channels) {
          const channel = channels.get(channelName);
          if (channel) channel.send(message);
        }
      }
    },

    removeRule(name) {
      const idx = rules.findIndex(r => r.name === name);
      if (idx > -1) rules.splice(idx, 1);
    },

    removeChannel(name) {
      channels.delete(name);
    }
  };
}
解説
  • Strategy パターン: 各チャネルが send メソッドを持ち、通知方法を交換可能にしています。
  • Observer パターン: ルールが match でイベントを購読し、条件に合致した場合にのみ処理を実行します。
  • transform でイベントをチャネル向けのメッセージに変換し、関心の分離を実現しています。
  • Sentry、PagerDuty、GitHub Actions の通知ルーティングと同じ設計原理です。
