穴埋め — @Componentと@Autowired

verified
実務ポイント

Spring BootではDI（依存性注入）でオブジェクトの生成と管理をフレームワークに任せます。@ComponentでクラスをBean登録し、@Autowiredで自動注入します。

assignment課題

以下のコードの /* ① */ 〜 /* ③ */ を埋めて、DIが正しく動作するようにしてください。

// GreetingService.java
/* ① */
public class GreetingService {
    public String greet(String name) {
        return "Hello, " + name + "!";
    }
}

// GreetingController.java
@RestController
public class GreetingController {

    /* ② */
    private /* ③ */ greetingService;

    @GetMapping("/greet")
    public String greet(@RequestParam String name) {
        return greetingService.greet(name);
    }
}
terminalGET /greet?name=Taro
Hello, Taro!
lightbulbヒント

① Beanとして登録するアノテーションは @Component または @Service です。② フィールド注入には @Autowired を使います。③ 注入する型名を書きます。

check_circle解答・解説 
// GreetingService.java
@Service
public class GreetingService {
    public String greet(String name) {
        return "Hello, " + name + "!";
    }
}

// GreetingController.java
@RestController
public class GreetingController {

    @Autowired
    private GreetingService greetingService;

    @GetMapping("/greet")
    public String greet(@RequestParam String name) {
        return greetingService.greet(name);
    }
}
解説
  • @Service は @Component の特殊化で、ビジネスロジック層であることを意味的に示します。
  • @Autowired はSpringが型に基づいて自動的にBeanを注入します。コンストラクタ注入が推奨ですが、まずフィールド注入で仕組みを理解しましょう。
  • @RestController は @Controller + @ResponseBody の合成アノテーションです。

写経+改造 — コンストラクタ注入への変換

verified
実務ポイント

フィールド注入（@Autowired）より、コンストラクタ注入が推奨されます。不変性が保証され、テスト時にモックを渡しやすくなります。

assignment課題

Step 1（写経）: 以下のフィールド注入コードを写経して動作確認してください。

@RestController
public class TaskController {

    @Autowired
    private TaskService taskService;

    @GetMapping("/tasks")
    public List<String> getTasks() {
        return taskService.findAll();
    }
}

Step 2（改造）: @Autowired フィールド注入を コンストラクタ注入 に書き換えてください。フィールドには final を付けてください。

lightbulbヒント

コンストラクタが1つだけの場合、@Autowired は省略可能です。private final TaskService taskService; として、コンストラクタで受け取ります。

check_circle解答・解説 
@RestController
public class TaskController {

    private final TaskService taskService;

    public TaskController(TaskService taskService) {
        this.taskService = taskService;
    }

    @GetMapping("/tasks")
    public List<String> getTasks() {
        return taskService.findAll();
    }
}
解説
  • コンストラクタが1つだけなら @Autowired は省略できます（Spring 4.3以降）。
  • final により、フィールドが不変（immutable）になり、NullPointerExceptionのリスクが減ります。
  • テスト時に new TaskController(mockService) のようにモックを直接渡せます。

T2 RESTコントローラ

穴埋め — GETエンドポイントのマッピング

verified
実務ポイント

REST APIでは@GetMapping, @PostMapping等のHTTPメソッド別アノテーションでエンドポイントを定義します。@PathVariableでURLパスの値、@RequestParamでクエリパラメータを受け取ります。

assignment課題

/* ① */ 〜 /* ③ */ を埋めて、ユーザー情報を返すRESTコントローラを完成させてください。

@RestController
/* ① */
public class UserController {

    // GET /api/users — 全ユーザー一覧
    /* ② */
    public List<User> getAllUsers() {
        return userService.findAll();
    }

    // GET /api/users/1 — IDで1件取得
    @GetMapping("/{id}")
    public User getUser(/* ③ */ Long id) {
        return userService.findById(id);
    }
}
lightbulbヒント

① クラスレベルの共通パスは @RequestMapping("/api/users") で指定します。② 一覧取得は @GetMapping です。③ URLの {id} 部分は @PathVariable で取得します。

check_circle解答・解説 
@RestController
@RequestMapping("/api/users")
public class UserController {

    @GetMapping
    public List<User> getAllUsers() {
        return userService.findAll();
    }

    @GetMapping("/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.findById(id);
    }
}
解説
  • @RequestMapping("/api/users") をクラスレベルに付けると、メソッドのパスのプレフィックスになります。
  • @GetMapping はパスなしで /api/users にマッチします。
  • @PathVariable はURL内の {id} プレースホルダに対応します。パラメータ名とプレースホルダ名が同じなら属性指定は不要です。

写経+改造 — POSTエンドポイントの追加

verified
実務ポイント

@PostMappingでリソース作成、@RequestBodyでJSONリクエストボディを受け取ります。ResponseEntityで適切なHTTPステータスコードを返すのが実務の基本です。

assignment課題

Step 1（写経）: 以下のGETエンドポイントを写経して動作確認してください。

@RestController
@RequestMapping("/api/items")
public class ItemController {

    private final ItemService itemService;

    public ItemController(ItemService itemService) {
        this.itemService = itemService;
    }

    @GetMapping
    public List<Item> getAll() {
        return itemService.findAll();
    }
}

Step 2（改造）: @PostMapping で新しいItemを作成するエンドポイントを追加してください。リクエストボディで Item を受け取り、ResponseEntity でステータス201を返してください。

lightbulbヒント

@PostMapping と @RequestBody を使います。ResponseEntity.status(HttpStatus.CREATED).body(saved) でステータス201を返せます。

check_circle解答・解説 
@PostMapping
public ResponseEntity<Item> create(@RequestBody Item item) {
    Item saved = itemService.save(item);
    return ResponseEntity.status(HttpStatus.CREATED).body(saved);
}
解説
  • @RequestBody はHTTPリクエストボディのJSONをJavaオブジェクトに変換（デシリアライズ）します。
  • ResponseEntity を使うと、ステータスコード・ヘッダー・ボディを細かく制御できます。
  • リソース作成時は 201 Created を返すのがRESTful APIの慣例です。

穴埋め — @Mapperインタフェースの基本

verified
実務ポイント

MyBatisでは@Mapperインタフェースにアノテーションで SQL を記述します。@Select, @Insert, @Update, @Delete でCRUD操作を定義します。

assignment課題

/* ① */ 〜 /* ③ */ を埋めて、TodoマッパーのCRUD基本操作を完成させてください。

/* ① */
public interface TodoMapper {

    /* ② */("SELECT * FROM todos")
    List<Todo> findAll();

    @Select("SELECT * FROM todos WHERE id = /* ③ */")
    Todo findById(Long id);

    @Insert("INSERT INTO todos (title, completed) VALUES (#{title}, #{completed})")
    void insert(Todo todo);
}
lightbulbヒント

① MyBatisのマッパーインタフェースには @Mapper を付けます。② 全件取得のSQLには @Select を使います。③ MyBatisのパラメータ参照は #{パラメータ名} です。

check_circle解答・解説 
@Mapper
public interface TodoMapper {

    @Select("SELECT * FROM todos")
    List<Todo> findAll();

    @Select("SELECT * FROM todos WHERE id = #{id}")
    Todo findById(Long id);

    @Insert("INSERT INTO todos (title, completed) VALUES (#{title}, #{completed})")
    void insert(Todo todo);
}
解説
  • @Mapper をインタフェースに付けると、MyBatisが実装クラスを自動生成してSpring Beanとして登録します。
  • #{id} はプリペアドステートメントのパラメータに変換され、SQLインジェクションを防ぎます。
  • ${} はリテラル置換で、SQLインジェクションの危険があるため原則使いません。

写経+改造 — UPDATEとDELETEの追加

verified
実務ポイント

@Updateと@Deleteで更新・削除操作を定義します。戻り値をintにすると、影響を受けた行数を取得できます。

assignment課題

Step 1（写経）: 以下のSELECTとINSERTのみのマッパーを写経してください。

@Mapper
public interface ProductMapper {

    @Select("SELECT * FROM products WHERE id = #{id}")
    Product findById(Long id);

    @Insert("INSERT INTO products (name, price) VALUES (#{name}, #{price})")
    void insert(Product product);
}

Step 2（改造）: 以下の2メソッドを追加してください。

  • update(Product product) — name と price を更新（戻り値: int）
  • deleteById(Long id) — IDで削除（戻り値: int）
lightbulbヒント

@Update("UPDATE products SET name = #{name}, price = #{price} WHERE id = #{id}") のように書きます。@Delete も同様のパターンです。

check_circle解答・解説 
@Update("UPDATE products SET name = #{name}, price = #{price} WHERE id = #{id}")
int update(Product product);

@Delete("DELETE FROM products WHERE id = #{id}")
int deleteById(Long id);
解説
  • 戻り値を int にすると影響行数が返ります。0なら対象レコードが存在しなかったことがわかります。
  • #{name}, #{price}, #{id} は引数のProductオブジェクトのフィールドに自動マッピングされます。
  • WHERE句を忘れると全行が更新・削除されるため、必ずIDで絞り込むのが鉄則です。

穴埋め — Bean Validationアノテーション

verified
実務ポイント

Bean Validation（jakarta.validation）のアノテーションでフィールドの制約を宣言的に定義します。@Valid を付けると自動でバリデーションが実行されます。

assignment課題

/* ① */ 〜 /* ③ */ を埋めて、ユーザー登録フォームのバリデーションを完成させてください。

public class UserForm {

    /* ① */(message = "名前は必須です")
    private String name;

    /* ② */(message = "メールアドレスの形式が正しくありません")
    private String email;

    /* ③ */(min = 8, message = "パスワードは8文字以上で入力してください")
    private String password;

    // getter/setter 省略
}
lightbulbヒント

① 必須チェックは @NotBlank（空白不可）です。② メール形式は @Email です。③ 文字数の最小値は @Size で指定します。

check_circle解答・解説 
public class UserForm {

    @NotBlank(message = "名前は必須です")
    private String name;

    @Email(message = "メールアドレスの形式が正しくありません")
    private String email;

    @Size(min = 8, message = "パスワードは8文字以上で入力してください")
    private String password;

    // getter/setter 省略
}
解説
  • @NotBlank は null・空文字・空白のみを拒否します。@NotNull は null のみ、@NotEmpty は null と空文字を拒否します。
  • @Email はRFC準拠のメールアドレス形式をチェックします。
  • @Size(min=8) は文字列の長さの最小値を制約します。max も指定可能です。

写経+改造 — @Validの適用

verified
実務ポイント

コントローラの引数に@Validを付けるとBean Validationが自動実行されます。バリデーションエラー時はMethodArgumentNotValidExceptionがスローされます。

assignment課題

Step 1（写経）: 以下のバリデーションなしのコントローラを写経してください。

@RestController
@RequestMapping("/api/users")
public class UserController {

    @PostMapping
    public ResponseEntity<User> create(@RequestBody UserForm form) {
        User user = userService.create(form);
        return ResponseEntity.status(HttpStatus.CREATED).body(user);
    }
}

Step 2（改造）: @Valid を追加してバリデーションを有効にしてください。また、@Min(1) で年齢フィールド（age）が1以上であることを UserForm に追加してください。

lightbulbヒント

@RequestBody の前に @Valid を付けます。@Min(value = 1, message = "...") で数値の最小値を制約します。

check_circle解答・解説 
// UserController.java
@PostMapping
public ResponseEntity<User> create(@Valid @RequestBody UserForm form) {
    User user = userService.create(form);
    return ResponseEntity.status(HttpStatus.CREATED).body(user);
}

// UserForm.java に追加
@Min(value = 1, message = "年齢は1以上で入力してください")
private Integer age;
解説
  • @Valid を付けないとバリデーションアノテーションは無視されます。忘れやすいポイントです。
  • @Min / @Max は数値型（Integer, Long等）に使います。文字列には @Size を使います。
  • バリデーションエラー時は400 Bad Requestが自動返却されます。エラー詳細をカスタマイズするには例外ハンドラを実装します。

T5 例外ハンドリング

穴埋め — @ExceptionHandlerの基本

verified
実務ポイント

@ExceptionHandlerでコントローラ内の例外をキャッチし、適切なエラーレスポンスを返します。@ControllerAdviceを使うとアプリ全体で共通の例外処理が可能です。

assignment課題

/* ① */ 〜 /* ③ */ を埋めて、例外ハンドラを完成させてください。

/* ① */
public class GlobalExceptionHandler {

    /* ② */(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            /* ③ */ ex) {
        ErrorResponse error = new ErrorResponse(
            HttpStatus.NOT_FOUND.value(),
            ex.getMessage()
        );
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }
}
lightbulbヒント

① アプリ全体の例外処理には @RestControllerAdvice を使います。② 特定の例外をキャッチするアノテーションは @ExceptionHandler です。③ キャッチする例外型を引数に指定します。

check_circle解答・解説 
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            ResourceNotFoundException ex) {
        ErrorResponse error = new ErrorResponse(
            HttpStatus.NOT_FOUND.value(),
            ex.getMessage()
        );
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }
}
解説
  • @RestControllerAdvice は @ControllerAdvice + @ResponseBody の合成アノテーションです。
  • @ExceptionHandler の引数に例外クラスを指定すると、その例外がスローされた時にメソッドが呼ばれます。
  • ErrorResponseは独自のDTO（Data Transfer Object）で、statusとmessageフィールドを持たせるのが一般的です。

写経+改造 — カスタム例外の作成

verified
実務ポイント

業務固有の例外をカスタム例外として定義し、@ResponseStatusで自動的にHTTPステータスをマッピングできます。

assignment課題

Step 1（写経）: 以下のカスタム例外クラスを写経してください。

@ResponseStatus(HttpStatus.NOT_FOUND)
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String message) {
        super(message);
    }
}

Step 2（改造）: 同様に DuplicateResourceException を作成してください。HTTPステータスは 409 Conflict にしてください。

lightbulbヒント

@ResponseStatus(HttpStatus.CONFLICT) でステータス409を指定します。クラス名以外の構造はResourceNotFoundExceptionと同じです。

check_circle解答・解説 
@ResponseStatus(HttpStatus.CONFLICT)
public class DuplicateResourceException extends RuntimeException {
    public DuplicateResourceException(String message) {
        super(message);
    }
}
解説
  • @ResponseStatus を付けた例外がスローされると、指定したHTTPステータスが自動で返ります。
  • 409 Conflictはリソースの重複（メールアドレスの重複登録等）に使います。
  • RuntimeException を継承すると非検査例外になり、throws宣言が不要になります。

穴埋め — SecurityFilterChainの基本構成

verified
実務ポイント

Spring Security 6ではSecurityFilterChainをBean定義してセキュリティルールを構成します。HttpSecurityのメソッドチェーンでURLごとのアクセス制御を設定します。

assignment課題

/* ① */ 〜 /* ③ */ を埋めて、セキュリティ設定を完成させてください。

@Configuration
@EnableWebSecurity
public class SecurityConfig {

    /* ① */
    public SecurityFilterChain filterChain(/* ② */ http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**")./* ③ */()
                .anyRequest().authenticated()
            )
            .httpBasic(Customizer.withDefaults());
        return http.build();
    }
}
lightbulbヒント

① Bean登録には @Bean を使います。② セキュリティ設定のビルダーは HttpSecurity 型です。③ 認証なしで許可するメソッドは permitAll です。

check_circle解答・解説 
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .httpBasic(Customizer.withDefaults());
        return http.build();
    }
}
解説
  • @Bean でSecurityFilterChainを定義するのがSpring Security 6の標準パターンです。
  • permitAll() は認証なしでアクセスを許可します。公開APIやログインページに使います。
  • anyRequest().authenticated() で、それ以外の全URLに認証を要求します。
  • ルールは上から順に評価されるため、許可ルールを先に書く必要があります。

写経+改造 — CORS設定の追加

verified
実務ポイント

フロントエンドとバックエンドが別オリジンの場合、CORS（Cross-Origin Resource Sharing）設定が必要です。Spring BootではWebMvcConfigurerでグローバルに設定できます。

assignment課題

Step 1（写経）: 以下のCORS設定を写経してください。

@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("http://localhost:3000")
            .allowedMethods("GET", "POST");
    }
}

Step 2（改造）: PUT, DELETE メソッドも許可し、allowedHeaders("*") と allowCredentials(true) を追加してください。

lightbulbヒント

.allowedMethods("GET", "POST", "PUT", "DELETE") でメソッドを追加します。メソッドチェーンで .allowedHeaders("*").allowCredentials(true) を続けます。

check_circle解答・解説 
@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("http://localhost:3000")
            .allowedMethods("GET", "POST", "PUT", "DELETE")
            .allowedHeaders("*")
            .allowCredentials(true);
    }
}
解説
  • allowedHeaders("*") は全HTTPヘッダーを許可します。本番では必要なヘッダーのみに絞るのが安全です。
  • allowCredentials(true) はCookieやAuthorizationヘッダーの送信を許可します。
  • CORSはブラウザのセキュリティ機構です。Postman等のツールからのリクエストには影響しません。

T7 テスト・MockMvc

穴埋め — @WebMvcTestの基本構成

verified
実務ポイント

@WebMvcTestはコントローラ層のみをテストするスライステストです。MockMvcでHTTPリクエストをシミュレートし、レスポンスを検証します。

assignment課題

/* ① */ 〜 /* ③ */ を埋めて、コントローラのテストを完成させてください。

/* ① */(UserController.class)
class UserControllerTest {

    @Autowired
    private /* ② */ mockMvc;

    /* ③ */
    private UserService userService;

    @Test
    void testGetAllUsers() throws Exception {
        when(userService.findAll()).thenReturn(List.of(
            new User(1L, "Taro")
        ));

        mockMvc.perform(get("/api/users"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$[0].name").value("Taro"));
    }
}
lightbulbヒント

① コントローラのスライステストは @WebMvcTest です。② HTTPリクエストをシミュレートするのは MockMvc 型です。③ Serviceのモックには @MockBean を使います。

check_circle解答・解説 
@WebMvcTest(UserController.class)
class UserControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private UserService userService;

    @Test
    void testGetAllUsers() throws Exception {
        when(userService.findAll()).thenReturn(List.of(
            new User(1L, "Taro")
        ));

        mockMvc.perform(get("/api/users"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$[0].name").value("Taro"));
    }
}
解説
  • @WebMvcTest はWebレイヤーのみをロードし、高速にテストできます。Service等はモック化が必要です。
  • @MockBean はSpringコンテキスト内のBeanをMockitoモックに置き換えます。
  • jsonPath でJSON応答の特定フィールドを検証できます。$[0] は配列の最初の要素です。

写経+改造 — 3層構造の基本骨格

verified
実務ポイント

Spring Bootアプリは Controller → Service → Mapper（Repository）の3層構造が基本です。各層の責務を分離することで保守性とテスト性が向上します。

assignment課題

Step 1（写経）: 以下の3ファイルを写経し、全体の構造を理解してください。

// TodoMapper.java
@Mapper
public interface TodoMapper {
    @Select("SELECT * FROM todos")
    List<Todo> findAll();
}

// TodoService.java
@Service
public class TodoService {
    private final TodoMapper todoMapper;
    public TodoService(TodoMapper todoMapper) {
        this.todoMapper = todoMapper;
    }
    public List<Todo> findAll() {
        return todoMapper.findAll();
    }
}

// TodoController.java
@RestController
@RequestMapping("/api/todos")
public class TodoController {
    private final TodoService todoService;
    public TodoController(TodoService todoService) {
        this.todoService = todoService;
    }
    @GetMapping
    public List<Todo> getAll() {
        return todoService.findAll();
    }
}

Step 2（改造）: findById(Long id) メソッドを3層すべてに追加してください。エンドポイントは GET /api/todos/{id} です。

lightbulbヒント

Mapper: @Select("SELECT * FROM todos WHERE id = #{id}")。Service: MapperのfindByIdを呼ぶ。Controller: @GetMapping("/{id}") + @PathVariable。

check_circle解答・解説 
// TodoMapper.java に追加
@Select("SELECT * FROM todos WHERE id = #{id}")
Todo findById(Long id);

// TodoService.java に追加
public Todo findById(Long id) {
    return todoMapper.findById(id);
}

// TodoController.java に追加
@GetMapping("/{id}")
public Todo getById(@PathVariable Long id) {
    return todoService.findById(id);
}
解説
  • 3層構造では、Controller→Service→Mapperの順にメソッドを追加していきます。
  • 各層が単一責務を持つため、テスト時にServiceだけモック化する等の柔軟なテストが可能です。
  • 実務ではfindByIdでnullが返った場合にResourceNotFoundExceptionをスローするのが一般的です。
