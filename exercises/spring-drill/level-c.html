インタフェースを用いたDI設計

verified
実務ポイント

インタフェースに依存させることで、実装の差し替えが容易になります。テスト時にモック実装を注入する際にも有効です。

assignment課題

以下の仕様に従い、インタフェースと実装クラスを作成してください。

  • NotificationService インタフェース: void send(String to, String message) メソッドを持つ
  • EmailNotificationService クラス: NotificationServiceを実装し、"[Email] To: {to} - {message}" をコンソール出力する
  • NotificationController: POST /api/notifications で NotificationServiceを呼び出す
lightbulbヒント

実装クラスに @Service を付けると、インタフェース型でDI注入が可能です。コントローラは NotificationService 型で受け取ります。

check_circle解答・解説 
// NotificationService.java
public interface NotificationService {
    void send(String to, String message);
}

// EmailNotificationService.java
@Service
public class EmailNotificationService implements NotificationService {
    @Override
    public void send(String to, String message) {
        System.out.println("[Email] To: " + to + " - " + message);
    }
}

// NotificationController.java
@RestController
@RequestMapping("/api/notifications")
public class NotificationController {
    private final NotificationService notificationService;

    public NotificationController(NotificationService notificationService) {
        this.notificationService = notificationService;
    }

    @PostMapping
    public ResponseEntity<String> send(@RequestBody NotificationRequest req) {
        notificationService.send(req.getTo(), req.getMessage());
        return ResponseEntity.ok("送信完了");
    }
}
解説
  • コントローラは NotificationService インタフェースに依存し、具象クラスを知りません。
  • 将来、Slack通知やSMS通知に切り替える場合、新しい実装クラスを @Service で登録するだけで済みます。
  • 複数の実装がある場合は @Primary や @Qualifier で使う実装を指定します。

@ConfigurationPropertiesで設定を外部化

verified
実務ポイント

@ConfigurationPropertiesでapplication.propertiesの値を型安全なJavaオブジェクトにバインドします。環境ごとの設定切り替えが容易になります。

assignment課題

以下の仕様に従い、アプリ設定を外部化してください。

  • AppProperties クラス: prefix app で name(String), version(String), maxRetry(int) をバインド
  • GET /api/info で上記設定をJSON返却するコントローラを作成
  • application.propertiesに app.name=MyApp, app.version=1.0.0, app.max-retry=3 を定義
lightbulbヒント

@ConfigurationProperties(prefix = "app") を付け、メインクラスに @EnableConfigurationProperties(AppProperties.class) を追加します。ケバブケース max-retry はキャメルケース maxRetry に自動マッピングされます。

check_circle解答・解説 
// AppProperties.java
@ConfigurationProperties(prefix = "app")
public class AppProperties {
    private String name;
    private String version;
    private int maxRetry;
    // getter/setter 省略
}

// InfoController.java
@RestController
public class InfoController {
    private final AppProperties appProperties;

    public InfoController(AppProperties appProperties) {
        this.appProperties = appProperties;
    }

    @GetMapping("/api/info")
    public AppProperties getInfo() {
        return appProperties;
    }
}

// application.properties
app.name=MyApp
app.version=1.0.0
app.max-retry=3
解説
  • @ConfigurationProperties はSpring Bootのリラックスバインディングにより、ケバブケース/スネークケース/キャメルケースを自動変換します。
  • @Value よりも型安全で、IDE補完も効くため実務では @ConfigurationProperties が推奨です。
  • プロファイル（application-dev.properties等）を使えば環境別に設定値を切り替えられます。

CRUD REST APIの設計と実装

verified
実務ポイント

RESTful APIではHTTPメソッドとURLでリソース操作を表現します。GET(取得), POST(作成), PUT(更新), DELETE(削除) の4操作をCRUDと呼びます。

assignment課題

以下の仕様に従い、書籍管理APIを作成してください。

  • GET /api/books — 全書籍一覧を返す
  • GET /api/books/{id} — IDで1件取得（なければ404）
  • POST /api/books — 新規作成（201 Created）
  • PUT /api/books/{id} — 更新（なければ404）
  • DELETE /api/books/{id} — 削除（204 No Content）

Book: id(Long), title(String), author(String), price(int)

lightbulbヒント

各メソッドに対応するアノテーション: @GetMapping, @PostMapping, @PutMapping, @DeleteMapping。削除は ResponseEntity.noContent().build() で204を返します。

check_circle解答・解説 
@RestController
@RequestMapping("/api/books")
public class BookController {
    private final BookService bookService;

    public BookController(BookService bookService) {
        this.bookService = bookService;
    }

    @GetMapping
    public List<Book> getAll() {
        return bookService.findAll();
    }

    @GetMapping("/{id}")
    public Book getById(@PathVariable Long id) {
        return bookService.findById(id);
    }

    @PostMapping
    public ResponseEntity<Book> create(@RequestBody Book book) {
        Book saved = bookService.save(book);
        return ResponseEntity.status(HttpStatus.CREATED).body(saved);
    }

    @PutMapping("/{id}")
    public Book update(@PathVariable Long id, @RequestBody Book book) {
        return bookService.update(id, book);
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        bookService.delete(id);
        return ResponseEntity.noContent().build();
    }
}
解説
  • REST API設計ではURLにリソース名（複数形: books）を使い、操作はHTTPメソッドで表現します。
  • ResponseEntity.noContent().build() は本文なしの204レスポンスを返します。
  • findByIdでリソースが見つからない場合、ServiceでResourceNotFoundExceptionをスローし、ExceptionHandlerで404を返すのが定石です。

ページネーション付き一覧APIの実装

verified
実務ポイント

大量データの一覧APIではページネーション（分割取得）が必須です。@RequestParamでpage/sizeを受け取り、LIMIT/OFFSETで制御します。

assignment課題

以下の仕様に従い、ページネーション付き商品一覧APIを作成してください。

  • GET /api/products?page=0&size=10 — ページ指定で取得
  • レスポンス: { "content": [...], "page": 0, "size": 10, "totalCount": 100 }
  • MyBatisで LIMIT #{size} OFFSET #{offset} を使うこと
lightbulbヒント

offset は page * size で計算します。レスポンス用DTOクラス PageResponse<T> を作ると再利用性が高まります。Mapperには findWithPaging(int size, int offset) と count() を定義します。

check_circle解答・解説 
// ProductMapper.java
@Mapper
public interface ProductMapper {
    @Select("SELECT * FROM products LIMIT #{size} OFFSET #{offset}")
    List<Product> findWithPaging(@Param("size") int size,
                                  @Param("offset") int offset);

    @Select("SELECT COUNT(*) FROM products")
    long count();
}

// ProductService.java
@Service
public class ProductService {
    private final ProductMapper productMapper;
    public ProductService(ProductMapper productMapper) {
        this.productMapper = productMapper;
    }
    public PageResponse<Product> findAll(int page, int size) {
        int offset = page * size;
        List<Product> content = productMapper.findWithPaging(size, offset);
        long totalCount = productMapper.count();
        return new PageResponse<>(content, page, size, totalCount);
    }
}

// ProductController.java
@RestController
@RequestMapping("/api/products")
public class ProductController {
    private final ProductService productService;
    public ProductController(ProductService productService) {
        this.productService = productService;
    }
    @GetMapping
    public PageResponse<Product> getAll(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "10") int size) {
        return productService.findAll(page, size);
    }
}
解説
  • @RequestParam(defaultValue = "0") でデフォルト値を指定でき、パラメータ省略時もエラーになりません。
  • MyBatisで複数パラメータを渡す場合は @Param で名前を明示します。
  • totalCountを別クエリで取得するのは基本パターンです。大規模データではCOUNTの最適化が課題になります。

検索条件付きSELECTの実装

verified
実務ポイント

検索APIでは複数の条件を動的に組み合わせる必要があります。MyBatisのXMLマッパーで<where>と<if>を使うと、条件の有無に応じたSQLを動的に生成できます。

assignment課題

以下の仕様に従い、社員検索APIを作成してください。

  • GET /api/employees?name=田中&department=営業
  • name, departmentは任意パラメータ。指定されたもののみWHERE条件に追加
  • XMLマッパーで動的SQLを記述すること
lightbulbヒント

XMLマッパーで <where><if test="name != null">AND name LIKE CONCAT('%', #{name}, '%')</if></where> のように記述します。<where>は先頭のANDを自動除去します。

check_circle解答・解説 
<!-- EmployeeMapper.xml -->
<mapper namespace="com.example.mapper.EmployeeMapper">
  <select id="search" resultType="Employee">
    SELECT * FROM employees
    <where>
      <if test="name != null and name != ''">
        AND name LIKE CONCAT('%', #{name}, '%')
      </if>
      <if test="department != null and department != ''">
        AND department = #{department}
      </if>
    </where>
  </select>
</mapper>

// EmployeeMapper.java
@Mapper
public interface EmployeeMapper {
    List<Employee> search(@Param("name") String name,
                           @Param("department") String department);
}

// EmployeeController.java
@GetMapping
public List<Employee> search(
        @RequestParam(required = false) String name,
        @RequestParam(required = false) String department) {
    return employeeService.search(name, department);
}
解説
  • <where> タグは内部に条件がある場合のみWHEREを付加し、先頭のAND/ORを自動除去します。
  • <if test="..."> でnullチェックと空文字チェックの両方を行うのが安全です。
  • LIKE検索は CONCAT で前後に%を付けるのがPostgreSQL向けの書き方です。

JOIN結果のネストマッピング

verified
実務ポイント

テーブル結合結果をネストしたJavaオブジェクトにマッピングするには、MyBatisのresultMapで<association>や<collection>を使います。

assignment課題

以下の仕様に従い、注文情報と注文明細をJOINで取得するマッパーを作成してください。

  • Order: id, customerName, orderDate
  • OrderItem: id, orderId, productName, quantity, price
  • Orderオブジェクトに List<OrderItem> items フィールドを含める
  • findById(Long id) で注文と明細を1回のクエリで取得
lightbulbヒント

XMLマッパーで <resultMap> を定義し、<collection property="items" ...> でネストマッピングします。JOINクエリでカラム名が衝突する場合はエイリアスを使います。

check_circle解答・解説 
<!-- OrderMapper.xml -->
<resultMap id="orderWithItems" type="Order">
  <id property="id" column="order_id"/>
  <result property="customerName" column="customer_name"/>
  <result property="orderDate" column="order_date"/>
  <collection property="items" ofType="OrderItem">
    <id property="id" column="item_id"/>
    <result property="productName" column="product_name"/>
    <result property="quantity" column="quantity"/>
    <result property="price" column="price"/>
  </collection>
</resultMap>

<select id="findById" resultMap="orderWithItems">
  SELECT o.id AS order_id, o.customer_name, o.order_date,
         oi.id AS item_id, oi.product_name, oi.quantity, oi.price
  FROM orders o
  LEFT JOIN order_items oi ON o.id = oi.order_id
  WHERE o.id = #{id}
</select>
解説
  • <collection> は1対多のリレーションをマッピングします。ofType でコレクション要素の型を指定します。
  • LEFT JOINにより、明細がない注文も取得できます。INNER JOINだと明細なしの注文が消えます。
  • カラム名のエイリアス（AS order_id等）で衝突を回避するのはJOINマッピングの必須テクニックです。

T4 バリデーション

商品登録フォームのバリデーション実装

verified
実務ポイント

複数のバリデーションアノテーションを組み合わせて、ビジネスルールに沿った入力チェックを実装します。エラーメッセージはユーザーにわかりやすく日本語で記述します。

assignment課題

以下の仕様に従い、商品登録のバリデーション付きAPIを作成してください。

  • name: 必須、1〜100文字
  • price: 必須、1以上（0円は不可）
  • category: 必須
  • description: 任意、最大500文字
  • バリデーションエラー時は400で { "errors": { "name": "メッセージ", ... } } を返す
lightbulbヒント

@NotBlank + @Size(max=100) を組み合わせます。エラーレスポンスのカスタマイズには @ExceptionHandler(MethodArgumentNotValidException.class) を使います。BindingResult からフィールドエラーを取得します。

check_circle解答・解説 
// ProductForm.java
public class ProductForm {
    @NotBlank(message = "商品名は必須です")
    @Size(max = 100, message = "商品名は100文字以内で入力してください")
    private String name;

    @NotNull(message = "価格は必須です")
    @Min(value = 1, message = "価格は1円以上で入力してください")
    private Integer price;

    @NotBlank(message = "カテゴリは必須です")
    private String category;

    @Size(max = 500, message = "説明は500文字以内で入力してください")
    private String description;
    // getter/setter 省略
}

// GlobalExceptionHandler.java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String, Object>> handleValidation(
            MethodArgumentNotValidException ex) {
        Map<String, String> errors = new HashMap<>();
        ex.getBindingResult().getFieldErrors().forEach(e ->
            errors.put(e.getField(), e.getDefaultMessage())
        );
        return ResponseEntity.badRequest().body(Map.of("errors", errors));
    }
}
解説
  • 数値の必須チェックには @NotNull を使います（@NotBlank はString型専用）。
  • MethodArgumentNotValidException は @Valid バリデーション失敗時にスローされます。
  • getFieldErrors() でフィールド単位のエラー情報を取得でき、フロントエンドでフィールドごとにエラー表示が可能です。

グループバリデーションの実装

verified
実務ポイント

新規作成と更新でバリデーションルールが異なる場合、グループバリデーションを使います。Create用/Update用のマーカーインタフェースでグループを切り替えます。

assignment課題

以下の仕様に従い、グループバリデーションを実装してください。

  • 新規作成時: nameとemailが必須
  • 更新時: idが必須（nameとemailは任意）
  • @Validated(Create.class) と @Validated(Update.class) で切り替え
lightbulbヒント

マーカーインタフェース interface Create {}, interface Update {} を定義し、@NotBlank(groups = Create.class) のように groups 属性で指定します。コントローラでは @Valid の代わりに @Validated(Create.class) を使います。

check_circle解答・解説 
// グループマーカー
public interface Create {}
public interface Update {}

// UserForm.java
public class UserForm {
    @NotNull(groups = Update.class, message = "IDは必須です")
    private Long id;

    @NotBlank(groups = Create.class, message = "名前は必須です")
    private String name;

    @NotBlank(groups = Create.class, message = "メールは必須です")
    @Email(message = "メール形式が正しくありません")
    private String email;
}

// UserController.java
@PostMapping
public ResponseEntity<User> create(
        @Validated(Create.class) @RequestBody UserForm form) {
    return ResponseEntity.status(HttpStatus.CREATED)
        .body(userService.create(form));
}

@PutMapping("/{id}")
public User update(@PathVariable Long id,
        @Validated(Update.class) @RequestBody UserForm form) {
    return userService.update(id, form);
}
解説
  • @Validated はSpring独自のアノテーションで、groups 指定に対応しています（@Valid はgroups非対応）。
  • グループ未指定の @Email はデフォルトグループに属し、どのグループでも適用されます。
  • 実務ではCreate/Update用に別々のDTOクラスを作る方がシンプルな場合もあります。要件に応じて使い分けましょう。

T5 例外ハンドリング

統一エラーレスポンス設計

verified
実務ポイント

APIのエラーレスポンスは統一フォーマットにします。status, error, message, timestampなどを含むErrorResponseクラスを定義し、全例外で共通利用します。

assignment課題

以下の仕様に従い、統一エラーレスポンスとグローバル例外ハンドラを作成してください。

  • ErrorResponse: status(int), error(String), message(String), timestamp(LocalDateTime)
  • ResourceNotFoundException → 404
  • MethodArgumentNotValidException → 400（バリデーションエラー詳細を含む）
  • その他のException → 500
lightbulbヒント

@RestControllerAdvice に複数の @ExceptionHandler メソッドを定義します。LocalDateTime.now() でタイムスタンプを生成します。

check_circle解答・解説 
// ErrorResponse.java
public class ErrorResponse {
    private int status;
    private String error;
    private String message;
    private LocalDateTime timestamp;

    public ErrorResponse(int status, String error, String message) {
        this.status = status;
        this.error = error;
        this.message = message;
        this.timestamp = LocalDateTime.now();
    }
    // getter 省略
}

// GlobalExceptionHandler.java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            ResourceNotFoundException ex) {
        ErrorResponse error = new ErrorResponse(
            404, "Not Found", ex.getMessage());
        return ResponseEntity.status(404).body(error);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
            MethodArgumentNotValidException ex) {
        String message = ex.getBindingResult().getFieldErrors()
            .stream()
            .map(e -> e.getField() + ": " + e.getDefaultMessage())
            .collect(Collectors.joining(", "));
        ErrorResponse error = new ErrorResponse(
            400, "Bad Request", message);
        return ResponseEntity.badRequest().body(error);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneral(Exception ex) {
        ErrorResponse error = new ErrorResponse(
            500, "Internal Server Error", "予期せぬエラーが発生しました");
        return ResponseEntity.status(500).body(error);
    }
}
解説
  • エラーレスポンスのフォーマットを統一することで、フロントエンドのエラーハンドリングが簡潔になります。
  • 500エラーではユーザーに内部情報を漏らさないよう、固定メッセージを返すのがセキュリティ上の鉄則です。
  • ExceptionHandlerの優先度は、より具体的な例外クラスが先にマッチします。

T5 例外ハンドリング

ビジネスロジック例外の設計と実装

verified
実務ポイント

業務ルール違反はカスタム例外で表現します。「在庫不足」「上限超過」などの業務エラーに対応する例外クラスを作成し、適切なHTTPステータスにマッピングします。

assignment課題

以下の仕様に従い、在庫管理の例外処理を実装してください。

  • InsufficientStockException: 在庫不足（HTTPステータス 422 Unprocessable Entity）
  • Serviceで在庫チェックを行い、不足時に例外をスロー
  • ExceptionHandlerで422レスポンスを返す
lightbulbヒント

カスタム例外に商品名と不足数をフィールドとして持たせると、エラーメッセージが詳細になります。HttpStatus.UNPROCESSABLE_ENTITY で422を指定します。

check_circle解答・解説 
// InsufficientStockException.java
public class InsufficientStockException extends RuntimeException {
    private final String productName;
    private final int requested;
    private final int available;

    public InsufficientStockException(String productName,
            int requested, int available) {
        super(productName + "の在庫が不足しています（要求: "
            + requested + ", 在庫: " + available + "）");
        this.productName = productName;
        this.requested = requested;
        this.available = available;
    }
    // getter 省略
}

// OrderService.java
@Service
public class OrderService {
    public void placeOrder(String productName, int quantity) {
        int stock = productMapper.getStock(productName);
        if (stock < quantity) {
            throw new InsufficientStockException(
                productName, quantity, stock);
        }
        // 注文処理...
    }
}

// GlobalExceptionHandler.java に追加
@ExceptionHandler(InsufficientStockException.class)
public ResponseEntity<ErrorResponse> handleInsufficientStock(
        InsufficientStockException ex) {
    ErrorResponse error = new ErrorResponse(
        422, "Unprocessable Entity", ex.getMessage());
    return ResponseEntity.status(HttpStatus.UNPROCESSABLE_ENTITY)
        .body(error);
}
解説
  • ビジネスロジック固有のエラーはカスタム例外で表現し、コントローラ層に業務判断を漏らさないのがクリーンな設計です。
  • 422 Unprocessable Entityは「構文は正しいが、意味的に処理できない」場合に使うHTTPステータスです。
  • 例外にコンテキスト情報（商品名、数量等）を含めることで、デバッグとエラーメッセージの質が向上します。

ロールベースアクセス制御の実装

verified
実務ポイント

URLパターンとロールに基づくアクセス制御はWebアプリの基本です。hasRole()やhasAuthority()でエンドポイントごとに必要な権限を指定します。

assignment課題

以下の仕様に従い、ロールベースのアクセス制御を実装してください。

  • /api/public/** — 認証不要
  • /api/admin/** — ADMINロールのみ
  • /api/users/** — USER または ADMINロール
  • それ以外 — 認証必須
lightbulbヒント

.requestMatchers("/api/admin/**").hasRole("ADMIN") でロール制限します。複数ロールは hasAnyRole("USER", "ADMIN") です。ルールは上から順に評価されます。

check_circle解答・解説 
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http)
            throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .requestMatchers("/api/users/**").hasAnyRole("USER", "ADMIN")
                .anyRequest().authenticated()
            )
            .httpBasic(Customizer.withDefaults())
            .csrf(csrf -> csrf.disable());
        return http.build();
    }
}
解説
  • hasRole("ADMIN") は内部的に ROLE_ADMIN 権限をチェックします。
  • ルールは上から順に評価されるため、permitAll() を最初に、anyRequest() を最後に書きます。
  • REST APIではCSRFトークンの管理が煩雑なため、csrf().disable() が一般的です（JWT認証と組み合わせる場合）。

パスワードエンコーディングの実装

実務ポイント

パスワードは必ずハッシュ化して保存します。Spring SecurityのBCryptPasswordEncoderは業界標準のハッシュアルゴリズムです。平文保存は絶対にNGです。

assignment課題

以下の仕様に従い、パスワードエンコーディングを実装してください。

  • BCryptPasswordEncoderをBean定義
  • ユーザー登録時にパスワードをハッシュ化して保存
  • ログイン時にBCryptで検証
lightbulbヒント

@Bean public PasswordEncoder passwordEncoder() { return new BCryptPasswordEncoder(); } でBean定義します。encoder.encode(rawPassword) でハッシュ化、encoder.matches(raw, encoded) で検証します。

check_circle解答・解説 
// SecurityConfig.java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}

// UserService.java
@Service
public class UserService {
    private final UserMapper userMapper;
    private final PasswordEncoder passwordEncoder;

    public UserService(UserMapper userMapper,
            PasswordEncoder passwordEncoder) {
        this.userMapper = userMapper;
        this.passwordEncoder = passwordEncoder;
    }

    public void register(UserForm form) {
        User user = new User();
        user.setName(form.getName());
        user.setEmail(form.getEmail());
        user.setPassword(passwordEncoder.encode(form.getPassword()));
        userMapper.insert(user);
    }
}
解説
  • BCryptは毎回異なるソルトを自動生成するため、同じパスワードでも異なるハッシュ値になります。
  • matches() メソッドが内部でソルトを復元して検証するため、ソルトを別管理する必要はありません。
  • パスワードの平文保存は情報漏洩時に全ユーザーのパスワードが流出する最悪の事態を招きます。

CRUD APIの網羅テスト

verified
実務ポイント

APIテストでは正常系だけでなく、404（リソース不在）やバリデーションエラーの異常系もテストします。テストメソッド名で何をテストしているか明確にします。

assignment課題

以下の仕様に従い、BookControllerのテストクラスを作成してください。

  • GET /api/books — 正常系（200、一覧返却）
  • GET /api/books/{id} — 正常系（200）と異常系（404）
  • POST /api/books — 正常系（201）とバリデーションエラー（400）
  • DELETE /api/books/{id} — 正常系（204）
lightbulbヒント

404テストでは when(...).thenThrow(new ResourceNotFoundException(...)) でServiceに例外を投げさせます。バリデーションエラーテストでは必須フィールドを空にしたJSONを送信します。

check_circle解答・解説 
@WebMvcTest(BookController.class)
class BookControllerTest {
    @Autowired private MockMvc mockMvc;
    @Autowired private ObjectMapper objectMapper;
    @MockBean private BookService bookService;

    @Test
    void getAll_returns200() throws Exception {
        when(bookService.findAll()).thenReturn(List.of(
            new Book(1L, "Java入門", "著者A", 3000)));
        mockMvc.perform(get("/api/books"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$", hasSize(1)));
    }

    @Test
    void getById_returns404_whenNotFound() throws Exception {
        when(bookService.findById(99L))
            .thenThrow(new ResourceNotFoundException("Book not found"));
        mockMvc.perform(get("/api/books/99"))
            .andExpect(status().isNotFound());
    }

    @Test
    void create_returns201() throws Exception {
        Book book = new Book(null, "Spring入門", "著者B", 4000);
        Book saved = new Book(1L, "Spring入門", "著者B", 4000);
        when(bookService.save(any())).thenReturn(saved);
        mockMvc.perform(post("/api/books")
                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(book)))
            .andExpect(status().isCreated());
    }

    @Test
    void delete_returns204() throws Exception {
        doNothing().when(bookService).delete(1L);
        mockMvc.perform(delete("/api/books/1"))
            .andExpect(status().isNoContent());
    }
}
解説
  • テストメソッド名は メソッド_期待結果_条件 形式にすると可読性が高まります。
  • doNothing().when() は戻り値がvoidのメソッドをモック化する書き方です。
  • 異常系テストは正常系と同じかそれ以上に重要です。本番で想定外のリクエストが来た際の挙動を保証します。

Serviceユニットテストの実装

verified
実務ポイント

Service層のテストではMapperをモック化し、ビジネスロジックだけを検証します。@ExtendWith(MockitoExtension.class)でSpringコンテキストなしの軽量テストが可能です。

assignment課題

以下の仕様に従い、TodoServiceのユニットテストを作成してください。

  • findAll(): マッパーが返すリストをそのまま返すことを検証
  • findById(): 存在する場合はTodoを返す
  • findById(): 存在しない場合はResourceNotFoundExceptionをスロー
lightbulbヒント

@Mock TodoMapper と @InjectMocks TodoService でDIを再現します。例外テストは assertThrows(ResourceNotFoundException.class, () -> ...) で検証します。

check_circle解答・解説 
@ExtendWith(MockitoExtension.class)
class TodoServiceTest {

    @Mock
    private TodoMapper todoMapper;

    @InjectMocks
    private TodoService todoService;

    @Test
    void findAll_returnsList() {
        List<Todo> todos = List.of(new Todo(1L, "タスク1", false));
        when(todoMapper.findAll()).thenReturn(todos);

        List<Todo> result = todoService.findAll();
        assertEquals(1, result.size());
        assertEquals("タスク1", result.get(0).getTitle());
    }

    @Test
    void findById_returnsTodo_whenExists() {
        Todo todo = new Todo(1L, "タスク1", false);
        when(todoMapper.findById(1L)).thenReturn(todo);

        Todo result = todoService.findById(1L);
        assertEquals("タスク1", result.getTitle());
    }

    @Test
    void findById_throwsException_whenNotExists() {
        when(todoMapper.findById(99L)).thenReturn(null);

        assertThrows(ResourceNotFoundException.class,
            () -> todoService.findById(99L));
    }
}
解説
  • @ExtendWith(MockitoExtension.class) はSpringコンテキストを起動しないため、テストが高速に実行されます。
  • @InjectMocks はモックオブジェクトをコンストラクタ経由で自動注入します。
  • assertThrows で例外の型とメッセージを検証できます。ビジネスロジックのエッジケースを漏れなくテストしましょう。

Todo REST APIの3層実装

verified
実務ポイント

仕様からController/Service/Mapperの3層を一気通貫で実装できることが実務スキルの基本です。各層の責務を意識して設計します。

assignment課題

以下の仕様に従い、Todo管理APIを3層構造で実装してください。

  • Todo: id(Long), title(String), completed(boolean), createdAt(LocalDateTime)
  • GET /api/todos — 全件取得
  • POST /api/todos — 新規作成（createdAtは自動設定）
  • PATCH /api/todos/{id}/complete — completedをtrueに更新
  • DELETE /api/todos/{id} — 削除
  • バリデーション: titleは必須・1〜200文字
lightbulbヒント

PATCHは部分更新に使います。@PatchMapping("/{id}/complete") でcompletedフラグのみを更新するMapperメソッドを呼びます。createdAtは LocalDateTime.now() でService内で設定します。

check_circle解答・解説 
// TodoMapper.java
@Mapper
public interface TodoMapper {
    @Select("SELECT * FROM todos ORDER BY created_at DESC")
    List<Todo> findAll();

    @Insert("INSERT INTO todos (title, completed, created_at) " +
            "VALUES (#{title}, #{completed}, #{createdAt})")
    @Options(useGeneratedKeys = true, keyProperty = "id")
    void insert(Todo todo);

    @Update("UPDATE todos SET completed = true WHERE id = #{id}")
    int complete(Long id);

    @Delete("DELETE FROM todos WHERE id = #{id}")
    int deleteById(Long id);
}

// TodoService.java
@Service
public class TodoService {
    private final TodoMapper todoMapper;
    public TodoService(TodoMapper todoMapper) {
        this.todoMapper = todoMapper;
    }
    public List<Todo> findAll() {
        return todoMapper.findAll();
    }
    public Todo create(TodoForm form) {
        Todo todo = new Todo();
        todo.setTitle(form.getTitle());
        todo.setCompleted(false);
        todo.setCreatedAt(LocalDateTime.now());
        todoMapper.insert(todo);
        return todo;
    }
    public void complete(Long id) {
        int rows = todoMapper.complete(id);
        if (rows == 0) throw new ResourceNotFoundException(
            "Todo not found: " + id);
    }
    public void delete(Long id) {
        int rows = todoMapper.deleteById(id);
        if (rows == 0) throw new ResourceNotFoundException(
            "Todo not found: " + id);
    }
}

// TodoController.java
@RestController
@RequestMapping("/api/todos")
public class TodoController {
    private final TodoService todoService;
    public TodoController(TodoService todoService) {
        this.todoService = todoService;
    }
    @GetMapping
    public List<Todo> getAll() { return todoService.findAll(); }

    @PostMapping
    public ResponseEntity<Todo> create(@Valid @RequestBody TodoForm form) {
        Todo todo = todoService.create(form);
        return ResponseEntity.status(HttpStatus.CREATED).body(todo);
    }
    @PatchMapping("/{id}/complete")
    public ResponseEntity<Void> complete(@PathVariable Long id) {
        todoService.complete(id);
        return ResponseEntity.ok().build();
    }
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        todoService.delete(id);
        return ResponseEntity.noContent().build();
    }
}
解説
  • @Options(useGeneratedKeys = true, keyProperty = "id") でINSERT後に自動採番されたIDをオブジェクトに反映します。
  • PATCHは部分更新に使い、PUTは全体更新に使うのがREST慣例です。
  • 更新・削除の影響行数（rows）が0の場合、対象リソースが存在しないため404を返します。

DTO変換パターンの実装

verified
実務ポイント

Entity（DBモデル）とDTO（API応答モデル）を分離することで、内部構造の変更がAPIに影響しなくなります。パスワード等の機密情報がレスポンスに漏れることも防げます。

assignment課題

以下の仕様に従い、DTO変換を含むユーザーAPIを実装してください。

  • User（Entity）: id, name, email, password, role, createdAt
  • UserResponse（DTO）: id, name, email, role（passwordとcreatedAtは含めない）
  • UserCreateRequest: name, email, password（バリデーション付き）
  • Service内でEntity↔DTO変換を行う
lightbulbヒント

DTOクラスにstaticファクトリメソッド UserResponse.from(User user) を定義すると変換がシンプルになります。パスワードはレスポンスDTOに絶対に含めないでください。

check_circle解答・解説 
// UserResponse.java
public class UserResponse {
    private Long id;
    private String name;
    private String email;
    private String role;

    public static UserResponse from(User user) {
        UserResponse dto = new UserResponse();
        dto.id = user.getId();
        dto.name = user.getName();
        dto.email = user.getEmail();
        dto.role = user.getRole();
        return dto;
    }
    // getter 省略
}

// UserService.java
public UserResponse createUser(UserCreateRequest req) {
    User user = new User();
    user.setName(req.getName());
    user.setEmail(req.getEmail());
    user.setPassword(passwordEncoder.encode(req.getPassword()));
    user.setRole("USER");
    user.setCreatedAt(LocalDateTime.now());
    userMapper.insert(user);
    return UserResponse.from(user);
}

public List<UserResponse> findAll() {
    return userMapper.findAll().stream()
        .map(UserResponse::from)
        .collect(Collectors.toList());
}
解説
  • staticファクトリメソッド from() パターンは変換ロジックをDTO側に閉じ込められるため、Serviceが簡潔になります。
  • Entity直接返却は「パスワードがJSONに含まれる」等の事故の元です。DTOを必ず介在させましょう。
  • Stream APIの map(UserResponse::from) でリストの一括変換が1行で書けます。
