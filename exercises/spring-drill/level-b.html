T1 Spring Boot基礎・DI

DI+プロファイルで環境別ロジック切替

verified
実務ポイント

@Profileアノテーションで開発/本番環境ごとに異なる実装をDIで切り替えられます。メール送信を開発時はコンソール出力、本番時は実際に送信するパターンは実務で頻出です。

assignment課題

以下の仕様に従い、環境別メール送信機能を実装してください。

  • MailSender インタフェース: void send(String to, String subject, String body)
  • ConsoleMailSender: @Profile("dev") — コンソールに出力
  • SmtpMailSender: @Profile("prod") — "SMTP送信: {to}" を出力（実際の送信は模擬）
  • コントローラからMailSenderを呼び出してテストできるエンドポイント
lightbulbヒント

@Profile("dev") を @Service と一緒に付けます。spring.profiles.active=dev でプロファイルを切り替えます。

check_circle解答・解説 
// MailSender.java
public interface MailSender {
    void send(String to, String subject, String body);
}

// ConsoleMailSender.java
@Service
@Profile("dev")
public class ConsoleMailSender implements MailSender {
    @Override
    public void send(String to, String subject, String body) {
        System.out.println("[DEV] To: " + to);
        System.out.println("[DEV] Subject: " + subject);
        System.out.println("[DEV] Body: " + body);
    }
}

// SmtpMailSender.java
@Service
@Profile("prod")
public class SmtpMailSender implements MailSender {
    @Override
    public void send(String to, String subject, String body) {
        System.out.println("SMTP送信: " + to);
        // 実際のSMTP送信処理
    }
}

// MailController.java
@RestController
@RequestMapping("/api/mail")
public class MailController {
    private final MailSender mailSender;
    public MailController(MailSender mailSender) {
        this.mailSender = mailSender;
    }
    @PostMapping("/test")
    public ResponseEntity<String> sendTest() {
        mailSender.send("test@example.com", "テスト", "本文");
        return ResponseEntity.ok("送信完了");
    }
}
解説
  • @Profile でBeanの有効化条件を環境プロファイルに紐付けます。コード変更なしで挙動を切り替えられます。
  • 開発環境で実際のメール送信を防ぐのは実務の鉄則です。誤送信による事故を防ぎます。
  • DI+プロファイルの組み合わせはストラテジーパターンの実践例です。

T1 Spring Boot基礎・DI

複数Bean登録と@Qualifierによる選択

verified
実務ポイント

同じインタフェースの実装が複数ある場合、@Qualifierで注入するBeanを明示的に選択します。ログ出力先やキャッシュ戦略の切り替えに使われます。

assignment課題

以下の仕様に従い、複数のストレージ実装を切り替えるファイルアップロード機能を実装してください。

  • StorageService インタフェース: String store(String filename, byte[] data)
  • LocalStorageService: ローカルファイルシステムに保存（Bean名: "localStorage"）
  • S3StorageService: S3にアップロード（模擬、Bean名: "s3Storage"）
  • コントローラで @Qualifier("localStorage") を使って選択
lightbulbヒント

@Service("localStorage") でBean名を指定します。注入側で @Qualifier("localStorage") を使ってBean名で選択します。

check_circle解答・解説 
// StorageService.java
public interface StorageService {
    String store(String filename, byte[] data);
}

// LocalStorageService.java
@Service("localStorage")
public class LocalStorageService implements StorageService {
    @Override
    public String store(String filename, byte[] data) {
        // ローカル保存処理
        return "/uploads/" + filename;
    }
}

// S3StorageService.java
@Service("s3Storage")
public class S3StorageService implements StorageService {
    @Override
    public String store(String filename, byte[] data) {
        // S3アップロード処理（模擬）
        return "https://s3.example.com/" + filename;
    }
}

// FileController.java
@RestController
@RequestMapping("/api/files")
public class FileController {
    private final StorageService storageService;

    public FileController(
            @Qualifier("localStorage") StorageService storageService) {
        this.storageService = storageService;
    }

    @PostMapping("/upload")
    public ResponseEntity<String> upload(@RequestParam String filename) {
        String url = storageService.store(filename, new byte[0]);
        return ResponseEntity.ok(url);
    }
}
解説
  • @Qualifier はBean名（デフォルトはクラス名のキャメルケース）で注入先を特定します。
  • @Primary を使うとデフォルトのBeanを指定でき、@Qualifier省略時はそちらが注入されます。
  • 実務ではプロファイルと組み合わせて、dev=ローカル、prod=S3 のように切り替えるのが一般的です。

T2 RESTコントローラ

検索+ページネーション+ソートAPIの統合

verified
実務ポイント

実務のAPIでは検索・ページネーション・ソートを組み合わせるのが一般的です。クエリパラメータで柔軟に制御できるAPIを設計します。

assignment課題

以下の仕様に従い、商品検索APIを実装してください。

  • GET /api/products?keyword=Java&category=book&sort=price&order=desc&page=0&size=10
  • keyword: 商品名の部分一致検索（任意）
  • category: カテゴリ完全一致（任意）
  • sort: ソート対象カラム（デフォルト: id）
  • order: asc/desc（デフォルト: asc）
  • MyBatis XMLマッパーで動的SQLを記述
lightbulbヒント

ORDER BY句は動的に組み立てる必要があるため、${sort}（リテラル置換）を使います。ただしSQLインジェクション防止のため、Serviceでsortの値をホワイトリスト検証してください。

check_circle解答・解説 
<!-- ProductMapper.xml -->
<select id="search" resultType="Product">
  SELECT * FROM products
  <where>
    <if test="keyword != null and keyword != ''">
      AND name LIKE CONCAT('%', #{keyword}, '%')
    </if>
    <if test="category != null and category != ''">
      AND category = #{category}
    </if>
  </where>
  ORDER BY ${sort} ${order}
  LIMIT #{size} OFFSET #{offset}
</select>

// ProductService.java
private static final Set<String> ALLOWED_SORT =
    Set.of("id", "name", "price", "created_at");

public PageResponse<Product> search(String keyword,
        String category, String sort, String order,
        int page, int size) {
    if (!ALLOWED_SORT.contains(sort)) sort = "id";
    if (!"desc".equals(order)) order = "asc";
    int offset = page * size;
    List<Product> content = productMapper.search(
        keyword, category, sort, order, size, offset);
    long total = productMapper.countSearch(keyword, category);
    return new PageResponse<>(content, page, size, total);
}
解説
  • ${sort} はSQLにそのまま展開されるため、ユーザー入力を直接渡すとSQLインジェクションになります。ホワイトリスト検証が必須です。
  • ORDER BYのカラム名は #{}（プリペアドステートメント）では使えないため、${} を使う必要があります。
  • 検索件数とデータ取得を別クエリにすることで、ページネーションのtotalCountを正確に返せます。

T2 RESTコントローラ

REST API + バリデーション + 例外処理の統合

verified
実務ポイント

実務のAPIでは入力バリデーション、ビジネスルールチェック、例外ハンドリングを統合的に実装します。各層で適切なエラーを返すことでフロントエンドが適切に対応できます。

assignment課題

以下の仕様に従い、ユーザー登録APIを完全実装してください。

  • POST /api/users — ユーザー新規作成
  • バリデーション: name必須・2〜50文字、email必須・Email形式、password必須・8文字以上
  • ビジネスルール: emailの重複チェック（重複時は409 Conflict）
  • パスワードはBCryptハッシュ化して保存
  • レスポンスはUserResponse DTO（passwordを含めない）
  • エラーレスポンスは統一フォーマット
lightbulbヒント

Service内で userMapper.findByEmail(email) を呼び、nullでなければ DuplicateResourceException をスローします。バリデーション → ビジネスルール → 保存 の順に処理します。

check_circle解答・解説 
// UserCreateRequest.java
public class UserCreateRequest {
    @NotBlank @Size(min = 2, max = 50)
    private String name;
    @NotBlank @Email
    private String email;
    @NotBlank @Size(min = 8)
    private String password;
}

// UserService.java
@Service
public class UserService {
    private final UserMapper userMapper;
    private final PasswordEncoder passwordEncoder;

    public UserResponse create(UserCreateRequest req) {
        // 重複チェック
        if (userMapper.findByEmail(req.getEmail()) != null) {
            throw new DuplicateResourceException(
                "このメールアドレスは既に登録されています");
        }
        User user = new User();
        user.setName(req.getName());
        user.setEmail(req.getEmail());
        user.setPassword(passwordEncoder.encode(req.getPassword()));
        user.setCreatedAt(LocalDateTime.now());
        userMapper.insert(user);
        return UserResponse.from(user);
    }
}

// UserController.java
@PostMapping
public ResponseEntity<UserResponse> create(
        @Valid @RequestBody UserCreateRequest req) {
    UserResponse response = userService.create(req);
    return ResponseEntity.status(HttpStatus.CREATED).body(response);
}
解説
  • 処理順序: @Valid(400) → ビジネスルール(409) → 保存(201) → エラー(500) のレイヤーで責務を分離します。
  • DTOパターンにより、パスワードハッシュがレスポンスに漏れることを構造的に防止しています。
  • この「バリデーション→重複チェック→保存→DTO変換」は実務CRUD APIの王道パターンです。

T3 データベース・MyBatis

トランザクション管理 + 複数テーブル操作

verified
実務ポイント

複数テーブルへの書き込みは@Transactionalで1つのトランザクションにまとめます。途中で例外が発生した場合、全ての変更がロールバックされてデータの整合性が保たれます。

assignment課題

以下の仕様に従い、注文処理を実装してください。

  • 注文作成時に orders テーブルと order_items テーブルの両方にINSERT
  • 在庫テーブル（products.stock）を減算し、在庫不足なら例外でロールバック
  • @Transactionalで全操作をアトミックにする
lightbulbヒント

Service メソッドに @Transactional を付けます。在庫確認→在庫減算→注文INSERT→明細INSERT の順に実行し、在庫不足時は InsufficientStockException をスローします。

check_circle解答・解説 
@Service
public class OrderService {
    private final OrderMapper orderMapper;
    private final OrderItemMapper orderItemMapper;
    private final ProductMapper productMapper;

    @Transactional
    public Order placeOrder(OrderRequest req) {
        // 在庫チェック&減算
        for (OrderItemRequest item : req.getItems()) {
            Product product = productMapper.findById(item.getProductId());
            if (product.getStock() < item.getQuantity()) {
                throw new InsufficientStockException(
                    product.getName(),
                    item.getQuantity(),
                    product.getStock());
            }
            productMapper.decreaseStock(
                item.getProductId(), item.getQuantity());
        }
        // 注文作成
        Order order = new Order();
        order.setCustomerName(req.getCustomerName());
        order.setOrderDate(LocalDateTime.now());
        orderMapper.insert(order);

        // 明細作成
        for (OrderItemRequest item : req.getItems()) {
            OrderItem oi = new OrderItem();
            oi.setOrderId(order.getId());
            oi.setProductId(item.getProductId());
            oi.setQuantity(item.getQuantity());
            orderItemMapper.insert(oi);
        }
        return order;
    }
}
解説
  • @Transactional は RuntimeException 発生時に自動ロールバックします。検査例外はデフォルトではロールバックされません。
  • 在庫チェック→減算を先に行うことで、在庫不足時に注文が作成されることを防ぎます。
  • 並行処理での在庫の競合は楽観的ロック（バージョン番号）や悲観的ロック（SELECT FOR UPDATE）で対処します。

T3 データベース・MyBatis

バッチINSERTとforeachタグ

verified
実務ポイント

大量データの登録では1件ずつINSERTするとパフォーマンスが悪化します。MyBatisの<foreach>タグで複数レコードを1回のSQLでINSERTするバッチ処理を実装します。

assignment課題

以下の仕様に従い、CSVインポート機能のバッチINSERTを実装してください。

  • リクエスト: POST /api/products/batch で商品リスト（最大100件）を一括登録
  • MyBatis XMLマッパーで <foreach> を使った1SQL複数行INSERT
  • @Transactionalでアトミックに処理
lightbulbヒント

<foreach collection="list" item="p" separator=",">(#{p.name}, #{p.price})</foreach> で VALUES句を動的に生成します。

check_circle解答・解説 
<!-- ProductMapper.xml -->
<insert id="batchInsert">
  INSERT INTO products (name, price, category)
  VALUES
  <foreach collection="list" item="p" separator=",">
    (#{p.name}, #{p.price}, #{p.category})
  </foreach>
</insert>

// ProductMapper.java
@Mapper
public interface ProductMapper {
    void batchInsert(@Param("list") List<Product> products);
}

// ProductService.java
@Transactional
public int batchImport(List<ProductForm> forms) {
    List<Product> products = forms.stream()
        .map(f -> new Product(f.getName(), f.getPrice(), f.getCategory()))
        .collect(Collectors.toList());
    productMapper.batchInsert(products);
    return products.size();
}
解説
  • <foreach> の separator="," で VALUES 句のカンマ区切りを自動生成します。
  • バッチINSERTは個別INSERTの10〜100倍高速です。ネットワーク往復回数が1回に減るためです。
  • PostgreSQLでは1SQLのパラメータ上限があるため、大量データは1000件ごとに分割するのが安全です。

T4 バリデーション

カスタムバリデーションアノテーションの作成

verified
実務ポイント

標準アノテーションでは対応できない業務固有のバリデーション（例: 電話番号形式、郵便番号形式）は、カスタムバリデーションアノテーションを作成して対応します。

assignment課題

以下の仕様に従い、電話番号バリデーションのカスタムアノテーションを作成してください。

  • @PhoneNumber アノテーション: 正規表現 ^0[0-9]{9,10}$ でチェック
  • PhoneNumberValidator: ConstraintValidatorを実装
  • ContactForm の phone フィールドに適用
lightbulbヒント

@Constraint(validatedBy = PhoneNumberValidator.class) でバリデータを紐付けます。ConstraintValidator<PhoneNumber, String> を実装し、isValid() メソッドで正規表現マッチを行います。

check_circle解答・解説 
// PhoneNumber.java
@Target({ElementType.FIELD})
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = PhoneNumberValidator.class)
public @interface PhoneNumber {
    String message() default "電話番号の形式が正しくありません";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

// PhoneNumberValidator.java
public class PhoneNumberValidator
        implements ConstraintValidator<PhoneNumber, String> {
    private static final Pattern PATTERN =
        Pattern.compile("^0[0-9]{9,10}$");

    @Override
    public boolean isValid(String value,
            ConstraintValidatorContext context) {
        if (value == null || value.isEmpty()) return true;
        return PATTERN.matcher(value).matches();
    }
}

// ContactForm.java
public class ContactForm {
    @NotBlank
    private String name;

    @PhoneNumber
    private String phone;
}
解説
  • isValid() で null/空文字を true にしているのは、必須チェックは @NotBlank の責務とするためです（責務の分離）。
  • カスタムアノテーションは @Target, @Retention, @Constraint の3つが必須です。
  • groups() と payload() はBean Validation仕様で必須の属性です。

T4 バリデーション

クロスフィールドバリデーションの実装

verified
実務ポイント

「パスワード確認」や「開始日 < 終了日」のように、複数フィールドを跨ぐバリデーションはクラスレベルのカスタムバリデーションで実装します。

assignment課題

以下の仕様に従い、日付範囲バリデーションを実装してください。

  • @DateRange: クラスレベルアノテーション
  • startDate < endDate を検証（同日は不可）
  • SearchForm の startDate/endDate に適用
lightbulbヒント

@Target(ElementType.TYPE) でクラスレベルにします。Validatorの isValid() で SearchForm を受け取り、startDate.isBefore(endDate) で検証します。

check_circle解答・解説 
// DateRange.java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = DateRangeValidator.class)
public @interface DateRange {
    String message() default "開始日は終了日より前の日付を指定してください";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

// DateRangeValidator.java
public class DateRangeValidator
        implements ConstraintValidator<DateRange, SearchForm> {
    @Override
    public boolean isValid(SearchForm form,
            ConstraintValidatorContext context) {
        if (form.getStartDate() == null
                || form.getEndDate() == null) return true;
        return form.getStartDate().isBefore(form.getEndDate());
    }
}

// SearchForm.java
@DateRange
public class SearchForm {
    @NotNull private LocalDate startDate;
    @NotNull private LocalDate endDate;
    private String keyword;
}
解説
  • クラスレベルのバリデーションは @Target(ElementType.TYPE) で定義し、Validatorのジェネリクスにフォームクラスを指定します。
  • null チェックを isValid() 内で行い、null の場合は true を返します（@NotNull の責務と分離）。
  • 実務では「パスワード一致確認」「金額範囲チェック」等、多くのクロスフィールドバリデーションが発生します。

B-09
T5 例外ハンドリング

@ControllerAdviceによるグローバル例外ハンドラ

verified
実務ポイント

@ControllerAdviceで全コントローラ共通の例外処理を一元管理します。バリデーションエラー(400)、リソース未発見(404)、業務エラー(409)、システムエラー(500)を統一フォーマットで返すのが実務の基本です。

assignment課題

以下の仕様に従い、グローバル例外ハンドラと統一エラーレスポンスを実装してください。

  • ErrorResponse DTO: timestamp, status, error, message, path フィールド
  • ResourceNotFoundException → 404
  • MethodArgumentNotValidException（@Valid失敗）→ 400（フィールドエラー一覧付き）
  • DuplicateResourceException → 409
  • Exception（その他）→ 500
lightbulbヒント

@RestControllerAdvice で全REST APIを対象にし、@ExceptionHandler(ResourceNotFoundException.class) で例外の種類ごとにメソッドを定義します。HttpServletRequest から getRequestURI() でパスを取得します。

check_circle解答・解説 
// ErrorResponse.java
public class ErrorResponse {
    private LocalDateTime timestamp;
    private int status;
    private String error;
    private String message;
    private String path;
    private List<FieldError> errors; // バリデーション用

    public ErrorResponse(int status, String error,
            String message, String path) {
        this.timestamp = LocalDateTime.now();
        this.status = status;
        this.error = error;
        this.message = message;
        this.path = path;
    }

    public record FieldError(String field, String message) {}
}

// GlobalExceptionHandler.java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            ResourceNotFoundException ex, HttpServletRequest req) {
        ErrorResponse body = new ErrorResponse(
            404, "Not Found", ex.getMessage(), req.getRequestURI());
        return ResponseEntity.status(404).body(body);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
            MethodArgumentNotValidException ex,
            HttpServletRequest req) {
        ErrorResponse body = new ErrorResponse(
            400, "Bad Request", "入力値に誤りがあります",
            req.getRequestURI());
        body.setErrors(ex.getBindingResult().getFieldErrors()
            .stream()
            .map(e -> new ErrorResponse.FieldError(
                e.getField(), e.getDefaultMessage()))
            .toList());
        return ResponseEntity.badRequest().body(body);
    }

    @ExceptionHandler(DuplicateResourceException.class)
    public ResponseEntity<ErrorResponse> handleDuplicate(
            DuplicateResourceException ex,
            HttpServletRequest req) {
        ErrorResponse body = new ErrorResponse(
            409, "Conflict", ex.getMessage(),
            req.getRequestURI());
        return ResponseEntity.status(409).body(body);
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneral(
            Exception ex, HttpServletRequest req) {
        ErrorResponse body = new ErrorResponse(
            500, "Internal Server Error",
            "予期しないエラーが発生しました",
            req.getRequestURI());
        return ResponseEntity.status(500).body(body);
    }
}
解説
  • @RestControllerAdvice は @ControllerAdvice + @ResponseBody の合成アノテーションで、JSON形式のエラーレスポンスを返します。
  • バリデーションエラーではフィールド名とメッセージの一覧を返すことで、フロントエンドが各フィールドにエラーを表示できます。
  • Exception.class のハンドラを最後に置くことで、想定外の例外もキャッチしてユーザーにスタックトレースが漏れるのを防ぎます。
  • 本番環境ではログレベルを分けて、500エラーはERROR、400/404はWARNで記録するのが一般的です。

T5 例外ハンドリング

業務例外クラス階層とProblemDetail対応

verified
実務ポイント

Spring Boot 3.xではRFC 9457準拠のProblemDetailが標準サポートされています。業務例外をカスタム例外クラスの階層で整理し、ProblemDetail形式で返すのが現在のベストプラクティスです。

assignment課題

以下の仕様に従い、ProblemDetail形式のエラーレスポンスを実装してください。

  • 基底クラス BusinessException（RuntimeException継承）にHTTPステータスとtypeURIを持たせる
  • OrderNotFoundException → 404, type: "/problems/order-not-found"

ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage()) でProblemDetailを生成し、setType(URI.create(...)) でtype URIを設定します。setProperty("key", value) で追加情報を埋め込めます。

check_circle解答・解説 
// BusinessException.java
public abstract class BusinessException extends RuntimeException {
    private final HttpStatus status;
    private final String typeUri;

    protected BusinessException(String message,
            HttpStatus status, String typeUri) {
        super(message);
        this.status = status;
        this.typeUri = typeUri;
    }
    public HttpStatus getStatus() { return status; }
    public String getTypeUri() { return typeUri; }
}

// InsufficientStockException.java
public class InsufficientStockException extends BusinessException {
    private final String productName;
    private final int requested;
    private final int available;

    public InsufficientStockException(String productName,
            int requested, int available) {
        super(productName + "の在庫が不足しています",
            HttpStatus.UNPROCESSABLE_ENTITY,
            "/problems/insufficient-stock");
        this.productName = productName;
        this.requested = requested;
        this.available = available;
    }
    // getter省略
}

// GlobalExceptionHandler.java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(BusinessException.class)
    public ProblemDetail handleBusiness(BusinessException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
            ex.getStatus(), ex.getMessage());
        pd.setType(URI.create(ex.getTypeUri()));
        pd.setTitle(ex.getStatus().getReasonPhrase());
        if (ex instanceof InsufficientStockException ise) {
            pd.setProperty("productName", ise.getProductName());
            pd.setProperty("requested", ise.getRequested());
            pd.setProperty("available", ise.getAvailable());
        }
        return pd;
    }
}
解説
  • ProblemDetail はSpring Boot 3.xで導入されたRFC 9457準拠のエラーレスポンス形式で、type, title, status, detail, instance の標準フィールドを持ちます。
  • setProperty() で任意の追加情報を埋め込めるため、在庫数や商品名などの業務固有データをクライアントに返せます。
  • 基底クラス BusinessException にステータスとtype URIを持たせることで、ハンドラを1メソッドに集約できます。
  • instanceof パターンマッチング（Java 21）で型に応じた追加情報を安全に取り出しています。

Spring Security基本設定とロールベース認可

verified
実務ポイント

Spring Securityで認証・認可を一元管理します。SecurityFilterChainでURLパターンごとのアクセス制御を設定し、ロール（ADMIN, USER）で操作権限を制御するのが基本パターンです。

assignment課題

以下の仕様に従い、ロールベースのアクセス制御を実装してください。

  • /api/public/** → 認証不要（permitAll）
  • /api/users/** → ROLE_USER または ROLE_ADMIN
  • /api/admin/** → ROLE_ADMIN のみ
  • インメモリユーザー2名: user(USER権限), admin(ADMIN権限)
  • CSRF無効化（REST API用）、セッションレス設定
lightbulbヒント

SecurityFilterChain をBean定義し、http.authorizeHttpRequests() でURLパターンごとの権限を設定します。httpBasic(Customizer.withDefaults()) でHTTP Basic認証を有効化します。

check_circle解答・解説 
// SecurityConfig.java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http)
            throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(sm -> sm
                .sessionCreationPolicy(
                    SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .requestMatchers("/api/admin/**")
                    .hasRole("ADMIN")
                .requestMatchers("/api/users/**")
                    .hasAnyRole("USER", "ADMIN")
                .anyRequest().authenticated())
            .httpBasic(Customizer.withDefaults());
        return http.build();
    }

    @Bean
    public InMemoryUserDetailsManager userDetailsService() {
        UserDetails user = User.withDefaultPasswordEncoder()
            .username("user")
            .password("password")
            .roles("USER")
            .build();
        UserDetails admin = User.withDefaultPasswordEncoder()
            .username("admin")
            .password("password")
            .roles("ADMIN")
            .build();
        return new InMemoryUserDetailsManager(user, admin);
    }
}
解説
  • REST APIではCSRFトークンが不要（ブラウザフォーム送信を使わない）なため、csrf.disable() が一般的です。
  • SessionCreationPolicy.STATELESS でサーバー側にセッションを持たず、リクエストごとに認証するステートレス設計になります。
  • requestMatchers() の順序が重要です。より具体的なパターンを先に記述しないと、anyRequest() に先にマッチしてしまいます。
  • withDefaultPasswordEncoder() は開発用のみです。本番では BCryptPasswordEncoder を使います。

認証情報の取得とCORS設定

verified
実務ポイント

認証済みユーザーの情報をコントローラで取得し、操作の所有者チェックを行います。また、フロントエンドとバックエンドが別ドメインの場合はCORS設定が必須です。

assignment課題

以下の仕様に従い、認証ユーザー取得とCORS設定を実装してください。

  • コントローラで @AuthenticationPrincipal から現在のユーザーを取得
  • 自分のプロフィールのみ編集可能（他人のIDを指定したら403）
  • CORS: http://localhost:3000 からのリクエストを許可
  • 許可メソッド: GET, POST, PUT, DELETE
lightbulbヒント

@AuthenticationPrincipal UserDetails userDetails でログインユーザーを取得します。CORS設定は CorsConfigurationSource Beanを定義してSecurityFilterChainに渡します。

check_circle解答・解説 
// ProfileController.java
@RestController
@RequestMapping("/api/users")
public class ProfileController {
    private final UserService userService;

    @PutMapping("/{id}/profile")
    public ResponseEntity<UserResponse> updateProfile(
            @PathVariable Long id,
            @Valid @RequestBody ProfileForm form,
            @AuthenticationPrincipal UserDetails userDetails) {
        User currentUser = userService.findByUsername(
            userDetails.getUsername());
        if (!currentUser.getId().equals(id)) {
            throw new ForbiddenException(
                "他のユーザーのプロフィールは編集できません");
        }
        UserResponse response = userService.updateProfile(id, form);
        return ResponseEntity.ok(response);
    }
}

// SecurityConfig.java（CORS設定を追加）
@Bean
public SecurityFilterChain filterChain(HttpSecurity http)
        throws Exception {
    http
        .cors(cors -> cors.configurationSource(
            corsConfigurationSource()))
        .csrf(csrf -> csrf.disable())
        .authorizeHttpRequests(auth -> auth
            .anyRequest().authenticated())
        .httpBasic(Customizer.withDefaults());
    return http.build();
}

@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(
        List.of("http://localhost:3000"));
    config.setAllowedMethods(
        List.of("GET", "POST", "PUT", "DELETE"));
    config.setAllowedHeaders(List.of("*"));
    config.setAllowCredentials(true);
    UrlBasedCorsConfigurationSource source =
        new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/api/**", config);
    return source;
}
解説
  • @AuthenticationPrincipal でSpring Securityが管理する認証済みユーザーをメソッド引数で受け取れます。
  • 所有者チェック（自分のリソースのみ操作可能）は認可の基本パターンで、IDOR（Insecure Direct Object Reference）脆弱性を防ぎます。
  • CORS設定はSpring Securityのフィルタチェーンに統合する必要があります。@CrossOriginアノテーションよりもSecurityFilterChainでの一元管理が推奨です。
  • setAllowCredentials(true) でCookieやAuthorizationヘッダーを含むクロスオリジンリクエストを許可します。

MockMvc + @MockBeanによる統合テスト

verified
実務ポイント

@WebMvcTestでコントローラ層のみを起動し、Service層は@MockBeanでモック化してテストします。HTTPリクエスト→レスポンスの流れを実際のHTTP層を通して検証できます。

assignment課題

以下の仕様に従い、商品APIのMockMvcテストを実装してください。

  • GET /api/products/{id} — 正常系(200)とリソース未発見(404)のテスト
  • POST /api/products — 正常系(201)とバリデーションエラー(400)のテスト
  • @MockBean ProductService でService層をモック化
  • レスポンスボディのJSON項目を jsonPath で検証
lightbulbヒント

when(productService.findById(1L)).thenReturn(...) でモックの戻り値を設定し、mockMvc.perform(get("/api/products/1")).andExpect(status().isOk()) で検証します。404テストでは thenThrow(new ResourceNotFoundException(...)) を使います。

check_circle解答・解説 
// ProductControllerTest.java
@WebMvcTest(ProductController.class)
class ProductControllerTest {

    @Autowired private MockMvc mockMvc;
    @Autowired private ObjectMapper objectMapper;
    @MockBean private ProductService productService;

    @Test
    @DisplayName("GET /api/products/{id} - 正常系: 200")
    void findById_returns200() throws Exception {
        when(productService.findById(1L))
            .thenReturn(new ProductResponse(1L, "Java本", 3000));

        mockMvc.perform(get("/api/products/1"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.id").value(1))
            .andExpect(jsonPath("$.name").value("Java本"))
            .andExpect(jsonPath("$.price").value(3000));
    }

    @Test
    @DisplayName("GET /api/products/{id} - 404: リソース未発見")
    void findById_returns404() throws Exception {
        when(productService.findById(999L))
            .thenThrow(new ResourceNotFoundException(
                "Product not found: 999"));

        mockMvc.perform(get("/api/products/999"))
            .andExpect(status().isNotFound())
            .andExpect(jsonPath("$.message")
                .value("Product not found: 999"));
    }

    @Test
    @DisplayName("POST /api/products - 正常系: 201")
    void create_returns201() throws Exception {
        ProductForm form = new ProductForm("Spring本", 3500);
        when(productService.create(any()))
            .thenReturn(new ProductResponse(2L, "Spring本", 3500));

        mockMvc.perform(post("/api/products")
                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(form)))
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.name").value("Spring本"));
    }

    @Test
    @DisplayName("POST /api/products - 400: バリデーションエラー")
    void create_returns400_whenInvalid() throws Exception {
        ProductForm form = new ProductForm("", -100);

        mockMvc.perform(post("/api/products")
                .contentType(MediaType.APPLICATION_JSON)
                .content(objectMapper.writeValueAsString(form)))
            .andExpect(status().isBadRequest())
            .andExpect(jsonPath("$.errors").isArray())
            .andExpect(jsonPath("$.errors.length()").value(
                org.hamcrest.Matchers.greaterThanOrEqualTo(1)));
    }
}
解説
  • @WebMvcTest はコントローラ層だけを起動するため、全Bean起動の @SpringBootTest より高速です。
  • @MockBean はSpringコンテキスト内のBeanをMockitoモックに差し替えます。Service層のDBアクセスを回避できます。
  • jsonPath でレスポンスJSONの各フィールドを個別に検証できます。ネストしたオブジェクトも $.items[0].name のようにアクセスできます。
  • @DisplayName でテストの意図を日本語で明示すると、テストレポートの可読性が上がります。

多層CRUD Serviceの統合実装

verified
実務ポイント

実務のCRUD APIはController→Service→Mapperの3層構造で、バリデーション、DTO変換、トランザクション管理、例外処理を統合的に実装します。この基本パターンを確実に身につけることが重要です。

assignment課題

以下の仕様に従い、書籍管理の完全CRUD APIを実装してください。

  • エンティティ: Book (id, title, author, isbn, price, publishedDate, createdAt, updatedAt)
  • POST /api/books — 登録（バリデーション: title必須, isbn形式チェック, price > 0）
  • GET /api/books/{id} — 1件取得（未発見時404）
  • GET /api/books?keyword=&sort=title&page=0&size=10 — 検索+ページネーション
  • PUT /api/books/{id} — 更新（楽観的ロック付き）
  • DELETE /api/books/{id} — 削除
  • レスポンスはBookResponse DTO、リスト取得はPageResponseでラップ
lightbulbヒント

Controller層でバリデーション、Service層でビジネスロジック(重複チェック・楽観ロック)、Mapper層でSQL実行と、各層の責務を明確に分離します。DTOはリクエスト用とレスポンス用を分けるのが基本です。

check_circle解答・解説 
// BookForm.java (リクエストDTO)
public class BookForm {
    @NotBlank private String title;
    @NotBlank private String author;
    @Pattern(regexp = "^\\d{13}$",
        message = "ISBNは13桁の数字で入力してください")
    private String isbn;
    @Positive private int price;
    private LocalDate publishedDate;
    private Integer version; // 更新時のみ使用
}

// BookResponse.java (レスポンスDTO)
public record BookResponse(
    Long id, String title, String author,
    String isbn, int price, LocalDate publishedDate,
    int version) {
    public static BookResponse from(Book b) {
        return new BookResponse(b.getId(), b.getTitle(),
            b.getAuthor(), b.getIsbn(), b.getPrice(),
            b.getPublishedDate(), b.getVersion());
    }
}

// BookService.java
@Service
public class BookService {
    private final BookMapper bookMapper;

    @Transactional
    public BookResponse create(BookForm form) {
        if (bookMapper.findByIsbn(form.getIsbn()) != null) {
            throw new DuplicateResourceException(
                "ISBN " + form.getIsbn() + " は既に登録されています");
        }
        Book book = new Book();
        book.setTitle(form.getTitle());
        book.setAuthor(form.getAuthor());
        book.setIsbn(form.getIsbn());
        book.setPrice(form.getPrice());
        book.setPublishedDate(form.getPublishedDate());
        book.setCreatedAt(LocalDateTime.now());
        bookMapper.insert(book);
        return BookResponse.from(book);
    }

    @Transactional
    public BookResponse update(Long id, BookForm form) {
        Book book = bookMapper.findById(id);
        if (book == null) throw new ResourceNotFoundException(
            "Book not found: " + id);
        book.setTitle(form.getTitle());
        book.setAuthor(form.getAuthor());
        book.setPrice(form.getPrice());
        int rows = bookMapper.updateWithVersion(book);
        if (rows == 0) throw new OptimisticLockException(
            "データが他のユーザーに更新されました");
        return BookResponse.from(bookMapper.findById(id));
    }

    public PageResponse<BookResponse> search(
            String keyword, String sort, int page, int size) {
        int offset = page * size;
        List<BookResponse> content = bookMapper
            .search(keyword, sort, size, offset)
            .stream().map(BookResponse::from).toList();
        long total = bookMapper.countSearch(keyword);
        return new PageResponse<>(content, page, size, total);
    }
}

// BookController.java
@RestController
@RequestMapping("/api/books")
public class BookController {
    private final BookService bookService;

    @PostMapping
    public ResponseEntity<BookResponse> create(
            @Valid @RequestBody BookForm form) {
        return ResponseEntity.status(HttpStatus.CREATED)
            .body(bookService.create(form));
    }
    @GetMapping("/{id}")
    public BookResponse findById(@PathVariable Long id) {
        return bookService.findById(id);
    }
    @GetMapping
    public PageResponse<BookResponse> search(
            @RequestParam(required = false) String keyword,
            @RequestParam(defaultValue = "id") String sort,
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "10") int size) {
        return bookService.search(keyword, sort, page, size);
    }
    @PutMapping("/{id}")
    public BookResponse update(@PathVariable Long id,
            @Valid @RequestBody BookForm form) {
        return bookService.update(id, form);
    }
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        bookService.delete(id);
        return ResponseEntity.noContent().build();
    }
}
解説
  • Controller層はHTTPの関心事（ステータスコード、パラメータ取得）、Service層はビジネスロジック（重複チェック、楽観ロック）、Mapper層はSQL実行に責務を分離しています。
  • record型の BookResponse は不変オブジェクトで、getter/equals/hashCode/toStringが自動生成されるJava 16以降の機能です。
  • @Pattern でISBN形式チェック、@Positive で正の数チェックと、宣言的バリデーションで入力検証のコードを削減しています。

CRUD + バリデーション + エラーハンドリングの統合

verified
実務ポイント

実務の完成度が高いAPIは、正常系だけでなく異常系（バリデーションエラー、業務エラー、システムエラー）の全パターンで統一されたレスポンスを返します。フロントエンドとの連携がスムーズになります。

assignment課題

以下の仕様に従い、タスク管理APIを統合実装してください。

  • エンティティ: Task (id, title, description, status, priority, dueDate, assigneeId, createdAt)
  • CRUD: 登録(201) / 取得(200) / 一覧(200) / 更新(200) / 削除(204)
  • バリデーション: title必須(2-100文字), status: TODO/IN_PROGRESS/DONE, priority: LOW/MEDIUM/HIGH, dueDate: 未来日
  • 業務ルール: DONE→TODO への状態戻しは不可(422)、担当者の存在チェック(404)

    @Pattern(regexp = "LOW|MEDIUM|HIGH")

    @FutureOrPresent(message = "期限は今日以降を指定してください")

// TaskService.java
@Service
public class TaskService {
    private static final Map<String, Set<String>> TRANSITIONS = Map.of(
        "TODO", Set.of("IN_PROGRESS"),
        "IN_PROGRESS", Set.of("TODO", "DONE"),
        "DONE", Set.of("IN_PROGRESS")
    );

    @Transactional
    public TaskResponse create(TaskForm form) {
        if (form.getAssigneeId() != null) {
            if (userMapper.findById(form.getAssigneeId()) == null) {
                throw new ResourceNotFoundException(
                    "担当者が見つかりません: " + form.getAssigneeId());
            }
        }
        Task task = new Task();
        task.setTitle(form.getTitle());
        task.setDescription(form.getDescription());
        task.setStatus("TODO");
        task.setPriority(
            form.getPriority() != null ? form.getPriority() : "MEDIUM");
        task.setDueDate(form.getDueDate());
        task.setAssigneeId(form.getAssigneeId());
        task.setCreatedAt(LocalDateTime.now());
        taskMapper.insert(task);
        return TaskResponse.from(task);
    }

    @Transactional
    public TaskResponse updateStatus(Long id, String newStatus) {
        Task task = taskMapper.findById(id);
        if (task == null) throw new ResourceNotFoundException(
            "Task not found: " + id);
        Set<String> allowed = TRANSITIONS.getOrDefault(
            task.getStatus(), Set.of());
        if (!allowed.contains(newStatus)) {
            throw new BusinessRuleException(
                task.getStatus() + " → " + newStatus
                + " への遷移はできません");
        }
        taskMapper.updateStatus(id, newStatus);
        return TaskResponse.from(taskMapper.findById(id));
    }
}

// GlobalExceptionHandler.java（抜粋）
@ExceptionHandler(BusinessRuleException.class)
public ResponseEntity<ErrorResponse> handleBusinessRule(
        BusinessRuleException ex, HttpServletRequest req) {
    ErrorResponse body = new ErrorResponse(
        422, "Unprocessable Entity",
        ex.getMessage(), req.getRequestURI());
    return ResponseEntity.status(422).body(body);
}
解説
  • 状態遷移マップで「DONE→TODOは不可」等のビジネスルールを宣言的に管理しています。新しい状態を追加する際もマップを更新するだけです。
  • @FutureOrPresent は Bean Validation 標準のアノテーションで、今日以降の日付のみを受け付けます。
  • 担当者の存在チェックはService層で行い、「誰が見つからないか」を明示するエラーメッセージを返しています。
  • 422 Unprocessable Entity は「構文は正しいが業務ルール違反」を表すHTTPステータスで、400 Bad Requestと使い分けます。
