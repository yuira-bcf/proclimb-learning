Section 5-1

イベント駆動の考え方

イベント駆動とは、「何かが起きた」というイベント（出来事）を通知し、それに反応する処理を実行するアーキテクチャスタイルです。Observerパターンは、この考え方の基盤となるパターンです。

5-1-1 直接呼び出しの問題

注文確定時に「在庫更新」「メール送信」「ポイント付与」を行う場合を考えましょう。直接呼び出しでは、処理が増えるたびにOrderServiceの修正が必要です。

直接呼び出しの問題 
@Service
public class OrderService {
    private final StockService stockService;
    private final MailService mailService;
    private final PointService pointService;
    // 処理が増えるたびにフィールドと依存が増える...

    public void confirmOrder(Order order) {
        order.confirm();
        stockService.decreaseStock(order);    // 在庫更新
        mailService.sendConfirmMail(order);   // メール送信
        pointService.addPoints(order);        // ポイント付与
        // ログ記録、分析データ送信...まだまだ増える
    }
}

error 問題点

  • OrderServiceが多数のサービスに依存し、肥大化する
  • 後続処理を追加するたびにOrderServiceを修正する必要がある
  • 後続処理の1つが失敗すると、全体に影響する

5-1-2 イベント駆動のメリット

イベント駆動にすると、OrderServiceは「注文が確定した」というイベントを発行するだけで、後続処理の追加や変更を気にしなくてよくなります。

Section 5-2

Observer（Spring Event）

Spring BootにはApplicationEventPublisherと@EventListenerという仕組みが標準搭載されており、Observerパターンをシンプルに実装できます。

5-2-1 イベントクラスの定義

OrderConfirmedEvent.java 
public class OrderConfirmedEvent {

    private final Long orderId;
    private final String customerEmail;
    private final int totalAmount;
    private final LocalDateTime confirmedAt;

    public OrderConfirmedEvent(Long orderId, String customerEmail,
                                int totalAmount) {
        this.orderId = orderId;
        this.customerEmail = customerEmail;
        this.totalAmount = totalAmount;
        this.confirmedAt = LocalDateTime.now();
    }

    // getter省略
}

5-2-2 イベントの発行（Publisher）

OrderService.java（イベント発行版） 
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final ApplicationEventPublisher eventPublisher;

    public OrderService(OrderRepository orderRepository,
                        ApplicationEventPublisher eventPublisher) {
        this.orderRepository = orderRepository;
        this.eventPublisher = eventPublisher;
    }

    @Transactional
    public void confirmOrder(Long orderId) {
        Order order = orderRepository.findById(orderId)
                .orElseThrow(() -> new RuntimeException("注文が見つかりません"));

        order.confirm();
        orderRepository.save(order);

        // イベントを発行するだけ（後続処理を知らなくてよい）
        eventPublisher.publishEvent(
            new OrderConfirmedEvent(
                order.getId(),
                order.getCustomerEmail(),
                order.getTotalAmount()
            )
        );
    }
}

5-2-3 イベントリスナーの実装

各種EventListener 
@Component
public class StockEventListener {

    private final StockService stockService;

    public StockEventListener(StockService stockService) {
        this.stockService = stockService;
    }

    @EventListener
    public void handleOrderConfirmed(OrderConfirmedEvent event) {
        stockService.decreaseStock(event.getOrderId());
        System.out.println("在庫を更新しました: 注文ID=" + event.getOrderId());
    }
}

@Component
public class MailEventListener {

    private final MailService mailService;

    public MailEventListener(MailService mailService) {
        this.mailService = mailService;
    }

    @EventListener
    public void handleOrderConfirmed(OrderConfirmedEvent event) {
        mailService.sendConfirmationMail(
            event.getCustomerEmail(),
            event.getOrderId()
        );
    }
}

@Component
public class PointEventListener {

    @EventListener
    public void handleOrderConfirmed(OrderConfirmedEvent event) {
        int points = (int)(event.getTotalAmount() * 0.01);
        System.out.println("ポイント付与: " + points + "pt");
    }
}

check_circle Spring Eventのポイント

  • 新しい後続処理を追加するときは、新しい@EventListenerクラスを作るだけ
  • OrderServiceの修正は一切不要（Open-Closed原則）
  • @Asyncを併用すると非同期で実行できる
Section 5-3

Commandパターン

Commandパターンは、操作（リクエスト）をオブジェクトとしてカプセル化するパターンです。操作の記録、取り消し（Undo）、キューイングなどが実現できます。

5-3-1 Commandパターンの構造

schema Commandパターンの構成要素

Command

操作をインターフェースとして定義
execute() / undo()

ConcreteCommand

具体的な操作の実装
実行に必要なデータを保持

Invoker

Commandを受け取り実行
履歴の管理も担当

5-3-2 Commandの実装例（在庫操作）

StockCommand.java / IncreaseStockCommand.java 
// Commandインターフェース
public interface StockCommand {
    void execute();
    void undo();
    String getDescription();
}

// 在庫追加コマンド
public class IncreaseStockCommand implements StockCommand {
    private final Product product;
    private final int quantity;

    public IncreaseStockCommand(Product product, int quantity) {
        this.product = product;
        this.quantity = quantity;
    }

    @Override
    public void execute() {
        product.setStock(product.getStock() + quantity);
        System.out.println(product.getName() + ": +" + quantity + "個 追加");
    }

    @Override
    public void undo() {
        product.setStock(product.getStock() - quantity);
        System.out.println(product.getName() + ": " + quantity + "個 取消");
    }

    @Override
    public String getDescription() {
        return product.getName() + " に " + quantity + " 個追加";
    }
}

// 在庫削減コマンド
public class DecreaseStockCommand implements StockCommand {
    private final Product product;
    private final int quantity;

    public DecreaseStockCommand(Product product, int quantity) {
        this.product = product;
        this.quantity = quantity;
    }

    @Override
    public void execute() {
        product.setStock(product.getStock() - quantity);
    }

    @Override
    public void undo() {
        product.setStock(product.getStock() + quantity);
    }

    @Override
    public String getDescription() {
        return product.getName() + " から " + quantity + " 個削減";
    }
}

5-3-3 Invoker（履歴管理付き）

StockCommandInvoker.java 
public class StockCommandInvoker {

    private final Deque<StockCommand> history = new ArrayDeque<>();

    public void executeCommand(StockCommand command) {
        command.execute();
        history.push(command);  // 履歴に追加
    }

    public void undoLast() {
        if (history.isEmpty()) {
            System.out.println("取り消す操作がありません");
            return;
        }
        StockCommand lastCommand = history.pop();
        lastCommand.undo();
        System.out.println("取消: " + lastCommand.getDescription());
    }

    public void printHistory() {
        System.out.println("=== 操作履歴 ===");
        history.forEach(cmd -> System.out.println("  " + cmd.getDescription()));
    }
}

// 使用例
StockCommandInvoker invoker = new StockCommandInvoker();
invoker.executeCommand(new IncreaseStockCommand(productA, 100));
invoker.executeCommand(new DecreaseStockCommand(productA, 30));
invoker.undoLast();  // 30個削減を取り消し → 在庫が元に戻る
Exercise

5章 練習問題

Q1

Spring Eventを使ったObserverパターンにおいて、新しい後続処理（例: 注文確定時のSlack通知）を追加するために必要な作業を選んでください。

A. OrderServiceにSlackServiceの依存を追加し、confirmOrderメソッドを修正する
B. 新しい@EventListenerクラスを作成するだけでよい
C. OrderConfirmedEventクラスにSlack通知用フィールドを追加する
D. ApplicationEventPublisherを再設定する

Q2

Commandパターンで undo() メソッドを実装するために、ConcreteCommandクラスが保持すべき情報は何ですか。

Q3

以下の要件では、Observer/Commandのどちらが適切ですか。理由も述べてください。

「管理画面での操作（商品追加・価格変更・在庫修正）を履歴として記録し、誤操作時に1つずつ元に戻したい。」

Answer

5章 解説・解答

Q1 解答

正解: B. 新しい@EventListenerクラスを作成するだけでよい

これがObserverパターン（Spring Event）の最大のメリットです。イベント発行側（OrderService）を修正せず、新しいリスナーを追加するだけで後続処理を拡張できます。Open-Closed原則に従った設計です。

Q2 解答

undo()を実装するためには、操作対象と操作内容（実行前の状態または操作の逆を算出するための情報）を保持する必要があります。例えば在庫追加コマンドなら「対象の商品（Product）」と「追加した数量（quantity）」を保持します。undo時は保持した数量を逆方向に適用（追加した分を減らす）することで元に戻せます。

Q3 解答

Commandパターンが適切。

この要件のポイントは「操作の履歴記録」と「元に戻す（Undo）」です。Commandパターンは操作をオブジェクトとしてカプセル化し、履歴管理とUndoを自然に実現できます。Observerパターンはイベント通知には適していますが、操作の取り消し機能は持っていません。
