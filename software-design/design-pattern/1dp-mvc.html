Section 1-1

なぜパターンを学ぶのか

デザインパターンとは、ソフトウェア開発で繰り返し現れる設計上の課題に対する「定番の解決策」です。パターンを知っていると、コードの見通しが良くなり、チームでの開発やAIプロンプトの精度が大幅に向上します。

1-1-1 デザインパターンとは

デザインパターンは1994年にGoF（Gang of Four）が書籍『Design Patterns』で23のパターンを体系化したことで広く知られるようになりました。しかし、本コースではGoFの全パターンを暗記するのではなく、Spring Boot開発で実際に頻出するパターンに絞って学びます。

lightbulb 本コースで扱うパターン

  • 構造系：MVC、Repository、DTO、Facade、Decorator
  • 生成系：Factory Method、Builder
  • 振る舞い系：Strategy、State、Observer、Command、Template Method
  • Spring固有：DI/IoC、AOP、Front Controller

1-1-2 パターン名がプロンプトの精度を上げる

AIにコード生成を依頼する際、パターン名を知っているかどうかで出力品質に大きな差が出ます。以下はプロンプトの比較例です。

compare プロンプト比較: パターン名なし vs パターン名あり

パターン名なし

"ユーザー情報を取得してHTMLに表示するSpring Bootのコードを書いて"

結果: 1ファイルにDB接続・ビジネスロジック・表示処理が混在した「神クラス」が生成される

パターン名あり

"MVC + 3層アーキテクチャで、Controller→Service→Repositoryに分離したユーザー一覧機能を作って"

結果: 各層が適切に分離された保守性の高いコードが生成される

Section 1-2

MVCの3つの役割

MVCはModel-View-Controllerの略で、アプリケーションを3つの役割に分離するアーキテクチャパターンです。Spring Bootでは、さらにService層とRepository層を加えた「3層アーキテクチャ」として実装します。

1-2-1 Model / View / Controllerとは

MVCの各要素には明確な責務があります。

1-2-2 3層アーキテクチャとの関係

Spring Bootの実プロジェクトでは、MVCのModel部分をさらにService層とRepository層に分離し、3層アーキテクチャとして構成します。

layers 3層アーキテクチャの全体像

1-2-3 各層の責務ルール

3層アーキテクチャでは「上位層は下位層のみを呼び出す」という一方向の依存ルールがあります。

warning 守るべきルール

  • ControllerはServiceを呼ぶ。直接Repositoryを呼んではいけない。
  • ServiceはRepositoryを呼ぶ。Controllerの情報（HttpServletRequestなど）に依存しない。
  • RepositoryはDBアクセスのみ。ビジネスロジックを持たない。
Section 1-3

Spring BootでのMVC実装

ここでは「ユーザー一覧を表示する」機能を題材に、Spring BootでのMVC + 3層アーキテクチャの実装を見ていきましょう。

1-3-1 Controllerの実装

Controllerはブラウザからのリクエストを受け取り、Serviceを呼び出して結果をViewに渡します。@Controllerアノテーションと@GetMappingでURLマッピングを行います。

UserController.java 
@Controller
@RequestMapping("/users")
public class UserController {

    private final UserService userService;

    // コンストラクタインジェクション
    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping
    public String list(Model model) {
        // Serviceからデータを取得してModelに詰める
        List<UserDto> users = userService.findAll();
        model.addAttribute("users", users);
        return "user/list";  // Viewのテンプレート名
    }
}

1-3-2 Serviceの実装

Service層はビジネスロジックを担当します。Controllerから呼ばれ、必要に応じてRepositoryにデータの取得や保存を依頼します。

UserService.java 
@Service
public class UserService {

    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public List<UserDto> findAll() {
        // RepositoryからEntityを取得
        List<User> entities = userRepository.findAll();

        // EntityをDTOに変換して返す
        return entities.stream()
                .map(e -> new UserDto(e.getId(), e.getName(), e.getEmail()))
                .collect(Collectors.toList());
    }
}

1-3-3 Repositoryの実装

Repository層はデータベースとのやり取りを担当します。Spring Data JPAを使えば、インターフェースを定義するだけで基本的なCRUD操作が自動生成されます。

UserRepository.java 
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    // JpaRepositoryを継承するだけで
    // findAll(), findById(), save(), deleteById() などが使える

    // カスタムクエリの追加も可能
    List<User> findByNameContaining(String keyword);
}

1-3-4 ファイル構成のベストプラクティス

Spring Bootプロジェクトでは、以下のようなパッケージ構成が推奨されます。

パッケージ構成例

├── controller/
│   └── UserController.java     ← リクエスト受付
├── service/

Exercise

1章 練習問題

Q1

Spring Bootの3層アーキテクチャにおいて、ビジネスロジック（例：税込価格の計算）を記述すべき層はどれですか。

A. Controller層   B. Service層   C. Repository層   D. View層

Q2

以下のコードは3層アーキテクチャのルールに違反しています。どの部分が問題で、どのように修正すべきか説明してください。

@Controller
public class OrderController {
    @Autowired
    private OrderRepository orderRepository; // 直接Repositoryを利用

    @GetMapping("/orders")
    public String list(Model model) {
        model.addAttribute("orders", orderRepository.findAll());
        return "order/list";
    }
}
Q3

MVCパターンにおけるControllerの役割として適切でないものを選んでください。

A. ブラウザからのリクエストを受け取る
B. 表示するViewテンプレートを決定する
C. データベースに直接SQLを発行する
D. Modelにデータを詰めてViewに渡す

Answer

1章 解説・解答

Q1 解答

正解: B. Service層

ビジネスロジック（税込計算、割引適用、在庫チェックなど）はService層に記述します。Controller層はリクエストの受付と振り分けのみ、Repository層はデータベースアクセスのみを担当します。

Q2 解答

問題点：ControllerがRepositoryを直接呼んでいる

3層アーキテクチャのルールでは、ControllerはServiceのみを呼び出し、Repositoryには直接アクセスしません。正しい実装は以下のとおりです。

@Controller
public class OrderController {
    private final OrderService orderService; // Serviceを注入

    public OrderController(OrderService orderService) {
        this.orderService = orderService;
    }

    @GetMapping("/orders")
    public String list(Model model) {
        model.addAttribute("orders", orderService.findAll());
        return "order/list";
    }
}
Q3 解答

正解: C. データベースに直接SQLを発行する

データベースへの直接アクセスはRepository層の責務です。Controllerはリクエストの受付（A）、Viewの決定（B）、Modelへのデータ受け渡し（D）を担当しますが、データベースに直接触れてはいけません。
