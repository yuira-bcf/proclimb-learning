Section 4-1

if-elseの肥大化問題

条件分岐が増えるとコードが読みにくくなり、バグの温床になります。StrategyパターンとStateパターンは、if-else/switchの肥大化を解消する定番のテクニックです。

4-1-1 よくある肥大化パターン

ECサイトの割引計算を例に見てみましょう。会員種別によって割引率が異なります。

肥大化したif-else文 
public int calculateDiscount(String memberType, int totalPrice) {
    if (memberType.equals("normal")) {
        // 通常会員: 割引なし
        return totalPrice;
    } else if (memberType.equals("silver")) {
        // シルバー会員: 5%割引
        return (int)(totalPrice * 0.95);
    } else if (memberType.equals("gold")) {
        // ゴールド会員: 10%割引 + ポイント2倍
        int discounted = (int)(totalPrice * 0.90);
        addBonusPoints(totalPrice, 2);
        return discounted;
    } else if (memberType.equals("platinum")) {
        // プラチナ会員: 15%割引 + ポイント3倍 + 送料無料
        int discounted = (int)(totalPrice * 0.85);
        addBonusPoints(totalPrice, 3);
        applyFreeShipping();
        return discounted;
    }
    // セール時の特別割引を追加すると...さらに膨らむ
    return totalPrice;
}

warning 問題点

  • 会員種別が増えるたびにメソッドが肥大化する
  • 各分岐のロジックが密結合でテストしにくい
  • 「割引率」「ポイント倍率」「送料ルール」が1箇所に混在している

4-1-2 StrategyとStateの使い分け

どちらもif-elseを解消するパターンですが、使い分けの基準があります。

compare Strategy vs State

Strategy

アルゴリズムを外部から選択して切り替える

例: 割引計算方法を会員種別で選ぶ、ソートアルゴリズムを選ぶ

State

オブジェクトの内部状態に応じて振る舞いが自動で変わる

例: 注文ステータス（新規→確定→発送→完了）で許可される操作が変わる

Section 4-2

Strategyパターン

Strategyパターンは、アルゴリズムをインターフェースとして切り出し、実行時に差し替え可能にするパターンです。割引計算の例でStrategyを適用してみましょう。

4-2-1 インターフェースの定義

DiscountStrategy.java 
public interface DiscountStrategy {
    int applyDiscount(int totalPrice);
    String getMemberType();
}

4-2-2 各Strategyの実装

各会員種別の割引Strategy 
@Component("normal")
public class NormalDiscount implements DiscountStrategy {
    @Override
    public int applyDiscount(int totalPrice) {
        return totalPrice;  // 割引なし
    }
    @Override
    public String getMemberType() { return "normal"; }
}

@Component("silver")
public class SilverDiscount implements DiscountStrategy {
    @Override
    public int applyDiscount(int totalPrice) {
        return (int)(totalPrice * 0.95);  // 5%割引
    }
    @Override
    public String getMemberType() { return "silver"; }
}

@Component("gold")
public class GoldDiscount implements DiscountStrategy {
    @Override
    public int applyDiscount(int totalPrice) {
        return (int)(totalPrice * 0.90);  // 10%割引
    }
    @Override
    public String getMemberType() { return "gold"; }
}

4-2-3 Serviceでの利用

PricingService.java 
@Service
public class PricingService {

    private final Map<String, DiscountStrategy> strategies;

    public PricingService(Map<String, DiscountStrategy> strategies) {
        this.strategies = strategies;
    }

    public int calculateFinalPrice(String memberType, int totalPrice) {
        DiscountStrategy strategy = strategies.get(memberType);
        if (strategy == null) {
            throw new IllegalArgumentException("未対応の会員種別: " + memberType);
        }
        return strategy.applyDiscount(totalPrice);
    }
}

check_circle Strategyパターンのメリット

  • 新しい会員種別を追加する際、既存コードを修正せず新しいクラスを追加するだけ
  • 各割引ロジックを個別にテストできる
  • if-else が完全に消える
Section 4-3

Stateパターン

Stateパターンは、オブジェクトの「状態」ごとにクラスを作り、状態に応じた振る舞いの変化をif文なしで実現するパターンです。注文ステータスの管理を例に学びましょう。

4-3-1 状態遷移図

timeline 注文ステータスの状態遷移

NEW
新規注文
→ confirm()
CONFIRMED
確定済み
→ ship()
SHIPPED
発送済み
→ deliver()
DELIVERED
配達完了

各状態で許可される操作が異なる（例: SHIPPED状態でconfirm()は不可）

4-3-2 Stateインターフェースと実装

OrderState.java（インターフェース） 
public interface OrderState {
    OrderState confirm();
    OrderState ship();
    OrderState deliver();
    OrderState cancel();
    String getStatusName();
}
NewOrderState.java / ConfirmedOrderState.java 
public class NewOrderState implements OrderState {
    @Override
    public OrderState confirm() {
        System.out.println("注文を確定しました");
        return new ConfirmedOrderState();  // 次の状態に遷移
    }

    @Override
    public OrderState ship() {
        throw new IllegalStateException("未確定の注文は発送できません");
    }

    @Override
    public OrderState deliver() {
        throw new IllegalStateException("未確定の注文は配達完了にできません");
    }

    @Override
    public OrderState cancel() {
        System.out.println("注文をキャンセルしました");
        return new CancelledOrderState();
    }

    @Override
    public String getStatusName() { return "NEW"; }
}

public class ConfirmedOrderState implements OrderState {
    @Override
    public OrderState confirm() {
        throw new IllegalStateException("既に確定済みです");
    }

    @Override
    public OrderState ship() {
        System.out.println("注文を発送しました");
        return new ShippedOrderState();
    }

    @Override
    public OrderState deliver() {
        throw new IllegalStateException("発送前に配達完了にはできません");
    }

    @Override
    public OrderState cancel() {
        System.out.println("確定済み注文をキャンセルしました");
        return new CancelledOrderState();
    }

    @Override
    public String getStatusName() { return "CONFIRMED"; }
}

4-3-3 コンテキストクラス

注文（Order）は内部にStateオブジェクトを保持し、操作を委譲します。

Order.java（コンテキスト） 
public class Order {
    private OrderState state;

    public Order() {
        this.state = new NewOrderState();  // 初期状態
    }

    public void confirm() {
        this.state = state.confirm();  // 状態遷移を委譲
    }

    public void ship() {
        this.state = state.ship();
    }

    public void deliver() {
        this.state = state.deliver();
    }

    public void cancel() {
        this.state = state.cancel();
    }

    public String getStatus() {
        return state.getStatusName();
    }
}

// 使用例
Order order = new Order();          // NEW
order.confirm();                     // NEW → CONFIRMED
order.ship();                        // CONFIRMED → SHIPPED
order.deliver();                     // SHIPPED → DELIVERED
Exercise

4章 練習問題

Q1

Strategyパターンを適用すべき場面として最も適切なものはどれですか。

A. オブジェクトの内部状態に応じて振る舞いを自動的に変えたい
B. 同じ処理に対して複数のアルゴリズムを外部から選択して切り替えたい
C. オブジェクトの生成を専用クラスに委譲したい
D. イベント発生時に複数のリスナーに通知したい

Q2

Stateパターンで「SHIPPED（発送済み）」の状態のときにconfirm()メソッドを呼ぶと何が起こるべきですか。その理由も説明してください。

Q3

以下の要件では、StrategyとStateのどちらが適切ですか。理由も述べてください。

「ブログ記事には「下書き」「公開中」「非公開」のステータスがあり、ステータスによって編集・削除・公開の可否が変わる。」

Answer

4章 解説・解答

Q1 解答

正解: B. 同じ処理に対して複数のアルゴリズムを外部から選択して切り替えたい

Strategyパターンは「どのアルゴリズムを使うか」を利用側が選択するパターンです。Aは State、CはFactory、DはObserverパターンの場面です。

Q2 解答

IllegalStateExceptionがスローされるべき。

SHIPPED状態でconfirm()を呼ぶのは不正な操作です。すでに確定・発送済みの注文を再度「確定」することは業務ロジックとして矛盾しています。Stateパターンでは各状態クラスが自身で許可されない操作を拒否（例外をスロー）することで、不正な状態遷移を防ぎます。

Q3 解答

Stateパターンが適切。

ブログ記事のステータスは「内部状態」であり、状態に応じて許可される操作が自動的に変わります（例：「下書き」では削除可能だが「公開中」では直接削除不可など）。これは「外部からアルゴリズムを選択する」Strategyではなく、「内部状態に応じて振る舞いが変わる」Stateパターンの典型的な適用場面です。
