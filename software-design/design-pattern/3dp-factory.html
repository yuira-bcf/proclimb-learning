Section 3-1

オブジェクト生成の課題

オブジェクトの生成をnewで直接行うと、クラス間の結合度が上がり、条件分岐が複雑化します。Factoryパターンは、オブジェクトの生成を専用クラスに委譲することで、この問題を解決します。

3-1-1 newの直接利用が引き起こす問題

通知機能を例に考えてみましょう。メール、SMS、プッシュ通知の3種類があり、種類に応じて異なるクラスを生成する必要があります。

問題のあるコード（newの直接利用） 
// 通知を送る箇所ごとにif文が必要になる
public void sendNotification(String type, String message) {
    if (type.equals("email")) {
        EmailNotification notification = new EmailNotification();
        notification.send(message);
    } else if (type.equals("sms")) {
        SmsNotification notification = new SmsNotification();
        notification.send(message);
    } else if (type.equals("push")) {
        PushNotification notification = new PushNotification();
        notification.send(message);
    }
    // 新しい通知手段を追加するたびに、ここに分岐を追加...
}

error この設計の問題点

  • 通知を送る箇所が複数あると、同じif-else分岐がコピペされる
  • 新しい通知手段を追加するたびに、全ての箇所を修正する必要がある
  • 変更漏れによるバグが発生しやすい

3-1-2 Factoryで解決するイメージ

Factoryパターンでは、「どのオブジェクトを作るか」の判断を専用のFactoryクラスに集約します。利用側はFactory.create(type)と呼ぶだけで、具体的なクラスを知らなくてよくなります。

Section 3-2

Factory Methodパターン

Factory Methodパターンは、オブジェクトの生成をサブクラスやメソッドに委譲するパターンです。共通のインターフェースを通じてオブジェクトを受け取るため、利用側のコードは具象クラスに依存しません。

3-2-1 クラス構造

schema Factory Methodパターンのクラス図

3-2-2 インターフェースの定義

まず、通知の共通インターフェースを定義します。全ての通知クラスはこのインターフェースを実装します。

Notification.java（インターフェース） 
public interface Notification {
    void send(String to, String message);
    String getType();
}

3-2-3 具象クラスの実装

EmailNotification.java / SmsNotification.java 
public class EmailNotification implements Notification {
    @Override
    public void send(String to, String message) {
        // メール送信ロジック
        System.out.println("メール送信: " + to + " - " + message);
    }

    @Override
    public String getType() {
        return "email";
    }
}

public class SmsNotification implements Notification {
    @Override
    public void send(String to, String message) {
        // SMS送信ロジック
        System.out.println("SMS送信: " + to + " - " + message);
    }

    @Override
    public String getType() {
        return "sms";
    }
}

3-2-4 Factoryクラスの実装

NotificationFactory.java 
public class NotificationFactory {

    public static Notification create(String type) {
        switch (type) {
            case "email":
                return new EmailNotification();
            case "sms":
                return new SmsNotification();
            case "push":
                return new PushNotification();
            default:
                throw new IllegalArgumentException("未対応の通知タイプ: " + type);
        }
    }
}

これで利用側のコードは非常にシンプルになります。

利用側のコード 
// Factoryを使えば、利用側は具象クラスを知らなくてよい
Notification notification = NotificationFactory.create("email");
notification.send("user@example.com", "注文完了のお知らせ");
Section 3-3

Spring Bootでの活用例

Spring BootではDIコンテナを活用することで、さらに柔軟なFactoryパターンを実現できます。インターフェースの実装クラスを全てBean登録しておき、Mapでまとめてインジェクションするテクニックがよく使われます。

3-3-1 SpringのDIを活用したFactory

各通知クラスに@Componentを付けてBeanとして登録し、FactoryでMapを使って自動インジェクションします。

Spring Boot版 NotificationFactory 
// 各通知クラスに@Componentを付ける
@Component("email")
public class EmailNotification implements Notification { ... }

@Component("sms")
public class SmsNotification implements Notification { ... }

@Component("push")
public class PushNotification implements Notification { ... }

// FactoryでMapインジェクション
@Component
public class NotificationFactory {

    private final Map<String, Notification> notificationMap;

    // Springが自動的にBean名をキーにMapを生成する
    public NotificationFactory(Map<String, Notification> notificationMap) {
        this.notificationMap = notificationMap;
    }

    public Notification create(String type) {
        Notification notification = notificationMap.get(type);
        if (notification == null) {
            throw new IllegalArgumentException("未対応の通知タイプ: " + type);
        }
        return notification;
    }
}

check_circle Spring版Factoryのメリット

  • 新しい通知クラスを追加しても、Factoryの修正が不要（@Componentを付けるだけで自動登録）
  • switch文が不要になり、Open-Closed原則（拡張に開き、修正に閉じる）に従える
  • 各通知クラスに他のBeanを注入できる（DBアクセスや外部API連携が可能）

3-3-2 ServiceからFactoryを利用する

NotificationService.java 
@Service
public class NotificationService {

    private final NotificationFactory factory;

    public NotificationService(NotificationFactory factory) {
        this.factory = factory;
    }

    public void notify(String type, String to, String message) {
        // Factoryから適切な通知オブジェクトを取得
        Notification notification = factory.create(type);
        notification.send(to, message);
    }
}
Exercise

3章 練習問題

Q1

Factoryパターンの目的として最も適切なものを選んでください。

A. オブジェクトの継承関係を定義する
B. オブジェクトの生成を専用クラスに委譲し、利用側と具象クラスの結合を下げる
C. データベースへのアクセスを抽象化する
D. アルゴリズムを実行時に切り替える

Q2

Spring BootでFactoryパターンを実装する際、Map<String, Notification>をコンストラクタインジェクションすると、MapのキーとBeanの対応はどのように決まりますか。

Q3

以下の要件に対して、Factoryパターンを適用する設計を簡潔に説明してください。

「ECサイトで、支払い方法（クレジットカード、銀行振込、コンビニ払い）に応じた決済処理クラスを切り替えたい。」

Answer

3章 解説・解答

Q1 解答

正解: B. オブジェクトの生成を専用クラスに委譲し、利用側と具象クラスの結合を下げる

Factoryパターンは「生成」に関するパターンです。利用側は共通インターフェースを通じてオブジェクトを受け取るため、具象クラスの変更や追加に強い設計になります。

Q2 解答

MapのキーはBean名（@Componentの引数、またはクラス名の先頭小文字）になります。例えば @Component("email") と指定した場合はキーが "email" になり、引数を省略すると "emailNotification"（クラス名のキャメルケース）がキーになります。

Q3 解答

以下のように設計します。

  1. PaymentProcessor インターフェースを定義し、process(Order order) メソッドを持たせる
  2. CreditCardProcessor、BankTransferProcessor、ConveniProcessor の3つの具象クラスがインターフェースを実装
  3. PaymentFactory がString型の支払い方法キーを受け取り、対応する PaymentProcessor を返す
  4. Spring Bootなら各ProcessorにBean名付きの@Componentを付け、Factory側でMap<String, PaymentProcessor>をインジェクション
