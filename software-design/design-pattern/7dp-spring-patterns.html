Section 7-1

DI/IoC（制御の反転）

DI（Dependency Injection / 依存性の注入）は、Spring Frameworkの最も重要な基盤技術です。オブジェクトの生成と依存関係の解決をフレームワーク（DIコンテナ）に任せることで、疎結合な設計を実現します。

7-1-1 IoC（制御の反転）とは

通常のプログラミングでは、自分のコードがライブラリを呼び出します（呼び出し側が制御権を持つ）。IoCでは、フレームワークが自分のコードを呼び出します（制御権がフレームワークに反転する）。

compare 通常の制御 vs IoC

通常の制御

自分のコードがnewでオブジェクトを生成し、依存先を自分で管理する

UserService svc = new UserService(
    new UserRepository()
);
IoC（Spring DI）

DIコンテナがオブジェクトを生成し、依存関係を自動で注入してくれる

@Service // Springが管理
class UserService {
    // 自動注入される
    UserService(UserRepository repo) {}
}

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }
}
Section 7-2

AOP（@Transactional, @Aspect）

AOP（Aspect-Oriented Programming / アスペクト指向プログラミング）は、ログ出力、トランザクション管理、セキュリティチェックなど、複数のクラスに横断的に適用する処理を「アスペクト」として分離する技術です。

7-2-1 横断的関心事とは

ビジネスロジックの本質とは関係ないが、多くのメソッドで必要になる処理を「横断的関心事」と呼びます。

info 横断的関心事の例

  • トランザクション管理：DB操作の成功/失敗でcommit/rollback
  • ログ出力：メソッドの開始・終了・例外を記録
  • パフォーマンス計測：メソッドの実行時間を測定
  • 認証・認可：権限チェック

7-2-2 @Transactionalの仕組み

@TransactionalはSpring AOPの最も身近な利用例です。メソッドの開始時にトランザクションを開始し、正常終了でcommit、例外発生でrollbackします。

@Transactionalの利用 
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final StockRepository stockRepository;

    @Transactional  // このメソッド全体が1つのトランザクション
    public void placeOrder(OrderRequest request) {
        // 注文保存
        Order order = new Order(request);
        orderRepository.save(order);

        // 在庫更新（ここで例外が起きたら注文保存もrollback）
        stockRepository.decrease(request.getProductId(), request.getQuantity());
    }
    // メソッドが正常終了 → 自動でcommit
    // 例外が発生 → 自動でrollback
}

7-2-3 カスタムAspectの作成

自分でAspectを定義すれば、メソッドの実行前後に任意の処理を差し込めます。

LoggingAspect.java 
@Aspect
@Component
public class LoggingAspect {

    private static final Logger log = LoggerFactory.getLogger(LoggingAspect.class);

    // Serviceパッケージの全メソッドに適用
    @Around("execution(* com.example.app.service.*.*(..))")
    public Object logExecutionTime(ProceedingJoinPoint joinPoint) throws Throwable {
        String methodName = joinPoint.getSignature().getName();

        log.info("開始: {}", methodName);
        long start = System.currentTimeMillis();

        try {
            Object result = joinPoint.proceed();  // 本来のメソッドを実行
            long elapsed = System.currentTimeMillis() - start;
            log.info("完了: {} ({}ms)", methodName, elapsed);
            return result;
        } catch (Exception e) {
            log.error("エラー: {} - {}", methodName, e.getMessage());
            throw e;
        }
    }
}
Section 7-3

Template Method

Template Methodパターンは、処理の骨格（テンプレート）を親クラスに定義し、具体的な処理ステップをサブクラスに委ねるパターンです。Spring内部のJdbcTemplateやRestTemplateがこのパターンを使っています。

7-3-1 Template Methodの構造

データエクスポートのTemplate Method 
// 抽象クラスに処理の骨格を定義
public abstract class DataExporter {

    // テンプレートメソッド（処理の流れを固定）
    public final void export(List<Map<String, Object>> data) {
        String header = createHeader();
        String body = createBody(data);
        String footer = createFooter();
        writeOutput(header + body + footer);
    }

    // サブクラスが実装する抽象メソッド
    protected abstract String createHeader();
    protected abstract String createBody(List<Map<String, Object>> data);
    protected abstract String createFooter();

    // 共通処理（オーバーライド可能）
    protected void writeOutput(String content) {
        System.out.println(content);
    }
}

// CSV出力
public class CsvExporter extends DataExporter {
    @Override
    protected String createHeader() { return "id,name,price\n"; }

    @Override
    protected String createBody(List<Map<String, Object>> data) {
        StringBuilder sb = new StringBuilder();
        for (Map<String, Object> row : data) {
            sb.append(row.get("id")).append(",")
              .append(row.get("name")).append(",")
              .append(row.get("price")).append("\n");
        }
        return sb.toString();
    }

    @Override
    protected String createFooter() { return ""; }
}

// JSON出力
public class JsonExporter extends DataExporter {
    @Override
    protected String createHeader() { return "[\n"; }

    @Override
    protected String createBody(List<Map<String, Object>> data) {
        // JSON変換ロジック
        return new ObjectMapper().writeValueAsString(data);
    }

    @Override
    protected String createFooter() { return "\n]"; }
}
Section 7-4

Front Controller（DispatcherServlet）

Front Controllerパターンは、全てのリクエストを1つの入口（エントリポイント）で受け取り、適切なハンドラに振り分けるパターンです。Spring MVCではDispatcherServletがこの役割を担います。

7-4-2 Front Controllerのメリット

check_circle Front Controller（DispatcherServlet）のメリット

  • 共通処理の一元化：認証、ログ、エラーハンドリングを1箇所で管理
  • URLルーティングの統一：HandlerMappingで@RequestMappingのURL解決を一元管理
  • フィルターの適用：CharacterEncodingFilter, CorsFilter等を全リクエストに適用
  • View技術の切り替え：ViewResolverを変えるだけでThymeleaf→JSP等に切替可能

7章 練習問題

Q1

Spring Bootでコンストラクタインジェクションが推奨される理由を2つ挙げてください。

Q2

@Transactionalをメソッドに付けると、内部でどのような処理が自動で行われますか。正常時と例外発生時のそれぞれの動作を説明してください。

Q3

DispatcherServletが担う「Front Controller」パターンの主な役割として誤っているものを選んでください。

A. 全てのHTTPリクエストを一元的に受け取る
B. HandlerMappingを使って適切なControllerを検索する
C. ビジネスロジックの実行を直接行う
D. ViewResolverを使ってレスポンスのView名を解決する

Answer

7章 解説・解答

Q1 解答

1. 不変性の保証：コンストラクタで注入されたフィールドをfinalにできるため、実行中に依存先が差し替わるバグを防げます。

2. テストの容易さ：コンストラクタの引数でモックオブジェクトを渡すだけで単体テストが書けます。@Autowiredフィールドインジェクションではリフレクションが必要になります。

Q2 解答

正常時：メソッドの開始前にトランザクションが開始（begin）され、メソッドが例外なく完了すると自動でコミット（commit）されます。

例外発生時：RuntimeException（非チェック例外）が発生すると、トランザクション全体がロールバック（rollback）されます。メソッド内で行ったDB操作は全て取り消されます。

Q3 解答

正解: C. ビジネスロジックの実行を直接行う

DispatcherServlet（Front Controller）はリクエストの受付・振り分け・View解決を行いますが、ビジネスロジックは実行しません。ビジネスロジックの実行はControllerから呼ばれるService層の責務です。Front Controllerは「交通整理役」であり、実際の業務処理は行いません。
