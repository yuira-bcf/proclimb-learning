Section 8-1

課題の概要

この章では、ECサイトの注文処理を題材に、これまで学んだデザインパターンを組み合わせてリファクタリングする総合演習に取り組みます。

8-1-1 題材: ECサイトの注文処理

あるECサイトの注文処理には以下の機能があります。

  • 割引計算：会員種別（通常/シルバー/ゴールド）に応じた割引率を適用
  • 決済処理：支払い方法（クレジットカード/銀行振込/コンビニ払い）の切り替え
  • 通知送信：注文確定後にメール送信、在庫更新、ポイント付与を実行

8-1-2 リファクタリングの目標

flag 目標

  • 新しい会員種別や支払い方法を追加するとき、既存コードの修正が不要になること
  • 注文確定後の後続処理を、OrderServiceを修正せず追加できること
  • 各処理を個別にテストできること
Section 8-2

Before: パターン未適用コード

まず、パターンを適用していない「よくある初期実装」を見てみましょう。すべての処理がOrderServiceに集中しています。

8-2-1 全部入りのOrderService

OrderService.java（Before: パターン未適用） 
@Service
public class OrderService {

    @Autowired private OrderRepository orderRepository;
    @Autowired private StockRepository stockRepository;
    @Autowired private MailService mailService;
    @Autowired private PointService pointService;

    @Transactional
    public OrderResult placeOrder(String memberType, String paymentMethod,
                                   List<OrderItem> items) {
        // 1. 合計金額の計算
        int totalPrice = items.stream()
                .mapToInt(item -> item.getPrice() * item.getQuantity())
                .sum();

        // 2. 割引計算（if-else地獄）
        int discountedPrice;
        if (memberType.equals("normal")) {
            discountedPrice = totalPrice;
        } else if (memberType.equals("silver")) {
            discountedPrice = (int)(totalPrice * 0.95);
        } else if (memberType.equals("gold")) {
            discountedPrice = (int)(totalPrice * 0.90);
        } else {
            discountedPrice = totalPrice;
        }

        // 3. 決済処理（if-else地獄）
        if (paymentMethod.equals("credit")) {
            // クレジットカード決済ロジック...
            System.out.println("クレジットカード決済: " + discountedPrice);
        } else if (paymentMethod.equals("bank")) {
            // 銀行振込ロジック...
            System.out.println("銀行振込: " + discountedPrice);
        } else if (paymentMethod.equals("convenience")) {
            // コンビニ払いロジック...
            System.out.println("コンビニ払い: " + discountedPrice);
        }

        // 4. 注文保存
        Order order = new Order();
        order.setItems(items);
        order.setTotalAmount(discountedPrice);
        order.setPaymentMethod(paymentMethod);
        order.setStatus("CONFIRMED");
        orderRepository.save(order);

        // 5. 後続処理（直接呼び出し → 追加のたびに修正が必要）
        for (OrderItem item : items) {
            stockRepository.decreaseStock(item.getProductId(), item.getQuantity());
        }
        mailService.sendConfirmationMail(order);
        pointService.addPoints(order.getCustomerId(), discountedPrice);

        return new OrderResult(order.getId(), "注文完了", discountedPrice);
    }
}

error このコードの問題点

  • if-else地獄：割引計算と決済処理にif-else分岐が大量にある
  • 単一責任の原則違反：1つのServiceに割引・決済・通知・在庫管理が集中
  • 拡張性の欠如：新しい会員種別や支払い方法の追加時にOrderServiceを修正する必要がある
  • テスト困難：全機能が密結合で個別テストが不可能

8-4-2 決済処理のFactory + Strategy化

PaymentProcessor.java + Factory 
public interface PaymentProcessor {
    PaymentResult process(int amount, PaymentInfo info);
}

@Component("credit")
public class CreditCardProcessor implements PaymentProcessor {
    @Override
    public PaymentResult process(int amount, PaymentInfo info) {
        // クレジットカード決済の実装
        System.out.println("クレジットカード決済: ¥" + amount);
        return new PaymentResult(true, "CC-" + UUID.randomUUID());
    }
}

@Component("bank")
public class BankTransferProcessor implements PaymentProcessor {
    @Override
    public PaymentResult process(int amount, PaymentInfo info) {
        System.out.println("銀行振込: ¥" + amount);
        return new PaymentResult(true, "BK-" + UUID.randomUUID());
    }
}

@Component("convenience")
public class ConvenienceProcessor implements PaymentProcessor {
    @Override
    public PaymentResult process(int amount, PaymentInfo info) {
        System.out.println("コンビニ払い: ¥" + amount);
        return new PaymentResult(true, "CV-" + UUID.randomUUID());
    }
}

// Factory（MapインジェクションでOCP準拠）
@Component
public class PaymentFactory {
    private final Map<String, PaymentProcessor> processors;

    public PaymentFactory(Map<String, PaymentProcessor> processors) {
        this.processors = processors;
    }

    public PaymentProcessor getProcessor(String method) {
        PaymentProcessor processor = processors.get(method);
        if (processor == null) {
            throw new IllegalArgumentException("未対応の決済方法: " + method);
        }
        return processor;
    }
}

8-4-3 後続処理のObserver化（Spring Event）

OrderConfirmedEvent.java + EventListeners 
// イベントクラス
public class OrderConfirmedEvent {
    private final Long orderId;
    private final Long customerId;
    private final int totalAmount;
    private final List<OrderItem> items;

    public OrderConfirmedEvent(Long orderId, Long customerId,
                                int totalAmount, List<OrderItem> items) {
        this.orderId = orderId;
        this.customerId = customerId;
        this.totalAmount = totalAmount;
        this.items = items;
    }
    // getter省略
}

// 在庫リスナー
@Component
public class StockEventListener {
    private final StockRepository stockRepository;

    public StockEventListener(StockRepository stockRepository) {
        this.stockRepository = stockRepository;
    }

    @EventListener
    public void onOrderConfirmed(OrderConfirmedEvent event) {
        for (OrderItem item : event.getItems()) {
            stockRepository.decreaseStock(item.getProductId(), item.getQuantity());
        }
    }
}

// メールリスナー

public class MailEventListener {
    private final MailService mailService;

8-4-4 リファクタリング後のOrderService

OrderService.java（After: パターン適用後） 
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final Map<String, DiscountStrategy> discountStrategies;
    private final PaymentFactory paymentFactory;
    private final ApplicationEventPublisher eventPublisher;

    public OrderService(OrderRepository orderRepository,
                        Map<String, DiscountStrategy> discountStrategies,
                        PaymentFactory paymentFactory,
                        ApplicationEventPublisher eventPublisher) {
        this.orderRepository = orderRepository;
        this.discountStrategies = discountStrategies;
        this.paymentFactory = paymentFactory;
        this.eventPublisher = eventPublisher;
    }

    @Transactional
    public OrderResult placeOrder(OrderRequest request) {
        // 1. 合計金額の計算
        int totalPrice = request.getItems().stream()
                .mapToInt(item -> item.getPrice() * item.getQuantity())
                .sum();

        // 2. 割引計算（Strategy）
        DiscountStrategy discount = discountStrategies.get(request.getMemberType());
        int discountedPrice = discount.applyDiscount(totalPrice);

        // 3. 決済処理（Factory + Strategy）
        PaymentProcessor processor = paymentFactory.getProcessor(
            request.getPaymentMethod()
        );
        PaymentResult payment = processor.process(discountedPrice,
            request.getPaymentInfo());

        // 4. 注文保存（Repository）
        Order order = Order.builder()
                .customerId(request.getCustomerId())
                .items(request.getItems())
                .totalAmount(discountedPrice)
                .paymentId(payment.getTransactionId())
                .status("CONFIRMED")
                .build();
        orderRepository.save(order);

        // 5. イベント発行（Observer）→ 後続処理は知らなくてよい
        eventPublisher.publishEvent(
            new OrderConfirmedEvent(
                order.getId(), request.getCustomerId(),
                discountedPrice, request.getItems()
            )
        );

        return new OrderResult(order.getId(), "注文完了", discountedPrice);
    }
}

check_circle リファクタリングの成果

  • if-else完全消滅：割引も決済もMapベースの動的ディスパッチに
  • Open-Closed原則：新規追加は新クラス作成のみ、既存コード修正なし
  • 疎結合：OrderServiceは後続処理（在庫・メール・ポイント）を知らない
  • テスト容易：各Strategy / Processor / Listener を個別にテスト可能

Exercise

8章 練習問題

Q1

以下の要件に対して、どのデザインパターンを適用すべきか選んでください。

「配送方法（通常配送/速達/コンビニ受取）に応じて配送料の計算ロジックが異なる。新しい配送方法の追加が頻繁にある。」

A. Observer   B. Factory   C. Strategy   D. Decorator

Q2

リファクタリング後のOrderServiceにおいて、ApplicationEventPublisherを使ってイベントを発行することの利点を2つ挙げてください。

Q3

以下の要件に対して、適用すべきパターンの組み合わせを選んでください。

「レポート生成システムで、出力形式（PDF/Excel/CSV）の切り替えと、レポート生成後の通知（メール/Slack）を行いたい。形式も通知先も今後追加される予定がある。」

A. Factory + Observer   B. Strategy + Decorator   C. Builder + Facade   D. State + Command

Answer

8章 解説・解答

Q1 解答

正解: C. Strategy

配送方法ごとに計算ロジックが異なり、外部から選択して切り替える場面はStrategyパターンが適切です。各配送方法のクラスがShippingStrategyインターフェースを実装し、MapインジェクションやFactoryと組み合わせて使います。新しい配送方法は新クラスを追加するだけで対応できます。

Q2 解答

1. 疎結合：OrderServiceは後続処理（在庫更新、メール送信、ポイント付与）のクラスを知らなくてよく、依存関係が切れます。OrderServiceのフィールドにStockServiceやMailServiceが不要になります。

2. 拡張性：新しい後続処理を追加する際、OrderServiceを修正する必要がありません。新しいEventListenerクラスを作成して@EventListenerメソッドを実装するだけで、自動的にイベントを受信できます（Open-Closed原則）。

Q3 解答

正解: A. Factory + Observer

Factory：出力形式（PDF/Excel/CSV）の切り替えにはFactoryパターンが適切です。ReportGeneratorFactoryが形式に応じたReportGeneratorを返します。新形式の追加はクラスを追加するだけで済みます。

Observer：レポート生成後の通知はObserver（Spring Event）が適切です。「レポート生成完了」イベントを発行し、各通知リスナー（メール、Slack）が独立して受信します。通知先の追加は新しいリスナーを作るだけで済みます。
