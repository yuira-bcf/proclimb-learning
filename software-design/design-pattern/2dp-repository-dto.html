Section 2-1

データアクセスの抽象化

アプリケーションが成長すると、データの取得方法や保存先が変わることがあります。Repositoryパターンは、データアクセスの詳細をビジネスロジックから隠蔽し、変更に強い設計を実現します。

2-1-1 なぜデータアクセスを抽象化するのか

ビジネスロジックの中にSQLやJDBC接続コードが直接書かれていると、以下の問題が発生します。

  • テストが困難：実際のDBに接続しないとテストできない
  • DB変更時の影響範囲が広い：MySQL→PostgreSQLの変更がアプリ全体に波及する
  • コードの重複：同じようなSQLが様々な場所にコピペされる

2-1-2 抽象化のイメージ

Repositoryパターンでは「何のデータが欲しいか」だけをServiceに伝え、「どうやって取得するか」はRepositoryに任せます。

info たとえ話

レストランでメニューを選ぶとき、お客さん（Service）は「カレーください」と注文するだけで、厨房（Repository）がどの鍋を使い、どの手順で調理するかは気にしません。これがデータアクセスの抽象化です。

Section 2-2

Repositoryパターン

Repositoryパターンは、データの永続化操作をインターフェースとして定義し、その実装をビジネスロジックから分離するパターンです。Spring Data JPAでは、このパターンが標準で組み込まれています。

2-2-1 Spring Data JPAのRepository

Spring Data JPAでは、JpaRepositoryを継承したインターフェースを定義するだけで、基本的なCRUD操作が自動的に提供されます。

ProductRepository.java 
@Repository
public interface ProductRepository extends JpaRepository<Product, Long> {

    // メソッド名からSQLが自動生成される
    List<Product> findByCategory(String category);

    List<Product> findByPriceGreaterThan(int price);

    // 複雑なクエリは@Queryで記述
    @Query("SELECT p FROM Product p WHERE p.stock > 0 AND p.category = :cat")
    List<Product> findAvailableByCategory(@Param("cat") String category);
}

2-2-3 Entity（エンティティ）の定義

Entityは、データベースのテーブルに1対1で対応するJavaクラスです。@Entityアノテーションを付け、各フィールドがテーブルのカラムに対応します。

Product.java（Entity） 
@Entity
@Table(name = "products")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 100)
    private String name;

    @Column(nullable = false)
    private int price;

    @Column(length = 50)
    private String category;

    private int stock;

    // コンストラクタ、Getter、Setter
    public Product() {}

    public Product(String name, int price, String category, int stock) {
        this.name = name;
        this.price = price;
        this.category = category;
        this.stock = stock;
    }

    // getter/setter省略
}
Section 2-3

DTOとEntityの使い分け

DTO（Data Transfer Object）は、画面やAPIに必要なデータだけを詰め替えるオブジェクトです。Entityをそのまま画面に渡すと、不要な情報の漏洩やセキュリティリスクが生じます。

2-3-1 なぜEntityをそのまま使わないのか

Entityをそのままフロントエンドに渡すと、以下の問題があります。

  • 情報漏洩：パスワードハッシュなど、表示不要なカラムも含まれる
  • 循環参照：JPA関連エンティティのJSON変換で無限ループが起きる
  • 画面仕様変更の影響：画面レイアウトの変更がEntityに波及する

2-3-2 DTOクラスの実装

DTOは画面に必要なフィールドだけを持つシンプルなクラスです。Entityのパスワードや内部ID、関連テーブルの情報をDTOには含めません。

ProductDto.java 
public class ProductDto {

    private Long id;
    private String name;
    private int price;
    private String category;
    private String priceDisplay;  // "¥1,200" のような表示用文字列

    public ProductDto(Long id, String name, int price, String category) {
        this.id = id;
        this.name = name;
        this.price = price;
        this.category = category;
        this.priceDisplay = String.format("¥%,d", price);
    }

    // getter省略
}

2-3-3 Entity→DTO変換メソッド

変換処理はService層に置くのが一般的です。専用のMapperクラスを作る方法もあります。

ProductService.java（変換処理） 
@Service
public class ProductService {

    private final ProductRepository productRepository;

    public ProductService(ProductRepository productRepository) {
        this.productRepository = productRepository;
    }

    public List<ProductDto> findAll() {
        return productRepository.findAll().stream()
                .map(this::toDto)
                .collect(Collectors.toList());
    }

    public ProductDto findById(Long id) {
        Product entity = productRepository.findById(id)
                .orElseThrow(() -> new RuntimeException("商品が見つかりません"));
        return toDto(entity);
    }

    // Entity → DTO 変換メソッド
    private ProductDto toDto(Product entity) {
        return new ProductDto(
            entity.getId(),
            entity.getName(),
            entity.getPrice(),
            entity.getCategory()
        );
    }
}

Exercise

2章 練習問題

Q1

Repositoryパターンを採用する最大のメリットとして最も適切なものを選んでください。

A. コード量が減る   B. データアクセスの詳細をビジネスロジックから隠蔽できる   C. SQLが不要になる   D. Entityが不要になる

Q2

Userエンティティに password（パスワードハッシュ）と loginCount（内部管理用ログイン回数）フィールドがあります。ユーザー一覧画面用のDTOにこれらのフィールドを含めるべきですか。理由とともに答えてください。

Q3

Entity→DTO の変換処理は、3層アーキテクチャのどの層に記述するのが適切ですか。

A. Controller層   B. Service層   C. Repository層   D. View層

Answer

2章 解説・解答

Q1 解答

正解: B. データアクセスの詳細をビジネスロジックから隠蔽できる

Repositoryパターンの核心は「抽象化」です。ビジネスロジック（Service層）はデータの取得方法を気にせず、必要なデータの種類だけを指定します。これにより、DB変更やテスト時のモック差し替えが容易になります。

Q2 解答

含めるべきではない。

passwordはセキュリティ上の理由から画面に渡すべきではありません。万が一レスポンスに含まれるとパスワードハッシュが漏洩するリスクがあります。loginCountは画面に表示する必要がない内部管理用データなので、DTOには不要です。DTOは「画面に必要な情報だけ」を厳選して定義します。

Q3 解答

正解: B. Service層

Entity→DTO変換はService層のメソッド内で行います。Repository層はEntityの永続化に専念し、Controller層はDTOを受け取ってViewに渡すだけです。変換ロジックをService層に集約することで、同じEntityから異なるDTOを生成する場合も管理しやすくなります。
