Section 6-1

Builderパターン

Builderパターンは、複雑なオブジェクトの組み立て手順を分離し、段階的に構築するパターンです。コンストラクタの引数が多い場合や、オプション項目がある場合に特に有効です。

6-1-1 コンストラクタ引数の爆発問題

商品オブジェクトに多数のフィールドがある場合、コンストラクタの引数が増え、順番を間違えやすくなります。

引数が多すぎるコンストラクタ 
// どの引数が何を意味するか分かりにくい...
Product product = new Product(
    "ノートPC", 89800, "electronics", 50, "高性能ノートPC",
    true, 1.5, "made in Japan", null, null
);

6-1-2 Builderによる段階的構築

Builderパターンでは、メソッドチェーンで可読性高くオブジェクトを構築します。

Product.java（Builderパターン） 
public class Product {
    private final String name;
    private final int price;
    private final String category;
    private final int stock;
    private final String description;
    private final boolean featured;

    // privateコンストラクタ（Builderからのみ生成可能）
    private Product(Builder builder) {
        this.name = builder.name;
        this.price = builder.price;
        this.category = builder.category;
        this.stock = builder.stock;
        this.description = builder.description;
        this.featured = builder.featured;
    }

    // staticなBuilderクラス
    public static class Builder {
        // 必須パラメータ
        private final String name;
        private final int price;

        // オプションパラメータ（デフォルト値あり）
        private String category = "general";
        private int stock = 0;
        private String description = "";
        private boolean featured = false;

        public Builder(String name, int price) {
            this.name = name;
            this.price = price;
        }

        public Builder category(String category) {
            this.category = category;
            return this;
        }

        public Builder stock(int stock) {
            this.stock = stock;
            return this;
        }

        public Builder description(String description) {
            this.description = description;
            return this;
        }

        public Builder featured(boolean featured) {
            this.featured = featured;
            return this;
        }

        public Product build() {
            return new Product(this);
        }
    }
}
Builderの使用例 
// 可読性が高く、オプション項目も柔軟に指定できる
Product product = new Product.Builder("ノートPC", 89800)
        .category("electronics")
        .stock(50)
        .description("高性能ノートPC")
        .featured(true)
        .build();

6-1-3 Lombok @Builderによる簡略化

Lombokライブラリを使えば、@Builderアノテーション1つでBuilderパターンを自動生成できます。

Lombok @Builder 
@Builder
@Getter
public class Product {
    private String name;
    private int price;
    @Builder.Default private String category = "general";
    @Builder.Default private int stock = 0;
    private String description;
    @Builder.Default private boolean featured = false;
}

// 使い方は同じ
Product product = Product.builder()
        .name("ノートPC")
        .price(89800)
        .category("electronics")
        .stock(50)
        .build();

Section 6-2

Facadeパターン

Facadeパターンは、複数のサブシステムへの操作を1つの統合窓口にまとめるパターンです。「注文処理」のように複数サービスを横断する処理をシンプルなAPIとして提供します。

6-2-1 Facadeが解決する問題

ECサイトの注文処理では、在庫確認・決済・メール送信・ポイント付与など複数のサービスを順番に呼ぶ必要があります。Controllerがこれらを直接呼ぶと複雑になります。

door_front Facadeパターンの構造

▼

6-2-2 Facadeの実装

OrderFacade.java 
@Service
public class OrderFacade {

    private final StockService stockService;
    private final PaymentService paymentService;
    private final MailService mailService;
    private final PointService pointService;
    private final OrderRepository orderRepository;

    public OrderFacade(StockService stockService,
                       PaymentService paymentService,
                       MailService mailService,
                       PointService pointService,
                       OrderRepository orderRepository) {
        this.stockService = stockService;
        this.paymentService = paymentService;
        this.mailService = mailService;
        this.pointService = pointService;
        this.orderRepository = orderRepository;
    }

    @Transactional
    public OrderResult placeOrder(OrderRequest request) {
        // 1. 在庫確認
        stockService.checkAndReserve(request.getItems());

        // 2. 決済処理
        PaymentResult payment = paymentService.process(
            request.getPaymentMethod(), request.getTotalAmount()
        );

        // 3. 注文保存
        Order order = Order.builder()
                .customerId(request.getCustomerId())
                .items(request.getItems())
                .totalAmount(request.getTotalAmount())
                .paymentId(payment.getPaymentId())
                .build();
        orderRepository.save(order);

        // 4. メール送信
        mailService.sendOrderConfirmation(order);

        // 5. ポイント付与
        pointService.addPoints(request.getCustomerId(),
                               request.getTotalAmount());

        return new OrderResult(order.getId(), "注文完了");
    }
}
OrderController.java（Facade利用側） 
@Controller
public class OrderController {
    private final OrderFacade orderFacade;  // Facadeだけに依存

    public OrderController(OrderFacade orderFacade) {
        this.orderFacade = orderFacade;
    }

    @PostMapping("/orders")
    public String placeOrder(@ModelAttribute OrderRequest request, Model model) {
        OrderResult result = orderFacade.placeOrder(request);
        model.addAttribute("result", result);
        return "order/complete";
    }
}
Section 6-3

Decoratorパターン

Decoratorパターンは、既存のオブジェクトに動的に機能を追加するパターンです。継承を使わずに、ラッピングによって機能を重ね掛けできます。

6-3-1 Decoratorの考え方

コーヒーショップを例に考えましょう。ベースのコーヒーに「ミルク追加」「シロップ追加」「ホイップ追加」をトッピングとして重ね掛けできます。

layers Decoratorの重ね掛けイメージ

BasePrice
基本料金: 10,000
→
TaxDecorator
+10%消費税
→
DiscountDecorator
-会員割引5%
→
最終価格
10,450円

6-3-2 Decoratorの実装（料金計算）

PriceCalculator.java（インターフェース + 実装） 
// 共通インターフェース
public interface PriceCalculator {
    int calculate(int basePrice);
    String getDescription();
}

// ベース（何もしない）
public class BasePriceCalculator implements PriceCalculator {
    @Override
    public int calculate(int basePrice) {
        return basePrice;
    }
    @Override
    public String getDescription() {
        return "基本価格";
    }
}

// Decoratorの抽象クラス
public abstract class PriceDecorator implements PriceCalculator {
    protected final PriceCalculator wrapped;

    public PriceDecorator(PriceCalculator wrapped) {
        this.wrapped = wrapped;
    }
}

// 消費税Decorator
public class TaxDecorator extends PriceDecorator {
    private final double taxRate;

    public TaxDecorator(PriceCalculator wrapped, double taxRate) {
        super(wrapped);
        this.taxRate = taxRate;
    }

    @Override
    public int calculate(int basePrice) {
        int subtotal = wrapped.calculate(basePrice);
        return (int)(subtotal * (1 + taxRate));
    }

    @Override
    public String getDescription() {
        return wrapped.getDescription() + " + 消費税";
    }
}

// 割引Decorator
public class DiscountDecorator extends PriceDecorator {
    private final double discountRate;

    public DiscountDecorator(PriceCalculator wrapped, double discountRate) {
        super(wrapped);
        this.discountRate = discountRate;
    }

    @Override
    public int calculate(int basePrice) {
        int subtotal = wrapped.calculate(basePrice);
        return (int)(subtotal * (1 - discountRate));
    }

    @Override
    public String getDescription() {
        return wrapped.getDescription() + " - 割引";
    }
}

6-3-3 Decoratorの組み合わせ

Decoratorの組み合わせ例 
// 基本 → 消費税10% → 会員割引5%
PriceCalculator calculator = new BasePriceCalculator();
calculator = new TaxDecorator(calculator, 0.10);
calculator = new DiscountDecorator(calculator, 0.05);

int finalPrice = calculator.calculate(10000);
// 10000 → 11000(税込) → 10450(割引後)

System.out.println(calculator.getDescription());
// "基本価格 + 消費税 - 割引"

System.out.println("最終価格: " + finalPrice);
// "最終価格: 10450"
Exercise

6章 練習問題

Q1

Builderパターンを適用すべき場面として最も適切なものはどれですか。

A. アルゴリズムを実行時に切り替えたい
B. オブジェクトのコンストラクタ引数が多く、オプション項目がある
C. 複数のサービスへの呼び出しを1つにまとめたい
D. オブジェクトに動的に機能を追加したい

Q2

Facadeパターンの「統合窓口」の役割を1文で説明してください。また、FacadeとServiceの違いは何ですか。

Q3

以下の要件で適用すべきパターンはBuilder、Facade、Decoratorのどれですか。理由も述べてください。

「ログ出力処理に、環境に応じてタイムスタンプ付与・JSON整形・ファイル書き出しの機能を自由に組み合わせて追加したい。」

Answer

6章 解説・解答

Q1 解答

正解: B. オブジェクトのコンストラクタ引数が多く、オプション項目がある

Builderパターンは「生成」に関するパターンで、複雑なオブジェクトの段階的構築を可読性高く行うためのものです。AはStrategy、CはFacade、DはDecoratorの場面です。

Q2 解答

Facadeの統合窓口は、複数のサブシステム（Service）への操作手順をカプセル化し、利用側にシンプルなAPIを提供するクラスです。

Serviceとの違いは、Serviceが1つのドメイン領域のビジネスロジックを担うのに対し、Facadeは複数のServiceを横断的に組み合わせる「オーケストレーター」の役割を持つ点です。

Q3 解答

Decoratorパターンが適切。

「機能を自由に組み合わせて追加したい」という要件はDecoratorの典型的な適用場面です。ベースのログ出力にタイムスタンプDecorator、JSON整形Decorator、ファイル書き出しDecoratorを重ね掛けすることで、環境に応じた柔軟な組み合わせが実現できます。継承では全組み合わせ分のサブクラスが必要になり、クラス爆発が起こります。
