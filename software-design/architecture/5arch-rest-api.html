Section 5-1

RESTful原則

REST（Representational State Transfer）は、Web APIの設計スタイルです。HTTPプロトコルの仕組みを最大限に活用し、リソースの操作をシンプルで統一的なインタフェースで表現します。

5-1-1 RESTの6つの制約

api REST 6原則

1. クライアント・サーバ
UI(フロント)とデータ(バック)を分離
2. ステートレス
各リクエストは独立、サーバに状態を保持しない
3. キャッシュ可能
レスポンスにキャッシュ可否を明示
4. 統一インタフェース
HTTPメソッドとURLで操作を表現
5. 階層化システム
LB, キャッシュ等の中間層を挿入可能
6. コードオンデマンド
必要に応じてコードを送信（任意）

5-1-2 HTTPメソッドとCRUD

HTTPメソッド CRUD操作 例 冪等性
GET Read（取得） GET /api/users/1 あり
POST Create（作成） POST /api/users なし
PUT Update（全体更新） PUT /api/users/1 あり
PATCH Update（部分更新） PATCH /api/users/1 場合による
DELETE Delete（削除） DELETE /api/users/1 あり
lightbulb 冪等性（べきとうせい）とは

同じリクエストを何度送っても結果が変わらない性質です。GETは何度取得しても同じ結果。DELETEも既に削除済みなら再度送っても変わりません。一方POSTは送るたびに新しいリソースが作られるため、冪等ではありません。

Section 5-2

URL設計

REST APIのURLは「リソース」を表します。動詞ではなく名詞を使い、階層構造でリソース間の関係を表現するのがポイントです。

5-2-1 URL設計のルール

ルール 悪い例 良い例
名詞（複数形）を使う /getUsers /users
動詞を含めない /deleteUser/1 DELETE /users/1
小文字ケバブケース /orderItems /order-items
階層でリソース関係を表す /getUserOrders?userId=1 /users/1/orders
フィルタはクエリパラメータ /orders/pending /orders?status=pending

5-2-2 ECサイトのURL設計例

link ECサイト API エンドポイント一覧

メソッド URL 説明
GET/api/products商品一覧を取得
GET/api/products/{id}商品詳細を取得
POST/api/orders注文を作成
GET/api/users/{id}/ordersユーザーの注文一覧
PATCH/api/orders/{id}注文を部分更新
DELETE/api/orders/{id}注文を削除
Section 5-3

HTTPステータスコード

ステータスコードはAPIの「返事」です。適切なコードを返すことで、クライアントはレスポンスの意味を正しく理解できます。

5-3-1 よく使うステータスコード

コード 意味 使用場面
200 OK 成功 GET, PUT, PATCHの成功時
201 Created 作成成功 POSTでリソース作成成功時
204 No Content 成功（本文なし） DELETEの成功時
400 Bad Request リクエスト不正 バリデーションエラー
401 Unauthorized 認証エラー ログインしていない
403 Forbidden 認可エラー 権限がない
404 Not Found 未検出 指定IDのリソースが存在しない
409 Conflict 競合 重複登録、楽観ロック失敗
500 Internal Server Error サーバエラー 予期しない例外発生時
Section 5-4

リクエスト/レスポンス設計

JSON形式のリクエスト/レスポンスの設計は、API利用者の使いやすさに直結します。Spring Bootの@RestControllerで実装する方法を見ていきましょう。

5-4-1 Spring Boot での REST API 実装

ProductController.java Java 
@RestController
@RequestMapping("/api/products")
public class ProductController {

    private final ProductService productService;

    public ProductController(ProductService productService) {
        this.productService = productService;
    }

    // GET /api/products - 一覧取得
    @GetMapping
    public ResponseEntity<List<ProductResponse>> getAll(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "20") int size) {
        List<ProductResponse> products = productService.findAll(page, size);
        return ResponseEntity.ok(products);
    }

    // GET /api/products/{id} - 1件取得
    @GetMapping("/{id}")
    public ResponseEntity<ProductResponse> getById(@PathVariable Long id) {
        ProductResponse product = productService.findById(id);
        return ResponseEntity.ok(product);
    }

    // POST /api/products - 新規作成
    @PostMapping
    public ResponseEntity<ProductResponse> create(
            @Valid @RequestBody ProductCreateRequest request) {
        ProductResponse created = productService.create(request);
        URI location = URI.create("/api/products/" + created.getId());
        return ResponseEntity.created(location).body(created);
    }

    // PUT /api/products/{id} - 全体更新
    @PutMapping("/{id}")
    public ResponseEntity<ProductResponse> update(
            @PathVariable Long id,
            @Valid @RequestBody ProductUpdateRequest request) {
        ProductResponse updated = productService.update(id, request);
        return ResponseEntity.ok(updated);
    }

    // DELETE /api/products/{id} - 削除
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        productService.delete(id);
        return ResponseEntity.noContent().build();
    }
}
リクエスト/レスポンスDTO Java 
// リクエストDTO（入力）
public class ProductCreateRequest {
    @NotBlank(message = "商品名は必須です")
    private String name;

    @NotNull @Min(value = 0, message = "価格は0以上")
    private Integer price;

    private String description;

    // getter/setter
}

// レスポンスDTO（出力）
public class ProductResponse {
    private Long id;
    private String name;
    private int price;
    private String description;
    private LocalDateTime createdAt;

    public static ProductResponse from(Product product) {
        ProductResponse res = new ProductResponse();
        res.id = product.getId();
        res.name = product.getName();
        res.price = product.getPrice();
        res.description = product.getDescription();
        res.createdAt = product.getCreatedAt();
        return res;
    }
}

GlobalExceptionHandler.java Java 
@RestControllerAdvice
public class GlobalExceptionHandler {

    // バリデーションエラー
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ErrorResponse> handleValidation(
            MethodArgumentNotValidException ex) {
        List<FieldError> details = ex.getBindingResult()
            .getFieldErrors().stream()
            .map(e -> new FieldError(e.getField(), e.getDefaultMessage()))
            .toList();

        ErrorResponse error = new ErrorResponse(
            400, "Bad Request", "VALIDATION_ERROR",
            "入力値に問題があります", details
        );
        return ResponseEntity.badRequest().body(error);
    }

    // リソース未検出
    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            ResourceNotFoundException ex) {
        ErrorResponse error = new ErrorResponse(
            404, "Not Found", "RESOURCE_NOT_FOUND",
            ex.getMessage(), null
        );
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }

    // 予期しないエラー
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleUnexpected(Exception ex) {
        ErrorResponse error = new ErrorResponse(
            500, "Internal Server Error", "INTERNAL_ERROR",
            "予期しないエラーが発生しました", null
        );
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(error);
    }
}

5章 練習問題

Q1

以下のURLのうち、RESTfulな設計として適切なものをすべて選んでください。

  1. GET /api/users/1/orders
  2. POST /api/createUser
  3. DELETE /api/users/1
  4. GET /api/getUserById?id=1
Q2

次の操作に対して、適切なHTTPメソッドとステータスコードの組み合わせを答えてください。

  1. 新しいユーザーを作成し、作成されたユーザー情報を返す
  2. 存在しないIDの商品を取得しようとした
  3. 注文を削除し、何も返さない
Q3

「ユーザーが自分の注文をキャンセルする」機能のエンドポイントを設計してください。URL、HTTPメソッド、リクエスト/レスポンスのJSON例を示してください。

Answer

5章 解説・解答

Q1の解答: A, C

A: リソースの階層関係（ユーザーの注文一覧）を正しく表現しています。C: DELETEメソッドで削除を表現し、URLは名詞のみです。B: URLに動詞「create」が含まれています（POST /api/usersが正しい）。D: URLに動詞「getUserById」が含まれています（GET /api/users/1が正しい）。

Q2の解答

1: POST /api/users → 201 Created（Locationヘッダーで作成されたリソースのURLも返す）。2: GET /api/products/{id} → 404 Not Found。3: DELETE /api/orders/{id} → 204 No Content。

Q3の解答

PATCH /api/orders/{orderId} でステータスを更新する方法が一般的です。

// リクエスト
PATCH /api/orders/123
Content-Type: application/json

{
    "status": "CANCELLED",
    "reason": "注文を間違えたため"
}

// レスポンス (200 OK)
{
    "id": 123,
    "status": "CANCELLED",
    "cancelledAt": "2026-03-04T15:30:00",
    "reason": "注文を間違えたため"
}

別の設計として POST /api/orders/123/cancel（サブリソースとしてのアクション）もありです。RESTの純粋さよりも分かりやすさを優先する場面もあります。
