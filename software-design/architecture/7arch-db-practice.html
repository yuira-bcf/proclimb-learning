Section 7-1

論理設計から物理設計へ

前章で学んだ正規化やER図は「論理設計」の段階です。本章では論理設計を実際のRDBMSに実装する「物理設計」のノウハウを学びます。

7-1-1 論理設計と物理設計の違い

観点 論理設計 物理設計
目的 データの構造を整理する パフォーマンスと運用を考慮して実装する
テーブル名 日本語OK（例: 注文テーブル） 英語スネークケース（例: orders）
データ型 概念的（数値, 文字列） 具体的（BIGINT, VARCHAR(255)）
正規化 厳密に正規化する パフォーマンスのため意図的に非正規化する場合も
インデックス 考慮しない クエリパターンに合わせて設計

7-1-2 命名規約

text_format テーブル/カラム命名規約
  • テーブル名: 複数形のスネークケース（users, order_items）
  • カラム名: スネークケース（created_at, user_id）
  • 主キー: id（シンプルに）
  • 外部キー: {参照先テーブル名の単数形}_id（例: user_id）
  • 日時カラム: xxx_at（created_at, updated_at）
  • 真偽値: is_xxx（is_active, is_deleted）
Section 7-2

テーブル設計パターン

実務で頻出するテーブル設計のパターンを紹介します。これらを知っておくと、設計の引き出しが増え、適切な判断ができるようになります。

7-2-1 よく使うパターン

pattern テーブル設計パターン一覧

履歴テーブル

変更を追跡するために、更新ではなくINSERTで記録。status_histories

論理削除

is_deletedフラグでDELETEの代わり。データ復旧可能。

区分値テーブル

ENUMの代わりにコードマスタテーブルで管理。拡張が容易。

楽観ロック

versionカラムで同時更新を検出。Spring JPAの@Version。

7-2-2 論理削除の実装

論理削除のDDLとクエリ SQL 
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) NOT NULL,
    is_deleted BOOLEAN NOT NULL DEFAULT FALSE,
    deleted_at TIMESTAMP NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 通常の取得: 削除されていないレコードのみ
SELECT * FROM users WHERE is_deleted = FALSE;

-- 論理削除の実行
UPDATE users SET is_deleted = TRUE, deleted_at = NOW() WHERE id = 1;

-- UNIQUEインデックスとの両立（削除済みは重複OK）
CREATE UNIQUE INDEX idx_users_email_active
    ON users(email) WHERE is_deleted = FALSE;  -- PostgreSQL
-- MySQLの場合: UNIQUE INDEX (email, is_deleted) で工夫
Section 7-3

共通カラム設計

全テーブルに共通で持たせるカラムを統一的に設計することで、監査ログやデバッグの効率が上がります。

7-3-1 共通カラム一覧

カラム名 型 用途 自動設定
id BIGINT 主キー（AUTO_INCREMENT） DB自動採番
created_at TIMESTAMP 作成日時 DEFAULT CURRENT_TIMESTAMP
updated_at TIMESTAMP 更新日時 ON UPDATE CURRENT_TIMESTAMP
created_by VARCHAR(100) 作成者（ユーザーID or システム名） アプリ側で設定
updated_by VARCHAR(100) 更新者 アプリ側で設定
version INT 楽観ロック用バージョン DEFAULT 0、更新時に+1
JPAの共通Entityクラス（BaseEntity） Java 
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @CreatedDate
    @Column(nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @LastModifiedDate
    @Column(nullable = false)
    private LocalDateTime updatedAt;

    @CreatedBy
    @Column(updatable = false)
    private String createdBy;

    @LastModifiedBy
    private String updatedBy;

    @Version
    private Integer version;

    // getter/setter
}

// 各エンティティはBaseEntityを継承するだけ
@Entity
@Table(name = "products")
public class ProductEntity extends BaseEntity {
    @Column(nullable = false)
    private String name;

    @Column(nullable = false)
    private Integer price;
}
Section 7-4

マスタ・トランザクション分離

テーブルは大きく「マスタテーブル」と「トランザクションテーブル」に分類されます。この分類を意識して設計することが、スケーラブルなDB設計の鍵です。

7-4-1 マスタとトランザクション

compare マスタ vs トランザクション

マスタテーブル
  • 基本情報を管理
  • 更新頻度が低い
  • レコード数が比較的少ない
  • 例: users, products, categories
トランザクションテーブル
  • 業務上の出来事を記録
  • INSERT中心（更新は少ない）
  • レコード数が急速に増加
  • 例: orders, payments, logs
warning 注文時の商品価格をどう保存するか

注文明細に商品マスタの価格をそのまま外部キーで参照すると、商品価格が変わったとき過去の注文金額も変わってしまいます。注文時点の価格をトランザクションテーブルにコピー（スナップショット）しておくのが正しい設計です。

7-5-2 ECサイトDDL

ECサイト DDL（MySQL） SQL 
-- マスタテーブル
CREATE TABLE categories (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    slug VARCHAR(100) NOT NULL UNIQUE,
    parent_id BIGINT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (parent_id) REFERENCES categories(id)
);

CREATE TABLE products (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    category_id BIGINT NOT NULL,
    name VARCHAR(200) NOT NULL,
    description TEXT,
    price INT NOT NULL CHECK (price >= 0),
    stock INT NOT NULL DEFAULT 0 CHECK (stock >= 0),
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    version INT NOT NULL DEFAULT 0,
    FOREIGN KEY (category_id) REFERENCES categories(id)
);

CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- トランザクションテーブル
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'PENDING',
    total_price INT NOT NULL,
    shipping_address TEXT NOT NULL,
    ordered_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE TABLE order_items (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    quantity INT NOT NULL CHECK (quantity > 0),
    unit_price INT NOT NULL,  -- 注文時点の価格をスナップショット
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);

CREATE TABLE payments (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id BIGINT NOT NULL UNIQUE,
    method VARCHAR(30) NOT NULL,
    amount INT NOT NULL,
    paid_at TIMESTAMP NULL,
    FOREIGN KEY (order_id) REFERENCES orders(id)
);

-- インデックス
CREATE INDEX idx_products_category ON products(category_id);
CREATE INDEX idx_orders_user ON orders(user_id);
CREATE INDEX idx_orders_status ON orders(status, ordered_at);
CREATE INDEX idx_order_items_order ON order_items(order_id);

7-5-3 JPA Entityマッピング

OrderEntity.java（JPA Entity） Java 
@Entity
@Table(name = "orders")
public class OrderEntity extends BaseEntity {

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    private UserEntity user;

    @Column(nullable = false, length = 20)
    @Enumerated(EnumType.STRING)
    private OrderStatus status;

    @Column(name = "total_price", nullable = false)
    private Integer totalPrice;

    @Column(name = "shipping_address", nullable = false)
    private String shippingAddress;

    @Column(name = "ordered_at", nullable = false)
    private LocalDateTime orderedAt;

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderItemEntity> items = new ArrayList<>();

    @OneToOne(mappedBy = "order", cascade = CascadeType.ALL)
    private PaymentEntity payment;
}
Exercise

7章 練習問題

Q1

注文明細テーブルのunit_priceカラムは、商品テーブルのpriceと重複しているように見えます。なぜ別途保存する必要があるのか、理由を説明してください。

Q2

「ブログサービス」のDB設計を行ってください。以下のテーブルのCREATE TABLE文を書いてください。共通カラム（created_at, updated_at）を含めること。

  • users（id, name, email）
  • posts（id, user_id, title, content, status）
  • tags（id, name）
  • post_tags（post_id, tag_id） - 多対多の中間テーブル
Q3

以下のテーブルのうち「マスタテーブル」と「トランザクションテーブル」に分類してください。

departments, employees, attendance_records, meeting_rooms, room_reservations, job_titles

Answer

7章 解説・解答

Q1の解答

商品の価格は時間とともに変更される可能性があります（値上げ・セール等）。注文明細にunit_priceを保存しておかないと、商品マスタの価格が変わったとき、過去の注文金額も連動して変わってしまいます。「注文時点の価格」を保持するスナップショットとして、unit_priceは不可欠です。

Q2の解答
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

CREATE TABLE posts (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'DRAFT',
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

CREATE TABLE tags (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL UNIQUE,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE post_tags (
    post_id BIGINT NOT NULL,
    tag_id BIGINT NOT NULL,
    PRIMARY KEY (post_id, tag_id),
    FOREIGN KEY (post_id) REFERENCES posts(id),
    FOREIGN KEY (tag_id) REFERENCES tags(id)
);
Q3の解答

マスタ: departments（部署）, employees（社員）, meeting_rooms（会議室）, job_titles（役職）。これらは基本情報であり、頻繁には変更されません。

トランザクション: attendance_records（勤怠記録）, room_reservations（会議室予約）。これらは日々の業務で発生する出来事の記録であり、INSERT中心でレコード数が増え続けます。
