Section 8-1

画面設計書とは

画面設計書は「何を・どこに・どう表示し、ユーザーが何を操作できるか」を定義するドキュメントです。開発者・デザイナー・クライアント間の認識を揃える、最も重要な設計成果物の1つです。

8-1-1 画面設計書の構成要素

description 画面設計書に含めるもの

1. 画面一覧
全画面のID・名前・概要を一覧化
2. 画面遷移図
画面間の遷移と条件を図示
3. ワイヤーフレーム
各画面のレイアウト設計図
4. 項目定義
入力項目の型・桁数・バリデーション
5. イベント定義
ボタンクリック時の動作を定義

Section 8-4

入力項目定義

各画面の入力フィールドについて、データ型・桁数・必須/任意・初期値・入力方法などを詳細に定義します。この定義は、フロント側のフォーム実装とバックエンド側のバリデーション実装の両方の基準になります。

Section 8-5

バリデーション設計

バリデーション（入力検証）は、フロントエンド（即座にフィードバック）とバックエンド（セキュリティ保証）の両方で実装します。画面設計の段階でバリデーションルールを明確に定義しておくことが重要です。

8-5-1 フロントエンドとバックエンドのバリデーション

verified_user バリデーションの二重防壁

フロントエンド
  • UX向上（リアルタイム表示）
  • サーバ負荷の軽減
  • HTML5のrequired, pattern
  • JavaScriptによる即時チェック

※ 開発者ツールで回避可能なため信頼しない

バックエンド（必須）
  • セキュリティの最終防壁
  • データ整合性の保証
  • Spring Validation（@Valid）
  • Bean Validationアノテーション

※ 必ずサーバ側でチェックする

8-5-2 Spring Boot バリデーション実装

CheckoutForm.java（Bean Validation） Java 
public class CheckoutForm {

    @NotBlank(message = "氏名は必須です")
    @Size(max = 50, message = "氏名は50文字以内で入力してください")
    private String recipientName;

    @NotBlank(message = "郵便番号は必須です")
    @Pattern(regexp = "\\d{7}", message = "郵便番号は半角数字7桁で入力してください")
    private String postalCode;

    @NotBlank(message = "都道府県を選択してください")
    private String prefecture;

    @NotBlank(message = "住所は必須です")
    @Size(max = 200, message = "住所は200文字以内で入力してください")
    private String address;

    @NotBlank(message = "電話番号は必須です")
    @Pattern(regexp = "\\d{10,11}", message = "電話番号は半角数字10〜11桁で入力してください")
    private String phoneNumber;

    @NotNull(message = "支払方法を選択してください")
    private PaymentMethod paymentMethod;
}

// Controllerでの利用
@PostMapping("/checkout")
public String checkout(@Valid @ModelAttribute CheckoutForm form,
                       BindingResult result, Model model) {
    if (result.hasErrors()) {
        return "checkout";  // バリデーションエラー時は画面を再表示
    }
    orderService.placeOrder(form);
    return "redirect:/order-complete";
}
lightbulb バリデーション設計のポイント
  • エラーメッセージはユーザーが理解できる日本語で書く（「Pattern mismatch」はNG）
  • フィールド単位のチェック（桁数、形式）と相関チェック（パスワード確認一致等）を分けて設計する
  • 画面設計書の段階でバリデーションルールを確定させると、フロント・バックの開発が並行で進められる

8章 練習問題

Q1

フロントエンドのバリデーションだけでは不十分な理由を、セキュリティの観点から説明してください。

Q2

「ユーザー登録画面」の以下の項目に対して、入力項目定義（型、桁数、必須/任意、バリデーション）を設計してください。

  • ユーザー名
  • メールアドレス
  • パスワード
  • パスワード確認
  • 利用規約への同意
Q3

Q2で設計した「ユーザー登録フォーム」のBean Validationアノテーションを使ったJavaのFormクラスを実装してください。パスワード確認の一致チェックも含めること。

Answer

8章 解説・解答

Q1の解答

フロントエンドのバリデーションはブラウザの開発者ツールで簡単に無効化できます。また、curlやPostman等のツールから直接APIを叩けば、フロントエンドのチェックは完全にバイパスされます。そのため、SQLインジェクションやXSS攻撃等を防ぐためにも、バックエンド側のバリデーションは必須です。フロントは「UX向上」、バックエンドは「セキュリティ保証」と役割が異なります。

Q2の解答
項目名型桁数必須バリデーション
ユーザー名文字列3〜20文字必須半角英数字とアンダースコアのみ
メールアドレス文字列最大255文字必須メール形式、重複不可
パスワード文字列8〜100文字必須英大文字・小文字・数字を各1文字以上含む
パスワード確認文字列-必須パスワードと一致すること
利用規約同意真偽値-必須trueであること
Q3の解答
@PasswordMatch  // カスタムバリデーション（相関チェック）
public class UserRegistrationForm {

    @NotBlank(message = "ユーザー名は必須です")
    @Size(min = 3, max = 20, message = "3〜20文字で入力してください")
    @Pattern(regexp = "^[a-zA-Z0-9_]+$",
             message = "半角英数字とアンダースコアのみ使用できます")
    private String username;

    @NotBlank(message = "メールアドレスは必須です")
    @Email(message = "正しいメール形式で入力してください")
    @Size(max = 255)
    private String email;

    @NotBlank(message = "パスワードは必須です")
    @Size(min = 8, max = 100, message = "8文字以上で入力してください")
    @Pattern(regexp = "^(?=.*[a-z])(?=.*[A-Z])(?=.*\\d).+$",
             message = "英大文字・小文字・数字を各1文字以上含めてください")
    private String password;

    @NotBlank(message = "パスワード確認は必須です")
    private String passwordConfirm;

    @AssertTrue(message = "利用規約に同意してください")
    private boolean agreeTerms;
}
