Section 4-1

DDDとは

DDD（Domain-Driven Design / ドメイン駆動設計）とは、ソフトウェアの複雑さに立ち向かうための設計手法です。ビジネスの問題領域（ドメイン）を深く理解し、それをソフトウェアモデルに反映させることを重視します。

4-1-1 DDDの基本思想

psychology DDDの核心: ビジネスの複雑さをコードで表現する

DDD以前: データ中心設計
  • まずテーブルを設計する
  • ビジネスロジックがServiceに散在
  • Entityはただのデータ入れ物
  • ドメイン知識がコードに表れない
DDD: ドメイン中心設計
  • まずビジネスルールを理解する
  • ロジックをドメインモデルに持たせる
  • Entityが自身のルールを知っている
  • コードがビジネスの仕様書になる
Section 4-2

ユビキタス言語

ユビキタス言語（Ubiquitous Language）とは、開発チームとビジネスの関係者（ドメインエキスパート）が共通して使う言葉のことです。コード内の変数名・クラス名・メソッド名に、ビジネスで使われる用語をそのまま使います。

4-2-1 ユビキタス言語の実践

ビジネス用語 悪い命名 ユビキタス言語に基づく命名
注文を確定する updateStatus() confirmOrder()
在庫を引き当てる decreaseCount() allocateStock()
商品を出荷する processItem() shipProduct()
返品する reverseTransaction() returnProduct()
lightbulb なぜユビキタス言語が重要か

ビジネスの人が「注文をキャンセルしたい」と言ったとき、コード上にもorder.cancel()というメソッドがあれば、開発者は即座に修正箇所を特定できます。命名がdata.setFlag(0)だったら、何をしているのか分からないままです。

Section 4-3

Entity・Value Object・Aggregate

DDDの戦術的パターンの中核をなす3つの概念を学びます。これらを正しく使い分けることで、ドメインモデルの表現力が飛躍的に向上します。

4-3-1 EntityとValue Object

category Entity vs Value Object

Entity（エンティティ）
  • 同一性（ID）で識別される
  • ライフサイクルがある（生成→更新→削除）
  • 可変（mutable）なことが多い
  • 例: User, Order, Product
Value Object（値オブジェクト）
  • 属性の値で識別される
  • IDを持たない
  • 不変（immutable）である
  • 例: Money, Email, Address
Value Object の例: Money Java 
// 値オブジェクト: 不変（final）, 値で等価判定
public class Money {
    private final int amount;
    private final String currency;

    public Money(int amount, String currency) {
        if (amount < 0) throw new IllegalArgumentException("金額は0以上");
        this.amount = amount;
        this.currency = currency;
    }

    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("通貨が異なります");
        }
        return new Money(this.amount + other.amount, this.currency);  // 新しいインスタンスを返す
    }

    public Money multiply(int quantity) {
        return new Money(this.amount * quantity, this.currency);
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Money)) return false;
        Money money = (Money) o;
        return amount == money.amount && currency.equals(money.currency);
    }

    @Override
    public int hashCode() {
        return Objects.hash(amount, currency);
    }
}
Entity の例: Order Java 
// エンティティ: IDで識別, ビジネスロジックを持つ
public class Order {
    private final OrderId id;            // 識別子
    private final UserId userId;
    private List<OrderLine> lines;
    private Money totalPrice;
    private OrderStatus status;

    public Order(OrderId id, UserId userId) {
        this.id = id;
        this.userId = userId;
        this.lines = new ArrayList<>();
        this.status = OrderStatus.DRAFT;
    }

    public void addItem(Product product, int quantity) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("確定済みの注文には追加できません");
        }
        Money lineTotal = product.getPrice().multiply(quantity);
        lines.add(new OrderLine(product.getId(), quantity, lineTotal));
        recalculateTotal();
    }

    public void confirm() {
        if (lines.isEmpty()) {
            throw new IllegalStateException("商品が空の注文は確定できません");
        }
        this.status = OrderStatus.CONFIRMED;
    }

    private void recalculateTotal() {
        this.totalPrice = lines.stream()
            .map(OrderLine::getSubtotal)
            .reduce(new Money(0, "JPY"), Money::add);
    }
}

4-3-2 Aggregate（集約）

集約（Aggregate）は、関連するEntityとValue Objectのまとまりです。外部からのアクセスは必ず集約ルート（Aggregate Root）を通じて行います。これによりデータの整合性を保証します。

Section 4-4

ドメインサービス

どのEntityにも属さないビジネスロジックはドメインサービスに配置します。「2つのEntityにまたがる処理」や「外部の情報が必要な計算」がその典型例です。

4-4-1 ドメインサービスの使い分け

ロジックの種類 配置先 例
1つのEntityの状態変更 Entity自身のメソッド order.cancel()
複数Entityにまたがる処理 ドメインサービス 在庫の引当（Order + Stock）
外部情報が必要な計算 ドメインサービス 税率を使った価格計算
ユースケースの調整 アプリケーションサービス 注文→在庫引当→通知の一連の流れ
ドメインサービスの例: 在庫引当 Java 
// ドメインサービス: 複数の集約にまたがるロジック
public class StockAllocationService {

    public void allocate(Order order, Stock stock) {
        for (OrderLine line : order.getLines()) {
            if (!stock.hasEnough(line.getProductId(), line.getQuantity())) {
                throw new InsufficientStockException(
                    line.getProductId() + "の在庫が不足しています"
                );
            }
            stock.reduce(line.getProductId(), line.getQuantity());
        }
    }
}

// アプリケーションサービスから呼び出す
@Service
@Transactional
public class OrderApplicationService {
    private final OrderRepository orderRepo;
    private final StockRepository stockRepo;
    private final StockAllocationService allocationService;

    public void confirmOrder(OrderId orderId) {
        Order order = orderRepo.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));
        Stock stock = stockRepo.findAll();

        allocationService.allocate(order, stock);  // ドメインサービス呼び出し
        order.confirm();

        orderRepo.save(order);
        stockRepo.save(stock);
    }
}
Section 4-5

境界づけられたコンテキスト

境界づけられたコンテキスト（Bounded Context）は、DDDの戦略的パターンの中核です。同じ言葉でもコンテキストによって意味が異なることを認め、明示的に境界を引きます。

4-5-1 コンテキストの境界

grid_view ECサイトのBounded Context例

lightbulb DDDの適用基準

DDDの全てのパターンを最初から適用する必要はありません。まずはユビキタス言語とEntity/Value Objectの使い分けから始めましょう。ビジネスロジックが複雑になってきたら、Aggregate や Bounded Context を検討します。

4章 練習問題

Q1

以下のうち、Value Object（値オブジェクト）として設計すべきものをすべて選んでください。

  1. メールアドレス（Email）
  2. ユーザー（User）
  3. 住所（Address）
  4. 注文（Order）
  5. 金額（Money）
Q2

以下のコードをDDDのEntityパターンに改善してください。現在はServiceにビジネスロジックが流出しています。

// 貧血ドメインモデル（アンチパターン）
public class Account {
    private int balance;
    public int getBalance() { return balance; }
    public void setBalance(int b) { this.balance = b; }
}

public class AccountService {
    public void withdraw(Account account, int amount) {
        if (account.getBalance() < amount) {
            throw new RuntimeException("残高不足");
        }
        account.setBalance(account.getBalance() - amount);
    }
}
Q3

「在庫を引き当てる」ロジックは、Order Entity・Stock Entity・ドメインサービスのどこに置くべきですか？ 理由を説明してください。

Answer

4章 解説・解答

Q1の解答: A, C, E

Email, Address, MoneyはIDで識別する必要がなく、属性の値で等価判定されます（同じメールアドレス文字列なら同じもの）。一方、UserとOrderはIDで識別されるエンティティです（同じ名前のUserでも別人）。

Q2の解答
// リッチドメインモデル: ビジネスロジックをEntity内に
public class Account {
    private int balance;

    public int getBalance() { return balance; }

    public void withdraw(int amount) {
        if (balance < amount) {
            throw new InsufficientBalanceException("残高不足");
        }
        this.balance -= amount;
    }

    public void deposit(int amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("入金額は正の値");
        }
        this.balance += amount;
    }
}

setBalance()を削除し、withdraw/depositといったビジネス操作のメソッドにしました。不正な状態遷移をEntity自身が防ぐことができます。

Q3の解答: ドメインサービス

在庫引当は「注文」と「在庫」の2つの集約にまたがる処理です。OrderがStockを直接操作するのは集約の境界を越えてしまいます。StockがOrderの詳細を知るのも不適切です。このような複数集約にまたがるロジックはドメインサービス（StockAllocationService等）に配置するのが適切です。
