Section 2-1

単一責任の原則 (SRP)

Single Responsibility Principle ―「クラスを変更する理由は、ただ1つであるべき」。1つのクラスが複数の役割を持つと、一方の変更がもう一方に影響を与えてしまいます。

2-1-1 SRPの概念

filter_1 SRP: 1クラス = 1責務

違反: 複数の責務
UserService
- authenticate()
- sendEmail()
- generateReport()
- formatCSV()

変更理由が4つもある

準拠: 1つの責務
AuthService - authenticate()
EmailService - sendEmail()
ReportService - generateReport()

2-1-2 Before / After コード

Before: SRP違反 ― 1クラスに複数責務 Java 
public class Employee {
    private String name;
    private double salary;

    // 責務1: 給与計算（経理部門の要件）
    public double calculatePay() {
        return salary * 1.1;  // 手当込み
    }

    // 責務2: 勤怠レポート生成（人事部門の要件）
    public String generateAttendanceReport() {
        return name + "の出勤日数: 20日";
    }

    // 責務3: DB保存（技術部門の要件）
    public void saveToDatabase() {
        // JDBC直接操作...
    }
}
After: SRP準拠 ― 責務を分離 Java 
// データ保持のみ
public class Employee {
    private String name;
    private double salary;

    public String getName() { return name; }
    public double getSalary() { return salary; }
}

// 給与計算の責務
public class PayCalculator {
    public double calculatePay(Employee employee) {
        return employee.getSalary() * 1.1;
    }
}

// レポート生成の責務
public class AttendanceReporter {
    public String generateReport(Employee employee) {
        return employee.getName() + "の出勤日数: 20日";
    }
}

// 永続化の責務
public class EmployeeRepository {
    public void save(Employee employee) {
        // JPA/MyBatis経由で保存
    }
}
Section 2-2

開放閉鎖の原則 (OCP)

Open-Closed Principle ―「拡張に対して開いていて、修正に対して閉じているべき」。新しい機能を追加するとき、既存のコードを変更せずに済む設計を目指します。

2-2-1 OCPの概念

lock_open OCP: 拡張に開き、修正に閉じる

add_circle
拡張: Open
新クラスの追加はOK
block
修正: Closed
既存コードの変更はNG

2-2-2 Before / After コード

Before: OCP違反 ― 新しい割引を追加するたびにif文を修正 Java 
public class DiscountCalculator {
    public double calculate(String type, double price) {
        if ("normal".equals(type)) {
            return price;
        } else if ("member".equals(type)) {
            return price * 0.9;
        } else if ("premium".equals(type)) {
            return price * 0.8;
        }
        // 新しい割引タイプを追加するたびにここを修正...
        // else if ("student".equals(type)) { ... }
        return price;
    }
}
After: OCP準拠 ― 新しい割引はクラス追加だけでOK Java 
// 割引戦略のインタフェース
public interface DiscountStrategy {
    double apply(double price);
}

// 各割引をクラスとして実装
public class NormalDiscount implements DiscountStrategy {
    @Override
    public double apply(double price) { return price; }
}

public class MemberDiscount implements DiscountStrategy {
    @Override
    public double apply(double price) { return price * 0.9; }
}

public class PremiumDiscount implements DiscountStrategy {
    @Override
    public double apply(double price) { return price * 0.8; }
}

// 新しい割引を追加しても、このクラスは変更不要！
public class DiscountCalculator {
    public double calculate(DiscountStrategy strategy, double price) {
        return strategy.apply(price);
    }
}

// 学生割引を追加: 新クラスを作るだけ
public class StudentDiscount implements DiscountStrategy {
    @Override
    public double apply(double price) { return price * 0.85; }
}
Section 2-3

リスコフの置換原則 (LSP)

Liskov Substitution Principle ―「子クラスは親クラスと置換可能であるべき」。親クラスを使っている場所で子クラスに差し替えても、プログラムが正しく動作しなければなりません。

2-3-1 LSPの概念

swap_horiz LSP: サブタイプは親と置換可能

有名な反例:「正方形は長方形の一種」と考えてしまうケース

違反例

Square(正方形) が Rectangle(長方形) を継承。setWidth()を呼ぶとheightも変わってしまい、Rectangleとして使えない。

準拠例

Shape インタフェースを定義し、Rectangle と Square を独立した実装にする。共通の振る舞いだけをインタフェースで規定。

2-3-2 Before / After コード

Before: LSP違反 ― Squareが親のRectangleの契約を破る Java 
public class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int w) { this.width = w; }
    public void setHeight(int h) { this.height = h; }
    public int getArea() { return width * height; }
}

public class Square extends Rectangle {
    @Override
    public void setWidth(int w) {
        this.width = w;
        this.height = w;  // 正方形なので同じにする → 親の契約違反！
    }

    @Override
    public void setHeight(int h) {
        this.width = h;
        this.height = h;
    }
}

// 利用側: Rectangleとして使うと予期しない結果に
Rectangle r = new Square();
r.setWidth(5);
r.setHeight(3);
System.out.println(r.getArea());  // 期待: 15、実際: 9
After: LSP準拠 ― 共通インタフェースで独立実装 Java 
public interface Shape {
    int getArea();
}

public class Rectangle implements Shape {
    private final int width;
    private final int height;

    public Rectangle(int width, int height) {
        this.width = width;
        this.height = height;
    }

    @Override
    public int getArea() { return width * height; }
}

public class Square implements Shape {
    private final int side;

    public Square(int side) {
        this.side = side;
    }

    @Override
    public int getArea() { return side * side; }
}

// 利用側: Shapeとして安全に使える
Shape rect = new Rectangle(5, 3);
Shape sq = new Square(4);
System.out.println(rect.getArea());  // 15
System.out.println(sq.getArea());    // 16
Section 2-4

インタフェース分離の原則 (ISP)

Interface Segregation Principle ―「クライアントに不要なメソッドへの依存を強制してはならない」。大きすぎるインタフェースは、小さな専用インタフェースに分割すべきです。

2-4-1 ISPの概念

call_split ISP: 太ったインタフェースを分割する

太ったIF
Machine
+ print()
+ scan()
+ fax()
+ staple()
arrow_forward
分割後
Printer: print()
Scanner: scan()
Fax: fax()

2-4-2 Before / After コード

Before: ISP違反 ― 使わないメソッドの実装を強制 Java 
public interface MultiFunctionDevice {
    void print(Document doc);
    void scan(Document doc);
    void fax(Document doc);
}

// シンプルなプリンターなのに、scan/faxも実装が必要...
public class SimplePrinter implements MultiFunctionDevice {
    @Override
    public void print(Document doc) { /* 印刷処理 */ }

    @Override
    public void scan(Document doc) {
        throw new UnsupportedOperationException("スキャン非対応");
    }

    @Override
    public void fax(Document doc) {
        throw new UnsupportedOperationException("FAX非対応");
    }
}
After: ISP準拠 ― 必要なインタフェースだけ実装 Java 
// 小さなインタフェースに分割
public interface Printer {
    void print(Document doc);
}

public interface Scanner {
    void scan(Document doc);
}

public interface Fax {
    void fax(Document doc);
}

// シンプルプリンター: Printerだけ実装
public class SimplePrinter implements Printer {
    @Override
    public void print(Document doc) { /* 印刷処理 */ }
}

// 複合機: 必要なものを全て実装
public class MultiFunctionPrinter implements Printer, Scanner, Fax {
    @Override
    public void print(Document doc) { /* 印刷処理 */ }

    @Override
    public void scan(Document doc) { /* スキャン処理 */ }

    @Override
    public void fax(Document doc) { /* FAX処理 */ }
}
Section 2-5

依存性逆転の原則 (DIP)

Dependency Inversion Principle ―「上位モジュールは下位モジュールに依存すべきでない。どちらも抽象に依存すべき」。Spring BootのDI（依存性注入）の根幹にある原則です。

2-5-1 DIPの概念

swap_vert DIP: 依存の方向を逆転させる

従来: 上位→下位に依存
DIP: 抽象に依存

2-5-2 Before / After コード（Spring Boot）

Before: DIP違反 ― 具象クラスに直接依存 Java 
// 上位モジュールが下位モジュールの具象クラスに依存
public class OrderService {
    private MySqlOrderRepository repository = new MySqlOrderRepository();

    public void placeOrder(Order order) {
        repository.save(order);
        // DBをPostgreSQLに変えたい → このクラスも修正が必要
    }
}
After: DIP準拠 ― インタフェース + Spring DI Java 
// 抽象（インタフェース）
public interface OrderRepository {
    void save(Order order);
    Optional<Order> findById(Long id);
}

// 下位モジュール: インタフェースを実装
@Repository
public class JpaOrderRepository implements OrderRepository {
    @Override
    public void save(Order order) { /* JPA経由で保存 */ }

    @Override
    public Optional<Order> findById(Long id) { /* JPA経由で検索 */ }
}

// 上位モジュール: 抽象に依存（Spring DIで注入）
@Service
public class OrderService {
    private final OrderRepository repository;

    @Autowired
    public OrderService(OrderRepository repository) {
        this.repository = repository;  // 実装の詳細を知らない
    }

    public void placeOrder(Order order) {
        repository.save(order);
        // DBを変えてもこのクラスは修正不要！
    }
}
lightbulb Spring BootとDIP

Spring Bootの@AutowiredによるDI（Dependency Injection）は、まさにDIPを実現する仕組みです。インタフェースを定義し、実装クラスに@Repositoryや@Serviceを付けるだけで、Springが自動的に依存関係を解決してくれます。

2章 練習問題

Q1

以下のコードはSOLID原則のどれに違反していますか？理由も答えてください。

public class ReportGenerator {
    public void generatePdf(Data data) { /* PDF生成 */ }
    public void generateExcel(Data data) { /* Excel生成 */ }
    public void sendEmail(String to, byte[] file) { /* メール送信 */ }
    public void uploadToS3(byte[] file) { /* S3アップロード */ }
}
Q2

OCP（開放閉鎖の原則）に従い、以下のif-else連鎖をStrategy パターンでリファクタリングしてください。

public double calculateShipping(String method, double weight) {
    if ("express".equals(method)) return weight * 500;
    else if ("standard".equals(method)) return weight * 200;
    else if ("economy".equals(method)) return weight * 100;
    return 0;
}
Q3

SOLID原則の5つをそれぞれ一言で説明してください。また、Spring Bootの開発で最も関わりが深い原則はどれですか？

Answer

2章 解説・解答

Q1の解答: SRP（単一責任の原則）違反

ReportGeneratorはレポート生成（PDF/Excel）、メール送信、ファイルアップロードという3つの異なる責務を持っています。「レポートの形式が変わった」「メール送信の仕様が変わった」「保存先がS3からGCSに変わった」など、変更理由が複数あるためSRP違反です。

Q2の解答
public interface ShippingStrategy {
    double calculate(double weight);
}

public class ExpressShipping implements ShippingStrategy {
    @Override public double calculate(double weight) { return weight * 500; }
}
public class StandardShipping implements ShippingStrategy {
    @Override public double calculate(double weight) { return weight * 200; }
}
public class EconomyShipping implements ShippingStrategy {
    @Override public double calculate(double weight) { return weight * 100; }
}

// 利用側
public double calculateShipping(ShippingStrategy strategy, double weight) {
    return strategy.calculate(weight);
}
Q3の解答

SRP: 1クラス1責務。OCP: 拡張に開き修正に閉じる。LSP: サブタイプは親と置換可能。ISP: インタフェースは小さく分割。DIP: 抽象に依存する。Spring Bootで最も関わりが深いのはDIPです。@AutowiredによるDIがDIPそのものであり、Controller→Service→Repositoryの層構造もインタフェースを介した依存関係で構築されています。
