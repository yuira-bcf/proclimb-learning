Section 6-2

ER図

ER図（Entity-Relationship Diagram）は、データベースのテーブル構造と関連を視覚的に表現する図です。エンティティ（テーブル）、属性（カラム）、リレーション（関連）の3要素で構成されます。

6-2-1 ER図の記法

カーディナリティ 記号 意味 例
1対1 1 --- 1 1つに対して1つ User --- UserProfile
1対多 1 --- * 1つに対して複数 User --- Order
多対多 * --- * 複数対複数（中間テーブル必要） Order --- Product

Section 6-3

正規化（1NF〜3NF）

正規化とは、データの冗長性を排除し、更新異常（挿入異常・更新異常・削除異常）を防ぐためにテーブル構造を整理することです。

6-3-1 非正規形から第1正規形へ

warning 非正規形: 繰り返し項目がある

注文ID 顧客名 商品名1 数量1 商品名2 数量2
1 田中太郎 ノートPC 1 マウス 2

第1正規形（1NF）: 繰り返し項目を排除し、全てのカラムが原子的（これ以上分割できない）な値を持つようにします。

check_circle 第1正規形: 繰り返しを行に分離

注文ID 顧客名 商品名 数量
1 田中太郎 ノートPC 1
1 田中太郎 マウス 2

6-3-2 第2正規形・第3正規形

第2正規形（2NF）: 部分関数従属を排除します。主キーの一部だけに依存するカラムを別テーブルに分離します。

第3正規形（3NF）: 推移的関数従属を排除します。非キーカラムが他の非キーカラムに依存する場合、別テーブルに分離します。

auto_fix_high 正規化の段階まとめ

1NF
繰り返し項目を排除。全カラムが原子値。
2NF
部分関数従属を排除。主キー全体に依存させる。
3NF
推移的関数従属を排除。非キー同士の依存をなくす。
第3正規形のテーブル（最終結果） SQL 
-- 顧客テーブル
CREATE TABLE customers (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE
);

-- 注文テーブル
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    customer_id BIGINT NOT NULL,
    ordered_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

-- 商品テーブル
CREATE TABLE products (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(200) NOT NULL,
    price INT NOT NULL
);

-- 注文明細テーブル（中間テーブル）
CREATE TABLE order_items (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    quantity INT NOT NULL DEFAULT 1,
    unit_price INT NOT NULL,
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);
Section 6-4

主キー・外部キー・インデックス

テーブル設計の3大制約を正しく使うことで、データの整合性を保証し、検索パフォーマンスを向上させます。

6-4-1 キーと制約

制約 役割 特徴
PRIMARY KEY 行を一意に識別する NOT NULL + UNIQUE。テーブルに1つだけ
FOREIGN KEY テーブル間の参照整合性を保証 親テーブルに存在しない値を入れられない
UNIQUE 重複を禁止する NULLは許可される（DB製品による）
NOT NULL NULL値を禁止する 必須項目に付与
INDEX 検索を高速化する WHERE句やJOINで使うカラムに設定

6-4-2 インデックスの仕組み

speed インデックスの効果

INDEXなし: 全件スキャン

100万行を1行ずつ確認
→ 最悪100万回の比較

INDEXあり: B-Tree検索

二分探索のようなツリー検索
→ 約20回の比較で到達

インデックスの作成例 SQL 
-- 単一カラムインデックス
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 複合インデックス（検索条件が複数のとき）
CREATE INDEX idx_orders_status_date ON orders(status, ordered_at);

-- ユニークインデックス
CREATE UNIQUE INDEX idx_users_email ON users(email);
lightbulb インデックスの注意点

インデックスは検索を速くしますが、INSERT/UPDATEは遅くなります（インデックスの更新が必要なため）。WHERE句やJOINで頻繁に使うカラムに絞って作成しましょう。全カラムにインデックスを張るのは逆効果です。

Exercise

6章 練習問題

Q1

以下のテーブルは第何正規形まで満たしていますか？ また、次の正規形にするにはどうすればよいですか。

社員ID社員名部署名部署長名
1田中開発部山田
2鈴木開発部山田
3佐藤営業部高橋
Q2

「ブログシステム」で以下のテーブルが必要です。CREATE TABLE文を書いてください。主キー、外部キー、NOT NULL制約を含めること。

  • authors テーブル（id, name, email）
  • posts テーブル（id, author_id, title, content, created_at）
Q3

postsテーブルで「特定のauthor_idの記事をcreated_atの降順で取得する」クエリが頻繁に実行されます。最適なインデックスを設計してください。

Answer

6章 解説・解答

Q1の解答: 第2正規形（2NF）

繰り返し項目はなく（1NF達成）、全カラムが主キー（社員ID）に完全従属しています（2NF達成）。しかし「部署長名」は「部署名」に依存しており、社員IDに直接は依存していません（推移的関数従属）。第3正規形にするには、部署テーブル（部署名, 部署長名）を分離し、社員テーブルには部署IDの外部キーだけを持たせます。

Q2の解答
CREATE TABLE authors (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE
);

CREATE TABLE posts (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    author_id BIGINT NOT NULL,
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (author_id) REFERENCES authors(id)
);
Q3の解答

複合インデックスを (author_id, created_at DESC) の順で作成します。CREATE INDEX idx_posts_author_date ON posts(author_id, created_at DESC); これにより、特定のauthor_idでフィルタした後、created_atの降順ソートもインデックスで処理できます（カバリングインデックス）。
