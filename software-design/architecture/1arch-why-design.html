Section 1-1

「動くコード」と「良いコード」

プログラミングを始めたばかりの頃は「とにかく動くこと」がゴールです。しかし実務では、コードは書いた後に何十回も読まれ、修正されます。「動くだけのコード」と「良いコード」の違いを理解することが、エンジニアとしての第一歩です。

1-1-1 動くコードの落とし穴

以下の2つのコードはどちらも同じ結果を返します。しかし、チーム開発でメンテナンスするとき、どちらが扱いやすいでしょうか。

悪い例: 全てが1つのメソッドに詰め込まれている Java 
public class OrderProcessor {
    public void process(String userId, String itemId, int qty) {
        // ユーザー検証
        Connection conn = DriverManager.getConnection("jdbc:mysql://...");
        PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE id=?");
        ps.setString(1, userId);
        ResultSet rs = ps.executeQuery();
        if (!rs.next()) throw new RuntimeException("User not found");
        String email = rs.getString("email");

        // 在庫チェック
        ps = conn.prepareStatement("SELECT stock FROM items WHERE id=?");
        ps.setString(1, itemId);
        rs = ps.executeQuery();
        rs.next();
        int stock = rs.getInt("stock");
        if (stock < qty) throw new RuntimeException("在庫不足");

        // 注文作成 + メール送信 + ログ出力...
        // さらに数百行が続く...
    }
}
良い例: 責務が分離されている Java 
public class OrderService {
    private final UserRepository userRepo;
    private final ItemRepository itemRepo;
    private final OrderRepository orderRepo;
    private final NotificationService notifier;

    public OrderService(UserRepository userRepo, ItemRepository itemRepo,
                        OrderRepository orderRepo, NotificationService notifier) {
        this.userRepo = userRepo;
        this.itemRepo = itemRepo;
        this.orderRepo = orderRepo;
        this.notifier = notifier;
    }

    public Order placeOrder(String userId, String itemId, int quantity) {
        User user = userRepo.findById(userId)
                .orElseThrow(() -> new UserNotFoundException(userId));
        Item item = itemRepo.findById(itemId)
                .orElseThrow(() -> new ItemNotFoundException(itemId));

        item.reduceStock(quantity);  // ドメインロジックをエンティティに
        Order order = Order.create(user, item, quantity);

        orderRepo.save(order);
        notifier.sendOrderConfirmation(user, order);
        return order;
    }
}
lightbulb 読みやすさ = 保守しやすさ

良いコードの基準はシンプルです。「半年後の自分（または他のメンバー）が読んで、すぐ理解できるか？」。1つのメソッドが100行を超えたら危険信号です。

1-1-2 良いコードの4つの特徴

star 良いコードの4条件

🔎
可読性
コードを読んで意図が分かる
⚙
変更容易性
修正が局所的で済む
🔬
テスト容易性
単体テストが書きやすい
🚀
拡張容易性

これらは相互に関連しています。責務が分離されたコードは読みやすく、変更しやすく、テストしやすく、拡張しやすいのです。

Section 1-2

技術的負債

「技術的負債（Technical Debt）」とは、短期的な利便性を優先して生じた、将来の修正コストの蓄積です。借金のように利子がつき、放置するほど返済（リファクタリング）コストが膨れ上がります。

1-2-1 技術的負債の種類

種類 例 影響
意図的・慎重な負債 納期優先でリファクタリングを後回し 計画的に返済可能
意図的・無謀な負債 「設計なんて不要」と突き進む 返済困難、破綻リスク大
無意識・慎重な負債 当時のベストが後から不適切と判明 学習で改善可能
無意識・無謀な負債 知識不足で悪い設計に気づかない 最も危険、発見が遅れる

1-2-2 技術的負債の成長曲線

trending_up 技術的負債の成長曲線

設計を怠ると、時間経過とともに変更コストが指数的に増大する

1-2-3 実際の失敗事例

warning よくある技術的負債の例
  • 神クラス（God Class）: 1つのクラスに3,000行以上のコード。修正のたびに全体が壊れるリスク
  • コピペ地獄: 同じ処理が10箇所に散在。1箇所の修正漏れでバグが発生
  • テストなしのリリース: 「動いているから触らないで」が口癖のコードベース
  • 密結合: DB変更でUI層まで修正が波及。1つの変更に3日かかる

check_circle きれいな設計

層が明確で依存は一方向

cancel 無秩序な設計

全てが全てに依存している

Section 1-3

変更容易性・テスト容易性

ソフトウェアの価値は「変更できること」にあります。ビジネス要件は常に変化するため、変更に強い設計が求められます。そしてテスト容易性は、変更に対する安全ネットです。

1-3-1 変更容易性とは

「変更容易性」とは、ある機能の修正が他の機能に影響を与えず、少ないコード変更で目的を達成できる性質です。鍵となる概念は疎結合（Loose Coupling）と高凝集（High Cohesion）です。

settings_suggest 疎結合 と 高凝集

疎結合 (Loose Coupling)

モジュール間の依存関係が最小限。インタフェースを通じて連携し、実装の詳細を隠蔽する。

高凝集 (High Cohesion)

1つのモジュールが1つの責務に集中。関連するデータと処理がまとまっている。

疎結合の実現: インタフェースによる抽象化 Java 
// インタフェースで抽象化
public interface NotificationSender {
    void send(String to, String message);
}

// メール実装
public class EmailSender implements NotificationSender {
    @Override
    public void send(String to, String message) {
        // SMTP経由でメール送信
    }
}

// Slack実装（後から追加しても既存コードに影響なし）
public class SlackSender implements NotificationSender {
    @Override
    public void send(String to, String message) {
        // Slack API経由で送信
    }
}

// 利用側はインタフェースに依存
public class OrderService {
    private final NotificationSender sender;

    public OrderService(NotificationSender sender) {
        this.sender = sender;  // どの実装が来ても動作する
    }
}

1-3-2 テスト容易性とは

テスト容易性が高いコードは、自動テストを書きやすく、変更した際に「壊れていないか」を即座に確認できます。テスト容易性の敵は密結合とグローバル状態です。

テストしにくいコード vs テストしやすいコード Java 
// --- テストしにくい: 内部でnewしている ---
public class ReportService {
    public String generate() {
        DatabaseConnection db = new DatabaseConnection(); // テスト時にDB不要でも接続
        List<Data> data = db.query("SELECT ...");
        return format(data);
    }
}

// --- テストしやすい: 依存を外部から注入 ---
public class ReportService {
    private final DataRepository repository;

    public ReportService(DataRepository repository) {
        this.repository = repository;  // テスト時にモックを渡せる
    }

    public String generate() {
        List<Data> data = repository.findAll();
        return format(data);
    }
}

// テストコード
@Test
void testGenerate() {
    DataRepository mockRepo = mock(DataRepository.class);
    when(mockRepo.findAll()).thenReturn(List.of(new Data("test")));
    ReportService service = new ReportService(mockRepo);

    String result = service.generate();
    assertThat(result).contains("test");
}

Exercise

1章 練習問題

Q1

以下のうち、「技術的負債」に該当するものをすべて選んでください。

  1. 納期に間に合わせるためにテストを書かずにリリースした
  2. 要件どおりに正しく動くプログラムを書いた
  3. 同じ処理を5つのクラスにコピペして実装した
  4. 1つのクラスにDB操作・ビジネスロジック・画面描画を全て書いた
Q2

「疎結合」を実現するためにJavaで最もよく使われる仕組みはどれですか。

  1. 継承（extends）
  2. インタフェース（interface）
  3. static メソッド
  4. public フィールド
Q3

以下のコードがテストしにくい理由を説明し、改善案を述べてください。

public class PriceCalculator {
    public int calculate(String productId) {
        Product p = new ProductDao().findById(productId);
        int tax = new TaxApi().getCurrentRate();
        return p.getPrice() * (100 + tax) / 100;
    }
}
Answer

1章 解説・解答

Q1の解答: A, C, D

A: テストを省略することは意図的な技術的負債です。C: コピペは重複コードを生み、修正漏れのリスクを高めます。D: 複数の責務を1クラスに詰め込むと変更のたびに広範囲に影響します。B: 正しく動くことは品質の一要素ですが、それだけでは負債とは言えません。

Q2の解答: B

インタフェースを使うことで、利用側は具体的な実装クラスに依存せず、抽象に依存できます。これにより実装の差し替えが容易になり、テスト時にモックを使うことも可能になります。継承は結合度が高くなりがちです。

Q3の解答

問題点: ProductDaoとTaxApiをメソッド内部で直接newしているため、テスト時に実際のDBやAPIに接続してしまいます。

改善案: コンストラクタインジェクションで依存を外部から注入する。ProductRepositoryとTaxServiceのインタフェースを作成し、テスト時にはモックを渡せるようにします。
