Section 3-1

なぜ層を分けるか

アプリケーションを役割ごとの「層（Layer）」に分割すると、各層が独立して変更・テスト・再利用できるようになります。レイヤードアーキテクチャは、Web開発で最も広く使われている設計パターンです。

3-1-1 層分けのメリット

layers レイヤー分割の3大メリット

build
変更の局所化
DB変更がUI層に波及しない
science
テスト容易性
各層を独立してテスト可能
group
チーム分業
フロント・バック・DBで分担

もし層を分けなければ、1つのServletクラスにHTMLの生成、ビジネスロジック、SQL文が混在する「スパゲッティコード」になります。これは第1章で学んだ「技術的負債」の典型例です。

3-3-2 各層の実装例

Presentation層: OrderController.java Java 
@RestController
@RequestMapping("/api/orders")
public class OrderController {

    private final OrderService orderService;

    public OrderController(OrderService orderService) {
        this.orderService = orderService;
    }

    @PostMapping
    public ResponseEntity<OrderResponse> createOrder(@Valid @RequestBody OrderForm form) {
        Order order = orderService.placeOrder(form.getUserId(), form.getItems());
        return ResponseEntity.status(HttpStatus.CREATED)
                .body(OrderResponse.from(order));
    }
}
Application層: OrderService.java Java 
@Service
@Transactional
public class OrderService {

    private final OrderRepository orderRepository;
    private final PricingService pricingService;

    public OrderService(OrderRepository orderRepository, PricingService pricingService) {
        this.orderRepository = orderRepository;
        this.pricingService = pricingService;
    }

    public Order placeOrder(Long userId, List<OrderItemRequest> items) {
        // ドメインオブジェクトを生成し、ドメインサービスを呼び出す
        Order order = Order.create(userId, items);
        Money totalPrice = pricingService.calculateTotal(order);
        order.setTotalPrice(totalPrice);

        return orderRepository.save(order);
    }
}
Domain層: Order.java（エンティティ） Java 
// ドメイン層はSpringに依存しない純粋なJavaクラス
public class Order {
    private Long id;
    private Long userId;
    private List<OrderItem> items;
    private Money totalPrice;
    private OrderStatus status;
    private LocalDateTime createdAt;

    public static Order create(Long userId, List<OrderItemRequest> itemRequests) {
        Order order = new Order();
        order.userId = userId;
        order.items = itemRequests.stream()
                .map(OrderItem::fromRequest)
                .collect(Collectors.toList());
        order.status = OrderStatus.PENDING;
        order.createdAt = LocalDateTime.now();
        return order;
    }

    public void cancel() {
        if (this.status != OrderStatus.PENDING) {
            throw new IllegalStateException("発送済みの注文はキャンセルできません");
        }
        this.status = OrderStatus.CANCELLED;
    }
}
Section 3-4

層間の依存ルール

レイヤードアーキテクチャで最も重要なのは「依存の方向」です。上の層は下の層に依存できますが、逆方向の依存は禁止です。これを守ることで各層の独立性が保たれます。

3-4-1 依存ルール一覧

rule 依存ルール: 何がOKで何がNGか

3-4-2 ドメイン層とインフラ層の関係

注意すべきはドメイン層とインフラストラクチャ層の関係です。ドメイン層にはRepositoryのインタフェースを置き、インフラ層にその実装を置きます。これにより、ドメイン層はインフラ層に依存しません（DIPの適用）。

domain/repository/OrderRepository.java（インタフェース） Java 
// ドメイン層: インタフェースのみ（実装の詳細は知らない）
package com.example.shop.domain.repository;

public interface OrderRepository {
    Order save(Order order);
    Optional<Order> findById(Long id);
    List<Order> findByUserId(Long userId);
}
infrastructure/repository/JpaOrderRepository.java（実装） Java 
// インフラ層: 具体的な永続化処理を実装
package com.example.shop.infrastructure.repository;

@Repository
public class JpaOrderRepository implements OrderRepository {
    private final JpaOrderEntityRepository jpa;

    @Override
    public Order save(Order order) {
        OrderEntity entity = OrderEntity.fromDomain(order);
        jpa.save(entity);
        return order;
    }

    @Override
    public Optional<Order> findById(Long id) {
        return jpa.findById(id).map(OrderEntity::toDomain);
    }
}
lightbulb 実務でのポイント

小規模なプロジェクトでは、厳密に4層にこだわる必要はありません。まずは「Controller→Service→Repository」の3層構造をしっかり守ることが大切です。プロジェクトが成長してビジネスロジックが複雑になってきたら、ドメイン層を分離していきましょう。

3章 練習問題

Q1

以下のコードはどの層に配置すべきですか？理由も答えてください。

public class TaxCalculator {
    public Money calculate(Money price, TaxRate rate) {
        return price.multiply(rate.getValue());
    }
}
  1. Presentation層
  2. Application層
  3. Domain層
  4. Infrastructure層
Q2

以下のimport文のうち、レイヤー依存ルール違反になるのはどれですか？

  1. Controller が Service をimport
  2. Service が Repository（インタフェース）をimport
  3. Repository実装 が Controller をimport
  4. Domain層のEntity が JPA の @Entity をimport
Q3

「Repositoryのインタフェースをドメイン層に、実装をインフラ層に置く」設計は、SOLID原則のどれを適用していますか？ その理由を説明してください。

Answer

3章 解説・解答

Q1の解答: C（Domain層）

税金計算はビジネスルールそのものであり、HTTPリクエスト（Presentation層）やDB操作（Infrastructure層）には依存していません。MoneyやTaxRateといった値オブジェクトを使った純粋な計算ロジックなので、Domain層のドメインサービスとして配置するのが適切です。

Q2の解答: C, D

C: Infrastructure層からPresentation層への依存は下から上への逆方向であり、ルール違反です。D: ドメイン層がJPA（フレームワーク）に依存すると、ドメイン層の独立性が損なわれます。厳密な4層ではドメインオブジェクトはPOJOとし、JPAアノテーションはインフラ層のEntityクラスに付けます。

Q3の解答: DIP（依存性逆転の原則）

上位モジュール（Domain層）がインタフェースを定義し、下位モジュール（Infrastructure層）がそれを実装します。これにより、Domain層はInfrastructure層の具体的な実装（MySQL, PostgreSQL等）に依存しません。依存の方向がInfra→Domainに逆転しており、DIPそのものです。
