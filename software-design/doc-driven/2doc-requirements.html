2-1

要件定義書の目的

2-1-1. 要件定義書とは

要件定義書は、「何を作るか」を明確にするためのドキュメントです。システム開発の最上流工程で作成され、後続のすべての設計書の土台となります。

lightbulb ポイント

要件定義書は「設計書の親」です。ここが曖昧だと、すべての設計書が曖昧になり、最終的に生成されるコードも曖昧になります。最初の要件定義が最も重要です。

2-1-2. 要件定義書に含まれる内容

セクション 内容 記述例
プロジェクト概要 背景・目的・対象ユーザー 「個人のタスク管理を効率化するWebアプリ」
機能要件 システムが「何をするか」 「タスクの追加・編集・削除・完了機能」
非機能要件 性能・セキュリティ・可用性 「応答時間3秒以内」「パスワード暗号化」
技術スタック 使用する技術 「Spring Boot 3.x + Thymeleaf + H2」
受け入れ基準 完成の判定基準 「全CRUDが動作すること」
2-2

機能要件の書き方

2-2-1. 機能要件一覧テーブル

機能要件はID・機能名・説明・優先度の4列テーブルで書きます。IDはFR-001のように「Functional Requirement」の略を使います。

table_chart 記入例: Todoアプリの機能要件

ID 機能名 説明 優先度
FR-001 タスク追加 タイトルと説明を入力してタスクを新規作成する 高
FR-002 タスク一覧表示 登録済みの全タスクを一覧表示する 高
FR-003 タスク編集 既存タスクのタイトル・説明を編集する 中
FR-004 タスク削除 タスクを削除する（確認ダイアログ表示後） 高
FR-005 タスク完了切替 タスクの完了/未完了を切り替える 高

2-2-2. ユーザーストーリーの書き方

ユーザーストーリーは「誰が・何を・なぜ」の3要素で書きます。機能要件を利用者の視点から表現したものです。

ユーザーストーリーの書き方 
## ユーザーストーリー

### US-001: タスクを追加する
- **誰が**: 一般ユーザーとして
- **何を**: 新しいタスクを追加したい
- **なぜ**: やるべきことを忘れないように記録するため

### US-002: タスクを完了にする
- **誰が**: 一般ユーザーとして
- **何を**: 完了したタスクにチェックを付けたい
- **なぜ**: 進捗状況を一目で把握するため
warning よくある間違い

「データベースにINSERTする」「POSTリクエストを送る」のような技術的な表現は要件定義では使わないでください。要件定義はあくまで「ユーザーから見た振る舞い」を記述します。技術的な詳細は後の設計書で記述します。

2-3

非機能要件

2-3-1. 非機能要件とは

非機能要件は、システムの「品質」に関する要件です。機能要件が「何をするか」であるのに対し、非機能要件は「どれくらいの品質で動くか」を定義します。

speed 非機能要件の4カテゴリ

speed パフォーマンス

  • 画面表示: 3秒以内
  • API応答: 1秒以内
  • 同時接続: 10ユーザー

security セキュリティ

  • 認証: セッション認証
  • パスワード: BCrypt暗号化
  • CSRF対策: 有効

backup 可用性

  • 稼働率: 99%
  • バックアップ: 日次
  • RTO: 4時間

devices 互換性

  • Chrome / Firefox / Safari
  • PC / タブレット
  • レスポンシブ対応

2-3-2. 訓練校レベルでの非機能要件

実務では非機能要件は非常に詳細に記述しますが、訓練校の演習レベルでは以下の最低限の項目を押さえておけば十分です。

school 訓練校レベルの非機能要件チェックリスト
  • 対応ブラウザ: Chrome最新版
  • DB: H2（開発用インメモリDB）
  • 認証: なし（または基本的なセッション認証）
  • 文字コード: UTF-8
  • 日本語対応: 必須
2-4

テンプレートの使い方

2-4-1. requirements-template.md の構造

本講座では、templates/requirements-template.md を用意しています。このテンプレートの {中括弧} 部分を自分のプロジェクトに合わせて埋めていきます。

downloadテンプレートをダウンロード

smart_toy Codexで対話的に作成する

ダウンロードしたテンプレートをVSCodeで開き、以下のプロンプトをCodexに投入すると、質問形式で1項目ずつ要件定義書を完成できます。

Codex用プロンプト 
requirements-template.md を読み込み、
各{中括弧}の項目について1つずつ質問してください。
私の回答をもとにテンプレートを埋めて、
最終的に完成した要件定義書を出力してください。

- 1項目ずつ順番に質問すること
- 選択肢がある場合は候補を提示すること
- 回答が曖昧な場合は具体例を示して確認すること
templates/requirements-template.md（抜粋） 
# 要件定義書: {プロジェクト名}

### 1.1 背景・目的

### 1.2 対象ユーザー

## 2. 機能要件
| ID | 機能名 | 説明 | 優先度 |
|----|--------|------|--------|
| FR-001 | {機能名} | {説明} | 高/中/低 |

## 3. 非機能要件

2-4-2. 記入例: Todoアプリ

テンプレートをTodoアプリ用に記入した例を示します。

requirements-todo.md（記入済み） 
# 要件定義書: Todoアプリ

## 1. プロジェクト概要
### 1.1 背景・目的
個人のタスク管理を効率化するためのWebアプリケーション。
日々のタスクを登録・管理し、完了状況を一覧で把握できる。

### 1.2 対象ユーザー
- 一般ユーザー: タスクを管理したい個人

## 2. 機能要件
| ID | 機能名 | 説明 | 優先度 |
|----|--------|------|--------|
| FR-001 | タスク追加 | タイトルと説明を入力して新規タスクを作成 | 高 |
| FR-002 | タスク一覧 | 登録済みの全タスクを一覧表示 | 高 |
| FR-003 | タスク編集 | 既存タスクのタイトル・説明を編集 | 中 |
| FR-004 | タスク削除 | タスクを削除（確認後） | 高 |
| FR-005 | 完了切替 | タスクの完了/未完了を切り替え | 高 |

## 3. 非機能要件
### 3.1 パフォーマンス
- 画面表示: 3秒以内
- DB: H2（インメモリ）

### 3.2 互換性
- 対応ブラウザ: Chrome最新版
- レスポンシブ対応: 不要（PC前提）

## 4. 技術スタック
| レイヤー | 技術 |
|----------|------|
| バックエンド | Spring Boot 3.4 |
| テンプレート | Thymeleaf |
| DB | H2 Database |
| ビルド | Gradle |

## 5. 受け入れ基準
- [ ] タスクの追加ができること
- [ ] タスク一覧が表示されること
- [ ] タスクの編集ができること
- [ ] タスクの削除ができること（確認ダイアログあり）
- [ ] タスクの完了/未完了の切り替えができること
2-5

プロンプトへの変換例

2-5-1. 要件定義書からプロンプトへ

記入済みの要件定義書は、そのままプロンプトの冒頭部分として使えます。以下のように「このドキュメントに基づいて実装してください」と付けるだけです。

smart_toy 要件定義書を使ったプロンプト例

以下の要件定義書に基づいて、Spring Boot + Thymeleaf + H2のTodoアプリのプロジェクト構成を提案してください。

# 要件定義書: Todoアプリ

## プロジェクト概要
個人のタスク管理を効率化するためのWebアプリケーション。

## 機能要件
| ID | 機能名 | 説明 | 優先度 |
|----|--------|------|--------|
| FR-001 | タスク追加 | タイトルと説明を入力して新規作成 | 高 |
| FR-002 | タスク一覧 | 全タスクを一覧表示 | 高 |
| FR-003 | タスク編集 | タイトル・説明を編集 | 中 |
| FR-004 | タスク削除 | 確認後に削除 | 高 |
| FR-005 | 完了切替 | 完了/未完了を切り替え | 高 |

## 技術スタック
- Spring Boot 3.4 / Thymeleaf / H2 / Gradle

まずはパッケージ構成とファイル一覧を提示してください。

2-5-2. プロンプト変換のコツ

tips_and_updates 変換のコツ3箇条

1

冒頭に指示文を付ける

「以下の要件定義書に基づいて〇〇してください」と明確に指示する

2

テンプレートのプレースホルダーを全て埋める

{中括弧}が残っているとAIが適切に解釈できない

3

末尾に期待する出力形式を指定する

「パッケージ構成を示してください」「Entityクラスを生成してください」など

edit_note

練習問題

1

要件定義書における「機能要件」と「非機能要件」の違いを説明してください。

2

以下のうち、機能要件に該当するものをすべて選んでください。

  1. ユーザーがログインできる
  2. 画面表示が3秒以内
  3. 商品を検索できる
  4. パスワードが暗号化される
  5. 商品をカートに追加できる
3

「メモ帳アプリ」の機能要件を3つ以上、FR-001の形式で書いてください。

解答を表示
問1の解答

機能要件は、システムが「何をするか」（振る舞い）を定義します。例: タスクの追加、一覧表示、削除。

非機能要件は、システムの「品質」を定義します。例: 応答速度、セキュリティ、対応ブラウザ。

問2の解答

A, C, E が機能要件です。

B（応答速度）とD（暗号化）は非機能要件（パフォーマンスとセキュリティ）です。

問3の解答例

| ID | 機能名 | 説明 | 優先度 |

| FR-001 | メモ作成 | タイトルと本文を入力してメモを新規作成 | 高 |

| FR-002 | メモ一覧 | 作成済みメモをタイトル一覧で表示 | 高 |

| FR-003 | メモ編集 | 既存メモのタイトル・本文を編集 | 中 |

| FR-004 | メモ削除 | メモを削除（確認ダイアログ付き） | 高 |
