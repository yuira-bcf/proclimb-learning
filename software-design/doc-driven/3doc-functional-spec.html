3-1

機能仕様書とは

3-1-1. 要件定義書との違い

要件定義書が「何を作るか」を定義するのに対し、機能仕様書は「どう動くか」を定義します。具体的には、ユーザーがどのような操作をし、システムがどう応答するかを詳細に記述します。

compare_arrows 要件定義書 vs 機能仕様書

assignment 要件定義書

What（何を作るか）

例: 「タスクの追加ができる」

fact_check 機能仕様書

How（どう動くか）

例: 「一覧画面で新規作成ボタンを押すと登録フォームが表示され、タイトル(必須・100文字以内)と説明を入力して登録ボタンを押すと一覧に戻る」

3-1-2. 機能仕様書に含まれる内容

機能仕様書は以下の要素で構成されます。

  • ユースケース一覧: システムの振る舞いを操作単位で定義
  • 画面一覧: 必要な画面とURLの対応
  • 画面遷移: 画面間の遷移ルール
  • ビジネスルール: バリデーションやデータの整合性ルール
  • メッセージ一覧: 成功・エラー時の表示メッセージ
3-2

ユースケース記述

3-2-1. ユースケースとは

ユースケースは「ユーザーがシステムを使って達成する目標」を1つの単位として記述したものです。各ユースケースには、アクター（操作者）、事前条件、操作手順（フロー）、事後条件を記載します。

3-2-2. ユースケース一覧テーブル

まず全体像を把握するために、ユースケースを一覧表で整理します。

table_chart 記入例: Todoアプリのユースケース一覧

UC-ID ユースケース名 アクター 概要 優先度
UC-001 タスクを追加する 一般ユーザー 新規タスクを登録する 高
UC-002 タスク一覧を見る 一般ユーザー 全タスクを一覧表示する 高
UC-003 タスクを編集する 一般ユーザー 既存タスクを修正する 中
UC-004 タスクを削除する 一般ユーザー 不要なタスクを削除する 高
UC-005 タスクを完了にする 一般ユーザー タスクの完了/未完了を切替 高

3-2-3. ユースケース詳細の書き方

各ユースケースの詳細を正常系フロー・代替フロー・例外フローの3つに分けて記述します。

UC-001: タスクを追加する（詳細） 
### UC-001: タスクを追加する

| 項目 | 内容 |
|------|------|
| アクター | 一般ユーザー |
| 事前条件 | ユーザーがタスク一覧画面を表示している |
| 事後条件 | 新しいタスクがDBに保存され一覧に表示される |

**正常系フロー:**
1. ユーザーが「新規作成」ボタンをクリックする
2. システムがタスク登録フォームを表示する
3. ユーザーがタイトル（必須）と説明（任意）を入力する
4. ユーザーが「登録」ボタンをクリックする
5. システムが入力値を検証する
6. システムがタスクをDBに保存する
7. システムがタスク一覧画面にリダイレクトし、
   成功メッセージ「タスクを登録しました」を表示する

**代替フロー:**
- 5a. タイトルが未入力の場合
  1. エラーメッセージ「タイトルは必須です」を表示
  2. 入力済みの値を保持したままフォームを再表示
  3. ステップ3に戻る
- 5b. タイトルが100文字を超える場合
  1. エラーメッセージ「100文字以内で入力してください」を表示
  2. ステップ3に戻る

**例外フロー:**
- E1. サーバーエラーが発生した場合
  1. エラーページを表示する
lightbulb なぜフローを3つに分けるのか

正常系はハッピーパス（すべてうまくいく場合）、代替フローはユーザーの入力ミスなどの想定内の異常、例外フローはシステム障害などの想定外の異常です。これを分けて書くことで、テストケースの抜け漏れを防げます。

3-3

画面一覧・機能一覧

3-3-1. 画面一覧テーブル

画面一覧は、システムに必要なすべての画面とそのURLを一覧化したものです。

画面ID 画面名 URL 概要
SCR-001 タスク一覧 /todos 全タスクを表形式で表示
SCR-002 タスク登録 /todos/new 新規タスク登録フォーム
SCR-003 タスク詳細 /todos/{id} タスクの詳細表示
SCR-004 タスク編集 /todos/{id}/edit タスク編集フォーム

3-3-2. 画面遷移の記述

画面遷移は、ユーザーがどの画面からどの画面に移動できるかを示します。Markdownでは矢印記法で簡潔に書けます。

account_tree Todoアプリ画面遷移図

SCR-001: タスク一覧
新規作成 ↓
SCR-002: 登録
詳細リンク ↓
SCR-003: 詳細
編集ボタン ↓
SCR-004: 編集

3-3-3. ビジネスルール

ビジネスルールは、データの整合性やバリデーションに関する業務上のルールです。

ルールID ルール名 説明 適用画面
BR-001 タイトル必須 タスクのタイトルは空欄不可、1〜100文字 SCR-002, SCR-004
BR-002 説明文字数制限 説明は最大500文字 SCR-002, SCR-004
BR-003 削除確認 削除実行前に確認ダイアログを表示する SCR-001, SCR-003

テンプレートの使い方

3-4-1. functional-spec-template.md の構造

templates/functional-spec-template.md は、ユースケース一覧、ユースケース詳細、画面一覧、ビジネスルール、メッセージ一覧のセクションで構成されています。

3-4-2. テンプレート記入のポイント

check_circle 記入のポイント
  • ユースケースIDは要件IDと対応させる: FR-001 に対して UC-001 が対応するように
  • 画面IDも紐付ける: ユースケース内で「SCR-002が表示される」のように参照する
  • 代替フローを忘れない: バリデーションエラー時の挙動は必ず記述する
  • メッセージは具体的に: 「エラーです」ではなく「タイトルは必須です」と書く
3-5

プロンプトへの変換

3-5-1. 機能仕様書からControllerの生成

機能仕様書のユースケースと画面一覧は、Spring BootのControllerクラスを生成するプロンプトに直接使えます。

smart_toy 機能仕様書を使ったプロンプト例

以下の機能仕様書に基づいて、Spring Boot + ThymeleafのControllerクラスを作成してください。

## 画面一覧
| 画面ID | 画面名 | URL | HTTPメソッド |
|--------|--------|-----|-------------|
| SCR-001 | タスク一覧 | /todos | GET |
| SCR-002 | タスク登録 | /todos/new (GET) / /todos (POST) | GET, POST |
| SCR-003 | タスク詳細 | /todos/{id} | GET |
| SCR-004 | タスク編集 | /todos/{id}/edit (GET) / /todos/{id} (PUT) | GET, PUT |

## ビジネスルール
- BR-001: タイトル必須、1〜100文字
- BR-002: 説明は最大500文字
- BR-003: 削除前に確認ダイアログ表示

## バリデーションエラー時
- 入力値を保持してフォームを再表示
- エラーメッセージをフィールド横に表示

TodoControllerクラスを生成してください。

3-5-2. 設計書の組み合わせ

実際のプロンプトでは、要件定義書と機能仕様書を組み合わせて使います。要件定義書で「全体像」を、機能仕様書で「詳細な振る舞い」を伝えることで、AIはより正確なコードを生成できます。

lightbulb プロンプト構成のパターン

パターン: 全体 → 詳細

  1. 冒頭に指示文（「以下の仕様書に基づいて〇〇を作成してください」）
  2. 要件定義書の技術スタック部分
  3. 機能仕様書のユースケース・画面一覧
  4. 末尾に出力形式の指定（「Controllerクラスを生成してください」）
edit_note

練習問題

1

ユースケースの「正常系フロー」「代替フロー」「例外フロー」の違いを、それぞれ1文で説明してください。

2

「メモ帳アプリ」の画面一覧を、画面ID・画面名・URL・概要の4列テーブルで作成してください（最低4画面）。

3

UC-001「メモを新規作成する」のユースケース詳細を、正常系フロー（5ステップ以上）と代替フロー（1つ以上）で書いてください。

解答を表示
問1の解答

正常系フロー: すべてが正しく動作する場合（ハッピーパス）の操作手順。

代替フロー: ユーザーの入力ミスなど、想定内の異常が発生した場合の処理。

例外フロー: サーバー障害など、想定外の異常が発生した場合の処理。

問2の解答例

| SCR-001 | メモ一覧 | /memos | 全メモをタイトル一覧で表示 |

| SCR-002 | メモ作成 | /memos/new | 新規メモ作成フォーム |

| SCR-003 | メモ詳細 | /memos/{id} | メモの全文表示 |

| SCR-004 | メモ編集 | /memos/{id}/edit | メモ編集フォーム |

問3の解答例

正常系: (1)ユーザーがメモ一覧で「新規作成」をクリック (2)システムが作成フォームを表示 (3)ユーザーがタイトルと本文を入力 (4)「保存」をクリック (5)システムが入力値を検証 (6)システムがDBに保存 (7)一覧画面にリダイレクト

代替フロー: 5a. タイトル未入力の場合 → 「タイトルは必須です」を表示し、入力値を保持してフォームを再表示
