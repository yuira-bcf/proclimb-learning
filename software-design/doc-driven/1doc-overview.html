1-1

設計書駆動開発の概要

設計書駆動開発（Document-Driven Development）とは

設計書駆動開発とは、設計書をまず作成し、その設計書をそのままAIプロンプトの入力として使う開発手法です。従来のソフトウェア開発では「設計書」と「実装」が分かれていましたが、AI時代においては設計書こそがプロンプトそのものになります。

lightbulb 核心的なアイデア

良い設計書 = 良いプロンプト。AIに「何を作るか」を正確に伝えるための最も確実な方法は、構造化された設計書を書くことです。

プロのエンジニアが現場で作成する設計書の書き方を学ぶことで、以下の2つのスキルが同時に身につきます。

school 設計書駆動開発で身につく2つのスキル

description

設計書を書く力

要件定義・仕様書・API設計など、プロの現場で使われる設計ドキュメントの書き方

smart_toy

プロンプトを書く力

AIに的確に指示を出して、高品質なコードを生成させるプロンプト作成スキル

1-1-2. 従来の開発とAI時代の開発

従来のソフトウェア開発では、設計書を書いた後に人間が手作業でコーディングしていました。AI時代の開発では、設計書を書いた後にAIがコードを生成します。つまり、設計書の品質がそのまま生成されるコードの品質に直結するのです。

比較項目 従来の開発 AI時代の設計書駆動開発
設計書の役割 人間が読むための仕様書 AIへの入力（プロンプト）
コーディング 人間が1行ずつ書く AIが一括生成
品質の決め手 プログラマーのスキル 設計書（プロンプト）の品質
開発速度 数日〜数週間 数時間〜1日

なぜ設計書からプロンプトを書くのか

1-2-1. 「いきなりプロンプト」の問題点

多くの初心者は、設計書を書かずにいきなりAIにプロンプトを投げてしまいます。これでは曖昧な指示になり、生成されるコードの品質が安定しません。

1-2-2. 設計書駆動の3つのメリット

stars 設計書駆動開発の3つのメリット

1

再現性が高い

同じ設計書からは、何度プロンプトを投げても同じ品質のコードが生成されます。チームメンバーが変わっても品質がブレません。

2

レビューしやすい

プロンプトの代わりに設計書をレビューできるため、プログラミングが分からない人でも仕様の確認ができます。

3

変更に強い

仕様変更が発生したら設計書を修正し、プロンプトを再実行するだけ。コードを1行ずつ修正する必要がありません。

1-3-2. 設計書のフォーマット

本講座では、すべての設計書をMarkdownファイル（.md）で作成します。Markdown形式には以下のメリットがあります。

  • テキストベース: Gitでバージョン管理できる
  • AIが理解しやすい: 構造化されたテキストはAIが正確に解釈できる
  • コピー&ペーストが簡単: そのままプロンプトに貼り付けられる
  • テンプレート化しやすい: プレースホルダーを置き換えるだけで使える
info Markdownとは

Markdown（マークダウン）は、見出し・箇条書き・表・コードブロックなどを簡単な記号で書けるテキスト形式です。# 見出し、- 箇条書き、| 表 |のように書きます。GitHubのREADMEやSlackのメッセージでも使われています。

edit_note

練習問題

1

設計書駆動開発の最大のメリットは何ですか。次の選択肢から正しいものを選んでください。

  1. コーディングが不要になる
  2. 設計書がそのままプロンプトとして使え、再現性の高い開発ができる
  3. 設計書を書く手間が省ける
  4. テストが不要になる
2

「いきなりプロンプト」の問題点を3つ挙げてください。

3

設計書駆動開発の6ステップを正しい順番で並べてください。

（テスト・検証 / 仕様書作成 / 要件定義 / コード生成 / 詳細設計 / プロンプト作成）

解答を表示
問1の解答

B. 設計書がそのままプロンプトとして使え、再現性の高い開発ができる

設計書駆動開発では、設計書がプロンプトの入力になるため、同じ設計書からは何度でも同品質のコードを生成できます。Aは不正解（テストやレビューは人間が行う）、Cは不正解（設計書は書く必要がある）、Dは不正解（テストは必要）。

問2の解答

以下の3つが主な問題点です。

  • 曖昧さ: 仕様が曖昧なため、AIが推測で実装してしまい、意図と異なるコードが生成される
  • 再現性の低さ: 毎回異なるプロンプトになりがちで、品質が安定しない
  • レビューの困難さ: 口頭やチャットベースの指示はレビューしにくく、仕様の認識ズレが起きやすい
問3の解答

要件定義 → 仕様書作成 → 詳細設計 → プロンプト作成 → コード生成 → テスト・検証

上流工程（何を作るか）から下流工程（どう実装するか）へと順に進めていきます。
