6-1

画面設計書の構成

6-1-1. 画面設計書とは

画面設計書は、各画面のレイアウト・入力項目・ボタン・バリデーションを定義するドキュメントです。Thymeleafテンプレートの設計に直結します。

6-1-2. 入力項目定義テーブル

フォーム画面では、各入力項目のHTML要素・type属性・name属性・必須/任意・バリデーションを定義します。

input SCR-002: タスク登録画面 入力項目定義

# 項目名 HTML要素 type name属性 必須 バリデーション 初期値
1 タイトル input text title はい 1-100文字 空
2 説明 textarea - description いいえ 最大500文字 空

6-1-3. レイアウト概要図

画面のレイアウトをテキストベースで記述します。

SCR-001: タスク一覧画面レイアウト 
+--------------------------------------------------+
| ヘッダー: Todoアプリ                               |
+--------------------------------------------------+
| タスク一覧            [新規作成ボタン]              |
+--------------------------------------------------+
| | # | タイトル    | 状態   | 作成日     | 操作  | |
| | 1 | 買い物に行く | 未完了 | 2026/03/01 | 詳細  | |
| | 2 | レポート     | 完了   | 2026/03/01 | 詳細  | |
+--------------------------------------------------+
| フッター                                          |
+--------------------------------------------------+
6-2

画面遷移図

6-2-1. 画面遷移の設計

画面遷移図は、ユーザーがどの画面からどの操作でどの画面に遷移するかを示します。

account_tree Todoアプリ 画面遷移図

遷移元 アクション 遷移先 HTTPメソッド
SCR-001 一覧 新規作成ボタン SCR-002 登録フォーム GET
SCR-002 登録 登録ボタン（成功） SCR-001 一覧 POST → redirect
SCR-002 登録 登録ボタン（エラー） SCR-002 登録フォーム（再表示） POST → return
SCR-001 一覧 詳細リンク SCR-003 詳細 GET
SCR-003 詳細 編集ボタン SCR-004 編集フォーム GET
SCR-003 詳細 削除ボタン（確認後） SCR-001 一覧 POST → redirect
lightbulb PRGパターン

POST送信が成功した後はredirect（リダイレクト）で一覧画面に戻ります。これはPRG（Post-Redirect-Get）パターンと呼ばれ、ブラウザの更新ボタンで二重送信されるのを防ぎます。

6-3

テスト仕様書の構成

6-3-1. テスト仕様書とは

テスト仕様書は、システムが仕様通りに動作するかを確認するためのドキュメントです。何をどのようにテストし、何が期待結果かを記載します。

6-3-2. テスト種別

テスト種別 テスト対象 例 訓練校での優先度
画面テスト 画面の表示・操作 一覧画面が表示されるか 高
バリデーションテスト 入力値の検証 空欄送信でエラーが出るか 高
CRUDテスト データの登録・更新・削除 登録後に一覧に表示されるか 高
境界値テスト 文字数の上限・下限 100文字ちょうど/101文字 中
6-4

テストケースの書き方

6-4-1. テストケーステーブル

テストケースはID・カテゴリ・テスト項目・前提条件・手順・期待結果・優先度で構成します。

fact_check Todoアプリ テストケース例

TC-ID カテゴリ テスト項目 手順 期待結果 優先度
TC-001 正常系 タスク一覧が表示される /todosにアクセス 一覧画面が表示される 高
TC-002 正常系 タスクを追加できる タイトル「テスト」で登録 一覧に追加される 高
TC-003 異常系 タイトル空欄で登録 タイトル未入力で送信 エラーメッセージ表示 高
TC-004 境界値 タイトル100文字で登録 100文字入力して送信 正常に登録される 中
TC-005 境界値 タイトル101文字で登録 101文字入力して送信 エラーメッセージ表示 中
TC-006 正常系 タスクを削除できる 詳細画面で削除ボタン 一覧から消える 高
6-5

テンプレートからプロンプトへ

6-5-1. 画面設計書からThymeleafテンプレートへ

smart_toy 画面設計書からThymeleafを生成するプロンプト

以下の画面設計書に基づいて、Thymeleafテンプレートを作成してください。

## SCR-002: タスク登録画面
- URL: /todos/new (GET) → フォーム表示
- 送信先: /todos (POST)
- テンプレート: todos/form.html

## 入力項目
| 項目名 | 要素 | name属性 | 必須 | バリデーション |
|--------|------|---------|------|-------------|
| タイトル | input[text] | title | はい | 1-100文字 |
| 説明 | textarea | description | いいえ | 最大500文字 |

## エラー表示
- 各フィールドの下にth:errorsでエラーメッセージ表示
- Bootstrap 5のフォームスタイルを使用

## ボタン
- 登録ボタン（type=submit）
- キャンセルリンク（/todosへ遷移）

Thymeleafのth:object, th:field, th:errorsを使用してください。

6-5-2. テスト仕様書からテストコード生成

テスト仕様書のテストケースは、将来的にJUnitテストコードの生成プロンプトとしても使えます。まずは手動テストの手順書として活用しましょう。

check_circle 画面設計 + テスト仕様 = 品質保証

画面設計書で「何を表示するか」を定義し、テスト仕様書で「正しく表示されるか」を検証します。この2つの設計書をセットで作ることで、開発と品質保証を同時に設計できます。

6-5-3. テンプレートを活用する

画面設計書・テスト仕様書のテンプレートをダウンロードし、Codexで対話的に作成できます。

download画面設計テンプレート downloadテスト仕様テンプレート

smart_toy Codexで対話的に作成する

ダウンロードしたテンプレートをVSCodeで開き、以下のプロンプトをCodexに投入すると、質問形式で1項目ずつ画面設計書・テスト仕様書を完成できます。

Codex用プロンプト 
screen-design-template.md と test-spec-template.md を読み込み、
各{中括弧}の項目について1つずつ質問してください。
私の回答をもとにテンプレートを埋めて、
最終的に完成した画面設計書とテスト仕様書を出力してください。

- 1項目ずつ順番に質問すること
- 画面ごとに入力項目・表示項目・アクションを確認すること
- テストケースは正常系・異常系を網羅するよう質問すること
- 回答が曖昧な場合は具体例を示して確認すること

練習問題

1

「メモ帳アプリ」のメモ作成画面の入力項目定義テーブルを作成してください（項目名・HTML要素・必須・バリデーション）。

2

テストケースの「正常系」「異常系」「境界値」の違いを説明してください。

3

「メモ帳アプリ」のメモ作成機能に対するテストケースを、正常系2件・異常系2件・境界値1件の計5件書いてください。

解答を表示
問1の解答例

| タイトル | input[text] | title | はい | 1-100文字 |

| 本文 | textarea | body | はい | 1-10000文字 |

問2の解答

正常系: 正しい入力で正しく動作することを確認するテスト。

異常系: 不正な入力や操作に対して、適切にエラー処理されることを確認するテスト。

境界値: 入力値の上限・下限ちょうど/超過で正しく判定されることを確認するテスト。

問3の解答例

正常系: TC-001 タイトル・本文入力して正常に登録される / TC-002 登録後に一覧画面にリダイレクトされる

異常系: TC-003 タイトル空欄で「タイトルは必須です」が表示 / TC-004 本文空欄で「本文は必須です」が表示

境界値: TC-005 タイトル100文字で正常に登録される（101文字ならエラー）
