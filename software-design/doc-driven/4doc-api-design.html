4-1

API設計書の構成

4-1-1. API設計書とは

API設計書は、サーバーとクライアント間の通信仕様を定義するドキュメントです。どのURLにどのHTTPメソッドでアクセスすると何が返ってくるかを正確に記述します。

lightbulb Thymeleafアプリでも使うの?

本講座ではThymeleaf（サーバーサイドレンダリング）を使いますが、API設計書の考え方はControllerのエンドポイント設計にそのまま使えます。URLとHTTPメソッドの対応表は、Controllerクラスのメソッドと1対1で対応します。

4-1-2. API設計書の主要セクション

api API設計書の構成要素

1 基本情報 ベースURL、認証方式、レスポンス形式
2 エンドポイント一覧 HTTPメソッド、パス、概要
3 リクエスト/レスポンス パラメータ、JSONボディ、レスポンスJSON
4 エラーコード HTTPステータス、エラーコード、メッセージ
4-2

エンドポイント一覧

4-2-1. CRUD対応のエンドポイント設計

REST APIの基本であるCRUD操作は、以下のようにHTTPメソッドとURLパスを対応させます。

table_chart Todoアプリ エンドポイント一覧

# メソッド パス 概要 対応CRUD
1 GET /todos タスク一覧画面を表示 Read（一覧）
2 GET /todos/new 新規登録フォームを表示 -
3 POST /todos タスクを新規作成 Create
4 GET /todos/{id} タスク詳細を表示 Read（詳細）
5 GET /todos/{id}/edit 編集フォームを表示 -
6 PUT /todos/{id} タスクを更新 Update
7 DELETE /todos/{id} タスクを削除 Delete

4-2-2. URL設計のルール

rule URL設計の基本ルール
  • リソース名は複数形: /todos（/todoではない）
  • IDはパスパラメータ: /todos/{id} でリソースを特定
  • 動詞はHTTPメソッドで表現: GET=取得、POST=作成、PUT=更新、DELETE=削除
  • ネストは浅く: /users/{userId}/todos は最大2階層まで
4-3

リクエスト/レスポンス定義

4-3-1. リクエストボディの定義

POSTやPUTで送信するデータの形式を定義します。各フィールドの型・必須/任意・バリデーションを明記します。

POST /todos リクエスト定義 
// リクエストボディ（JSON形式で記述）
{
  "title": "買い物に行く",
  "description": "牛乳とパンを買う"
}

// フィールド定義テーブル
| フィールド   | 型     | 必須 | バリデーション | 説明         |
|-------------|--------|------|--------------|-------------|
| title       | String | はい  | 1-100文字     | タスクタイトル |
| description | String | いいえ | 最大500文字    | タスク説明    |

4-3-2. レスポンスの定義

APIが返すデータの形式を、HTTPステータスコードごとに定義します。

GET /todos レスポンス（200 OK） 
[
  {
    "id": 1,
    "title": "買い物に行く",
    "description": "牛乳とパンを買う",
    "completed": false,
    "createdAt": "2026-03-01T10:00:00",
    "updatedAt": "2026-03-01T10:00:00"
  },
  {
    "id": 2,
    "title": "レポートを書く",
    "description": null,
    "completed": true,
    "createdAt": "2026-03-01T09:00:00",
    "updatedAt": "2026-03-01T14:30:00"
  }
]
lightbulb Thymeleafの場合

Thymeleafを使う場合、APIはJSONではなくHTMLを返します。しかし、Controllerが受け取るパラメータと返すModelの属性は同じ構造です。API設計書のフィールド定義は、Formクラスのフィールドとして直接使えます。

エラーコード定義

4-4-1. エラーコード一覧

システムで発生しうるエラーと、その際のHTTPステータスコード・エラーメッセージを定義します。

エラーコード HTTPステータス 説明 対処法
VALIDATION_ERROR 400 入力値バリデーションエラー エラーフィールドを確認
NOT_FOUND 404 指定されたリソースが存在しない IDを確認

4-5

テンプレートからプロンプトへ

4-5-1. API設計書からSpring Boot Controllerへ

API設計書のエンドポイント一覧は、Spring BootのControllerクラスにほぼそのまま変換できます。

smart_toy API設計書からControllerを生成するプロンプト

以下のAPI設計書に基づいて、Spring Boot + ThymeleafのTodoControllerクラスを作成してください。

## エンドポイント一覧
| メソッド | パス | 概要 | Controllerメソッド名 |
|---------|------|------|-------------------|
| GET | /todos | 一覧表示 | list() |
| GET | /todos/new | 登録フォーム | showCreateForm() |
| POST | /todos | 新規作成 | create() |
| GET | /todos/{id} | 詳細表示 | show() |
| GET | /todos/{id}/edit | 編集フォーム | showEditForm() |
| POST | /todos/{id} | 更新 | update() |
| POST | /todos/{id}/delete | 削除 | delete() |

## リクエストパラメータ（POST /todos, POST /todos/{id}）
| フィールド | 型 | 必須 | バリデーション |
|-----------|-----|------|-------------|
| title | String | はい | @NotBlank, @Size(max=100) |
| description | String | いいえ | @Size(max=500) |

## バリデーションエラー時
- BindingResultでエラーを検出
- エラーがあればフォームを再表示（入力値保持）

@Controllerアノテーション、TodoServiceへの依存注入、
各メソッドの@GetMapping/@PostMappingを含めてください。

4-5-2. 生成されるControllerのイメージ

TodoController.java（生成イメージ） 
@Controller
@RequestMapping("/todos")
public class TodoController {

    private final TodoService todoService;

    public TodoController(TodoService todoService) {
        this.todoService = todoService;
    }

    // GET /todos - 一覧表示
    @GetMapping
    public String list(Model model) {
        model.addAttribute("todos", todoService.findAll());
        return "todos/list";
    }

    // GET /todos/new - 登録フォーム
    @GetMapping("/new")
    public String showCreateForm(Model model) {
        model.addAttribute("todoForm", new TodoForm());
        return "todos/form";
    }

    // POST /todos - 新規作成
    @PostMapping
    public String create(
            @Valid TodoForm form,
            BindingResult result) {
        if (result.hasErrors()) {
            return "todos/form";
        }
        todoService.create(form);
        return "redirect:/todos";
    }
}
check_circle 設計書とコードの対応

エンドポイント一覧の各行が、Controllerの各メソッドに1対1で対応していることが分かります。設計書が正確であればあるほど、AIが生成するコードも正確になります。
