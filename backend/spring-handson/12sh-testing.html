Section 12-1

なぜテストを書くのか

ここまでTodoアプリの機能をすべてブラウザで手動確認してきました。しかし、機能が増えるにつれて「変更した箇所が別の機能を壊していないか」を毎回手動で確認するのは現実的ではありません。自動テストを書くことで、コードを変更するたびにワンコマンドで全機能を検証できるようになります。

12-1-1 手動テストの限界

compare 手動テスト vs 自動テスト

手動テスト
☐ ブラウザを開いてURLを入力
☐ フォームに値を入力して送信
☐ 画面を目視で確認
☐ 全ページ分を繰り返す...
所要時間: 10〜30分
自動テスト
☑ コマンド1つで全テスト実行
☑ 結果は合格/不合格で自動判定
☑ 変更のたびに何度でも実行可能
☑ CI/CDパイプラインに組み込める
所要時間: 数秒〜数十秒

info テスト環境について

Spring Bootプロジェクトには、JUnit5とMockitoがデフォルトで含まれています（spring-boot-starter-test）。追加の依存関係は不要で、すぐにテストを書き始められます。テストコードはsrc/test/java/配下に配置します。

Section 12-2

Repositoryテストを書く

最初に、データベースとのやり取りが正しく動作するかを検証するRepositoryテストを書きます。@DataJpaTestを使うと、JPAに関連するコンポーネントだけが起動するため、高速にテストできます。

12-2-1 テストクラスを作成する

src/test/java/com/example/todo/ディレクトリにTodoRepositoryTest.javaを新規作成してください。以下のコードを手で入力しましょう。

src/test/java/com/example/todo/TodoRepositoryTest.java 
package com.example.todo;

import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.autoconfigure.orm.jpa.DataJpaTest;
import org.springframework.boot.test.autoconfigure.orm.jpa.TestEntityManager;

import java.time.LocalDate;
import java.util.List;

import static org.assertj.core.api.Assertions.assertThat;

@DataJpaTest
class TodoRepositoryTest {

    @Autowired
    private TestEntityManager entityManager;

    @Autowired
    private TodoRepository todoRepository;

    @Test
    @DisplayName("ステータスで絞り込めること")
    void findByStatus_returnsMatchingTodos() {
        // 準備: テストデータを登録
        Todo todo1 = new Todo();
        todo1.setTitle("テスト1");
        todo1.setStatus(TodoStatus.NOT_STARTED);
        todo1.setDueDate(LocalDate.of(2026, 4, 1));
        entityManager.persist(todo1);

        Todo todo2 = new Todo();
        todo2.setTitle("テスト2");
        todo2.setStatus(TodoStatus.COMPLETED);
        todo2.setDueDate(LocalDate.of(2026, 4, 1));
        entityManager.persist(todo2);

        Todo todo3 = new Todo();
        todo3.setTitle("テスト3");
        todo3.setStatus(TodoStatus.NOT_STARTED);
        todo3.setDueDate(LocalDate.of(2026, 4, 1));
        entityManager.persist(todo3);

        entityManager.flush();

        // 実行: NOT_STARTEDで検索
        List<Todo> result = todoRepository.findByStatus(TodoStatus.NOT_STARTED);

        // 検証: 2件返ること
        assertThat(result).hasSize(2);
        assertThat(result).allMatch(t -> t.getStatus() == TodoStatus.NOT_STARTED);
    }

    @Test
    @DisplayName("タイトルの部分一致検索ができること")
    void findByTitleContaining_returnsMatchingTodos() {
        Todo todo1 = new Todo();
        todo1.setTitle("Spring Boot勉強");
        todo1.setStatus(TodoStatus.NOT_STARTED);
        todo1.setDueDate(LocalDate.of(2026, 4, 1));
        entityManager.persist(todo1);

        Todo todo2 = new Todo();
        todo2.setTitle("買い物リスト");
        todo2.setStatus(TodoStatus.NOT_STARTED);
        todo2.setDueDate(LocalDate.of(2026, 4, 1));
        entityManager.persist(todo2);

        entityManager.flush();

        List<Todo> result = todoRepository.findByTitleContaining("Spring");

        assertThat(result).hasSize(1);
        assertThat(result.get(0).getTitle()).contains("Spring");
    }
}
info コードの解説
  • @DataJpaTest -- JPA関連のBeanだけを起動する軽量テスト。インメモリH2データベースを使うため高速です。
  • TestEntityManager -- テスト用のEntityManager。persist()でテストデータを直接DBに登録します。
  • @DisplayName -- テスト結果レポートに表示される日本語名を付けます。
  • assertThat() -- AssertJライブラリの検証メソッド。hasSize(2)で件数、allMatch()で全件の条件を検証します。

timeline テストの3ステップ: 準備 → 実行 → 検証

1. 準備（Arrange）
テストデータをDBに登録する
2. 実行（Act）
テスト対象のメソッドを呼ぶ
3. 検証（Assert）
結果が期待通りか確認する
Section 12-3

Serviceテストを書く

Service層のテストでは、データベースへの実際のアクセスを避けるためにMockitoを使います。Repositoryの動作をモック（偽物）に置き換えることで、Serviceのビジネスロジックだけを純粋にテストできます。

12-3-1 モックとは何か

science モックを使ったServiceテストの仕組み

TodoService
テスト対象（本物）
→
TodoRepository
モック（偽物）
モックは「findById(1L)を呼ばれたらこのTodoを返す」と
事前にプログラムされた偽オブジェクト

モック（Mock）は「呼ばれたら決まった値を返す偽物のオブジェクト」です。本物のRepositoryはDBアクセスが必要ですが、モックを使えばDB不要でServiceだけをテストできます。

12-3-2 TodoServiceTestを作成する

src/test/java/com/example/todo/にTodoServiceTest.javaを作成してください。

src/test/java/com/example/todo/TodoServiceTest.java 
package com.example.todo;

import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;

import java.util.Optional;

import static org.assertj.core.api.Assertions.assertThat;
import static org.assertj.core.api.Assertions.assertThatThrownBy;
import static org.mockito.Mockito.verify;
import static org.mockito.Mockito.when;

@ExtendWith(MockitoExtension.class)
class TodoServiceTest {

    @Mock
    private TodoRepository todoRepository;

    @InjectMocks
    private TodoService todoService;

    @Test
    @DisplayName("IDで1件取得できること")
    void findById_returnsTodo() {
        // 準備: モックの振る舞いを定義
        Todo todo = new Todo();
        todo.setId(1L);
        todo.setTitle("テストTodo");
        when(todoRepository.findById(1L)).thenReturn(Optional.of(todo));

        // 実行
        Todo result = todoService.findById(1L);

        // 検証
        assertThat(result.getTitle()).isEqualTo("テストTodo");
        verify(todoRepository).findById(1L);
    }

    @Test
    @DisplayName("存在しないIDで例外がスローされること")
    void findById_throwsException_whenNotFound() {
        // 準備: 空のOptionalを返すモック
        when(todoRepository.findById(99L)).thenReturn(Optional.empty());

        // 実行 & 検証: 例外がスローされることを確認
        assertThatThrownBy(() -> todoService.findById(99L))
                .isInstanceOf(IllegalArgumentException.class)
                .hasMessageContaining("不正なID");
    }

    @Test
    @DisplayName("ステータスが次の状態に切り替わること")
    void toggleStatus_changesStatus() {
        // 準備
        Todo todo = new Todo();
        todo.setId(1L);
        todo.setTitle("テスト");
        todo.setStatus(TodoStatus.NOT_STARTED);
        when(todoRepository.findById(1L)).thenReturn(Optional.of(todo));

        // 実行
        todoService.toggleStatus(1L);

        // 検証: NOT_STARTED → IN_PROGRESS に変わったこと
        assertThat(todo.getStatus()).isEqualTo(TodoStatus.IN_PROGRESS);
        verify(todoRepository).save(todo);
    }
}
info アノテーションの解説
  • @ExtendWith(MockitoExtension.class) -- JUnit5でMockitoを使うための設定。@Mockや@InjectMocksが動作するようになります。
  • @Mock -- モックオブジェクトを自動生成します。TodoRepositoryのメソッドは全てnullや空コレクションを返す状態になります。
  • @InjectMocks -- テスト対象のクラスを作成し、@Mockオブジェクトを自動注入します。
  • when(...).thenReturn(...) -- 「このメソッドが呼ばれたら、この値を返せ」というモックの振る舞いを定義します。
  • verify(...) -- メソッドが実際に呼ばれたことを検証します。save()が呼ばれずにテストが通ってしまう事故を防ぎます。
Section 12-4

Controllerテストを書く

Controllerテストでは、実際にHTTPリクエストを送信せずにMockMvcを使ってリクエストを模擬します。ControllerがServiceを呼んで正しいレスポンスを返すかを検証します。

12-4-1 TodoControllerTestを作成する

src/test/java/com/example/todo/にTodoControllerTest.javaを作成し、以下のコードを入力してください。

src/test/java/com/example/todo/TodoControllerTest.java 
package com.example.todo;

import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.autoconfigure.web.servlet.WebMvcTest;
import org.springframework.boot.test.mock.bean.MockBean;
import org.springframework.test.web.servlet.MockMvc;

import java.util.List;

import static org.mockito.Mockito.when;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.get;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.post;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.*;

@WebMvcTest(TodoController.class)
class TodoControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private TodoService todoService;

    @Test
    @DisplayName("一覧ページが正常に表示されること")
    void list_returnsListView() throws Exception {
        // 準備
        Todo todo = new Todo();
        todo.setTitle("テストTodo");
        todo.setStatus(TodoStatus.NOT_STARTED);
        when(todoService.findAll(null)).thenReturn(List.of(todo));

        // 実行 & 検証
        mockMvc.perform(get("/todos"))
                .andExpect(status().isOk())
                .andExpect(view().name("layout"))
                .andExpect(model().attributeExists("todos"));
    }

    @Test
    @DisplayName("新規登録ページが表示されること")
    void create_returnsCreateView() throws Exception {
        mockMvc.perform(get("/todos/create"))
                .andExpect(status().isOk())
                .andExpect(view().name("layout"))
                .andExpect(model().attributeExists("todo"));
    }

    @Test
    @DisplayName("削除後に一覧にリダイレクトされること")
    void delete_redirectsToList() throws Exception {
        mockMvc.perform(post("/todos/1/delete"))
                .andExpect(status().is3xxRedirection())
                .andExpect(redirectedUrl("/todos"));
    }
}

api MockMvcの処理フロー

MockMvc
→
GET /todos
→
Controller
→
Service(Mock)

実際のHTTP通信もDB接続もなし。Controllerのルーティングと戻り値だけをテスト。

info アノテーションの解説
  • @WebMvcTest(TodoController.class) -- 指定したControllerだけを起動する軽量テスト。ServiceやRepositoryは起動しません。
  • @MockBean -- Spring のApplicationContextにモックBeanを登録します。Controllerが依存するServiceをモックで置き換えます。
  • mockMvc.perform(get("/todos")) -- GET /todos リクエストを模擬的に送信します。
  • andExpect(status().isOk()) -- HTTPステータスが200であることを検証します。
  • andExpect(view().name("layout")) -- 返されたビュー名が"layout"であることを検証します。

12章 練習問題

Q1

JUnit5でMockitoを使ったServiceテストにおいて、@Mockと@InjectMocksの組み合わせについて正しい記述はどれですか。

  1. @Mockはテスト対象のクラスに付け、@InjectMocksは依存先に付ける
  2. @Mockは依存先（モック化する対象）に付け、@InjectMocksはテスト対象のクラスに付ける
  3. @Mockと@InjectMocksは同じクラスに同時に付けることができる
  4. @InjectMocksを使う場合、@Autowiredも併記する必要がある
Q2

MockMvcを使ったControllerテストで、mockMvc.perform(get("/todos")).andExpect(status().isOk()) が検証していることはどれですか。

  1. ブラウザが正常にページを表示できること
  2. GET /todos に対するHTTPレスポンスのステータスコードが200であること
  3. データベースにTodoが正常に保存されること
  4. Thymeleafテンプレートが正しくレンダリングされること
Q3

@DataJpaTestアノテーションについて正しい記述はどれですか。

  1. アプリケーション全体を起動してテストを実行する
  2. JPA関連のコンポーネントのみを起動し、インメモリDBを使ってRepositoryをテストする
  3. Controllerのエンドポイントをテストするためのアノテーションである
  4. テスト実行時に本番データベースに接続してデータを検証する
Answer

12章 解説・解答

Q1の解答: B

@Mockはモック化（偽物化）する依存先のクラスに付けます。@InjectMocksはテスト対象のクラスに付け、そのクラスのコンストラクタに@Mockオブジェクトが自動注入されます。例えば、TodoServiceをテストする場合、@Mock TodoRepositoryと@InjectMocks TodoServiceの組み合わせで、モックのRepositoryが注入されたServiceが作成されます。

Q2の解答: B

status().isOk()は、HTTPレスポンスのステータスコードが200（OK）であることを検証します。MockMvcは実際のHTTPサーバーを起動せず、Spring MVCのDispatcherServletを通じてリクエストを処理します。ブラウザの表示やDBへの保存は検証対象外です。テンプレートのレンダリングは別途view().name()で検証します。

Q3の解答: B

@DataJpaTestはJPA関連のコンポーネント（Entity、Repository、TestEntityManager等）だけを起動する軽量なテスト環境を提供します。デフォルトでインメモリのH2データベースを使用するため、本番DBに影響を与えません。ControllerやServiceは起動されないため、Repositoryの動作検証に特化しています。アプリケーション全体を起動するテストには@SpringBootTestを使います。
