Section 11-1

なぜServiceが必要か

ここまでのTodoアプリでは、Controllerがデータの取得・保存・ステータスの切り替えなど、あらゆる処理を直接行っています。機能が増えるにつれてControllerが肥大化し、コードの見通しが悪くなります。この章では、ビジネスロジックをServiceクラスに分離する「3層アーキテクチャ」のリファクタリングを行います。

11-1-1 Controller肥大化の問題

現在のTodoControllerを振り返ってみましょう。1つのクラスに以下の全てが詰め込まれています。

warning 現在のTodoController -- 責務が多すぎる

TodoController
HTTPリクエストの受付・レスポンス
Todoの検索・ソート処理
ステータスの切り替えロジック
期限切れチェック
データベースアクセス（Repository呼び出し）
1クラスに全部入りで見通しが悪い

11-1-2 3層アーキテクチャとは

Spring Bootの標準的な設計パターンでは、アプリケーションを3つの層に分けて、それぞれに明確な役割を持たせます。

layers 3層アーキテクチャ

Controller層（プレゼンテーション）
HTTPリクエスト/レスポンスの処理、画面遷移
@Controller / @RestController
↓ 呼び出す ↓
Service層（ビジネスロジック）
業務ルール、データ加工、複雑な処理の実行
@Service
↓ 呼び出す ↓
Repository層（データアクセス）
データベースへの読み書き（CRUD操作）
@Repository / JpaRepository

ControllerはServiceだけを知り、ServiceはRepositoryだけを知る

lightbulb なぜ層を分けるのか
  • テストしやすい -- Serviceだけを単体テストできる（次章で実践）
  • 再利用できる -- 同じServiceをWebコントローラとAPIコントローラの両方から呼べる
  • 変更の影響を限定できる -- 画面変更はController、ロジック変更はServiceだけを修正すればよい
Section 11-2

TodoServiceクラスを作成する

新しいファイルTodoService.javaを作成し、データの取得・保存・削除のメソッドを定義します。

11-2-1 基本的なCRUDメソッドを作る

com.example.todoパッケージにTodoService.javaを新規作成し、以下のコードを手で入力してください。

src/main/java/com/example/todo/TodoService.java 
package com.example.todo;

import org.springframework.stereotype.Service;
import java.util.List;

@Service
public class TodoService {

    private final TodoRepository todoRepository;

    // コンストラクタインジェクション
    public TodoService(TodoRepository todoRepository) {
        this.todoRepository = todoRepository;
    }

    // 全件取得
    public List<Todo> findAll() {
        return todoRepository.findAll();
    }

    // ID指定で1件取得
    public Todo findById(Long id) {
        return todoRepository.findById(id)
                .orElseThrow(() -> new IllegalArgumentException("不正なID: " + id));
    }

    // 保存（新規・更新共通）
    public Todo save(Todo todo) {
        return todoRepository.save(todo);
    }

    // 削除
    public void deleteById(Long id) {
        todoRepository.deleteById(id);
    }
}
info コードの解説
  • @Service -- このクラスがService層のコンポーネントであることをSpringに伝えます。@Componentと同等ですが、役割を明示するため@Serviceを使います。
  • コンストラクタインジェクション -- finalフィールドとコンストラクタでDI（依存性注入）を行います。Spring Bootはコンストラクタが1つの場合、@Autowiredを省略できます。
  • orElseThrow() -- findById()はOptionalを返すため、値がない場合に例外をスローします。この判定ロジックをServiceに閉じ込めることで、Controllerがシンプルになります。

11-2-2 DI（依存性注入）の仕組み

sync_alt Springの依存性注入（DI）の流れ

1
起動時: Spring が @Repository 付きの TodoRepository を検出し、インスタンスを作成
2
起動時: Spring が @Service 付きの TodoService を検出し、コンストラクタに TodoRepository を注入して作成
3
起動時: Spring が @Controller 付きの TodoController を検出し、コンストラクタに TodoService を注入して作成
4
実行時: HTTPリクエスト → Controller → Service → Repository → DB

自分で new TodoService() する必要がない。Springが自動で生成・接続してくれる。

Section 11-3

ControllerをService呼び出しに変更する

TodoControllerが直接TodoRepositoryを呼んでいた部分を、TodoServiceの呼び出しに段階的に置き換えていきます。

11-3-1 Controller の依存関係を変更する

まず、TodoControllerのフィールドとコンストラクタを変更します。TodoRepositoryをTodoServiceに差し替えてください。

TodoController.java - 変更前 
@Controller
@RequestMapping("/todos")
public class TodoController {

    private final TodoRepository todoRepository;

    public TodoController(TodoRepository todoRepository) {
        this.todoRepository = todoRepository;
    }
    // ...
}
TodoController.java - 変更後 
@Controller
@RequestMapping("/todos")
public class TodoController {

    private final TodoService todoService;

    public TodoController(TodoService todoService) {
        this.todoService = todoService;
    }
    // ...
}

11-3-2 各メソッドをService呼び出しに書き換える

Controllerの各メソッドを順番に書き換えましょう。まずはlist()メソッドです。

TodoController.java - list()の変更 
    @GetMapping
    public String list(@RequestParam(required = false) TodoStatus status,
                       Model model) {
        List<Todo> todos = todoService.findAll(status);
        model.addAttribute("todos", todos);
        model.addAttribute("currentStatus", status);
        model.addAttribute("statuses", TodoStatus.values());
        model.addAttribute("viewName", "list");
        model.addAttribute("pageTitle", "Todo一覧");
        return "layout";
    }

次に、詳細・編集・削除メソッドも書き換えます。

TodoController.java - edit/delete の変更 
    // 編集画面表示
    @GetMapping("/{id}/edit")
    public String edit(@PathVariable Long id, Model model) {
        model.addAttribute("todo", todoService.findById(id));
        model.addAttribute("viewName", "edit");
        model.addAttribute("pageTitle", "Todo編集");
        return "layout";
    }

    // 更新処理
    @PostMapping("/{id}/update")
    public String update(@PathVariable Long id,
                         @Valid @ModelAttribute Todo todo,
                         BindingResult result, Model model) {
        if (result.hasErrors()) {
            model.addAttribute("viewName", "edit");
            model.addAttribute("pageTitle", "Todo編集");
            return "layout";
        }
        todo.setId(id);
        todoService.save(todo);
        return "redirect:/todos";
    }

    // 削除処理
    @PostMapping("/{id}/delete")
    public String delete(@PathVariable Long id) {
        todoService.deleteById(id);
        return "redirect:/todos";
    }

    // ステータス切り替え
    @PostMapping("/{id}/toggle-status")
    public String toggleStatus(@PathVariable Long id) {
        todoService.toggleStatus(id);
        return "redirect:/todos";
    }

Controllerのコードがとてもシンプルになりました。各メソッドは「リクエストを受け取る → Serviceに処理を委譲する → 画面を返す」という役割だけに集中しています。

Section 11-4

ビジネスロジックをServiceに集約する

Controllerから切り出したロジックをTodoServiceに追加していきましょう。

11-4-1 フィルタ付き検索メソッドを追加する

TodoService.javaに、ステータスによるフィルタ検索メソッドを追加してください。

TodoService.java に追加 
    // ステータスでフィルタして取得（nullの場合は全件）
    public List<Todo> findAll(TodoStatus status) {
        if (status != null) {
            return todoRepository.findByStatus(status);
        }
        return todoRepository.findAll();
    }

条件分岐がServiceの中に移動したことで、Controllerのlist()メソッドはtodoService.findAll(status)の1行で済むようになりました。

11-4-2 ステータス切り替えロジックを追加する

ステータスのトグル処理もServiceに移動しましょう。

TodoService.java に追加 
    // ステータスを次の状態に切り替える
    public void toggleStatus(Long id) {
        Todo todo = findById(id);
        todo.setStatus(todo.getStatus().next());
        todoRepository.save(todo);
    }
lightbulb Serviceメソッドの設計ポイント

toggleStatus()は内部でfindById()を再利用しています。Serviceクラス内のメソッド同士を組み合わせることで、コードの重複を避けられます。将来「ステータス変更時にメール通知を送る」といった機能追加も、このメソッドだけを修正すれば済みます。

11章 練習問題

Q1

Spring Bootの3層アーキテクチャにおける@Serviceアノテーションの役割について正しいものはどれですか。

  1. HTTPリクエストを受け付けるエンドポイントを定義する
  2. データベースアクセス用のクラスであることをSpringに伝える
  3. ビジネスロジックを担当するクラスであることをSpringに伝え、DIコンテナに登録する
  4. テンプレートエンジンにデータを渡すためのクラスを定義する
Q2

以下のコンストラクタインジェクションについて正しい記述はどれですか。

TodoService.java 
private final TodoRepository todoRepository;

public TodoService(TodoRepository todoRepository) {
    this.todoRepository = todoRepository;
}
  1. @Autowiredアノテーションが省略されているためコンパイルエラーになる
  2. コンストラクタが1つしかない場合、Spring Bootは自動的に@Autowiredを適用する
  3. finalフィールドはDIできないため、finalを外す必要がある
  4. このコードではフィールドインジェクションが行われる
Q3

Service層を導入するメリットとして誤っているものはどれですか。

  1. ビジネスロジックを単体テストしやすくなる
  2. 同じロジックをWebコントローラとAPIコントローラから再利用できる
  3. Service層を導入すると処理速度が向上し、レスポンスが速くなる
  4. Controller が HTTP処理に集中でき、コードの見通しが良くなる
Answer

11章 解説・解答

Q1の解答: C

@Serviceはクラスがビジネスロジックを担当するService層のコンポーネントであることを示すアノテーションです。内部的には@Componentの特殊化であり、SpringのDIコンテナにBeanとして登録されます。HTTPリクエストの受付は@Controller、データアクセスは@Repositoryの役割です。

Q2の解答: B

Spring Boot（Spring Framework 4.3以降）では、クラスにコンストラクタが1つしかない場合、@Autowiredを省略しても自動的に依存性注入が行われます。finalフィールドを使うことで、注入後にフィールドが再代入されることを防ぎ、不変性を保証できます。これが現在推奨されるDIパターンです。

Q3の解答: C

Service層の導入は設計の改善（保守性・テスタビリティ・再利用性の向上）が目的であり、処理速度には直接的な影響はありません。むしろメソッド呼び出しが1段増えるため、わずかなオーバーヘッドが加わりますが、実用上は無視できるレベルです。パフォーマンス改善には、クエリの最適化やキャッシュの導入といった別のアプローチが必要です。
