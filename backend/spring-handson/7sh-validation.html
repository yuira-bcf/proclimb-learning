Section 7-1

なぜバリデーションが必要か

前章まででCRUD操作が完成しましたが、フォームに空文字や不正な値を入力してもそのまま保存されてしまいます。この章では入力チェック（バリデーション）を追加し、不正なデータがデータベースに保存されるのを防ぎます。

7-1-1 バリデーションがない場合の問題

バリデーションなしで起きる問題
warning 空のタイトル

タイトルなしのTodoが保存され、一覧画面で何のTodoかわからない

warning 極端に長い文字列

数万文字の入力でデータベースエラーや画面崩れが発生

warning 過去の期限日

すでに過ぎた日付が設定され、意味のないTodoが作られる

7-1-2 バリデーションの実装方針

Spring Bootではバリデーションを3つのステップで実装します。

バリデーション実装の3ステップ
1
Entity にバリデーションアノテーションを追加
（どのフィールドにどんなルールを適用するか）
2
Controller に @Valid と BindingResult を追加
（エラーがあったらフォームに戻す）
3
HTMLテンプレート にエラーメッセージ表示を追加
（ユーザーに何が問題かを伝える）
Section 7-2

Entityにバリデーションアノテーションを追加する

まずはTodo Entityの各フィールドにバリデーションルールを設定します。

7-2-1 主なバリデーションアノテーション

Spring Bootのspring-boot-starter-validation（プロジェクト作成時に追加済み）が提供するアノテーションの中から、よく使うものを確認しましょう。

アノテーション 説明 対象
@NotBlank null、空文字、空白のみを拒否 String
@NotNull nullを拒否（空文字はOK） 全型
@Size(min, max) 文字列の長さを制限 String, Collection
@Future 未来の日付のみ許可 日付型
@Email メールアドレス形式チェック String

7-2-2 Todo.javaにアノテーションを追加する

Todo.java を開いてください。まず、必要なimport文を追加します。

Todo.java のimportに追加 
import jakarta.validation.constraints.Future;
import jakarta.validation.constraints.NotBlank;
import jakarta.validation.constraints.Size;

次に、各フィールドの上にアノテーションを追加してください。既存のコードを以下のように変更しましょう。

Todo.java のフィールド部分を変更 
@NotBlank(message = "タイトルは必須です")
@Size(max = 100, message = "タイトルは100文字以内で入力してください")
private String title;

@Size(max = 500, message = "説明は500文字以内で入力してください")
private String description;

@Future(message = "期限日は未来の日付を指定してください")
private LocalDate dueDate;
フィールドとバリデーションの対応
title
@NotBlank @Size(max=100)
description
@Size(max=500)
dueDate
@Future
lightbulb @NotBlank vs @NotNull vs @NotEmpty

@NotNullはnullのみ拒否（空文字""はOK）、@NotEmptyはnullと空文字を拒否（空白" "はOK）、@NotBlankはnull・空文字・空白を全て拒否します。文字列フィールドには@NotBlankを使うのが最も安全です。

Section 7-3

Controllerでバリデーションを実行する

Entityにルールを定義しただけでは、バリデーションは実行されません。Controllerでバリデーションを有効にし、エラー時の処理を記述します。

7-3-1 @ValidとBindingResultを追加する

TodoController.java を開き、まずimport文を追加してください。

TodoController.java のimportに追加 
import jakarta.validation.Valid;
import org.springframework.validation.BindingResult;

次に、新規作成のcreateTodoメソッドを以下のように変更してください。

TodoController.java の createTodo メソッドを変更 
// 新規作成処理（バリデーション付き）
@PostMapping("/todo/create")
public String createTodo(@Valid @ModelAttribute Todo todo,
                         BindingResult result) {
    if (result.hasErrors()) {
        return "todo/create";
    }
    todoRepository.save(todo);
    return "redirect:/todo";
}
バリデーションの処理フロー
1
フォームから送信されたデータを @ModelAttribute でTodoに変換
2
@Valid によりEntity上のバリデーションアノテーションを実行
3
エラー結果が BindingResult に格納される
エラーあり → フォーム画面に戻す
エラーなし → save() で保存
warning BindingResultの位置に注意

BindingResultは@Validを付けた引数の直後に書く必要があります。間に他の引数を挟むと正しく動作しません。

7-3-2 更新メソッドにもバリデーションを追加する

同様に、updateTodoメソッドにもバリデーションを追加しましょう。

TodoController.java の updateTodo メソッドを変更 
// 更新処理（バリデーション付き）
@PostMapping("/todo/update/{id}")
public String updateTodo(@PathVariable Long id,
                         @Valid @ModelAttribute Todo todo,
                         BindingResult result) {
    if (result.hasErrors()) {
        todo.setId(id);
        return "todo/edit";
    }
    todo.setId(id);
    todoRepository.save(todo);
    return "redirect:/todo";
}

更新時のエラー処理では todo.setId(id) を呼んでからフォームに戻しています。IDを保持しておかないと、edit.htmlのフォーム送信先URLが正しく生成されません。

Section 7-4

エラーメッセージをフォームに表示する

バリデーションエラーが発生したとき、ユーザーにどの入力が間違っているかを伝える必要があります。Thymeleafのth:errorsを使ってエラーメッセージを表示しましょう。

7-4-1 create.htmlにエラー表示を追加する

create.html を開き、各入力欄の下にエラーメッセージ表示用の要素を追加してください。

create.html のフォーム部分を変更 
<form th:action="@{/todo/create}" th:object="${todo}" method="post">
  <div>
    <label for="title">タイトル:</label>
    <input type="text" id="title" th:field="*{title}"
           th:classappend="${#fields.hasErrors('title')} ? 'error-input'">
    <span th:if="${#fields.hasErrors('title')}" th:errors="*{title}"
          style="color:red; font-size:small;"></span>
  </div>
  <div>
    <label for="description">説明:</label>
    <textarea id="description" th:field="*{description}"
              th:classappend="${#fields.hasErrors('description')} ? 'error-input'"></textarea>
    <span th:if="${#fields.hasErrors('description')}" th:errors="*{description}"
          style="color:red; font-size:small;"></span>
  </div>
  <div>
    <label for="dueDate">期限日:</label>
    <input type="date" id="dueDate" th:field="*{dueDate}"
           th:classappend="${#fields.hasErrors('dueDate')} ? 'error-input'">
    <span th:if="${#fields.hasErrors('dueDate')}" th:errors="*{dueDate}"
          style="color:red; font-size:small;"></span>
  </div>
  <button type="submit">登録する</button>
</form>
Thymeleaf エラー表示の仕組み
th:if="${#fields.hasErrors('title')}"

titleフィールドにエラーがある場合のみ、この要素を表示します。

th:errors="*{title}"

titleフィールドのエラーメッセージを出力します。Entityのmessage属性の値が使われます。

th:classappend="${...} ? 'error-input'"

エラーがある入力欄にCSSクラスを追加し、赤枠などのスタイルを適用できます。

7-4-2 実行して確認しましょう

アプリケーションを再起動し、新規作成画面でタイトルを空のまま「登録する」を押してみましょう。

monitor エラー表示のイメージ
Todoを追加する
タイトル:
タイトルは必須です
説明:
期限日:
2020-01-01
期限日は未来の日付を指定してください

エラーメッセージが表示され、入力した値も保持されていることを確認してください。edit.htmlにも同様のエラー表示を追加しておきましょう。

Section 7-5

日本語エラーメッセージをカスタマイズする

アノテーションのmessage属性に直接日本語を書く方法は簡単ですが、メッセージを一括管理したい場合はmessages.propertiesファイルを使う方法もあります。

7-5-1 メッセージ管理の2つの方法

メッセージ管理方法の比較
方法A: message属性（今回の方式）
@NotBlank(message = "タイトルは必須です")
  • シンプルで直感的
  • 小規模アプリに最適
  • メッセージがコードに散らばる
方法B: messages.properties
NotBlank.todo.title=タイトルは必須です
  • メッセージを一括管理
  • 多言語対応が容易
  • 大規模アプリ向け

今回のTodoアプリではmessage属性を直接書く方法（方法A）で十分です。将来的にアプリが大きくなった場合は、messages.propertiesに移行することを検討しましょう。

lightbulb messages.propertiesの作り方（参考）

src/main/resources/messages.propertiesを作成し、NotBlank.todo.title=タイトルは必須ですのように「アノテーション名.オブジェクト名.フィールド名=メッセージ」の形式で記述します。Entityのmessage属性は省略できます。

7章 練習問題

Q1

Stringフィールドに対して、null・空文字・空白のみの入力を全て拒否するアノテーションはどれですか。

  1. @NotNull
  2. @NotEmpty
  3. @NotBlank
  4. @Required
Q2

Controllerメソッドで BindingResult を使う際の注意点として正しいものはどれですか。

  1. BindingResultはメソッドの最初の引数に置く必要がある
  2. BindingResultは@Validを付けた引数の直後に置く必要がある
  3. BindingResultはメソッドの最後の引数に置く必要がある
  4. BindingResultはどこに置いても動作する
Q3

バリデーションエラー時に return "todo/create"; としたとき、ユーザーが入力した値はどうなりますか。

  1. 全てクリアされ、空のフォームが表示される
  2. 入力した値が保持されたまま、エラーメッセージとともにフォームが再表示される
  3. 一覧画面にリダイレクトされる
  4. 500エラーページが表示される
Answer

7章 解説・解答

Q1の解答: C

@NotBlankはnull、空文字("")、空白のみ(" ")の全てを拒否します。@NotNullはnullのみ拒否し空文字は通します。@NotEmptyはnullと空文字を拒否しますが空白のみの文字列は通します。@RequiredはSpring Bootのバリデーションアノテーションには存在しません。

Q2の解答: B

BindingResultは@Validを付けた引数の直後に配置する必要があります。Spring Bootは引数の順序を見て、どの引数のバリデーション結果をどのBindingResultに格納するかを判断します。間に別の引数を挟むと、バリデーションエラーが発生した時点で例外がスローされてしまいます。

Q3の解答: B

エラー時にテンプレート名を返す（リダイレクトではない）場合、@ModelAttributeで受け取ったTodoオブジェクトがそのままテンプレートに渡されます。そのため、ユーザーが入力した値は保持されたまま、th:fieldによってフォームに再表示されます。th:errorsがエラーメッセージを表示するので、ユーザーはどこを修正すればよいか分かります。
