Section 6-1

CRUD操作の全体像

前章でデータの新規登録（Create）ができるようになりました。一覧表示（Read）は第4章で実装済みです。この章では残りの更新（Update）と削除（Delete）を実装し、CRUDを完成させます。

6-1-1 CRUDの進捗を確認する

CRUD操作と実装状況
check_circle Create
新規登録 （第5章）
@PostMapping + save()
check_circle Read
一覧表示 （第4章）
@GetMapping + findAll()
edit Update
更新 （この章）
findById() + save()
delete Delete
削除 （この章）
deleteById()

6-1-2 更新・削除に必要なもの

更新と削除では、どのデータを対象にするかを指定する必要があります。そのためにTodoのidをURLに含めます。

操作 HTTPメソッド URL 処理
編集画面表示 GET /todo/edit/{id} 既存データを表示
更新処理 POST /todo/update/{id} データを上書き保存
削除処理 POST /todo/delete/{id} データを削除
lightbulb {id} はパス変数

URLに含まれる{id}はパス変数（Path Variable）と呼ばれ、@PathVariableアノテーションで受け取ります。例えば/todo/edit/3にアクセスするとid=3のTodoが対象になります。

Section 6-2

編集画面を作る

まずは編集画面のHTMLテンプレートと、それを表示するControllerメソッドを作成します。

6-2-1 edit.htmlを作成する

src/main/resources/templates/todo ディレクトリに edit.html を新規作成し、以下のコードを手で入力してください。

src/main/resources/templates/todo/edit.html 
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<head>
  <meta charset="UTF-8">
  <title>Todo - 編集</title>
</head>
<body>
  <h1>Todoを編集する</h1>

      <input type="text" id="title" th:field="*{title}">
    </div>
    <div>
      <label for="description">説明:</label>
      <textarea id="description" th:field="*{description}"></textarea>
    </div>
    <div>
      <label for="dueDate">期限日:</label>
      <input type="date" id="dueDate" th:field="*{dueDate}">
    </div>

  </form>

</body>
</html>

create.htmlとほぼ同じ構造ですが、以下の2点が異なります。

create.html との違い

6-2-2 編集画面表示メソッドを追加する

TodoController.java を開き、以下のメソッドを追加してください。importにPathVariableを追加するのを忘れないでください。

TodoController.java のimportに追加 
import org.springframework.web.bind.annotation.PathVariable;
TodoController.java にメソッド追加 
// 編集画面表示
@GetMapping("/todo/edit/{id}")
public String showEditForm(@PathVariable Long id, Model model) {
    Todo todo = todoRepository.findById(id)
            .orElseThrow(() -> new IllegalArgumentException("不正なID: " + id));
    model.addAttribute("todo", todo);
    return "todo/edit";
}
findById() と Optional
findById(3)
→

→
.orElseThrow() で Todoを取り出す
存在しない場合

IllegalArgumentException を投げる
warning なぜOptionalなのか

findById()は指定したIDのデータが見つからない可能性があるため、Optional<Todo>を返します。.orElseThrow()を使うことで、データが見つからなかった場合にエラーを発生させます。.get()を直接呼ぶのは推奨されません。

Section 6-3

更新処理を実装する

編集画面で変更した内容をデータベースに保存する処理を追加します。

6-3-1 更新メソッドを追加する

TodoController.java に以下のメソッドを追加してください。

TodoController.java にメソッド追加 
// 更新処理
@PostMapping("/todo/update/{id}")
public String updateTodo(@PathVariable Long id, @ModelAttribute Todo todo) {
    todo.setId(id);
    todoRepository.save(todo);
    return "redirect:/todo";
}

todo.setId(id) でURLから受け取ったIDをTodoオブジェクトに設定しています。JPAのsave()メソッドは、IDが設定されている場合はUPDATE、IDがnullの場合はINSERTを実行します。

save() の動作判定
id = null
INSERT（新規登録）
id = 3
UPDATE（上書き更新）

Section 6-4

削除機能を実装する

次に削除機能を追加します。削除はHTMLテンプレートの追加は不要で、Controllerにメソッドを1つ追加するだけです。

6-4-1 削除メソッドを追加する

TodoController.java に以下のメソッドを追加してください。

TodoController.java にメソッド追加 
// 削除処理
@PostMapping("/todo/delete/{id}")
public String deleteTodo(@PathVariable Long id) {
    todoRepository.deleteById(id);
    return "redirect:/todo";
}

deleteById(id) は指定したIDのデータを削除します。削除後は一覧画面にリダイレクトします。

削除処理の流れ
削除ボタン
POST /todo/delete/3
→
Controller
deleteById(3)
→
DB
DELETE WHERE id=3
→
リダイレクト
redirect:/todo
warning 削除にGETを使わない理由

削除をGETリクエスト（リンク）で実装すると、検索エンジンのクローラーやブラウザのプリフェッチ機能がURLにアクセスしただけでデータが削除されてしまいます。データを変更する操作は必ずPOSTメソッドで行いましょう。

6章 練習問題

Q1

todoRepository.findById(id) の戻り値の型として正しいものはどれですか。

  1. Todo
  2. Optional<Todo>
  3. List<Todo>
  4. null
Q2

todoRepository.save(todo) でUPDATEが実行される条件はどれですか。

  1. Todoオブジェクトのtitleフィールドが空文字のとき
  2. Todoオブジェクトのidフィールドがnullのとき
  3. Todoオブジェクトのidフィールドに値がセットされているとき
  4. TodoオブジェクトのcreatedAtフィールドが過去の日付のとき
Q3

削除処理をGETリクエスト（リンク）ではなくPOSTリクエスト（フォーム）で実装する理由として最も適切なものはどれですか。

  1. GETリクエストではURLにパラメータを含められないため
  2. POSTリクエストの方がGETリクエストより処理速度が速いため
  3. GETリクエストはクローラーやプリフェッチで意図しない削除が起きる可能性があるため
  4. POSTリクエストでないとSpring Bootがエラーを出すため
Answer

6章 解説・解答

Q1の解答: B

findById()の戻り値はOptional<Todo>です。指定したIDのデータが存在しない可能性があるため、nullを直接返す代わりにOptionalで包んで返します。.orElseThrow()や.orElse()を使って値を取り出すのが推奨される方法です。

Q2の解答: C

JPAのsave()メソッドは、エンティティのIDフィールドの状態で動作が変わります。IDがnullの場合は新規レコードとしてINSERT文が実行され、IDに値がセットされている場合は既存レコードの更新としてUPDATE文が実行されます。これがcreateとupdateで同じsave()メソッドを使える理由です。

Q3の解答: C

GETリクエストはブックマークやリンクの共有、検索エンジンのクローラー、ブラウザのプリフェッチ（先読み）機能などで、ユーザーの意図とは関係なくアクセスされる可能性があります。削除のような副作用のある操作をGETで実装すると、これらの機能が動作しただけでデータが削除されてしまいます。POSTリクエストはフォーム送信でしか発生しないため安全です。
