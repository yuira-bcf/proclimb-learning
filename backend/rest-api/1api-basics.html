Section 1-1

APIの概念

スマホで天気予報を見る、Googleアカウントで他のサービスにログインする、Uber Eatsで注文する。これらの裏側では、すべてAPIが動いています。APIとは何なのか、身近な体験から理解していきましょう。

1.1 APIとは何か

API（Application Programming Interface）とは、あるサービスの機能を、外部のプログラムから呼び出せるようにした「窓口」です。

まず、具体例から見てみましょう。あなたがスマホの天気アプリを開いた瞬間、裏側ではこんなことが起きています。

1

天気アプリ「東京の天気を教えて」

アプリが気象データを持っている別のサーバーに問い合わせを送ります。

2

気象サーバー「東京は晴れ、最高気温25度です」

サーバーが天気データを探し、結果を返します。

3

天気アプリが画面に表示

受け取ったデータをアプリが見やすく画面に表示します。

この「問い合わせの窓口」がAPIです。天気アプリは気象データの計算方法を知りません。ただ「APIに聞けば答えが返ってくる」ということだけ知っていれば十分なのです。

では、「東京の天気を教えて」という問い合わせは、実際にはどんな形で送られるのでしょうか？

天気アプリが実際に送っているリクエスト HTTP 
GET https://api.weather.example.com/forecast?city=Tokyo

これがAPIへのリクエスト（お願い）です。分解すると次のようになります。

要素 この例での値 意味
メソッド GET 「データをください」という操作の種類
URL（エンドポイント） api.weather.example.com/forecast APIの住所。どのサーバーの、どの機能を呼ぶか
パラメータ city=Tokyo 「東京の」という条件の指定

APIは、このリクエストを受け取ると結果をレスポンス（返事）として返します。

APIから返ってくるレスポンス JSON 
{
  "city": "Tokyo",
  "weather": "晴れ",
  "temperature": 25
}

APIという言葉はとても広い意味を持ちますが、この講座ではWeb API（HTTPを使ってインターネット越しに通信するAPI）に焦点を当てて学習します。上の天気アプリの例がまさにWeb APIです。

1.2 APIが必要な理由

もしAPIがなかったら？天気アプリを作るには、自分で気象衛星のデータを分析し、天気予報のアルゴリズムを開発し…と、途方もない作業が必要です。APIがあるおかげで、他社がすでに作った機能を「借りる」ことができます。

1

機能の再利用 — 車輪の再発明をしない

例：ECサイトにクレジットカード決済を導入したい場合、セキュリティ基準を満たす決済システムを自前で作るには数年かかります。Stripe APIを使えば、数行のコードで決済機能を追加できます。

2

開発の分業 — フロントとバックを分離

例：Aさんは画面のデザインとUI（フロントエンド）、Bさんはデータ処理（バックエンド）を担当。2人の間をAPIで「こういう形式でデータをやりとりしよう」と決めておくことで、それぞれ独立して開発できます。

3

マルチプラットフォーム — 1つのAPIで複数の画面

例：Instagramの投稿データは1つのAPIから提供されています。Web版、iPhoneアプリ、Androidアプリのすべてが同じAPIからデータを取得しているため、どの端末でも同じ投稿が表示されます。

Section 1-3

Web APIの仕組み

Web APIは、HTTPプロトコルを使ってクライアント（利用する側）とサーバー（提供する側）が通信する仕組みです。インターネットを通じてデータをやりとりします。

3.1 クライアント-サーバーモデル

Web APIの通信は、クライアントとサーバーの2つの役割で成り立ちます。

🔌 Web API通信の全体像
devices クライアント ブラウザ / アプリ
HTTPリクエスト → ← JSONレスポンス
cloud APIサーバー 処理 & データ返却
データ取得 ↔
storage データベース データ保管
役割 説明 例
クライアント APIにリクエストを送る側 ブラウザ、スマホアプリ、別のサーバー
サーバー リクエストを受け取り、処理して返す側 Webサーバー、APIサーバー、データベース

3.2 リクエストとレスポンスの流れ

1

クライアントがリクエストを送信

「ユーザーID=1の情報をください」というリクエストをHTTPで送ります。

2

サーバーがリクエストを処理

サーバーはリクエスト内容を解析し、データベースから該当データを取得します。

3

サーバーがレスポンスを返送

処理結果をJSON形式などでクライアントに返します。

4

クライアントがデータを表示

受け取ったデータを画面に表示したり、次の処理に使います。

3.3 実際のAPI通信の例

ユーザー情報を取得するAPIリクエストとレスポンスの例を見てみましょう。

レスポンス（サーバー → クライアント） JSON

  "id": 1,

Section 1-4

APIがもたらす価値

APIの活用は、現代のソフトウェア開発に欠かせないものとなっています。APIを使うことでどのようなメリットがあるのかを理解しましょう。

4.1 APIの4つのメリット

⚡

開発効率の向上

既存サービスの機能をAPIで呼び出すだけで、決済・地図・認証などの複雑な機能を自分で一から開発する必要がなくなります。

🔄

データ連携

異なるシステム間でデータを安全かつ効率的にやりとりできます。社内の業務システム連携やサービス間のデータ共有に活用されます。

🚀

マイクロサービス

大規模なシステムを小さなサービス（API）の集合体として構築できます。各サービスを独立して開発・デプロイ・スケーリングできるため、チーム開発に向いています。

🌎

エコシステムの形成

APIを公開することで、サードパーティの開発者がサービスを拡張できます。Twitter API、GitHub APIなどは多くの外部アプリを生み出しています。

4.2 APIの種類

APIはアクセス範囲によって3つに分類されます。

種類 公開範囲 説明 例
公開API（Open API） 誰でも利用可能 一般に公開されているAPI。登録やAPIキー取得が必要な場合が多い OpenWeatherMap、GitHub API
パートナーAPI 特定のパートナー企業 ビジネス契約を結んだ相手にのみ提供されるAPI 決済API、物流API
プライベートAPI 社内のみ 自社サービス内でのみ使用するAPI。マイクロサービス間の連携に使われる 社内システム連携
warning 注意

公開APIであっても、利用規約（Terms of Service）やレート制限（一定時間内のリクエスト上限）が設けられています。APIを利用する際は必ず利用規約を確認しましょう。

1章 練習問題

Q1

APIの説明として最も適切なものを選んでください。

  1. プログラミング言語の一種
  2. ソフトウェア同士がやりとりするための窓口（インターフェース）
  3. データベースの管理システム
  4. Webページのデザインツール
Q2

Web APIの通信の流れとして正しい順序を選んでください。

  1. レスポンス返送 → リクエスト送信 → サーバー処理 → データ表示
  2. リクエスト送信 → サーバー処理 → レスポンス返送 → データ表示
  3. サーバー処理 → リクエスト送信 → データ表示 → レスポンス返送
  4. データ表示 → リクエスト送信 → レスポンス返送 → サーバー処理
Q3

社内システム間の連携のみに使用されるAPIの種類を選んでください。

  1. 公開API
  2. パートナーAPI
  3. プライベートAPI
  4. オープンソースAPI
Answer

1章 解説・解答

Q1の解答: B

APIはApplication Programming Interfaceの略で、ソフトウェア同士がデータや機能をやりとりするための窓口（インターフェース）です。プログラミング言語やツールではなく、通信の「仕組み」を指します。

Q2の解答: B

Web APIの通信は「リクエスト送信 → サーバー処理 → レスポンス返送 → データ表示」の順序で行われます。クライアントがまずリクエストを送り、サーバーが処理してレスポンスを返し、クライアントが受け取ったデータを表示します。

Q3の解答: C

プライベートAPIは、自社サービス内でのみ使用されるAPIです。マイクロサービスアーキテクチャにおけるサービス間連携などに活用されます。外部には公開されません。
