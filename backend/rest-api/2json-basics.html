Section 2-1

JSONとは

JSON（JavaScript Object Notation）は、データを交換するための軽量なテキスト形式です。2001年にDouglas Crockfordが仕様を整理し、現在ではWeb APIのデータ交換形式として事実上の標準となっています。

1.1 JSONの特徴

📄

軽量でシンプル

XMLと比べてデータサイズが小さく、構文がシンプルです。余計なタグがないためネットワーク帯域を節約できます。

👀

人間にも読みやすい

テキストベースのフォーマットなので、人間が直接読んで内容を理解できます。

🌐

言語非依存

JavaScript由来ですが、Python、Java、PHP、Ruby、Goなどほぼ全てのプログラミング言語で読み書きできます。

lightbulb 身近なJSONファイル

JSONはAPIのデータ交換だけでなく、設定ファイルとしても広く使われています。Node.jsのpackage.json、VSCodeのsettings.json、TypeScriptのtsconfig.jsonなど、開発現場のいたるところにJSONがあります。

Section 2-2

JSONの文法ルール

JSONにはいくつかの厳密な文法ルールがあります。これらを正しく理解しないと、パースエラーの原因になります。

🔍 JSONの構造を分解してみよう
{ ← オブジェクトの開始
  "name": "田中太郎", ← キー: 値（文字列）
  "age": 28, ← キー: 値（数値）
  "isActive": true, ← キー: 値（真偽値）
  "skills": ["HTML", "CSS"] ← キー: 値（配列）
} ← オブジェクトの終了

2.1 基本ルール

  • キー（プロパティ名）は必ずダブルクォート（"）で囲む
  • 文字列値もダブルクォートで囲む（シングルクォート不可）
  • 要素の区切りにはカンマ（,）を使う
  • 末尾のカンマ（trailing comma）は許可されない
  • コメントは書けない（// も /* */ も不可）
  • オブジェクトは {} 、配列は [] で囲む
正しいJSON JSON 
{
  "name": "田中太郎",
  "age": 28,
  "isStudent": false,
  "hobbies": ["読書", "映画鑑賞", "プログラミング"]
}
間違ったJSON（よくあるエラー） JSON（エラーあり） 
{
  name: "田中太郎",        // NG: キーにダブルクォートがない
  'age': 28,              // NG: シングルクォート
  "isStudent": false,
  "hobbies": ["読書",],   // NG: 末尾カンマ
  // これはコメント        // NG: コメントは書けない
}
warning JavaScriptオブジェクトとの違い

JavaScriptではキーにクォートなし、シングルクォート、末尾カンマ、コメントが使えますが、JSONではこれらは全て文法エラーになります。JSONとJavaScriptオブジェクトは似ていますが別物です。

Section 2-3

JSONのデータ型

JSONで使えるデータ型は6種類です。これだけでほとんどのデータ構造を表現できます。

3.1 6つのデータ型

データ型 説明 例
文字列（string） ダブルクォートで囲んだテキスト "Hello"、"東京"
数値（number） 整数または小数（クォート不要） 42、3.14、-10
真偽値（boolean） true または false true、false
null 値がないことを示す null
オブジェクト（object） キーと値のペアの集合 {"key": "value"}
配列（array） 値の順序付きリスト [1, 2, 3]
🎨 JSONの6つのデータ型
Aa文字列"Hello"
123数値42, 3.14
✓✗真偽値true / false
∅null値なし
{ }オブジェクト{"key": "val"}
[ ]配列[1, 2, 3]
全データ型を含むJSON例 JSON 
{
  "string": "これは文字列です",
  "number_int": 42,
  "number_float": 3.14,
  "boolean_true": true,
  "boolean_false": false,
  "null_value": null,
  "object": {
    "nested_key": "ネストされた値"
  },
  "array": [1, "two", true, null]
}
lightbulb JSONで使えない値

undefined、関数（function）、Dateオブジェクト、NaN、Infinity はJSONでは使えません。日付はISO 8601形式の文字列（例: "2025-04-01T10:30:00Z"）で表現するのが一般的です。

Section 2-4

JSONの実例

実際にAPIやアプリケーションで使われるJSON形式のデータ例を見てみましょう。

4.1 ユーザー情報

ユーザープロフィール JSON 
{
  "id": 1,
  "name": "田中太郎",
  "email": "tanaka@example.com",
  "age": 28,
  "isActive": true,
  "address": {
    "prefecture": "東京都",
    "city": "渋谷区",
    "zipCode": "150-0001"
  },
  "skills": ["HTML", "CSS", "JavaScript", "Java"]
}

オブジェクトの中にオブジェクト（address）や配列（skills）を含む、ネスト構造が使われています。

4.2 商品カタログ

商品一覧（配列の中にオブジェクト） JSON 
{
  "products": [
    {
      "id": 101,
      "name": "ワイヤレスマウス",
      "price": 3980,
      "inStock": true,
      "category": "周辺機器"
    },
    {
      "id": 102,
      "name": "メカニカルキーボード",
      "price": 12800,
      "inStock": false,
      "category": "周辺機器"
    }
  ]
}

4.3 APIレスポンスの典型的な構造

APIレスポンス（メタ情報付き） JSON 
{
  "status": "success",
  "data": {
    "users": [
      { "id": 1, "name": "田中太郎" },
      { "id": 2, "name": "佐藤花子" }
    ]
  },
  "meta": {
    "total": 50,
    "page": 1,
    "perPage": 10
  }
}

APIレスポンスでは、status（処理結果）、data（実データ）、meta（ページ情報などの付帯情報）を含む構造がよく使われます。

Section 2-5

他のデータ形式との比較

データ交換には JSON 以外にもいくつかの形式があります。それぞれの特徴を比較してみましょう。

5.1 形式の比較表

形式 可読性 データ構造 サイズ 主な用途
JSON 高い ネスト可能 小さい Web API、設定ファイル
XML やや低い ネスト可能 大きい SOAP API、文書定義
CSV 高い 表形式のみ 最小 データのインポート/エクスポート
YAML 最も高い ネスト可能 小さい 設定ファイル（Docker、CI/CD）

5.2 同じデータを各形式で表現

JSON形式 JSON 
{
  "name": "田中太郎",
  "age": 28,
  "skills": ["HTML", "CSS"]
}
XML形式 XML 
<user>
  <name>田中太郎</name>
  <age>28</age>
  <skills>
    <skill>HTML</skill>
    <skill>CSS</skill>
  </skills>
</user>
CSV形式 CSV 
name,age,skills
田中太郎,28,"HTML,CSS"
lightbulb 現在のWeb APIの標準

現在のWeb APIではJSONが事実上の標準です。XMLベースのSOAP APIは主にレガシーシステムで使われており、新規のAPIはほぼJSONを採用しています。CSVはネスト構造を表現できないため、API通信には不向きです。

Exercise

2章 練習問題

Q1

次のうち、正しいJSON形式はどれですか？

  1. {name: "太郎", age: 20}
  2. {"name": "太郎", "age": 20}
  3. {'name': '太郎', 'age': 20}
  4. {"name": "太郎", "age": 20,}
Q2

JSONのデータ型に含まれないものはどれですか？

  1. 文字列（string）
  2. undefined
  3. 配列（array）
  4. null
Q3

JSONの特徴として正しくないものを選んでください。

  1. 軽量なテキスト形式である
  2. コメントを記述できる
  3. ほぼ全てのプログラミング言語で扱える
  4. Web APIのデータ交換形式として広く使われている
Answer

2章 解説・解答

Q1の解答: B

JSONではキーを必ずダブルクォートで囲む必要があります。Aはクォートなし、Cはシングルクォート、Dは末尾カンマがあるため、いずれもJSONとして無効です。

Q2の解答: B

undefinedはJavaScriptの値であり、JSONのデータ型には含まれません。JSONで使えるのは文字列、数値、真偽値、null、オブジェクト、配列の6つです。

Q3の解答: B

JSONにはコメントを記述する仕組みがありません。// や /* */ のいずれも使えません。これはJavaScriptオブジェクトとの重要な違いの一つです。
