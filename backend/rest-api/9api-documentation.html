Section 9-1

良いAPIドキュメントとは

APIドキュメントは、開発者がAPIを正しく利用するための説明書です。良いドキュメントがあれば、開発者はスムーズにAPIを理解し、素早く実装できます。

良いAPIドキュメントに必要な6つの要素
🔗 エンドポイント URLパスとベースURL
🔧 HTTPメソッド GET / POST / PUT / DELETE
📝 パラメータ 名前・型・必須/任意
📄 レスポンス例 成功 / エラーのJSON
🔒 認証情報 APIキー / Bearer Token
📖 利用例 コピペで試せるサンプル

1.1 APIドキュメントに必要な情報

優れたAPIドキュメントには、以下の情報が漏れなく記載されている必要があります。

🔗

エンドポイント（URL）

各機能にアクセスするためのURLパスとベースURL。例: GET /api/v1/users/{id}

🔧

HTTPメソッド

各エンドポイントで使用するHTTPメソッド（GET, POST, PUT, DELETE等）の明示。

📝

パラメータ

パスパラメータ、クエリパラメータ、リクエストボディの各フィールドの名前・型・必須/任意・説明。

📄

レスポンス例

成功時・エラー時の具体的なJSONレスポンス例とステータスコード。

🔒

認証情報

必要な認証方法（APIキー、Bearer Token等）とその送信方法。

📖

利用例・チュートリアル

実際のコード例やクイックスタートガイドで、すぐに試せる内容。

Section 9-2

OpenAPI (Swagger) 仕様

OpenAPI仕様（旧Swagger仕様）は、REST APIの構造をYAMLまたはJSON形式で記述する標準規格です。この定義ファイルから、自動的にドキュメントやクライアントコードを生成できます。

OpenAPI 定義ファイルから自動生成するワークフロー
📄 OpenAPI定義 YAML / JSON
→
📖 Swagger UI インタラクティブ文書
→
💻 コード生成 クライアント / サーバー

1つの定義ファイルからドキュメントもコードも自動生成できる

2.1 OpenAPI定義ファイルの例

以下は、シンプルな /users エンドポイントをOpenAPI形式で定義した例です。

OpenAPI定義（YAML形式） YAML 
openapi: 3.0.0
info:
  title: ユーザー管理API
  description: ユーザー情報を管理するREST API
  version: 1.0.0
servers:
  - url: https://api.example.com/v1

paths:
  /users:
    get:
      summary: ユーザー一覧を取得
      description: 登録されている全ユーザーの一覧を返します
      parameters:
        - name: limit
          in: query
          description: 取得件数の上限
          required: false
          schema:
            type: integer
            default: 20
      responses:
        '200':
          description: 成功
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/User'
        '401':
          description: 認証エラー

  /users/{id}:
    get:
      summary: 特定のユーザーを取得
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: 成功
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
        '404':
          description: ユーザーが見つからない

components:
  schemas:
    User:
      type: object
      properties:
        id:
          type: integer
          example: 1
        name:
          type: string
          example: "田中太郎"
        email:
          type: string
          example: "tanaka@example.com"

2.2 Swagger UIで可視化

OpenAPI定義ファイルを読み込ませると、Swagger UIが自動的にインタラクティブなドキュメントを生成します。ブラウザ上でAPIを試すことも可能です。

lightbulb Swagger Editorを試してみよう

Swagger Editor（editor.swagger.io）では、ブラウザ上でOpenAPI定義を書きながらリアルタイムでドキュメントをプレビューできます。学習に最適なので、上記のYAMLを貼り付けて試してみましょう。

Section 9-3

公開APIリファレンスを読む

実際の公開APIドキュメントを読んで、必要な情報を見つける練習をしましょう。ここではGitHub APIを例に解説します。

3.1 GitHub API：ユーザー情報の取得

GitHub APIは認証なしでも一部のエンドポイントを利用できます。ユーザーのプロフィール情報を取得してみましょう。

3.2 APIドキュメントの探し方

1

公式サイトで「API」「Developer」を探す

多くのサービスはフッターやメニューに「Developers」「API」リンクを用意しています。

2

「サービス名 API documentation」で検索

Google検索で「GitHub API documentation」のように検索すると、公式ドキュメントがすぐに見つかります。

3

Getting Started / Quick Start を読む

まずはクイックスタートガイドを読んで、認証方法とベースURLを確認しましょう。

4

エンドポイント一覧から目的の機能を探す

APIリファレンスのエンドポイント一覧から、目的の操作に対応するエンドポイントを見つけます。

Section 9-4

APIドキュメントの書き方

自分でAPIを作成する際には、他の開発者が理解しやすいドキュメントを書くことが重要です。エンドポイント仕様書のテンプレートを紹介します。

4.1 エンドポイント仕様書テンプレート

各エンドポイントについて、以下の情報をテーブル形式でまとめると分かりやすくなります。

項目 内容
エンドポイント POST /api/v1/users
説明 新しいユーザーを作成する
認証 Bearer Token（必須）
Content-Type application/json
リクエストボディ name (string, 必須), email (string, 必須), role (string, 任意, デフォルト: "user")
成功レスポンス 201 Created
エラーレスポンス 400 Bad Request / 401 Unauthorized / 409 Conflict

4.2 サンプルリクエスト・レスポンスの重要性

仕様書だけでなく、コピー&ペーストで試せる具体的なリクエスト・レスポンス例を必ず用意しましょう。

サンプルリクエスト Terminal 
curl -X POST https://api.example.com/v1/users \
  -H "Authorization: Bearer your_token_here" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "田中太郎",
    "email": "tanaka@example.com",
    "role": "editor"
  }'
成功レスポンス（201 Created） JSON 
{
  "id": 42,
  "name": "田中太郎",
  "email": "tanaka@example.com",
  "role": "editor",
  "created_at": "2025-01-15T10:30:00Z"
}
エラーレスポンス（400 Bad Request） JSON 
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力データにエラーがあります",
    "details": [
      { "field": "email", "message": "有効なメールアドレスを入力してください" }
    ]
  }
}
Section 9-5

APIの発見と選び方

世の中には数多くの公開APIが存在します。自分のプロジェクトに適したAPIを見つけ、安全に利用するための方法を学びましょう。

5.1 公開APIディレクトリ

以下のサイトで、カテゴリ別に公開APIを探すことができます。

📚

public-apis (GitHub)

無料で使える公開APIのリスト。カテゴリ分類されており、認証要否やHTTPS対応なども記載されています。

🔍

RapidAPI Hub

数万件のAPIが登録されたマーケットプレイス。APIの検索・テスト・利用が一つのプラットフォームでできます。

🌐

API Directory (ProgrammableWeb)

長年運営されているAPIディレクトリ。ニュースやトレンド情報も提供しています。

5.2 API利用時の確認ポイント

確認項目 チェック内容 注意点
料金プラン 無料枠の範囲と制限 無料枠を超えると課金される場合がある
レート制限 1日/1時間あたりのリクエスト数 制限超過でサービスが停止する
利用規約 商用利用の可否、データの二次利用 規約違反でアカウント停止のリスク
認証方法 APIキー、OAuth等の認証方式 認証が不要なAPIは手軽だが制限が多い
ドキュメントの品質 情報の充実度と更新頻度 ドキュメントが不十分なAPIは開発コスト増
サービスの安定性 稼働率、メンテナンス頻度 突然のサービス終了リスクも考慮する
warning レート制限とコストに注意

APIには必ずレート制限（リクエスト数の上限）があります。開発中にループ処理でAPIを大量に呼び出すと、制限に達してしばらく利用できなくなることがあります。また、有料APIでは意図せず高額な請求が発生する可能性もあるため、必ず料金プランを確認し、テスト時は無料枠の範囲内で利用しましょう。

9章 練習問題

Q1

良いAPIドキュメントに含まれるべき情報として、最も重要度が低いものはどれですか？

  1. エンドポイントのURLとHTTPメソッド
  2. 開発チームのメンバー紹介
  3. リクエスト/レスポンスの具体例
  4. 認証方法の説明
Q2

OpenAPI仕様の説明として正しいものはどれですか？

  1. APIのソースコードを公開する仕組みである
  2. REST APIの構造をYAMLやJSONで定義する標準規格である
  3. APIのレスポンス速度を測定するツールである
  4. APIのセキュリティを自動テストするフレームワークである
Q3

公開APIを選ぶ際に確認すべき項目として、適切でないものはどれですか？

  1. レート制限（リクエスト数の上限）
  2. APIサーバーのプログラミング言語
  3. 料金プランと無料枠の範囲
  4. 利用規約と商用利用の可否
Answer

9章 解説・解答

Q1の解答: B

APIドキュメントに必要な情報は、エンドポイント・HTTPメソッド・パラメータ・レスポンス例・認証方法などの技術的な情報です。開発チームのメンバー紹介はAPIの利用に直接関係なく、APIドキュメントとしての重要度は低いです。

Q2の解答: B

OpenAPI仕様（旧Swagger仕様）は、REST APIの構造をYAMLまたはJSON形式で記述する標準規格です。この定義ファイルからSwagger UIなどのツールを使って、インタラクティブなドキュメントを自動生成できます。ソースコードの公開やセキュリティテストとは異なります。

Q3の解答: B

公開APIを利用する際、サーバー側のプログラミング言語を知る必要はありません。APIはHTTPプロトコルを通じて通信するため、サーバーの実装言語に関係なく利用できます。一方、レート制限・料金プラン・利用規約は利用前に必ず確認すべき項目です。
