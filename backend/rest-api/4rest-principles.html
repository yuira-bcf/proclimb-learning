Section 4-1

RESTとは

REST（REpresentational State Transfer）は、2000年にRoy Fieldingが博士論文で提唱したWebのアーキテクチャスタイルです。HTTPの設計思想を活かし、シンプルで拡張性の高いAPI設計を実現する考え方です。

1.1 RESTの基本概念

RESTは「プロトコル」や「仕様書」ではなく、設計の「スタイル」「考え方」です。以下の点が重要です。

  • Web上の全てのものをリソース（資源）として捉える
  • 各リソースに固有のURI（アドレス）を割り当てる
  • HTTPメソッド（GET, POST等）でリソースを操作する
  • データの表現形式（JSON等）でリソースの状態をやりとりする
lightbulb RESTは仕様ではない

RESTはHTTPのような厳密な仕様書ではなく、API設計の「ガイドライン」に近い概念です。そのため、実装には解釈の幅があり、完全にRESTに準拠したAPIは実務ではあまり多くありません。

Section 4-2

RESTの6つの制約

Roy Fieldingは、RESTアーキテクチャを定義する6つの制約を提唱しました。これらの制約に従うことで、スケーラブルで保守しやすいシステムを構築できます。

2.1 6つの制約の詳細

💻

1. クライアント-サーバー分離

UIの関心事（クライアント）とデータ処理の関心事（サーバー）を分離します。フロントエンドとバックエンドを独立して開発・デプロイでき、それぞれが進化しやすくなります。

📄

2. ステートレス

サーバーはクライアントの状態を保持しません。各リクエストには処理に必要な全ての情報を含める必要があります。これにより、サーバーのスケーラビリティが向上します。

📦

3. キャッシュ可能

レスポンスにはキャッシュ可否の情報を含めます。キャッシュを活用することで、ネットワーク通信を削減し、レスポンス速度を向上させます。

🔗

4. 統一インターフェース

全てのリソースに対して統一された方法（URI + HTTPメソッド）でアクセスします。これにより、APIの使い方がシンプルで予測可能になります。

📑

5. 階層化システム

クライアントは、直接サーバーと通信しているのか、中間サーバー（ロードバランサー、プロキシ等）を経由しているのかを意識する必要がありません。

💻

6. コードオンデマンド（任意）

サーバーがクライアントに実行可能なコード（JavaScriptなど）を送信できます。これは唯一のオプション制約で、必ずしも適用する必要はありません。

実務で特に重要なのはステートレスと統一インターフェースです。ステートレスにより水平スケーリング（サーバー台数の増加）が容易になり、統一インターフェースによりAPIの学習コストが下がります。

Section 4-3

リソースとURI

REST APIにおいて、リソースはAPIが扱うデータや概念を指し、URIはそのリソースを一意に識別するアドレスです。

3.1 リソース指向設計

💡 RESTful URI設計 — 良い例 vs 悪い例
check_circle 良い設計（名詞ベース）
GET /users
GET /users/123
POST /users
DELETE /users/123
cancel 悪い設計（動詞ベース）
GET /getUsers
GET /getUserById?id=123
POST /createUser
POST /deleteUser/123

REST APIのURLは名詞（リソース名）を使って設計します。「何を」操作するかはURLで、「どう」操作するかはHTTPメソッドで表現します。

良い設計（名詞ベース） URI 
GET    /users          ← ユーザー一覧を取得
GET    /users/123      ← ID=123のユーザーを取得
POST   /users          ← 新しいユーザーを作成
PUT    /users/123      ← ID=123のユーザーを更新
DELETE /users/123      ← ID=123のユーザーを削除
悪い設計（動詞ベース） URI 
GET    /getUsers             ← NG: 動詞を使っている
GET    /getUserById?id=123   ← NG: 動詞を使っている
POST   /createUser           ← NG: 動詞を使っている
POST   /deleteUser/123       ← NG: メソッドと意味が不一致
warning URI設計の鉄則

URLにはリソース名（名詞）を使い、操作（get, create, delete等の動詞）は含めません。操作はHTTPメソッドで表現するのがRESTの基本です。

3.2 コレクションと個別リソース

URI 種類 説明
/users コレクション ユーザー全体の集合を表す
/users/123 個別リソース ID=123の特定ユーザーを表す

コレクション名は複数形を使うのが慣例です（/users、/products、/orders）。

Section 4-4

RESTfulなAPI設計パターン

実際のアプリケーションでは、リソース同士が関連を持つことが多くあります。ネストしたURIで親子関係を表現します。

4.1 ネストしたリソース

親子関係のあるリソース URI 
/users/{userId}/orders           ← ユーザーの注文一覧
/users/{userId}/orders/{orderId} ← 特定の注文
/products/{productId}/reviews    ← 商品のレビュー一覧
/teams/{teamId}/members          ← チームのメンバー一覧
lightbulb ネストの深さ

ネストが3階層以上（例: /users/1/orders/2/items/3）になる場合は、設計を見直した方がよいことが多いです。深いネストはURLが長くなり、使いづらくなります。

4.2 CRUD操作の対応表

操作 HTTPメソッド URI例 説明
一覧取得 GET /users 全ユーザーの一覧を取得
個別取得 GET /users/1 ID=1のユーザーを取得
新規作成 POST /users 新しいユーザーを作成
全体更新 PUT /users/1 ID=1のユーザーを全体更新
部分更新 PATCH /users/1 ID=1のユーザーを部分更新
削除 DELETE /users/1 ID=1のユーザーを削除
Section 4-5

RESTとRESTfulの違い

「REST」と「RESTful」は似た言葉ですが、意味が少し異なります。その違いと、実務での使われ方を理解しましょう。

5.1 定義の違い

用語 意味
REST Roy Fieldingが提唱したアーキテクチャスタイル（理論・概念）
RESTful RESTの原則に従って設計・実装されたAPI（実装物）

5.2 実務での適用状況

完全にREST制約に準拠したAPIは現実には少なく、多くの「RESTful API」はRESTの一部の原則を適用しています。

REST制約 実務での適用度 備考
クライアント-サーバー ほぼ全てで適用 SPA + API構成が主流
ステートレス ほぼ全てで適用 JWT等のトークン認証で実現
統一インターフェース 多くで適用 URI + HTTPメソッドの規約
キャッシュ可能 部分的に適用 Cache-Controlヘッダーの活用
階層化システム 部分的に適用 CDN、ロードバランサーの利用
HATEOAS ほとんど適用されない レスポンスに次のアクションリンクを含める
コードオンデマンド ほとんど適用されない オプション制約
lightbulb 実務での考え方

実務では「完全なREST準拠」よりも「実用的で分かりやすいAPI」を優先する場面が多くあります。RESTの原則を理解した上で、プロジェクトの要件に合わせて柔軟に設計することが大切です。

Exercise

4章 練習問題

Q1

RESTの6つの制約のうち、唯一のオプション（任意）制約はどれですか？

  1. ステートレス
  2. キャッシュ可能
  3. コードオンデマンド
  4. 統一インターフェース
Q2

RESTfulなURI設計として最も適切なものはどれですか？

  1. GET /getUser?id=1
  2. POST /createUser
  3. GET /users/1
  4. GET /api/fetchUserData/1
Q3

「ステートレス」の説明として正しいものはどれですか？

  1. サーバーがクライアントの過去のリクエスト情報を全て記憶する
  2. 各リクエストは独立しており、処理に必要な情報を全て含める
  3. クライアントがサーバーの状態を管理する
  4. 通信が暗号化されていること
Answer

4章 解説・解答

Q1の解答: C

コードオンデマンド（サーバーからクライアントに実行可能コードを送信する機能）は、RESTの6つの制約の中で唯一のオプション制約です。他の5つは必須の制約です。

Q2の解答: C

GET /users/1は、名詞ベースのリソース名（users）とHTTPメソッド（GET）で操作を表現しており、RESTfulな設計です。A、B、Dは動詞（get, create, fetch）を含んでいるためRESTfulではありません。

Q3の解答: B

ステートレスとは、サーバーがクライアントの状態を保持せず、各リクエストが処理に必要な全ての情報を自己完結的に含むことを意味します。これによりサーバーのスケーラビリティが向上します。
