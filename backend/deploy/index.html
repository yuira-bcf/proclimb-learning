Step 1

DockerfileとDockerignoreを作成

Renderでビルド・公開するために必要なファイルを作成します。Renderは言語としてJavaを直接サポートしていないため、Dockerという仮想コンテナ技術を利用します。ビルドはRender側でDockerfileを読み取って行われるため、ローカルにDockerをインストールしていなくても問題ありません。

lightbulb Docker とは？

アプリケーションの実行環境をまるごとパッケージングする技術です。「どのOSでも同じように動く」状態を実現できます。詳しくは Docker構築入門 を参照してください。

1-1 VSCodeでプロジェクトを開く

1 VSCodeを立ち上げ、対象のJavaプロジェクトを開きます。

1-2 AIにプロンプトを投入

2 Claude Code / Copilot 等のAIツールに以下のプロンプトを投入し、Dockerfile と .dockerignore を自動生成します。
smart_toy AIプロンプト — Dockerfile 自動生成 
Spring Boot（Maven）プロジェクトを Render（Docker デプロイ）で確実に動作させる前提で、必要なファイル作成・修正・検証まで実施してください。

【目的】
- Render でビルド/起動エラーなくデプロイできる状態にする
- 過去に発生した以下を再発させない
  - mvnw 実行権限エラー（Permission denied）
  - Java ファイル先頭の BOM によるコンパイルエラー（illegal character: '\ufeff'）
  - ルート URL `/` で 404 になる問題（必要なら `/todo` へ誘導）

【必須対応】
1. ルート直下に Dockerfile を作成/更新
   - Maven Wrapper を使ってビルド
   - Linux 上で `mvnw` が必ず実行できるように `chmod +x ./mvnw` を入れる
   - Render の待受ポート 10000 で起動する（`-Dserver.port=10000`）
   - マルチステージビルドにする（build + runtime）

2. ルート直下に .dockerignore を作成/更新
   - 必須: .git / .idea / .vscode / target / build / out

3. 文字コードチェック
   - `src/main/java/**/*.java` を走査し、UTF-8 BOM があれば除去

4. ルーティング確認
   - `/` アクセス時に 404 になるなら、`/todo` へリダイレクトする最小修正を追加

5. 検証
   - `./mvnw test` を実行して成功を確認
   - 可能なら `docker build` も実行し、Docker ビルド成功を確認

【出力してほしい内容】
- 変更したファイル一覧
- Dockerfile と .dockerignore の最終内容
- BOM 除去を行ったファイル一覧（なければ「なし」）
- テスト結果（成功/失敗）
- Render 側で確認すべきポイント

1-3 コミット・プッシュ

3 問題なく作成されたら、Sourcetree等でコミット・プッシュします。事前準備はこれだけで完了です。
Step 2

Renderアカウント作成とGitHub連携

2-1 Renderにアクセス

1 https://render.com/ にアクセスし、「Start for free」をクリックします。

2-2 アカウント作成

2 任意の方法でアカウントを作成します（Google OAuth 等）。

2-3 Web Service を新規作成

3 ワークスペース画面が開いたら、右上の「+New」→「Web Service」をクリックします。

2-4 GitHub連携

4 「Git Provider」→「GitHub」をクリックし、「Authorize Render」で連携を許可します。

2-5 リポジトリを選択

5 Renderにインストールするリポジトリを選択します。
info リポジトリの選択方式
  • All Repositories — GitHubの全リポジトリをRenderに接続
  • Only Select Repositories — 任意の1つだけを接続（推奨）

2-6 接続確認

6 Installボタンをクリックし、New Web Service画面にリポジトリが表示されたらOKです。
info この段階では…

まだデプロイ（公開）はされていません。Render側に、デプロイ対象となるリポジトリを登録しただけの状態です。

Step 3

RenderでJavaアプリケーションをデプロイ

3-1 リポジトリを選択

1 インストール済みリポジトリの中から、デプロイするリポジトリをクリックします。

3-2 サービス設定

2 設定画面で以下の項目を登録します。
項目設定値備考
Name任意デフォルトはリポジトリ名
LanguageDockerJavaではなくDockerを選択
Branchmainデフォルトブランチ
RegionOregon (US West)
Root Directory（空欄）入力しない
Instance TypeFreeデフォルトはStarterなので注意
Environment Variables（空欄）必要に応じてAIに確認
warning Instance Type に注意

デフォルトでは Starter（有料） が選択されています。学習用途では必ず Free に変更してください。

3-3 デプロイ実行

3 下部の「Deploy Web Service」をクリックします。

3-4 デプロイ結果の確認

4 サービス画面が表示され、赤い×が出ていなければデプロイ成功です。
error デプロイが失敗した場合

画面中央付近に赤い×アイコンが表示されます。下部のログをメモ帳等に保存し、AIにファイル添付して修正依頼 → 修正後コミット・プッシュで自動再デプロイされます。

Step 4

URLにアクセスし、表示・機能のテスト

4-1 公開URLにアクセス

1 サービス画面上部に表示されているURLをコピーし、ブラウザでアクセスします。
link URLについて
  • リダイレクト設定済みの場合 → 自動的に正しいページへ遷移します
  • リダイレクト未実装の場合 → URLの末尾に適切なパスを追加してください

例）https://todos-o31a.onrender.com → https://todos-o31a.onrender.com/todo

4-2 機能テスト

2 各下層ページ、CRUD操作、バリデーション等が正しく動作するかチェックします。

4-3 デプロイ完了

3 問題がなければデプロイは完了です。以降、機能を追加してプッシュするたびに自動で再デプロイされます。
Notice

Render Freeプランの注意事項

timer スリープ機能

15分間アクセスがないとServiceが停止します。次回アクセス時にService起動で1分程度かかります。

storage DBリセット

FreeプランではDBが停止するたびにリセットされます。永続的にデータを保存するには有料プランが必要です。

schedule 月間稼働時間

月間Service起動時間が750時間です。1Serviceなら問題ありませんが、複数Serviceでは不足する場合があります。

memory メモリ制限

Freeプランではメモリが小さいため、重いサービスでは動作が鈍くなる可能性があります。

warning Node.js 併用時の注意

Node.js等を導入している場合は動作の保証ができません。AIに確認しながら適切なデプロイ環境に整えましょう。
