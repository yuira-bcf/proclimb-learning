Section 7-1

この章のねらい

この章では、システム開発における環境構成について学びます。本番環境、検証環境、開発環境など、複数の環境を分けることの意味、それぞれの役割、環境ごとの設定の分離方法、デプロイの流れを理解することを目標とします。

この章を読み終わる頃の目標

  • なぜ複数の環境を分けるのか、その理由を説明できる
  • 本番環境、検証環境、開発環境それぞれの役割と特徴を理解している
  • 環境ごとの設定の分離方法（環境変数、設定ファイル）を理解している
  • 開発からリリースまでのデプロイの流れをイメージできる
重要なポイント

環境の分離は、システムの安定性とセキュリティを守るための基本です。「開発環境で動いたから本番でも動くはず」という考え方は危険です。環境の違いを理解し、適切に管理することが重要です。

Section 7-2

なぜ環境を分けるのか

1　本番データを守る

本番環境には、顧客の個人情報、注文データ、売上データなど、重要なデータが保存されています。開発中の不完全なコードを本番環境で動かすと、

  • データが破壊される
  • データが漏洩する
  • サービスが停止する

といった深刻な問題が発生するリスクがあります。環境を分けることで、開発やテストの影響が本番データに及ばないようにします。

2　品質を確保する

本番環境にリリースする前に、しっかりとテストを行う必要があります。テスト専用の環境があれば、

  • 様々なテストケースを試せる
  • 本番に影響を与えずに検証できる
  • 問題があっても本番には影響しない

という安心感を持ってテストできます。

3　開発効率を上げる

開発者が自由に使える環境があれば、

  • 試行錯誤しながら開発できる
  • 失敗しても他の人に影響しない
  • 新しい技術や手法を試せる

という形で、開発効率が向上します。

4　一般的な環境構成

環境の流れ（イメージ） 
開発環境（Development）
    ↓
検証環境（Staging / Test）
    ↓
本番環境（Production）

コードは、開発環境 → 検証環境 → 本番環境 の順に移動していきます。各環境でテストや確認を行い、問題がなければ次の環境へ進みます。

Section 7-3

本番環境（Production）の役割

1　本番環境とは

本番環境（Production Environment、略して「本番」「Prod」）は、実際のユーザーがアクセスし、実際のデータを扱う環境です。

本番環境 = ビジネスが動いている環境

  • 実際の顧客がアクセスする
  • 実際の取引が行われる
  • 実際のお金が動く
  • 問題が起きると、ビジネスに直接影響する

2　本番環境の特徴

  • 高い可用性：24時間365日稼働し続ける必要がある（場合が多い）
  • 高い信頼性：障害が起きにくく、起きても復旧が早い
  • 高いセキュリティ：アクセス制限が厳しく、監査ログが取られる
  • 高い性能：実際の負荷に耐えられる性能
  • 本物のデータ：顧客の個人情報など、重要なデータ

3　本番環境へのアクセス制限

本番環境は、厳格にアクセスを制限する必要があります。

  • アクセスできる人を限定する
  • アクセス時に認証・認可を厳密に行う
  • アクセスログを記録する
  • 変更作業は承認プロセスを経て行う
本番環境の鉄則
  • 本番環境で「試しに」実行しない
  • 本番環境で直接コードを編集しない
  • 本番データを開発・テストに使わない（匿名化が必要）
  • 本番環境への変更は、必ず検証環境でテストしてから
Section 7-4

検証環境・テスト環境の役割

1　検証環境とは

検証環境（Staging Environment、Test Environment）は、本番環境にリリースする前に、動作を確認するための環境です。

  • ステージング環境：本番環境に限りなく近い構成で、最終確認を行う
  • テスト環境：結合テストやシステムテストを行う
  • QA環境：品質保証チームがテストを行う

会社やプロジェクトによって、名称や役割分担は異なります。

2　検証環境の特徴

  • 本番に近い構成：本番環境と同じ（または近い）サーバー構成、ミドルウェア、設定
  • テスト用データ：本番データではなく、テスト用に作成されたデータ
  • アクセス制限：関係者のみアクセス可能
  • 自由なテスト：様々なテストシナリオを試せる

3　検証環境で行うこと

  • 結合テスト・システムテストの実施
  • リリース前の最終動作確認
  • 本番相当の負荷をかけた性能テスト
  • リリース手順のリハーサル
  • 障害発生時の復旧手順の確認
本番と検証の差異に注意

「検証環境で動いたのに、本番で動かない」というトラブルはよく発生します。原因としては、

  • 環境設定の違い
  • データ量の違い
  • ネットワーク構成の違い
  • 外部連携先の違い

などがあります。検証環境は、できるだけ本番に近づけることが重要です。

Section 7-5

開発環境の役割

1　開発環境とは

開発環境（Development Environment、略して「Dev」）は、エンジニアが日常的にコーディングやデバッグを行う環境です。

  • ローカル開発環境：各エンジニアのPC上に構築された環境
  • 共有開発環境：チームで共有する開発サーバー

2　開発環境の特徴

  • 自由度が高い：試行錯誤、実験が自由にできる
  • 簡易な構成：本番ほど厳密な構成は不要
  • ダミーデータ：テスト用のサンプルデータを使用
  • デバッグ機能：詳細なログ出力、デバッガの使用が可能

3　ローカル開発環境の構築

最近は、以下のようなツールを使ってローカル開発環境を構築することが一般的です。

  • Docker：コンテナ技術を使って、環境を簡単に構築・再現できる
  • 仮想マシン（VM）：VirtualBox、VMwareなどで仮想環境を構築
  • クラウド開発環境：GitHub Codespaces、AWS Cloud9など
Dockerのメリット
  • 「自分の環境では動くのに」問題を減らせる
  • 環境構築の手順をコード化（Infrastructure as Code）できる
  • 新しいメンバーの環境構築が簡単
  • 本番環境と同じ構成を再現しやすい
Section 7-6

その他の環境

1　サンドボックス環境

実験や検証のために、本番とは完全に分離された環境です。新しい技術の検証、セキュリティテスト、負荷テストなどに使用します。

2　デモ環境

営業活動やユーザー向けのデモンストレーションに使用する環境です。見せ方に最適化されたデータが入っていることが多いです。

3　DR環境（災害復旧環境）

本番環境に障害が発生した場合に切り替えるための環境です。別のデータセンターやリージョンに用意することが多いです。

4　環境の数と管理

環境が多すぎると管理コストが増えます。プロジェクトの規模や要件に応じて、適切な数の環境を設計することが重要です。

環境 主な用途 データ
本番 実際のサービス提供 本物のデータ
ステージング リリース前の最終確認 テストデータ
テスト 結合・システムテスト テストデータ
開発 日常的な開発作業 ダミーデータ
Section 7-7

環境ごとの構成要素

1　環境を構成する要素

システムの環境は、以下のような要素で構成されます。

  • サーバー：アプリケーションサーバー、Webサーバー、DBサーバーなど
  • ミドルウェア：データベース、キャッシュ、メッセージキューなど
  • ネットワーク：ファイアウォール、ロードバランサー、DNS
  • アプリケーション：デプロイされたプログラム
  • データ：データベース内のデータ
  • 設定：接続先URL、認証情報、各種パラメータ

2　環境による違い

環境によって、同じ構成要素でも設定が異なります。

項目 開発環境 本番環境
DB接続先 localhost:5432 prod-db.example.com:5432
ログレベル DEBUG INFO または WARN
外部API サンドボックスAPI 本番API
メール送信 無効化 or テスト用 本物のメール送信
セキュリティ 緩め 厳格
Section 7-8

設定の分離：環境変数と設定ファイル

1　なぜ設定を分離するのか

環境ごとに設定が異なるため、設定値をコードにハードコードしてしまうと、

  • 環境を切り替えるたびにコードを書き換える必要がある
  • パスワードなどの機密情報がコードに含まれてしまう
  • 設定ミスによる本番障害のリスクが高まる

といった問題が発生します。設定をコードから分離することで、同じコードを異なる環境で動かせるようになります。

2　環境変数

環境変数（Environment Variables）は、OSやコンテナレベルで設定される変数です。アプリケーションは、環境変数を読み取って設定を取得します。

環境変数の例 
# 環境変数の設定
export DATABASE_URL=postgresql://user:pass@localhost:5432/mydb
export API_KEY=your-api-key-here
export LOG_LEVEL=DEBUG

# アプリケーションから読み取り
String dbUrl = System.getenv("DATABASE_URL");
String apiKey = System.getenv("API_KEY");
環境変数のメリット
  • コードに設定を含めなくて良い
  • 環境ごとに異なる値を設定できる
  • 機密情報（パスワード、APIキー）をコードから分離できる
  • 12-Factor Appの原則に沿っている

3　設定ファイル

環境ごとの設定ファイルを用意し、起動時にどのファイルを使うかを指定する方法もあります。

設定ファイルの例（Spring Boot） 
application.properties        # 共通設定
application-dev.properties    # 開発環境用
application-stg.properties    # ステージング環境用
application-prod.properties   # 本番環境用

# 起動時にプロファイルを指定
java -jar app.jar --spring.profiles.active=prod

4　機密情報の管理

パスワードやAPIキーなどの機密情報は、特に慎重に管理する必要があります。

  • Gitにコミットしない：.gitignoreに設定ファイルを追加
  • シークレット管理サービス：AWS Secrets Manager、HashiCorp Vaultなど
  • 暗号化：設定ファイルを暗号化して管理
よくある事故

パスワードやAPIキーを含む設定ファイルをGitHubにプッシュしてしまい、機密情報が漏洩する事故が頻繁に発生しています。一度プッシュしてしまうと、履歴からの完全削除は困難です。

Section 7-9

デプロイの流れ

1　デプロイとは

デプロイ（Deploy）とは、開発したアプリケーションを、サーバー上で動作する状態にすることです。「リリース」「展開」とも呼ばれます。

2　一般的なデプロイの流れ

デプロイの流れ 
1. コードをコミット・プッシュ
       ↓
2. CIでビルド・テスト
       ↓
3. 開発環境にデプロイ・動作確認
       ↓
4. 検証環境にデプロイ・テスト
       ↓
5. 本番環境にデプロイ・動作確認

3　CI/CD

CI/CD（Continuous Integration / Continuous Delivery または Deployment）は、コードの変更から本番デプロイまでを自動化する仕組みです。

CI（継続的インテグレーション）

コードがプッシュされるたびに、自動でビルド・テストを実行する。問題を早期に発見できる。

CD（継続的デリバリー/デプロイ）

CIが成功したら、自動で（または承認後に）環境にデプロイする。リリースの頻度を上げ、品質を安定させる。

4　本番デプロイの注意点

  • バックアップ：デプロイ前にデータベースのバックアップを取る
  • 切り戻し計画：問題発生時に旧バージョンに戻す手順を用意
  • 段階的デプロイ：一部のサーバーから順に展開（カナリアリリース、ブルーグリーンデプロイ）
  • 監視：デプロイ後の監視を強化し、問題を早期に検知
  • メンテナンス時間：必要に応じてメンテナンス時間を設定

5　デプロイ方式

ローリングデプロイ

サーバーを1台ずつ順番に更新する方式。サービスを停止せずに更新できるが、新旧バージョンが混在する時間がある。

ブルーグリーンデプロイ

本番環境を2つ（Blue/Green）用意し、新バージョンを非稼働系にデプロイ後、トラフィックを切り替える方式。切り戻しが容易。

カナリアリリース

新バージョンを一部のユーザーにのみ公開し、問題がなければ徐々に範囲を広げる方式。リスクを最小化できる。

Section 7-10

環境管理のベストプラクティス

1　Infrastructure as Code（IaC）

環境の構成をコードで定義し、バージョン管理する考え方です。

  • Terraform、CloudFormation、Ansible、Dockerなど
  • 環境の再現性が高まる
  • 変更履歴が追跡できる
  • レビューが可能

2　環境の一貫性を保つ

  • 開発・検証・本番で、できるだけ同じ構成を使う
  • ミドルウェアのバージョンを揃える
  • Dockerを使って環境を統一する

3　本番データの取り扱い

  • 本番データを開発・テストに使う場合は、必ず匿名化・マスキングする
  • 個人情報を含むデータは特に注意
  • 法令（個人情報保護法など）を遵守する

4　アクセス権限の管理

  • 環境ごとにアクセスできる人を限定する
  • 本番環境へのアクセスは特に厳格に管理
  • 最小権限の原則（必要最低限の権限のみ付与）
  • アクセスログを記録・監査する
Section 7-11

まとめ

  • 環境を分けることで、本番データの保護、品質の確保、開発効率の向上が実現できる。
  • 本番環境は実際のユーザーがアクセスする環境であり、高い可用性、信頼性、セキュリティが求められる。
  • 検証環境は本番環境に近い構成で、リリース前の最終確認やテストを行う。
  • 開発環境はエンジニアが日常的に開発作業を行う環境で、自由度が高い。
  • 環境ごとに設定が異なるため、環境変数や設定ファイルを使って設定をコードから分離する。
  • 機密情報（パスワード、APIキー）は、Gitにコミットしない、シークレット管理サービスを使うなど、慎重に管理する。
  • デプロイは、開発環境→検証環境→本番環境の順に行い、各段階でテストや確認を行う。
  • CI/CDを導入することで、ビルド・テスト・デプロイを自動化し、品質と効率を向上できる。
  • 本番デプロイでは、バックアップ、切り戻し計画、段階的デプロイ、監視が重要である。
  • Infrastructure as Code（IaC）で環境をコード化し、一貫性と再現性を高めることがベストプラクティスである。
