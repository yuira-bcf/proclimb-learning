Section 15-1

工程概要

管理者・一般ユーザーの権限を設定し、ロールに基づいたアクセス制御を実装します。

学習テーマ

管理者・一般ユーザーの権限設定

主な技術

@PreAuthorize / hasRole / sec:authorize

関連レイヤー

Security / Controller / View

この機能で行うこと

toggle_on

メソッドセキュリティの有効化

@EnableMethodSecurityを設定

lock

URLレベルのアクセス制御

/admin/**パスの保護

verified_user

メソッドレベルの権限チェック

@PreAuthorizeでロール検証

visibility

画面の権限別表示

sec:authorizeで要素の表示制御

Section 15-2

成果物

この機能で作成・更新するファイルの一覧です。

ファイル 変更内容
SecurityConfig.java @EnableMethodSecurity、URLパターン制御
TodoController.java @PreAuthorize("hasRole('ADMIN')")
TodoService.java isOwner()所有者チェックメソッド
list.html sec:authorize="hasRole('ADMIN')"で管理者メニュー表示

@PreAuthorize式の例

  • hasRole('ADMIN') - ADMIN権限を持つ場合
  • hasAnyRole('USER', 'ADMIN') - いずれかの権限を持つ場合
  • isAuthenticated() - 認証済みの場合
  • #id - メソッド引数を参照
  • authentication.name - ログインユーザー名
  • @todoService.isOwner(#id, authentication.name) - カスタムメソッド呼び出し
Section 15-3

生成プロンプト

以下のプロンプトをAIアシスタント（Claude、ChatGPT等）に入力することで、実装コードを生成できます。

smart_toy AIへの入力プロンプト

あなたはJavaプログラマーです。

Spring Boot 3.xのToDoアプリにロール別権限制御を追加してください。

【要件】
- ロール：ADMIN（管理者）とUSER（一般ユーザー）
- ADMIN：全ユーザーのToDo閲覧・編集・削除可能、ユーザー管理可能
- USER：自分のToDoのみ閲覧・編集・削除可能
- 画面上でロールに応じたメニュー表示切り替え
- URLレベルでのアクセス制御

【実装箇所】

■ SecurityConfig.java
- @EnableMethodSecurityを追加
- /admin/**パスはROLE_ADMINのみ許可
- /todo/**パスはROLE_USER, ROLE_ADMIN許可

■ TodoController.java
- @PreAuthorize("hasRole('ADMIN')")で管理者専用エンドポイント
- @PreAuthorize("hasRole('ADMIN') or @todoService.isOwner(#id, authentication.name)")
  で削除処理を制御

■ TodoService.java
- isOwner(Integer todoId, String username)メソッドを追加
- ToDoの所有者かどうかを判定

■ list.html（Thymeleaf + Security）
- sec:authorize="hasRole('ADMIN')"で管理者メニューを表示
- 一括削除チェックボックスは管理者のみ表示
- 全ToDo一覧リンクは管理者のみ表示

【技術スタック】
- Spring Boot 3.x
- Spring Security (@PreAuthorize, hasRole)
- Thymeleaf (sec:authorize)

必要なコードを提示してください。
lightbulb ポイント

「管理者が一般ユーザーのToDoを編集する場合は、編集履歴を残すようにしてください」のように、監査要件を追加するとより堅牢な実装になります。
