Section 19-1

工程概要

@Transactionalでデータベース操作の整合性を保証し、エラー発生時のロールバックを実装します。

学習テーマ

データ整合性の保証

主な技術

@Transactional / rollbackFor / Propagation

関連レイヤー

Service / Repository

Section 19-2

成果物

この機能で作成・更新するファイルの一覧です。

ファイル 変更内容
TodoService.java @Transactional適用、ロールバック制御

@Transactionalの属性

  • readOnly - 読み取り専用（最適化のヒント）デフォルト: false
  • timeout - タイムアウト（秒）デフォルト: -1（制限なし）
  • rollbackFor - ロールバック対象の例外 デフォルト: RuntimeException
  • noRollbackFor - ロールバックしない例外 デフォルト: なし
  • propagation - トランザクションの伝播 デフォルト: REQUIRED

サンプルソースコード

TodoService.java 
@Service
public class TodoService {

    private final TodoMapper todoMapper;
    private final HistoryMapper historyMapper;
    private final AuditService auditService;

    // 基本的なトランザクション - 更新系処理
    @Transactional
    public void save(Todo todo) {
        todoMapper.insert(todo);
        historyMapper.insert(new History(todo, "CREATE"));
        // どちらかで例外発生 → 両方ロールバック
    }

    // 読み取り専用トランザクション - 参照系処理
    @Transactional(readOnly = true)
    public List<Todo> findAll() {
        return todoMapper.findAll();
    }

    // チェック例外もロールバック対象に含める
    @Transactional(rollbackFor = Exception.class)
    public void updateWithValidation(Todo todo) throws ValidationException {
        validate(todo);  // チェック例外でもロールバック
        todoMapper.update(todo);
    }
}

lightbulb コード解説

  • @Transactional - メソッド内の全DB操作を1つのトランザクションとして実行
  • readOnly = true - 読み取り専用を明示し、DBの最適化ヒントを提供
  • rollbackFor = Exception.class - チェック例外もロールバック対象に含める（デフォルトはRuntimeExceptionのみ）
  • 複数テーブル更新 - todo + historyを同時に更新し、どちらかが失敗したら両方ロールバック
AuditService.java（独立トランザクション） 
@Service
public class AuditService {

    private final AuditLogMapper auditLogMapper;

    // REQUIRES_NEW: 呼び出し元と独立したトランザクション
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(String action, String detail) {
        AuditLog log = new AuditLog(action, detail, LocalDateTime.now());
        auditLogMapper.insert(log);
        // 呼び出し元がロールバックしても、監査ログは残る
    }
}

lightbulb コード解説

  • Propagation.REQUIRES_NEW - 常に新しいトランザクションを開始（呼び出し元とは独立）
  • 監査ログの独立性 - 本体処理がロールバックしても、「処理を試みた」記録は残す
  • セキュリティ要件 - 監査ログは改ざん防止のため、本体と分離することが多い

schema トランザクション伝播の図解

┌─────────────────────────────────────────────────────────────┐
│  TodoService.save() @Transactional                          │
│  ┌────────────────────────────────────────────────────────┐ │
│  │  トランザクション A                                      │ │
│  │                                                        │ │
│  │  1. todoMapper.insert(todo)                            │ │
│  │  2. historyMapper.insert(history)                      │ │
│  │  3. auditService.log()                                 │ │
│  │        ↓                                               │ │
│  │  ┌──────────────────────────────────┐                  │ │
│  │  │  トランザクション B (REQUIRES_NEW) │                  │ │
│  │  │  auditLogMapper.insert()         │ ← 独立コミット   │ │
│  │  └──────────────────────────────────┘                  │ │
│  │                                                        │ │
│  │  4. 例外発生！→ トランザクションAのみロールバック        │ │
│  │     (todo, history は取り消し、auditLog は残る)        │ │
│  └────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘

【Propagation（伝播）の種類】
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 REQUIRED      │ 既存トランザクションがあれば参加、なければ新規作成（デフォルト）
 REQUIRES_NEW  │ 常に新しいトランザクションを作成（既存は一時停止）
 MANDATORY     │ 既存トランザクション必須（なければ例外）
 SUPPORTS      │ 既存があれば参加、なければトランザクションなしで実行
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Section 19-3

生成プロンプト

以下のプロンプトをAIアシスタント（Claude、ChatGPT等）に入力することで、実装コードを生成できます。

auto_awesome AIプロンプト

あなたはJavaプログラマーです。

Spring Boot 3.xのToDoアプリにトランザクション管理機能を追加してください。

【要件】
- TodoServiceに@Transactionalを適用してデータベース操作の整合性を保証
- ToDoと履歴を同一トランザクションで保存し、どちらかで例外が発生したら両方ロールバック
- 読み取り専用メソッドには@Transactional(readOnly = true)を適用
- rollbackFor属性でチェック例外もロールバック対象に設定
- 監査ログ記録にはREQUIRES_NEWを使用し、本体がロールバックしても監査ログは残す

【実装箇所】

■ TodoService.java
- save()メソッドに@Transactionalを追加
- ToDoの保存と履歴の保存を同一トランザクション内で実行
- どちらかで例外が発生すると両方ロールバック

- findAll(), findById()には@Transactional(readOnly = true)を追加
- 読み取り専用でDB接続を最適化

- @Transactional(rollbackFor = Exception.class)でチェック例外もロールバック
- @Transactional(noRollbackFor = BusinessException.class)で特定例外はコミット

■ AuditService.java
- @Transactional(propagation = Propagation.REQUIRES_NEW)
- 監査ログは本体のトランザクションとは独立して処理
- 本体がロールバックしても監査ログは残す

【技術スタック】
- Spring Boot 3.x
- Spring Transaction
- MyBatis
