info このChapterで学ぶこと

このChapterでは、一覧→新規登録→確認→完了という典型的な画面遷移フローを実装します。 ControllerとThymeleafテンプレートを使い、リンクやフォーム送信による画面遷移を学びましょう。

使用するアノテーション

  • @Controller - クラスがMVCのControllerであることを示す
  • @GetMapping - GETリクエストを処理するメソッドを定義
  • @PostMapping - POSTリクエストを処理するメソッドを定義
  • @RequestParam - リクエストパラメータを取得（例: フォームの入力値）

Section 2-1

一覧画面の作成

まずはControllerを作成し、ToDoの一覧画面を表示できるようにします。

psychology Controllerの役割とは？

Controllerは、ユーザーからのリクエストを受け取り、適切な処理を行って画面を返す役割を担います。 MVCアーキテクチャの「C」にあたり、ModelとViewの橋渡し役です。

1 リクエスト受付

HTTPリクエスト（GET/POST）を受け取る

2 Service呼び出し

ビジネスロジックをServiceクラスに委譲

3 Model設定

Viewに渡すデータをModelに追加

4 View名を返す

表示するテンプレート（HTML）名を返す

smart_toy AIプロンプト

Spring BootでToDoアプリ用のControllerを作成してください。

要件:
- クラス名: TodoController
- @Controllerアノテーションを使用
- 以下の画面遷移に対応するエンドポイントを作成:
  1. GET /todo → 一覧画面（list.html）
  2. GET /todo/new → 新規登録画面（new.html）
  3. POST /todo/confirm → 確認画面（confirm.html）
  4. POST /todo/create → 登録処理 → 完了画面へリダイレクト
  5. GET /todo/complete → 完了画面（complete.html）

また、一覧画面のThymeleafテンプレート（list.html）も作成してください。
- 場所: src/main/resources/templates/todo/
- ToDoリストをテーブルで表示
- 「新規登録」ボタンで /todo/new へ遷移
- Bootstrap 5のCDNを使用してスタイリング

auto_awesome 生成されるソースコード

TodoController.java 
package com.example.todo.controller;

import org.springframework.stereotype.Controller;
import org.springframework.ui.Model;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestParam;

/**
 * ToDoアプリのControllerクラス
 * 一覧→新規登録→確認→完了の画面遷移を管理
 */
@Controller
@RequestMapping("/todo")
public class TodoController {

    /**
     * 一覧画面を表示
     * GET /todo でアクセス
     */
    @GetMapping
    public String list(Model model) {
        // ToDoリストをModelに追加（後でServiceから取得）
        return "todo/list";
    }

    /**
     * 新規登録画面を表示
     * GET /todo/new でアクセス
     */
    @GetMapping("/new")
    public String showCreateForm() {
        return "todo/new";
    }

    /**
     * 確認画面を表示
     * POST /todo/confirm でアクセス
     */
    @PostMapping("/confirm")
    public String confirm(@RequestParam String title, Model model) {
        model.addAttribute("title", title);
        return "todo/confirm";
    }

    /**
     * 登録処理を実行し、完了画面へリダイレクト
     * POST /todo/create でアクセス
     */
    @PostMapping("/create")
    public String create(@RequestParam String title) {
        // ToDoを保存（後でServiceを実装）
        return "redirect:/todo/complete";
    }

    /**
     * 完了画面を表示
     * GET /todo/complete でアクセス
     */
    @GetMapping("/complete")
    public String complete() {
        return "todo/complete";
    }
}
lightbulb コード解説

@Controller と @RequestMapping

  • @Controller - このクラスがMVCのControllerであることをSpringに伝えます
  • @RequestMapping("/todo") - クラス全体のベースURLを /todo に設定

@GetMapping と @PostMapping

  • @GetMapping - リンククリックなどのGETリクエストを処理
  • @PostMapping - フォーム送信などのPOSTリクエストを処理

@RequestParam と redirect:

  • @RequestParam - フォームから送信された値をメソッドの引数で受け取る
  • return "redirect:/todo/complete" - 別のURLへリダイレクト（画面遷移）

link URLマッピングの一覧

@RequestMapping("/todo") ← クラスのベースURL
@GetMapping → /todo → 一覧画面
@GetMapping("/new") → /todo/new → 新規登録画面
@PostMapping("/confirm") → /todo/confirm → 確認画面
@PostMapping("/create") → redirect:/todo/complete → 登録処理
@GetMapping("/complete") → /todo/complete → 完了画面

lightbulb コード解説

Thymeleaf名前空間の宣言

  • xmlns:th="http://www.thymeleaf.org" - Thymeleaf属性を使うための宣言

リンク式 @{...}

  • th:href="@{/todo/new}" - 新規登録画面へのリンク

繰り返し処理 th:each

  • th:each="todo : ${todos}" - Controllerから渡されたtodosリストをループ
  • 各ループで変数 todo に1件ずつデータが入る

テキスト出力 th:text

  • th:text="${todo.title}" - todoオブジェクトのtitleプロパティを出力
  • HTMLタグ内のテキスト（サンプルタスク等）は、実行時にth:textの値で置き換えられる

<html xmlns:th="http://www.thymeleaf.org">

lightbulb コード解説

フォーム送信の仕組み

  • th:action="@{/todo/confirm}" - フォームの送信先URLを指定
  • method="post" - POST送信（@PostMappingで受け取る）

name属性と@RequestParam

  • name="title" - 送信されるパラメータ名
  • Controllerの @RequestParam String title で、この値を受け取る

画面遷移のパターン

  • 「確認」ボタン → フォーム送信（POST） → 確認画面へ
  • 「戻る」リンク → Aタグ（GET） → 一覧画面へ

compare_arrows GETリクエストとPOSTリクエストの違い

GET（リンククリック）
  • Aタグのクリックで発生
  • データの送信なし
  • 画面の表示に使用
  • 例: <a th:href="@{/todo}">
POST（フォーム送信）
  • フォームの送信で発生
  • 入力データを送信
  • データの登録・更新に使用
  • 例: <form method="post">

確認画面の作成

新規登録画面から送信された入力内容を表示し、登録前に確認する画面を作成します。

verified 確認画面のポイント

  • 新規登録画面からPOSTで受け取った値を th:text で表示する
  • 登録処理へ送るために hidden input でデータを保持する
  • 「戻る」リンクで新規登録画面に戻れるようにする

    <title>ToDo登録確認</title>

データの受け渡しの流れ

  • 新規登録画面のフォームでPOST → Controllerの@RequestParam String titleで受け取り
  • model.addAttribute("title", title) でViewに渡す
  • テンプレートのth:text="${title}"で画面に表示

hidden input でデータを保持

  • <input type="hidden" name="title" th:value="${title}">

  • 確認画面 → 登録処理へデータを引き継ぐために必要

th:value

  • th:value="${title}" - input要素のvalue属性に動的な値をセット

Section 2-4

完了画面の作成

登録処理の完了後に表示する完了画面を作成します。リダイレクトを使って画面遷移します。

directions リダイレクトとは？

return "redirect:/todo/complete" は、ブラウザに「/todo/complete に再度アクセスしてね」と指示します。

POST /todo/create → redirect → GET /todo/complete → 完了画面

なぜリダイレクト？ POST処理の後にリダイレクトすることで、ブラウザのリロード時に二重登録を防ぎます（PRGパターン）。

lightbulb コード解説

Controllerのリダイレクト処理

  • return "redirect:/todo/complete" - create()メソッドから完了画面へリダイレクト
  • return "todo/complete"（redirectなし）とは異なり、ブラウザのURLが変わる

PRGパターン（Post/Redirect/Get）

  • Post - フォームデータをPOSTで送信（/todo/create）
  • Redirect - 処理後にリダイレクト（redirect:/todo/complete）
  • Get - リダイレクト先をGETで表示（/todo/complete）
  • この流れにより、ブラウザの「戻る」や「リロード」での二重送信を防止できる

Thymeleaf基本構文まとめ

  • th:text - テキストを出力
  • th:each - 繰り返し処理
  • th:href - リンクURLを動的に生成
  • th:action - フォームの送信先URLを指定
  • th:value - input要素のvalue属性を動的に設定
